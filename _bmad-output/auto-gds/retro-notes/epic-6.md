# Epic 6 — Auto-GDS retro notes

## Story 6-1-item-loot-and-reward-definitions
- [Phase 3 — create-story] 6.1 is the epic's heaviest, highest-regression story: bundles 8 loot/reward definition+repository families, the item roll-model, the deterministic reward-offer fixture, AND the cross-cutting duplicate-id fail-loud retrofit (changes `ContentRepository.register_definition`, touching all 6 existing repos + `RunStartCommand` content gates). Isolate the retrofit and re-run the full suite first.
- [Phase 5 — dev-story] Ratified content-layer convention for Epic 6+: `ContentRepository.register_definition` now returns `ActionResult` (was `void`) and rejects duplicate `(type, id)` with `duplicate_<x>`; all 6 existing repos surface a per-type duplicate error and fail batches closed. New repos MUST follow this fail-loud-on-duplicate contract. Item roll-model ships range-live; affixes/enhancements/affinities are shape-only-inert (`*_mvp_deferred`) until Epic 7+. Reward-offer builder accepts an injected `RngStreamSet` (`rewards`/`loot` streams) — Story 6.3 threads the run-level streams, do not fork.

## Story 6-2-small-inventory-and-equipment-model
- [Phase 3 — create-story] Highest-regression trap for 6.2: `RunSnapshot` already ships `inventory: Array[Dictionary]` + `equipment: Dictionary` as Story-2.7 placeholder fields INSIDE the pinned 23-key no-surprise-key gate (round-trip + default-empty assertions). If 6.2 serializes inventory/equipment, compose into those EXISTING fields — do NOT add a new top-level snapshot key (the gate fails loud on surprise keys). `item_gained`/`ITEM_GAINED`, `InventoryState`, `PickupItemCommand`, `inventory_full` are all greenfield. New `RunState` constructor param goes LAST (after `new_rules_resolver`).

## Story 6-3-reward-offer-flow
- [Phase 5 — dev-story] Reward GENERATE is an explicit `RunOrchestrator.generate_reward_offer`/`generate_passive_reward_offer` method invoked by a caller at the node-completion boundary — deliberately NOT auto-wired into `run_to_completion`/`_resolve_combat`. Auto-generating on every combat node would advance the run-level `streams` mid-run with no auto-resolve policy (tripping the `reward_offer_pending` guard on the next node) and perturb the interrupted==uninterrupted determinism tests. A later HUD/orchestrator story can auto-wire generate-on-complete once it owns a resolution policy.
- [Phase 5 — dev-story] T2 inert run-level `RngStreamSet`: CLOSED for the reward-DRAW half (reward roll now draws through `RunOrchestrator.streams` via `RewardOfferBuilder`; the inert false-confidence test was replaced). The `LevelGenerator.generate`-injected-`RngStreamSet` half remains a tracked residual (no 6.3 AC requires it). Reward draws ALWAYS route through `RewardOfferBuilder.build_offer` + `RewardTableRepository.get_reward_table` (validated tables only) — never a hand-rolled draw (6.1 reward-table validate-before-draw `[Review][Decision]` satisfied).

## Story 6-4-passive-reward-modal-data-contract
- [Phase 3 — create-story] Latent regression trap pinned in the story: tightening `PassiveDefinition.validate()` to require the new FR47/FR77 reward-modal fields retroactively invalidates the SIX Story-5.4 baseline passives (referenced by 6.3's `passive_reward_choice` table). They MUST be extended with the new required fields in the SAME change, or `create_baseline_repository()` silently returns null and the class-start + reward-table suites break. Extend `PassiveDefinition` ADDITIVELY (no field renames/removals).

## Story 6-6-destroy-passive-command-and-outcome-distribution
- [Phase 3 — create-story] Story introduces a NEW content type `DestroyOutcomeTableDefinition` (mirrors `RewardTableDefinition`) with an exact-integer 70/20/10 distribution check + a visible `mvp_distribution_exception` tuning marker — but recommends a SINGLE defaulted baseline table with NO repository, diverging from the epic's prior table-with-repository pattern (`RewardTableRepository`) for scope minimalism. Worth sanity-checking the convention divergence in review / retro.
- [Phase 3 — create-story] `passive_destroyed` deliberately diverges from its sibling `passive_consumed`: it carries RNG draw provenance (`roll`/`draw_index`) plus `outcome_category`/`outcome_id`/`outcome_effect`/`explanation`, because Destroy DRAWS a named-RNG outcome (70/20/10) and records the rolled result, unlike the deterministic Consume. Outcome RNG uses the `rewards` named stream (consistency with 6.3 reward draws; AC offered `rewards` OR `events` — dev to record the choice). Append the new event at the enum end with NO renumber; extend the exhaustive `expected_ids` map.

## Story 6-7-loot-and-passive-build-smoke-run

Worth confirming in review that the new mechanic is in-scope for a capstone and the record-only decision is right vs pulling Epic-7 HP forward.
