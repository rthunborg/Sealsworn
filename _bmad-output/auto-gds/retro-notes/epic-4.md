# Epic 4 — Auto-GDS Retro Notes

Signal-only scratchpad for later Epic 4 stories and the epic retrospective. Each bullet is a
constraint, gotcha, or ratified convention surfaced by an earlier story in this epic.

## Story 4-1-run-state-and-route-node-model
- [Phase 3 — create-story] Story directs a NEW `scripts/run/` domain folder for the run-progression model (`RunState`/`RouteState`/`RouteNode`) and adding `run` to the project-context.md code-organization domain list. Architecture specifies the `RunState` machine but maps no directory for the run model (route *generation* → `scripts/generation/route/`, saves → `scripts/save/`). Defensible (mirrors how `settings` was added in Epic 2) but is a genuine convention extension later stories inherit — flag if a different placement is preferred.
- [Phase 5 — dev-story] Run phase is serialized as `run_phase` NESTED inside the existing `route_state` payload, deliberately NOT a new top-level `RunSnapshot` key — keeps the pinned 23-key no-surprise gate green untouched. Later route stories must keep run-progression fields under `route_state`, not flatten new top-level run keys onto `RunSnapshot`.
- [Phase 5 — dev-story] `run_started` `DomainEvent` is now fully wired (factory + payload validator, system event, no actor) and tested, but has NO emission site yet — no run-start command exists in 4.1. A later Epic 4 story (run-start command) must actually emit it, or the validator is dead weight.
- [Phase 7 — code review] `DomainEvent._has_decimal_string_payload` (the new `run_started` `root_seed` validator) uses loose `is_valid_int()` only — it accepts out-of-int64 strings that `to_int()` saturates/wraps, weaker than the strict lossless-decimal-string check Story 3.7 added to `ManualSeedLoader.parse_seed`. It is the project's first seed-string validator added AFTER 3.7 set the lossless idiom and it diverged. Deferred to the first `run_started` emitter (folds with the `node_count` defer). Epic-retro lesson: any new seed-string validator must adopt the 3.7 lossless/canonicalize idiom, not bare `is_valid_int()`.

## Story 4-2-seeded-8-12-node-route-generation
- [Phase 3 — create-story] 4.2 structurally bounds the route to 8-12 non-boss nodes (`run_started.node_count` ∈ [8,12]), so the eventual run-start emitter (4.3/4.6) can close the two `run_started` int64/lossless-decimal defers as permanently benign rather than re-litigating them.
- [Phase 5 — dev-story] `RouteGenerator.INTERIOR_COLUMN_COUNT = 6` (not 7) is load-bearing for the "≥1 branch point per seed" guarantee (AC4): min interior demand (7 nodes) > 6 columns forces ≥1 width-2 column even at the count=8 floor; 7 columns collapses to an all-linear route. Any column-count/band change must re-verify the branch-point guarantee and re-pin route fingerprints via `tools/dump_route_fingerprints.gd`.
- [Phase 5 — dev-story] Route result-shape pattern ratified: `RouteGenerator` returns a `GenerationResult` (`PHASE_ROUTE`) whose serializable `payload.route_state` rehydrates via `route_from_result` — payload carries the snapshot dict, not the live `RefCounted`. The run-start command (4.3/4.6) must consume it by rebuilding the `RouteState`, mirroring the level pipeline discipline.

## Story 4-3-route-choice-and-forward-commitment
- [Phase 3 — create-story] Reveal-on-arrival coupling is the load-bearing correctness trap in 4.3: the 4.2 generator only marks depths 0–1 `REVEAL_REVEALED` (everything deeper HIDDEN), so `RouteAdvanceCommand` MUST reveal the arrived node's forward neighbors on commit or the run soft-locks after the first choice. Later node-flow stories (4.4 entry, 4.5 resolution) inherit this: any advance/entry path must keep forward reveal in lockstep with commitment. Story mandates a multi-step start→boss-tier "no-soft-lock walk" test, not a single-commit test.
- [Phase 5 — dev-story] `RouteAdvanceCommand` ratified the first run-domain command shape: takes `RunState` directly (no `RunActionContext` wrapper), advances the route pointer only and stays in `ACTIVE_ROUTE` (the `ACTIVE_ROUTE → NODE_RESOLUTION` transition is 4.4's), supplies its event sequence id via the constructor (default 1) since the run domain still has no event sequencer. Error model: single `ineligible_route_choice` code with the precise reason in metadata (`unknown_node`/`not_linked`/`hidden_node`/`cleared_node`/`is_current_node`) — hyphenated node ids never embedded in a code. 4.4–4.6 should follow this command/error idiom.
- [Phase 5 — dev-story] Fixture trap for run-command tests: `RunState.new_run()` resets `current_node_id` + `cleared_node_ids` to fresh-run state, so any test needing a pre-cleared/parked route must build `RunState.new(PHASE_ACTIVE_ROUTE, ...)` directly, not `new_run()`. 4.4–4.6 will hit the same trap.

## Story 4-4-node-entry-and-exit-resolution
- [Phase 3 — create-story] Cross-command duplicate-clear bug surfaced: 4.3's `RouteAdvanceCommand` already appends the LEFT node to `cleared_node_ids` unconditionally, and `RouteState.validate()` rejects `duplicate_cleared_node`. So if 4.4 also clears the node on EXIT, the next advance double-clears and fails validation. Story mandates making 4.3's clear idempotent — a genuine epic-4 gotcha the 4.4→4.5→4.6 chain inherits.
- [Phase 3 — create-story] Hyphen trap is load-bearing: route node ids are hyphenated (`node-1-0`) but `GenerationRequest.node_id` is validated lower_snake (`_is_lower_snake_id` rejects hyphens). Node-entry must derive a lower_snake request id from the hyphenated node id or request validation fails at runtime.
- [Phase 3 — create-story] AC2 "between-level autosave can be requested" is only partially buildable in 4.4: `RunSnapshot.from_between_level` requires a live `BoardState` that 4.4 has none of, so 4.4 surfaces only the autosave seam (route-side `to_run_snapshot_fields` + `autosave_requested` flag); the board-bearing write moves to 4.6. Flagged so the 4.6 owner expects it.
- [Phase 5 — dev-story] Cross-command cleared-set bug confirmed and FIXED via a one-line idempotency guard in `RouteAdvanceCommand` (exit clears the current node, then the next advance re-clears it → `duplicate_cleared_node`). 4.5/4.6 owners must keep advance/enter/exit clear+reveal logic in lockstep and NOT shuffle cleared/reveal responsibilities between commands. `NodeEnterCommand` does NOT run `LevelGenerator` (builds + returns the validated `GenerationRequest` only; zero RNG); per-node-type resolution and the success-path `GenerationResult` consumption are 4.5/4.6.

## Story 4-5-mvp-node-types-and-boss-placeholder
- [Phase 3 — create-story] 4.5's real work is *resolution*, not the AC1 surface reading: `RouteNode` (4.1) + `RouteGenerator` (4.2) already define and assign all 8 node types incl. one terminal boss, so "the route can include every MVP type" is already true. The dev agent must NOT re-litigate generation — only add safe placeholder resolution for the 5 non-combat node types + boss (today `NodeEnterCommand` hard-rejects them with `unsupported_node_entry`).
- [Phase 3 — create-story] 4.5/4.6 scope seam ratified: the boss run-end event (a NEW `run_completed`) belongs to 4.5 (AC3 demands it at the boss-node boundary); `run_started` emission, the full start-to-end shell, pacing, and route-position save/resume stay 4.6, which CONSUMES (not re-creates) the `run_completed` boundary. Confirm at epic-4 retro that 4.6 consumes rather than duplicates it.
- [Phase 5 — dev-story] GDScript gotcha: a `const` initialized by a `static func` call cannot be referenced cross-script as `ClassName.CONST` ("Could not resolve external class member"). `PLACEHOLDER_NODE_TYPES` had to be an explicit compile-time Dictionary literal, with its complement-of-`NODE_TYPE_RECIPE` relationship proven by a test rather than computed in the const. Applies to any future derived-constant pattern shared across scripts.
- [Phase 5 — dev-story] 4.5 shipped two new additive system `DomainEvent`s — `node_placeholder_resolved` + `run_completed` — wired end-to-end (factory, payload validator, stable-id map, round-trip + malformed tests). Enum values appended at the end (not renumbered), so the snapshot/event schema stays back-compatible; 4.6 must EMIT `run_completed` from the real start-to-end shell, not redefine it.
