- [Phase 5 ‚Äî dev-story] CORRECTION to a misdescribed convention: the story Dev Notes (and a cited 1-3 deferred-work note) wrongly claimed `BoardState.try_from_snapshot` strips cell `occupant_id` and re-derives it from blocking entities, and told the dev to emit enemy cells with `occupant_id: ""`. ## Story 3-6-generator-validation-and-bounded-retry
- [Phase 3 ‚Äî create-story] Bounded retry COLLIDES with the established "level = pure function of (seed, recipe)" invariant: retrying the same seed reproduces the same rejected candidate, so the story specifies a per-attempt seed mix with **attempt 0 unperturbed** ‚Äî preserving the pinned terrain seed-regression fingerprints from 3.2/3.3/3.4. This is the non-obvious design constraint shaping 3.6, and 3.7 (manual-seed batch loader) will build on this retry engine. Flag at the epic retro.
- [Phase 3 ‚Äî create-story] Generated candidates are valid BY CONSTRUCTION (reserved corridor + sparse enemies), so the retry-failure branch and most validator-rejection branches are unreachable end-to-end without a test seam ‚Äî story mandates an injected-validator seam, the recurring Epic-3 "correct branch, integration coverage missing because unreachable by construction" pattern the reviews keep flagging.
- [Phase 5 ‚Äî dev-story] DEVIATION from the story's literal Dev Note: the recommended `unsafe_first_reveal` semantic (no enemy within Chebyshev <= 1 of the entrance) conflicts with the baseline generator, which legitimately places enemies orthogonally adjacent to the entrance ‚Äî enforcing it would fail clean attempt-0 candidates and force a re-roll that drifts the pinned terrain fingerprints. Narrowed (per the story's own 3.6.2/3.6.6 escape hatch) to entrance-cell occupancy/HAZARD only; adjacent threats are fair (within LoS radius 4). Documented in-code; both review rounds confirmed. Future first-reveal/safety tuning should treat entrance-cell-only as the ratified v0 contract.
- [Phase 5 ‚Äî dev-story] Validator `check_order()` deliberately runs `illegal_enemy_placement` (check b) BEFORE the entity-aware `soft_lock_detected` so an enemy on the exit reports the root-cause illegal placement, not a downstream soft-lock ‚Äî documented in `check_order()` and the `level_validator.gd` header. The `entity_on_entrance` sub-branch of the safe-first-reveal check (h) is consequently unreachable through `validate()` (check b catches it first) and is kept as documented defense-in-depth for direct callers / future check-ordering changes (code-review R2 decision).

## Story 3-7-manual-seed-level-loader-and-regression-tests
- [Phase 3 ‚Äî create-story] 3.7 is the only Epic-3 story whose ACs are met primarily by REUSE + harness coverage, not new generation logic: a thin pure-domain `ManualSeedLoader` that parses/normalizes a seed and CALLS the complete 3.6 `LevelGenerator.generate(...)` pipeline (it must NOT re-author generation/validation/retry), a `generate`-level batch seed-regression harness reusing the existing terrain `fingerprint(...)`, and a preserved annotated approved-seed catalog. Main risk is scope creep (a dev agent could wrongly build a seed-entry UI, a run-save write, or route/`map`-stream reproduction) ‚Äî scope boundary in the story calls each out explicitly. Watch for over-build at review.
- [Phase 3 ‚Äî create-story] Seed-sign provenance (latent cross-story bug from the 3.1 review Decision, only actionable here): `GenerationRequest.validate()` rejects `root_seed < 0`, so the loader MUST normalize a negative-decoding full-64-bit seed to non-negative before building the request (with a regression) rather than relax the validator. The normalization choice (mask vs unsigned-reinterpret) determines which concrete level a shared negative seed maps to ‚Äî confirm it at review. Also reaffirmed: manual-seed eligibility vocabulary already exists (`RunSnapshot.is_manual_seed` + `meta_progression_eligible`); the dropped `manual_seed_eligible_for_progression` key is test-pinned absent and must NOT be re-added. No second fingerprint format / no terrain re-pin ‚Äî attempt-0-unperturbed means batch fingerprints must AGREE with existing 3.2/3.3/3.4 fixtures.
- [Phase 5 ‚Äî dev-story] `GenerationResult.seed` (String) is populated ONLY on the error path; a successful result leaves `seed == ""` and carries the seed in `payload.level_seed`. Consumers must read `payload.level_seed` on success, not `result.seed` (cost a first-pass 12-test miss here). Worth a future `GenerationResult.seed` cleanup so the field is populated on success too ‚Äî flagged for the epic retro / next consumer.
- [Phase 5 ‚Äî dev-story] Decision: a FLOAT seed input is rejected as `non_integer_seed` (more restrictive than `RngStreamSet._int64_from_value`, which would accept an integral float) ‚Äî share/replay seed format is int-or-decimal-string and a float is ambiguous. Ratified strictness for the manual-seed loader; confirm at review.
- [Phase 7 ‚Äî code review] Loader now rejects an out-of-int64-range decimal string (e.g. `"99999999999999999999"`) as `non_integer_seed` rather than silently saturating via `String.to_int()` (Round 1 Low Patch). The far-over-int64 regression input makes `String.to_int()` emit ONE benign engine `ERROR: Cannot represent ‚Ä¶ too large` stderr line on the otherwise-green run ‚Äî like the documented save/settings parse-failure fixtures, NOT a regression. Future test-run readers should expect it.
