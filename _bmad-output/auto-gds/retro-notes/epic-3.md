## Story 3-4-tactical-wrinkles-blockers-and-hazards
- [Phase 3 — create-story] 3.4 is the first epic-3 story that deliberately RE-PINS both the Small and Medium seed-regression fingerprints (terrain grid gains wrinkle/HAZARD cells). Earlier stories never perturbed an existing generator's fingerprints — treat the fingerprint drift as intentional, not a regression, at review.
- [Phase 3 — create-story] HAZARD terrain is already fully board-valid AND walkable + sight-transparent (only WALL blocks occupancy/LOS, per board_cell.gd). So AC2 (entrance→exit progress with hazards present) needs ZERO board-model change, and hazard DANGER is correctly deferred to the rules kernel. A careless impl could wrongly make HAZARD block movement — don't.

## Story 3-5-enemy-and-reward-placement
- [Phase 3 — create-story] 3.5 closes the Story 3.1 deferral by deciding NO standalone `RewardTableDefinition` in v0: reward-placement rules stay on `LevelRecipeDefinition` and rewards are modeled as abstract payload markers (NOT board entities — `TacticalEntityState` has no reward type); concrete loot tables + repository deferred to Epic 6. Ratified reward-placement convention for the epic retro.
- [Phase 3 — create-story] AC3 "assigned generation stream" wording is a latent ambiguity: a dedicated `rewards`/`loot` stream exists in `RngStreamSet` but the whole Epic-3 pipeline (3.1–3.4) draws generation randomness through `GenerationRequest.draw_layout_int` → `STREAM_LEVEL` (3.1 retro mandates `level_seed()`/`level`-stream). Story resolves placement toward `STREAM_LEVEL` (regression-safe, convention-consistent; `rewards`/`loot` reserved for Epic 6 runtime resolution). Stories 3.6/3.7 + Epic 6 should note reward/loot stream ownership will resurface there.
- [Phase 5 — dev-story] CORRECTION to a misdescribed convention: the story Dev Notes (and a cited 1-3 deferred-work note) wrongly claimed `BoardState.try_from_snapshot` strips cell `occupant_id` and re-derives it from blocking entities, and told the dev to emit enemy cells with `occupant_id: ""`. Actual contract is the opposite — the validator records the cell occupant, strips it, then cross-checks against blocking entities and REJECTS a blocking entity whose cell carries `""` (`invalid_cell_occupant`; see test_board_state.gd:1014-1025). Correct pattern: `build_board_snapshot` sets the matching `occupant_id` on each blocking enemy's cell, mirroring `BoardState.to_snapshot()`. Future Epic-3/board stories that emit entities must follow `to_snapshot()`, not that note.
- [Phase 5 — dev-story] BREAKING: `LevelGenerator.generate(request, recipe_repository)` → `generate(request, recipe_repository, enemy_repository)` — enemy repository is now a REQUIRED 3rd param (null/empty → structured `PHASE_ENEMIES` error). All in-repo callers (tests + tools/dump_*) updated. Additive: success diagnostics gain `enemy_count`/`reward_count`/`optional_reward_count`; payload gains a `rewards` marker list; boards now carry enemy `entities`. Terrain seed-regression fingerprints unchanged (placement did not perturb terrain or blocker/wrinkle draw order).
