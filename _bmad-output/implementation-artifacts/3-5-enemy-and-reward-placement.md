# Story 3.5: Enemy and Reward Placement

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want generated levels to place enemies and rewards legally,
so that each seed creates fair tactical pressure and reachable rewards.

## Acceptance Criteria

1. Given a generated level has legal floor cells, when enemy placement runs, then enemies are placed on valid unoccupied cells according to recipe budgets, and no enemy starts on the entrance, exit, wall, blocker, or unreachable required cell.
2. Given intended rewards are placed, when reward reachability validation runs, then reachable rewards pass and unreachable intended rewards fail validation, and optional risky rewards are marked as optional if placed behind danger.
3. Given enemy and reward placement uses randomness, when the same seed and recipe are used, then placements are reproducible, and placement rolls use the assigned generation stream rather than global randomness.
4. Given enemy or reward placement applies a candidate to the level, when placement succeeds, then it updates the generation candidate or final snapshot through generation phase outputs, and it does not bypass validation by mutating active tactical state directly.

### Placement Determinism Contract (AC3 — same seed reproduces placements; the `level` stream is the only randomness source)

The single most important constraint, identical in spirit to 3.2/3.3/3.4: **enemy AND reward placement is a pure deterministic function of (root seed, recipe, starting `level`-stream state). Every placement-affecting choice (enemy count, enemy positions, enemy kinds, reward count, reward positions) flows through `GenerationRequest.draw_layout_int` (-> `RngStreamSet.STREAM_LEVEL`), the draw ORDER is fixed (placement draws APPENDED after the wrinkle draws), and no other randomness source touches placement.**

- Build the streams as `RngStreamSet.new(request.level_seed())` (the phase methods already do). Draw blocker count -> blocker positions -> wrinkle kinds -> wrinkle positions -> THEN enemy count -> enemy positions -> enemy kinds -> reward count -> reward positions, in a FIXED, documented order. Appending placement draws AFTER the wrinkle draws preserves the existing blocker+wrinkle behaviour and keeps the terrain fingerprints stable. [Source: Story 3.2/3.3/3.4 fixed-draw-order; project-context.md Determinism rules]
- Different seeds must produce meaningfully different placements (enemy/reward positions/kinds) — add a `>= 2 distinct` placement-divergence assertion (independent of the terrain fingerprint).
- Stream isolation (carried from Story 1.4 / 3.1-3.4): pre-advancing `cosmetic`/`combat`/`rewards`/`loot` draws before generation must NOT change placement. Extend the `_layout_draws_only_from_level_stream` assertion to cover the placement draws — ALL placement draws must advance ONLY `STREAM_LEVEL`.
- No global fallback: an unknown stream or invalid range surfaces a structured `ActionResult` error from `RngStreamSet`; placement must never silently fall back to `randi()`/`randf()`. The `GenerationRequest` helpers enforce this — just don't bypass them. [Source: project-context.md "Do not use one global RNG for gameplay"]

### RNG Stream Contract (placement uses `STREAM_LEVEL`, NOT the `rewards`/`loot` streams — resolves the AC3 "assigned generation stream" wording)

AC3 says "placement rolls use the assigned generation stream rather than global randomness." There is a genuine choice to make explicit, and the established convention resolves it decisively:

- The `RngStreamSet` defines a `rewards` stream and a `loot` stream. Those are reserved for RUNTIME reward/loot RESOLUTION (which reward contents drop, which affix rolls — Epic 6+ gameplay), NOT level-generation PLACEMENT. The Epic-3 generation pipeline (Stories 3.1-3.4) draws ALL layout/placement-affecting randomness EXCLUSIVELY through `GenerationRequest.draw_layout_int/float`, which is hardwired to `RngStreamSet.STREAM_LEVEL`.
- THEREFORE: in v0 generation, the **assigned generation stream IS `level`**. Enemy AND reward PLACEMENT draws use `GenerationRequest.draw_layout_int` (-> `STREAM_LEVEL`), exactly like blockers and wrinkles. Do NOT introduce a separate `rewards`/`loot` draw for placement — doing so would (a) break the established `_layout_draws_only_from_level_stream` / stream-isolation regression tests for 3.2/3.3/3.4, (b) split the generation seed across two streams with no benefit, and (c) front-run Epic 6's reward-resolution stream usage. The Epic-3 retro (Story 3.1) is explicit: downstream stories call `request.level_seed()` and verify seed stability through the `level` stream.
- This is the regression-safe, convention-consistent reading of "the assigned generation stream." Record the decision in the Completion Notes so the reviewer sees it was deliberate (it is also flagged as an open question for review awareness — see the Story Context open-questions note).

### AC1 Enemy-Legality Contract (legal unoccupied reachable cells; nothing on entrance/exit/wall/blocker/unreachable)

- AC1 verbatim: "Given a generated level has legal floor cells, when enemy placement runs, then enemies are placed on valid unoccupied cells according to recipe budgets, and no enemy starts on the entrance, exit, wall, blocker, or unreachable required cell." [Source: epics.md Story 3.5 AC1]
- **Legal unoccupied cells:** draw enemy cells from the residual candidate pool the wrinkle step leaves — that pool already excludes the border WALL ring, the reserved corridor, the entrance, the exit, every blocker, and every wrinkle cell, and the shrinking-pool draw removes each picked cell, so two enemies never collide. A HAZARD cell is occupiable (walkable) and MAY hold an enemy — do not exclude HAZARD unless documented. Wall/entrance/exit are non-occupiable AND excluded from the pool, so an enemy can never land on them (belt-and-suspenders: `BoardState.try_from_snapshot` would also reject a blocking entity on a WALL).
- **Budget:** enemy count is drawn over `[enemy_budget_min .. enemy_budget_max]` CLAMPED to the residual candidate count (mirror `_draw_blocker_count`'s `maximum = min(maximum, candidate_count)` / `minimum = min(minimum, maximum)` clamp — closes the 3.2-deferred "clamp-to-candidate-count unexercised" gap by actually exercising it for placement; add a test recipe whose enemy budget exceeds available cells if you want to prove the clamp).
- **Reachability ("unreachable required cell"):** every enemy cell must be reachable from the entrance over non-WALL cells (4-neighbour BFS — reuse the Medium `_flood_reachable`). Because enemies draw from the corridor-respecting pool and the corridor connects entrance->exit, generated enemies are reachable by construction; ASSERT it so a future change cannot silently strand an enemy behind a wall. (The phrase "unreachable required cell" in AC1 means: do not place an enemy on a cell the player cannot reach — i.e. it must be in the entrance-reachable region.)
- **Occupant invariant:** emit enemy cells with `occupant_id: ""` and append the entity dicts to `entities`; `try_from_snapshot` re-derives `occupant_id` from blocking entities and cross-validates. Do NOT hand-write `occupant_id` onto the cell snapshot (it would be stripped and re-derived; a mismatch would be REJECTED). [Source: board_state.gd `try_from_snapshot` occupant handling; deferred-work.md "code review of 1-3" occupant-consistency note]

### AC2 Reward-Reachability Contract (reachable passes, unreachable fails; behind-danger -> optional)

- AC2 verbatim: "Given intended rewards are placed, when reward reachability validation runs, then reachable rewards pass and unreachable intended rewards fail validation, and optional risky rewards are marked as optional if placed behind danger." [Source: epics.md Story 3.5 AC2]
- **Reachable passes / unreachable fails:** a focused 4-neighbour BFS from the entrance over non-WALL cells; every intended reward cell must be in the reachable set. A reachable reward PASSES; an unreachable intended reward FAILS with a structured error (e.g. `unreachable_reward`) + compact diagnostics (offending coordinate + reachable count) — NEVER a full grid dump. Generated rewards (drawn from the corridor-respecting pool) are reachable by construction, so the failure path is reachable today only via a hand-built malformed candidate; build that test (mirror the Medium AC2 hand-built rejection pattern).
- **Behind-danger -> optional:** a reward is `optional = true` IFF behind danger; mandatory rewards are `optional = false`. DOCUMENT the predicate (recommended v0: a reward cell is behind danger if the cell or any 4-neighbour is HAZARD terrain). `allow_reward_behind_danger` gates whether behind-danger placement is permitted (`false` for Small — do not place a behind-danger reward, or keep it off the mandatory route and mark it optional). An optional reward is still REACHABLE (optional means "guarded/skippable," NOT "stranded") — assert this.
- **SCOPE GUARD:** this is the focused reward-reachability + optional-flag check, NOT the 3.6 comprehensive validator. Do NOT add no-soft-lock / safe-first-reveal / legal-class-gate checks here; 3.6 owns them and will likely subsume this check.

### AC4 Phase-Output Contract (apply through generation outputs; never mutate live tactical state)

- AC4 verbatim: "Given enemy or reward placement applies a candidate to the level, when placement succeeds, then it updates the generation candidate or final snapshot through generation phase outputs, and it does not bypass validation by mutating active tactical state directly." [Source: epics.md Story 3.5 AC4]
- **Through generation phase outputs:** placement results live in the generated layout/board snapshot (`entities` for enemies) and the payload (`rewards` markers). The enemy-bearing board is built/validated via the strict `build_board_snapshot` -> `BoardState.try_from_snapshot` path (validate-then-reject, never coerce — the Story 1.3 precedent), survives a real `JSON.stringify`/`parse_string` round-trip, and rides `TacticalSnapshot.from_domain`/`parse`. The payload is pure serializable data (`assert_false(board is Node, ...)`; payload strings JSON-clean).
- **Never mutate live tactical state:** placement is a pure data operation on the GENERATED candidate. It must NOT execute commands, apply events to a live board, draw RNG from any non-`level` stream, or touch the run save / settings / progression. The generated board is data; do NOT apply it as live state. [Source: project-context.md "Do not let a snapshot compose/restore mutate source state"; NFR14]
