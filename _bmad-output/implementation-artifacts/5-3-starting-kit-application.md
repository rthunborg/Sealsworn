# Story 5.3: Starting Kit Application

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want my selected class to begin with the correct starting equipment and baseline stats,
so that each run starts consistently from class identity.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 5 -> Story 5.3 (verbatim AC, with the implementation-grounded detail this story adds).

1. **A run started with Warrior/Pyromancer/Ranger RESOLVES the class's starting kit and APPLIES it into domain state — the configured starting weapon, support item, baseline HP, class-passive reference, and equipment-synergy-passive reference are present on the run's domain state.**
   **Given** a run starts with Warrior, Pyromancer, or Ranger
   **When** starting kit application runs
   **Then** the hero receives the configured starting weapon, support item, baseline HP, class passive reference, and equipment-synergy passive reference (the `ClassDefinition` kit fields — `starting_weapon_id`/`starting_support_id`/`baseline_hp`/`class_passive_id`/`equipment_synergy_passive_id` — are RESOLVED through the content repositories and recorded as the run's applied starting kit)
   **And** the result is represented in domain state (the applied kit lives on the EXISTING `RunState` — recorded by `RunStartCommand` AFTER it records `selected_class_id` — NOT in UI-only state and NOT in a parallel run format; it is the by-id-reference kit Story 5.1 designed so 5.3 seats it into `RunState`).

2. **A starting kit that references missing equipment or passive data FAILS run start with a structured content error and leaves NO partial run state active.**
   **Given** a starting kit references missing equipment or passive data
   **When** validation runs
   **Then** run start fails with a structured content error (a `starting_weapon_id`/`starting_support_id` that does not resolve against `WeaponRepository`/`SupportRepository` makes `RunStartCommand` REJECT with a stable lower_snake `ActionResult.error` — e.g. `unknown_starting_weapon` / `unknown_starting_support` — carrying the offending id in metadata, the EXACT validate-then-mutate / byte-identical-no-mutation idiom the run commands follow)
   **And** no partial run state becomes active (the reject happens BEFORE any `RunState` is built — ZERO events, NO run, no half-applied kit; the orchestrator surfaces the error verbatim and seats nothing). NOTE: passive ids are SHAPE-only string references in v0 (no passive system exists yet) — "missing passive data" here means a SHAPE-invalid passive id, NOT a lookup against a (nonexistent) passive repository; do NOT resolve passive ids against anything (Story 5.4 / Epic 6 own that).

3. **Each playable class starting a run produces equipment + passives matching its `ClassDefinition`; locked classes still cannot start runs; the proofs run headlessly.**
   **Given** class start tests run headlessly
   **When** each playable class starts a run
   **Then** starting equipment and passives match the class definition (Warrior -> sword + shield + hp 18; Pyromancer -> staff + tome + hp 18; Ranger -> bow + `none` + hp 18; each with its two class/equipment-synergy passive-id references — asserted against the resolved `ClassRepository` baselines, not hardcoded literals)
   **And** locked classes cannot start runs (the Story-5.2 `class_not_selectable` / `unknown_class` fail-closed gate is PRESERVED — a locked class still rejects BEFORE any kit resolution; kit resolution only runs for a confirmed-selectable class).

### Acceptance-criteria interpretation notes (read before coding)

- **This story is the KIT-RESOLUTION-AND-APPLICATION extension of the run-start seam — NOT a new tactical player, NOT real combat, NOT a passive/rules system.** Story 5.2 made `RunStartCommand` carry + RECORD the selected `class_id` on `RunState` (fail-closed on unknown/locked). 5.3 does the next concrete step: when a confirmed-selectable class starts a run, RESOLVE that class's `starting_weapon_id` + `starting_support_id` against `WeaponRepository`/`SupportRepository` (fail closed on a missing item), read `baseline_hp` + the two passive-id references, and RECORD them as the run's APPLIED STARTING KIT in domain state. It does NOT instantiate a tactical `BoardState` player, does NOT run a level, does NOT wire passives into the rules kernel (5.4/Epic 6), and does NOT add active class skills (FR45). [Source: epics.md Story 5.3; 5-1 forward-seam note "Story 5.3 applies them into the EXISTING RunState without a parallel run format"; 5-2 Completion Notes Decision (c)]

- **CRITICAL — the class accessor is `ClassRepository.get_class_definition(class_id)`, NEVER `get_class(id)`.** `get_class()` is a reserved native `Object` method; defining/calling a custom `get_class` is a hard Godot parse error (Story 5.1 Decision E, human-ratified D3 — it compile-failed two 5.1 test files and produced a FALSE PASS). The epics.md Story 5.3 text predates this and is silent on the accessor; every class lookup in this story MUST call `ClassRepository.get_class_definition(id)`. The weapon/support accessors are `WeaponRepository.get_weapon(id)` and `SupportRepository.get_support(id)` (NO native collision). [Source: 5-1 Decision E; deferred-work.md Story 5-1 Decision 3 (D3 ratified); epic-5 retro-notes Story 5-1]

- **5.3 OWNS the cross-repository equipment-id resolution that 5.1 deferred (5.1 Decision A, Option A).** Story 5.1 validated `starting_weapon_id`/`starting_support_id` for lower_snake SHAPE only inside `ClassDefinition.validate()` and explicitly deferred resolving them against `WeaponRepository`/`SupportRepository` to THIS story (it kept an OPTIONAL integration safety-net test asserting the baseline equipment ids are real repo ids, but did NOT wire resolution). 5.3 is the cross-repository resolution: `WeaponRepository.get_weapon(starting_weapon_id) == null` -> `unknown_starting_weapon`; `SupportRepository.get_support(starting_support_id) == null` -> `unknown_starting_support`. Fail closed, byte-identical no-mutation, zero events, no run built — exactly the 5.2 class-gate idiom. [Source: 5-1 Completion Notes Decision A + Task 4.3 ("Story 5.3 owns cross-repository kit resolution"); epic-5 retro-notes Story 5-1 ("Equipment ids ... cross-repository resolution deferred to Story 5.3"); deferred-work.md note folded under 5-2 "ADJACENT (Story 5.3's concern)"]

- **`none` support is a REAL baseline support id — it RESOLVES; it is NOT a missing item.** Ranger's `starting_support_id` is `&"none"`, and `SupportRepository.BASELINE_SUPPORT_IDS = [none, tome, shield]` — `none` is a registered `SupportDefinition` (`SUPPORT_NONE`, armor 0 / block 0 / no bonus, "No off-hand modifier."). The kit resolution MUST resolve `none` through `SupportRepository.get_support(&"none")` (which returns the real no-op support definition) and treat it as a VALID resolved support — do NOT special-case `none` as "no support" by skipping resolution, and do NOT treat it as `unknown_starting_support`. This is the single most likely correctness trap in this story (Ranger would falsely fail run-start if `none` were treated as missing). [Source: support_repository.gd BASELINE_SUPPORT_IDS + `_baseline_definitions()` SUPPORT_NONE; class_repository.gd ranger baseline `starting_support_id = &"none"`]

- **Passive ids STAY string-shape references — do NOT resolve them against anything (there is no passive system yet).** `class_passive_id` / `equipment_synergy_passive_id` are lower_snake STRING-shape forward references with NOTHING behind them (grep-confirmed: 0 `.gd` files under `scripts/rules/{conditions,operations,resolver,triggers}`; no passive definition / passive repository anywhere). 5.3 RECORDS these two ids verbatim on the applied kit (so the kit is complete per AC1), but it MUST NOT resolve them against a (nonexistent) passive repository, build a passive system, or wire them into the rules kernel — that is **Story 5.4** + Epic 6 (the passive POOL, FR46). AC2's "missing passive data" therefore means a SHAPE-invalid passive id at most (and 5.1's `validate()` already guarantees selectable classes carry lower_snake passive ids, so this branch is unreachable for the baseline classes — record the ids; do not add a passive lookup). [Source: epic-5 retro-notes Story 5-1; 5-1 Completion Notes Decision B; epics.md Story 5.4 / FR46; deferred-work.md "passive-id resolution + rules-kernel wiring is Story 5.4 / Epic 6, NOT 5.3"]

- **Where the applied kit lives = on `RunState`, recorded by `RunStartCommand` (do NOT invent a parallel run format, a tactical player, or a new top-level RunSnapshot key) — this is the biggest design decision; record it as a `[Decision]`.** There is currently NO hero/player run-level domain class (`scripts/run/` holds only `RunState`/`RouteState`/`RouteNode`/`RunOrchestrator`; grep-confirmed no `Player`/`Hero`/`StartingKit`/`RunHero` class exists). Story 5.1's forward-seam note is explicit: the kit is a small set of by-id references (weapon/support/passives) + `baseline_hp` "so **Story 5.3 applies them into the EXISTING `RunState`** without a parallel run format." Follow that: add the applied kit to `RunState` (the cleanest fit, consistent with 5.2 recording `selected_class_id` there). Two shapes are acceptable — PICK ONE and document it as a `[Decision]`:
  - **(Shape A — RECOMMENDED) a small typed `StartingKit` value object** (`RefCounted`, scene-free, under `scripts/run/`) holding `class_id`, `weapon_id`, `support_id`, `baseline_hp`, `class_passive_id`, `equipment_synergy_passive_id`, with `to_dictionary()` / `try_from_dictionary` (lenient) + an exact-key contract, recorded as `RunState.starting_kit: StartingKit = null`. This mirrors the project's typed-DTO discipline and keeps `RunState` from sprouting six loose fields.
  - **(Shape B) six additive scalar fields on `RunState`** (`applied_weapon_id`/`applied_support_id`/`applied_baseline_hp`/`applied_class_passive_id`/`applied_equipment_synergy_passive_id`, alongside the existing `selected_class_id`). Simpler but spreads the kit across `RunState`'s `_init`/`copy()`/`to_dictionary()`.
  Either way the kit is ADDITIVE + LENIENT (default null/empty when absent so every pre-5.3 run dict / save still parses), is recorded only on a SUCCESSFUL selectable-class start (an empty-class legacy run records NO kit), and is NOT a required `validate()` field for the legacy/empty-class run. Do NOT build a tactical `BoardState` player from the kit (that is Story 5.5's smoke slice / a later HUD/combat-wiring story); 5.3 records the applied kit as run domain state only. [Source: 5-1 Completion Notes forward-seam note; run_state.gd `selected_class_id` precedent; project-context.md typed-DTO + scene-free `scripts/run/` rules]

- The kit gate runs ONLY when `class_id` is non-empty AND the resolved class `is_selectable()` (the 5.2 gate already rejected unknown/locked before this — kit resolution never runs for a locked/unknown class).
