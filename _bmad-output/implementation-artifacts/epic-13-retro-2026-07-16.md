## 1. Epic Summary

Epic 13 is the two-story epic that made the MVP **human-playable** — it closed the gap between "the domain and the interactive seams work (Epic 12 proved a human CAN win a generated fight, headlessly)" and "a person can actually SEE the board, CLICK it, and COLLECT what the fight earns." The MVP-readiness gate had passed `READY_WITH_GATES` and Epic 12's retro had declared the hands-on WIN loop built — but the first human desktop playtest after the Epic-10 close (2026-07-13) found the tactical board region rendered as ONE summary text label (`tactical_board_presenter.gd:139`) and no input path existed anywhere (zero `_gui_input`/`Button` handlers; every tap method took an already-resolved `Vector2i` that only a test could supply). The interactive combat layer was real and regression-enforced, but nobody could reach it with a mouse. The 2026-07-13 correct-course inserted Epic 13 to close that purely-presentation/input gap, in two deliberate layers each strictly additive over the pinned Epic-2/6/12 contracts:

- **13.1 — Live board render + pixel→cell tap input (DONE, merged 2026-07-14, PR #69, suite 193 PASS).** Replaced the text-only board region with a real tile-grid render from `TacticalBoardViewModel` reads (terrain, hero, enemies, blockers/hazards, fog `hidden`/`memory`/`visible` states, and the level's affinity treatment from the approved board art under `godot/assets/`), and added a `_gui_input` hit-test that maps a tap pixel → board `Vector2i` and routes it into the EXISTING `interactive_submit_move` / `interactive_tap_attack` / `interactive_inspect` seams. The crux was anti-reinvention: the pixel↔cell geometry (`TacticalBoardZoomState`, Story 2.4) AND the tap seams (`interactive_*` + `bind_interactive_session`, Story 12.1) already existed and were unit-tested — so 13.1 is a "draw the grid + wire a hit-test into existing seams" job, not build-from-scratch. Two NEW pure `RefCounted` seams carry all assertable geometry/decision logic: `TacticalBoardGridFit` (square-cell centered fit that builds the SHARED `TacticalBoardZoomState` used for BOTH draw and hit-test — so they can never disagree) and `TacticalBoardTapRouter` (cell-state → move/attack/inspect/none intent); a thin `tactical_board_grid.gd` Control draws the presenter-built op list and forwards the tap. ## 4. What Was Challenging / Growth Areas

**Amelia (Developer):** The genuine surprise of the epic was that the approved board art had NEVER been imported. The PNGs were on disk under `godot/assets/` and marked `approved` in the manifest, but there were no `.png.import` sidecars and no `.godot/` cache — so `preload`/`load` of the textures could not resolve them, and the "black board" wasn't only the text-label render, it was also that the art pipeline had never actually been exercised in-engine. This was the FIRST story to import the Godot project. The fix was three-part and each part matters for any future art-consuming story: run a headless `--import` to generate the sidecars, COMMIT them, and load textures DEFENSIVELY at runtime (guarded `load()`, never `preload`) so the presenter still compiles on a fresh checkout where the machine-local `.godot/` cache is absent (it degrades to flat terrain-colored fallback tiles rather than crashing). The import also emitted ~405 `.gd.uid` script-UID sidecars, which forced the version-control-convention decision.

**Charlie (Senior Dev / Reviewer):** The one 13.1 finding worth being honest about is the `.gd.uid` version-control convention, because it was a `[Review][Decision]` — not something the code could resolve, a project-policy call. Because 13.1 was the first import, Godot 4.6 generated ~405 `.gd.uid` sidecars, and `godot/.gitignore` ignored `.godot/` and `.import/` but NOT `*.gd.uid` — so the next editor-open or headless `--import` would regenerate all ~405 as untracked files and break the mandatory "git status clean" invariant on every future story. Left unresolved that would have been a recurring finalize-hygiene landmine. The human ratified committing them (Godot 4.4+ guidance, like `*.import`); the dev regenerated + verified they are not matched by any `.gitignore` rule and left them for the orchestrator to commit. It's now in `project-context.md` as a standing convention, so it's a one-time cost, not a per-story one. The lesson: the first story to cross an engine-tooling boundary (import, UID, cache) inherits a project-wide convention decision that has to be made explicitly, not defaulted.

## 5. Key Insights

1. **When the seams already exist and are tested, the on-screen story is render + hit-test GLUE — recognize it and don't rebuild.** 13.1's crux was that `TacticalBoardZoomState` (pixel↔cell geometry, Story 2.4) AND the `interactive_*` tap seams (Story 12.1) were already built and unit-tested — so the story is "draw a grid from the VM + wire a `_gui_input` hit-test into the existing seams," adding no domain/VM/event/RNG/autoload change. The 12-1 defer and the investigation slightly understated what was already built; reading the existing seams first turned a "build the input layer" job into a wiring job.
2. **Build the draw and the hit-test off ONE geometry object.** A tile grid that draws with one `cell_size`/`origin` and hit-tests with another lands taps on the wrong cell. Constructing ONE `TacticalBoardZoomState` per `render()` from the current region rect + board dims, and using `cell_rect` for the draw AND `screen_to_cell` for the tap, makes "the tile you see is the tile you tap" structurally impossible to violate.
3. **A "render the existing offer" HUD story is really a "wire the GENERATE" story — read the live flow before scoping it render-only.** Nothing in the live flow generated a reward offer (the interactive victory branch advanced straight to the route map). AC1's "a resolved node yields a reward offer" and AC3's "live-HUD-proven end to end" are only non-vacuous if the story WIRES the generate at the interactive-shell boundary. The easy under-scope ("just render `pending_reward_offer`") would have shipped a HUD that never fires.
4. **Wire new live RNG into the INTERACTIVE path only — never the hands-off driver — to keep every fingerprint byte-identical.** The reward GENERATE + the Destroy roll are real `rewards`/`combat`-stream draws, but they're wired into `gameplay_shell_presenter._begin_reward_step` (the human-driven path no seed-regression fixture drives), never `run_to_completion`/`_resolve_combat`/`auto_play_boss_fight`. That single boundary is what lets a presentation epic add live RNG and still move ZERO pinned fingerprint.
5. **The first story to cross an engine-tooling boundary inherits a project-wide convention decision — make it explicitly.** 13.1 was the first import, which forced (a) the art-import pipeline (sidecars must be generated + committed; runtime loads must be defensive) and (b) the `.gd.uid` version-control convention (commit, don't gitignore). Neither is a per-story cost once decided, but each is a landmine if defaulted silently.
6. **A separate composed HUD surface — not a widened VM contract — is how you add on-screen data without breaking a pinned contract.** The affinity treatment reads the separate `_affinity_id`; the reward HUD is a separate composed overlay (like the affinity read / G1 `RunHudViewModel`). Neither added a 17th board-VM key or a `MODAL_KEYS`/`DICTIONARY_KEYS` field. Adding a key to "carry render data" is the tempting mistake that breaks the pinned-contract tests.
7. **Follow the story's own recommended mapping unless you argue the deviation in the story file.** 13.2's dev deviated from the story's Task-1 recommended node→table mapping without recording the argument, and it cost a review round. A create-story recommendation is a scope decision already reasoned once; departing from it silently re-opens that reasoning under adversarial review.
8. **Making a latent domain residual human-reachable is itself a finding — log the HUD gap, not just the domain footnote.** The full-backpack `inventory_full` soft-lock existed in the domain before 13.2, but 13.2 is the story that made a person able to hit it. The right move was to log the missing HUD escape hatch against the later replacement-choice UX owner, so the gap is tracked at the layer where it's now reachable.
