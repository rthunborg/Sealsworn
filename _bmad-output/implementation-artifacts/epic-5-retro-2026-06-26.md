**Alice (Product Owner):** Scope held to FR42-FR45 exactly. **Rasmus (Project Lead):** The save design is the quiet win — persist class-id only, re-derive kit and resolver on restore. ## 4. What Was Challenging / Growth Areas

**Amelia (Developer):** The `get_class()` -> `get_class_definition()` reserved-method collision was the one hard wall. `get_class()` is a native `Object` method, so *defining* it is a hard parse error — the spec literally named an impossible accessor. It forced a rename (`ClassRepository.get_class_definition(class_id)`), and the cost was that 5-2/5-3 story text still referenced the impossible `get_class(id)` and had to be corrected at create-time. Lesson captured: any future `get_<thing>` content accessor must avoid reserved `Object` method names.

**Dana (QA Engineer):** The false-PASS incident is the uncomfortable one — a test can compile-fail and still print PASS. Comfortable that the grep guard catches it now, but it's a reminder that the summary line is not trustworthy on its own.

**Charlie (Senior Dev / Reviewer):** The recurring forward-residual is the duplicate-id last-write-wins trap. It is unguarded across all SIX content repositories now (`passive_repository.gd` folded in by 5-4): `register_*` overwrites by `(type, id)` while `*_ids()` keeps the FIRST insertion, so a duplicate id makes the id-list and `get_*` silently disagree about the canonical definition. Harmless today (baseline ids are distinct code constants), but a latent fail-quiet trap the moment content moves to a JSON pipeline — and it directly conflicts with our fail-closed/fail-loud rule. It has now ridden through five stories untouched.

**Alice (Product Owner):** The AC4 "did the class change a combat decision?" human-felt pass on 5-5 is a tester deliverable that is still open. v0 passives are explanation-only, so the felt difference at this milestone is the equipment shape + the explanation surface, NOT a mutated combat number. The structural differentiation is proven (each class enters with a different weapon/support + a disjoint passive-explanation pair), but the human play-pass note is the tester's to complete — same pattern as the 1.11 / 3.7 / 4.6 playtest ACs.

**Rasmus (Project Lead):** Status hygiene slipped — 5-5 sits at `review` despite being merged and twice-approved. Small, but it's the kind of drift that makes sprint-status untrustworthy if it accumulates.

### Challenge themes
1. **Reserved-name collisions in content accessors** — `get_class` was undefinable; corrected to `get_class_definition`; downstream story text lagged.
2. **Summary-line dishonesty in the test runner** — compile-fail can print PASS; mitigated only by the raw-output grep guard.
3. **Cross-cutting duplicate-id hardening still open** — six repositories fail-quiet on duplicate ids; benign now, latent trap under a future JSON content pipeline.
4. **Open human deliverable** — 5-5 AC4 felt-difference tester pass not yet done (expected v0 boundary, not a bug).
5. **Status-field lag** — 5-5 left at `review` after merge + dual APPROVE.

## 5. Key Insights

1. **The seam strategy from Epic 1-4 generalised cleanly to content+rules.** "Keep a by-id lookup, add the new param last, validate-then-mutate" scaled from a single command to a three-gate content pipeline without a rewrite. This is now the house pattern for adding a starting-content type.
2. **Null-by-design + re-derive is the right save shape for derived run services.** The kit (5-3) and the resolver (5-4) both followed the inert-`RngStreamSet` precedent: live `RefCounted` services that are NOT serialized and are rebuilt from the persisted class-id on restore. 5-5 unified this into one canonical re-derive (`re_derive_kit` + `re_derive_resolver`) — the single place a future live tactical resume will rebuild a class start.
3. **The rules kernel exists but is deliberately inert.** `RuleTrigger` (fixed ten-window vocabulary + `is_valid_window`) and `RulesResolver` (register-by-window + resolve + explain, pure read, zero RNG) are real; `conditions/` and `operations/` are empty Epic-6 scaffolding. The kernel resolves and *explains* but mutates nothing — exactly the v0 line FR44/FR45 require.
4. **Verification guards are load-bearing, not ceremony.** The false-PASS grep and the 23-key snapshot gate each caught/prevented a real regression class this epic. Keep both as standing gates.

## 7. Next-Epic Preview — Epic 6: Loot, Passives, and Consume/Destroy

**Objective (epics.md line 1631):** Players shape a run through loot, passive rewards, a small inventory, consumables, and meaningful Consume/Destroy choices. **FRs covered:** FR46-FR53. **Implementation note (epics.md line 451):** *"Passive effects and reward outcomes use the rules kernel with explicit trigger windows, stable ordering, conflict handling, and explanation output."*

### Dependencies on Epic 5 (all satisfied and stable)
- `PassiveDefinition` / `PassiveRepository` + the `RuleTrigger` window vocabulary + `RulesResolver` — Epic 6 builds its 20-30 passive pool (FR46) and effect model on exactly this kernel.
- The empty `scripts/rules/{conditions,operations}` scaffolding — Epic 6 fills it with the `RuleCondition`/`RuleTarget`/`RuleOperation` evaluation model + stacking/conflict/duration.
- `RunState.starting_kit` + the three-gate `RunStartCommand` — the inventory/loot model (FR51, FR52) extends from here.
- The re-derive-on-restore seam — any new run-level service Epic 6 adds (inventory, reward state) should follow the same null-by-design + re-derive pattern OR nest under `route_state`, never a new top-level `RunSnapshot` key.

### Preparation / risks Epic 6 must account for
1. **The explanation-only -> felt-effect transition is the core Epic-6 build.** v0 passives explain but don't mutate. Epic 6 owns turning a `before_attack`/`run_started` explanation into an actual combat-number change (the per-effect operation model + combat hook sites in `AttackCommand`/`MoveCommand`/attack-preview). The 5-4/5-5 deferral notes are the precise handoff.
2. **First run-affecting RNG lands here — and trips the inert-stream trap.** The 4.6 defer is explicit: when a passive/reward first ROLLS, that draw MUST route through the orchestrator's run-level `streams` (and `LevelGenerator.generate` must accept an injected run-level `RngStreamSet` instead of minting a local one), or the route-position save/resume seam silently persists a stream disconnected from where draws happen — breaking interrupted==uninterrupted determinism. This is a **critical-path prerequisite**, not a nice-to-have.
3. **Duplicate-id hardening is now most valuable here.** Epic 6 is where content volume (loot, the 20-30 passive pool) and likely a JSON/data pipeline arrive — exactly the conditions that turn the fail-quiet duplicate-id trap from latent to real. Strong recommendation: land the all-six-repos fail-loud hardening story *before or alongside* the first data-pipeline content, with tests.
4. **Stable ordering + conflict handling is a new correctness surface.** The kernel resolves by window today with simple registration order; Epic 6's FR46 rule-benders + multi-passive stacking demand explicit, tested ordering/conflict/duration semantics. Budget test design for trigger/order/stacking/conflict (per the project testing rules) up front.

No Epic-6 dependency from Epic 5 is incomplete or unstable.
