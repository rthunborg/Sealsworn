## Story

As a player,
I want a summary of what happened during the run,
so that death, victory, rewards, and mistakes are understandable.

## Acceptance Criteria

(Verbatim from `planning-artifacts/epics.md` lines 2044-2074, Story 8.2; FR60 — "Run summary must show cause of death or victory, nodes cleared, boss or elite progress, passives consumed and destroyed, notable loot, Oath Shards earned, Echoes discovered, unlock progress, and seed." Also serves FR28 boundary hygiene — "Manual seed runs must not grant meta progression" — at the SUMMARY level: the summary REPORTS eligibility, it does not award. The SECOND story of Epic 8, building directly on the run-END boundary Story 8.1 shipped: 8.1 emits `run_failed` (with a `cause`) + a broadened `run_completed` (with an `outcome`) + a terminal `RunState` phase; 8.2 AGGREGATES those run-END facts plus the accumulated run STATE + the run's domain EVENT stream into a single, scene-free, serializable run-summary read surface. It does NOT award Oath Shards / build the meta profile (8.3), does NOT build the outpost menu scene (8.6), and does NOT deliver the first-death narrative line (8.5).)

**AC1 — The summary aggregates every FR60 field from run STATE + events, with unavailable future fields represented safely**
- **Given** a run ends
- **When** the run summary snapshot is built
- **Then** it includes cause of death or victory, nodes cleared, boss or elite progress, consumed and destroyed passives, notable loot, Oath Shards earned, Echoes discovered, unlock progress, and seed where available
- **And** unavailable future fields are represented as zero, empty, or not-yet-supported without breaking the summary.

**AC2 — Summary source truth is domain STATE + domain EVENT records, NOT presentation logs; manual-seed eligibility is included**
- **Given** domain events exist for the run
- **When** summary data is derived
- **Then** it uses domain state and event records rather than presentation logs as source truth
- **And** manual-seed eligibility is included.

**AC3 — The summary reads across EXPLICIT boundaries between run-scoped, profile/meta, and content-unlock state; replay/debug cannot grant progress**
- **Given** run-scoped state, profile/meta state, and content unlock state are summarized
- **When** the run summary is built
- **Then** the summary reads from explicit boundaries between those state types
- **And** replay/debug state cannot accidentally grant profile or unlock progress.

**AC4 — Summary tests cover death, shell completion, and manual-seed-ineligible fixture runs with expected field values**
- **Given** summary tests run
- **When** fixture runs end in death, shell completion, and manual-seed ineligible paths
- **Then** the summary fields match expected values.

**AC5 — Every not-yet-supported summary field is a TRACKED placeholder for the Epic-10 readiness gate; no placeholder grants progress**
- **Given** unavailable future summary fields remain represented as zero, empty, or not-yet-supported
- **When** Epic 10 readiness reviews the run summary
- **Then** each placeholder field is either replaced, explicitly de-scoped for MVP, or kept with a visible limitation note
- **And** no placeholder summary value grants rewards, unlocks, or profile progress.

## Scope boundary (read FIRST)

This story builds the **RUN SUMMARY** — a single, scene-free, pure-read, serializable DTO that AGGREGATES "what happened during the run" from the run's DOMAIN STATE and DOMAIN EVENT STREAM into the FR60 field set, so a later outpost/run-summary UI (8.6) can present it and so a later meta-award path (8.3) has a canonical summary to read. It is the "review what happened" half of Epic 8's "die or finish a run, return to the last outpost, **review what happened**, receive eligible progress, start another descent."

**⭐ THE SINGLE MOST IMPORTANT ARCHITECTURAL FACT — 8.2 is a PURE READ AGGREGATOR. It builds NO new domain state, NO new event, NO new save key, NO command, NO mutation, and NO RNG.** It is a read-only projection over data that ALREADY EXISTS:
- The run-END **facts** 8.1 shipped: `run_failed.cause` / `run_completed.outcome` (the events) and the terminal `RunState.phase` (`completed` / `failed`). These are the "cause of death or victory" input.
- The accumulated run **STATE**: `RunState.route.cleared_node_ids` (nodes cleared), `RunState.route`'s node types (boss/elite progress), `RunState.root_seed` (seed), `RunState.meta_progression_eligible` (manual-seed eligibility), `RunState.risk_economy` (gold / curse_count / corruption — economy readouts), `RunState.assigned_affinities`, `RunState.selected_class_id`, `RunState.inventory`.
- The run's **DOMAIN EVENT stream**: `passive_consumed` / `passive_destroyed` (passives consumed & destroyed — AC1), `item_gained` / `reward_resolved` (notable loot — AC1). **⭐ These lists have NO persisted home in v0** (`RunState` has no consumed-passive list; `RunSnapshot.passives`/`curses` are EMPTY placeholders — VERIFIED). So the summary DERIVES them from the event records the run emitted. This is EXACTLY what AC2 mandates ("domain state and event records rather than presentation logs as source truth").

**`[Decision]` — THE RECOMMENDED DISPOSITION (confirm in Completion Notes):** model the run summary as a scene-free `RefCounted` value object + read surface under `godot/scripts/run/` (RECOMMENDED name `RunSummary`, file `run_summary.gd`), mirroring the **`RunEndOutcome` (8.1) exact-key discipline VERBATIM**: a factory `RunSummary.build(...)` (a PURE read that takes the terminal `RunState` + the run's ordered `Array[DomainEvent]` log + the resolved run-end cause/outcome and returns the aggregated summary) and a `to_dictionary()` that projects an EXACT pinned `DICTIONARY_KEYS` set (a key never silently appears/vanishes; pinned by test). It reads `RunState` / `RiskEconomyState` / `RouteState` / the event payloads, computes the FR60 fields, and returns plain serializable data. No live handle leaks out. Repeated builds/reads are byte-identical (pure).
- **Where the events come from (`[Decision]`, pick + record):** v0 has NO run-level event STORE — commands RETURN their events in `ActionResult.events`, and the `RunOrchestrator` threads sequence ids but does NOT accumulate a run log. So `RunSummary.build` must ACCEPT the ordered event list as an EXPLICIT parameter (the caller — a test today, a later HUD/run-flow layer tomorrow — supplies the events it collected across the run). Do NOT invent a persisted event-log field on `RunState`/`RunSnapshot` (that is a larger save-shape decision; not this story). Do NOT read presentation/`CombatExplanationLog` state as source truth (AC2 forbids it). The summary is a pure function of `(terminal RunState, ordered domain events, resolved cause/outcome)`.
- **The run-end cause/outcome input (`[Decision]`, pick + record):** the "cause of death or victory" is EITHER re-derivable from the supplied events (scan for the `run_failed` / `run_completed` event and read its `cause` / `outcome`) OR passed explicitly by the caller (it holds the `CompleteRunCommand` / `resolve_run_end` result). PREFER deriving it from the event stream when a terminal run-end event is present, with an explicit-parameter fallback / cross-check — so the summary is self-consistent with the events (mirror how `RunOrchestrator.resolve_run_end` captures it from `resolved.events`). Whatever you choose: a `failed` run's summary carries its `cause`; a `completed` run's summary carries its `outcome`; the terminal phase and the unified `outcome_or_cause` mirror `RunEndOutcome`'s vocabulary.

**AC1's future-field placeholders (the load-bearing "not-yet-supported" contract, AC1 + AC5):** several FR60 fields have NO v0 domain source yet. They MUST appear in the summary as a stable zero/empty/not-yet-supported value that CANNOT be mistaken for a real award, AND each must be a TRACKED Epic-10 readiness placeholder (AC5). The v0 disposition per field (`[Decision]` — confirm the exact shape you ship):
- **Oath Shards earned** → **0** in v0. Oath-Shard AWARDING is Story 8.3 (`RunSnapshot.oath_shards` stays the 0 AWARDED-count placeholder; v0 awards none — VERIFIED `run_snapshot.gd` lines 265-266). The summary REPORTS the AWARDED count (0) AND the ELIGIBILITY (`meta_progression_eligible`, AC2) — it does NOT compute/grant an award (AC3/AC5). Reading `RiskEconomyState.oath_shard_eligible` READ-ONLY for the eligibility readout is fine; awarding is 8.3.
- **Echoes discovered** → **empty** in v0. Echoes/Seal-Fragments/unlock-progress are Story 8.4. No Echo domain content exists yet → an empty list / not-yet-supported marker.
- **Unlock progress** → **empty / not-yet-supported** in v0 (Story 8.4). No unlock-progress domain state exists yet.
- **Boss or elite progress** → DERIVABLE NOW from the route: count/flag whether the boss node was cleared and how many `elite_combat` nodes were cleared (read `RunState.route` node types cross-referenced with `cleared_node_ids`). This is a REAL v0 field, not a placeholder (the boss placeholder resolves to `run_completed` at the boss boundary; a `completed`/`boss_placeholder` run cleared the boss).
- **Notable loot** → DERIVABLE NOW from the `item_gained` / `reward_resolved` events (the "notable loot" list is the loot the run gained). Keep it honest: it lists what the events RECORD (by-id/category), not a curated "notable" ranking (curation/ranking is a later content/UX concern — note it if you flag anything as deferred).
- **Passives consumed & destroyed** → DERIVABLE NOW from the `passive_consumed` / `passive_destroyed` events. REAL v0 fields.
- **Cause of death / victory, nodes cleared, seed, manual-seed eligibility** → REAL v0 fields (run-end event + `route.cleared_node_ids` + `root_seed` + `meta_progression_eligible`).

Mark EVERY placeholder field with a stable, machine-detectable "not-yet-supported"/limitation signal (`[Decision]` — pick the shape: e.g. a per-field default of `0`/`[]` PLUS a `not_yet_supported`/`limitations` list naming the deferred fields, so the Epic-10 readiness pass (AC5) can enumerate them and the outpost UI can render an honest limitation note). Whatever you choose, a placeholder value must be structurally impossible to mistake for a real award/unlock (AC5 second half).

**AC3 — read across EXPLICIT state boundaries; replay/debug cannot grant progress.** The summary READS three DISTINCT state kinds and must keep them SEPARABLE in the output:
1. **Run-scoped state** (this run's `RunState` + its events) — the source for cause/nodes-cleared/loot/passives/economy/seed.
2. **Profile/meta state** (the cross-run profile — Oath Shards awarded, mastery) — DOES NOT EXIST YET (Story 8.3). The summary reports the AWARDED count as 0 and reads NOTHING from a profile (there is none). It must NOT create, mutate, or grant profile state — it is a READ.
3. **Content-unlock state** (Echoes/Seal-Fragments/unlock flags) — DOES NOT EXIST YET (Story 8.4). Reported empty; read-only; granted NOTHING.
The AC3 "replay/debug state cannot accidentally grant profile or unlock progress" is satisfied STRUCTURALLY because the summary is a PURE READ that mutates nothing and awards nothing — a manual-seed (replay/practice) run's summary reports `meta_progression_eligible == false` and the same 0/empty award/unlock fields as any other v0 run, and building it grants nothing regardless of eligibility. Record in Completion Notes that the summary is the read seam the 8.3 award path sits IN FRONT OF (8.3 reads eligibility + the summary, THEN awards behind the 8.1 idempotency guard; 8.2 itself never awards).

Do NOT redefine established patterns — REUSE them. 8.2 reuses: the `RunEndOutcome` scene-free exact-key read-DTO discipline (the `AffinityViewModel`/`HeroSelectViewModel`/`ClassStartSummaryViewModel` precedent — pinned `DICTIONARY_KEYS`, fail-closed on a non-terminal/null run, no live-handle leak, pure repeated reads); the run-END vocabulary 8.1 ratified (`RUN_FAILED_CAUSES`, `RUN_COMPLETED_OUTCOME_COMPLETED` / `RUN_COMPLETED_OUTCOME_BOSS_PLACEHOLDER`, `PHASE_COMPLETED`/`PHASE_FAILED`); the domain event payload shapes (read them, don't re-emit); the 23-key `RunSnapshot` gate (add NOTHING); the named-RNG rule (draw ZERO RNG — a read aggregation, no roll); the difficulty-non-goal guard (nothing scales by difficulty). It builds NO command, NO new event, NO new `RunSnapshot` key, NO `.tscn`, NO content.
