# Story 8.3: Meta Profile and Oath Shard Awards

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want eligible runs to award shallow meta progress,
so that repeated descents expand options without becoming a stat grind.

## Acceptance Criteria

(Verbatim from `planning-artifacts/epics.md` lines 2076-2107, Story 8.3; GDD FR95 + implementation FR67 ‚Äî "capped, sparse meta power" (epics line 399); FR59 ‚Äî "The outpost loop must support Oath Shards, Echoes, Seal Fragments, class mastery progress, unlock progress, and a shallow MVP meta menu or tree"; FR28 ‚Äî "Manual seed runs must not grant meta progression." The THIRD story of Epic 8. 8.1 shipped the run-END boundary (`run_failed`/`run_completed` + terminal phase + the `run_already_terminal` idempotency guard); 8.2 shipped the DERIVED `RunSummary` read aggregator (which REPORTS `profile_meta.oath_shards_earned == 0` and `meta_progression_eligible` READ-ONLY). 8.3 is the FIRST story that AWARDS meta progress and creates the FIRST persistent CROSS-RUN state: a versioned `ProfileSnapshot` + `ProfileRepository`, an Oath-Shard award CALCULATION from approved rules, a deterministic meta-award domain EVENT, and the FR28 manual-seed denial ‚Äî all sitting BEHIND the 8.1 idempotency guard so a re-completion never re-awards.)

**AC1 ‚Äî An eligible non-manual run awards Oath Shards (+ eligible progress) from approved rules; the profile is updated through a versioned repository**
- **Given** a non-manual eligible run ends
- **When** meta award calculation runs
- **Then** Oath Shards and eligible progress are calculated from approved rules
- **And** the profile snapshot is updated through a versioned repository.

**AC2 ‚Äî Profile changes are recorded through deterministic profile/meta events or repository results; run/profile/unlock state stay separable**
- **Given** meta awards are applied
- **When** profile state changes
- **Then** changes are recorded through deterministic profile/meta events or repository results
- **And** run state, profile state, and unlock/content state remain separable for save, replay, and tests.

**AC3 ‚Äî Direct power gains are capped, sparse, and secondary; raw-stat ladders are REJECTED for MVP (unless capped onboarding smoothing)**
- **Given** meta progression awards or unlocks can affect future runs
- **When** meta content validation runs
- **Then** direct power gains are capped, sparse, and secondary to variety, knowledge, classes, loot pools, passives, secrets, codex entries, starting options, or class mastery
- **And** broad raw-stat ladders such as repeatable damage, max HP, armor, crit, or dodge upgrades are rejected for MVP unless explicitly approved as capped onboarding smoothing.

**AC4 ‚Äî A manual-seed run awards NOTHING; the summary shows a replay/practice warning**
- **Given** a manual seed run ends
- **When** meta award calculation runs
- **Then** Oath Shards, class mastery, unlock progress, and other meta rewards are not granted
- **And** the summary shows a replay/practice warning.

**AC5 ‚Äî A profile SAVE FAILURE exposes a structured recovery path and does NOT silently lose the run summary**
- **Given** profile save fails
- **When** the repository reports the failure
- **Then** the system exposes a structured recovery path
- **And** it does not silently lose current run summary data.

## Scope boundary (read FIRST)

This story builds the **META PROFILE + OATH-SHARD AWARD path** ‚Äî the "receive eligible progress" half of Epic 8 ("die or finish a run, return to the last outpost, review what happened, **receive eligible progress**, start another descent"). It ships FOUR tightly-scoped pieces:
1. A **versioned cross-run `ProfileSnapshot`** (a scene-free serializable value object) ‚Äî the FIRST persistent CROSS-RUN state in the project.
2. A **`ProfileRepository`** with atomic-write + versioned-parse + migration + structured-error semantics, mirroring `SaveRepository` VERBATIM but with its OWN path/snapshot/schema.
3. An **Oath-Shard award CALCULATION from approved rules** (deterministic, capped, sparse) that reads the terminal run + `RunSummary` + eligibility and produces an award amount BEHIND the idempotency + eligibility gates.
4. A **deterministic meta-award domain EVENT** (appended to `DomainEvent.Type`) that RECORDS the award (AC2), plus the FR28 manual-seed denial (AC4) and the AC5 save-failure structured recovery.

**‚≠ê THE SINGLE MOST IMPORTANT ARCHITECTURAL FACT ‚Äî this is the project's FIRST persistent CROSS-RUN state.** Everything before this persisted only WITHIN a run (`RunSnapshot` = the run autosave). The profile OUTLIVES a run: it accumulates Oath Shards across many descents. The retro-notes (Epic-7 ¬ß7 Action T2 heads-up, re-stated for 8.3) mandate: **the profile is its OWN snapshot (a SEPARATE `ProfileSnapshot` + `ProfileRepository` + save file), NOT nested under `route_state` and NOT the `RunSnapshot`** ‚Äî with **migration coverage from the START** (its OWN `SCHEMA_VERSION` + an `unsupported`/migration path baked in on day one). This is the ledger's + retro's most-repeated 8.3 constraint.

**‚≠ê THE AWARD SITS BEHIND TWO GATES ‚Äî a re-completion NEVER re-awards, a manual-seed run NEVER awards:**
- **Gate 1 (idempotency, the 8.1 seam):** the award must run BEHIND the `run_already_terminal` idempotency guard. A run-end is resolved exactly ONCE (`CompleteRunCommand`/`RunOrchestrator.resolve_run_end` reject a re-resolution of an already-terminal run with `run_already_terminal` ‚Äî ZERO second event, ZERO mutation). The award fires as part of / after that single successful run-end resolution ‚Äî never on a re-completion. `RunSnapshot.oath_shards` (the run-scoped AWARDED count) stays 0; the CROSS-RUN awarded total lives in the profile.
- **Gate 2 (eligibility, FR28/AC4):** the award reads `RunState.meta_progression_eligible` (top-level, lockstep with `not is_manual_seed`, asserted in `RunState.validate()`) and/or `RiskEconomyState.oath_shard_eligible` (kept in lockstep). A manual-seed run (`meta_progression_eligible == false`) awards **NOTHING** ‚Äî no Oath Shards, no class mastery, no unlock progress (AC4). The eligibility gate is a HARD invariant, not a soft check.

**‚≠ê AC3 IS A CONTENT-DESIGN CONSTRAINT WITH TEETH.** The award rule must be **capped + sparse + secondary to variety**. For v0 the RECOMMENDED award is a SMALL, CAPPED, deterministic Oath-Shard grant (e.g. a fixed small amount per eligible completed run, or a small amount scaled ONLY by a bounded, non-difficulty run signal like nodes-cleared with a hard CAP) ‚Äî NOT a raw-stat ladder. `[Decision]` ‚Äî pick + record the exact rule. It MUST be:
- **DETERMINISTIC** (ZERO RNG ‚Äî same terminal run ‚Üí same award; the named-RNG rule + the whole-epic determinism invariant). NO `randi`/`randf`/`RandomNumberGenerator`; `RngStreamSet.required_streams()` stays 7 streams.
- **CAPPED** (a hard maximum per run; the award cannot grow unbounded ‚Äî AC3 "capped").
- **SPARSE + SECONDARY** (a shallow trickle that expands OPTIONS over time, not a stat grind ‚Äî the user story "without becoming a stat grind" + AC3 "secondary to variety").
- **NOT a raw-stat ladder** (AC3 second half: repeatable damage / max-HP / armor / crit / dodge upgrades are REJECTED for MVP). Oath Shards are a currency toward variety/knowledge/options (Epic-8/9 unlock spend); they are NOT a direct combat-stat buff. v0 AWARDS the currency; it does NOT build the unlock-SPEND tree (8.6/later) or apply any stat.

**‚≠ê NOT-YET-SUPPORTED FIELDS STAY DEFERRED (8.4 owns them).** 8.3 awards **Oath Shards** (the currency). Echoes / Seal-Fragments / class-mastery-points / unlock-progress are **Story 8.4** ‚Äî 8.3 does NOT author that content or merge it into the profile. AC1's "eligible progress" and AC4's "class mastery, unlock progress" are named because the FR28 DENIAL must cover them (a manual-seed run grants none of them), and because the profile shape should have a HOME for them (an empty/0 placeholder the 8.4 merge fills) ‚Äî but 8.3 GRANTS only Oath Shards. Keep the `RunSummary`'s `not_yet_supported` list (`["oath_shards_earned", "echoes_discovered", "unlock_progress"]`) honest: after 8.3, `oath_shards_earned` in the summary can reflect the REAL awarded amount for the just-ended eligible run (if you choose to feed it ‚Äî see Task 5), but `echoes_discovered`/`unlock_progress` STAY not-yet-supported placeholders (8.4).

**‚≠ê AC5 IS THE `SaveRepository` STRUCTURED-ERROR PATTERN.** A profile save failure must return a STRUCTURED `ActionResult.error(...)` (a stable lower_snake code + diagnostic metadata) ‚Äî NEVER a silent swallow, NEVER a crash, NEVER a loss of the current run summary. Mirror `SaveRepository.write_run_snapshot`'s atomic temp‚Üíbackup‚Üíreplace write with structured `save_open_failed`/`save_backup_failed`/`save_replace_failed` codes. AC5's "does not silently lose current run summary data" means: the run `RunSummary` (a DERIVED read, 8.2) is INDEPENDENT of the profile save ‚Äî a failed profile write leaves the summary fully readable; the caller can surface the summary + the structured save-error to a recovery UI (8.6). 8.3 makes the FAILURE VISIBLE and RECOVERABLE; it does not build the recovery UI (8.6).

Do NOT redefine established patterns ‚Äî REUSE them. 8.3 reuses: the `SaveRepository` atomic-write + versioned-parse + structured-error discipline (its DIRECT template ‚Äî copy the write/read/parse shape, change the path + schema); the `RunSnapshot`/`RiskEconomyState`/`InventoryState` exact-key `to_dictionary()` + lenient `try_from_dictionary`/`parse` value-object discipline (a key never silently appears/vanishes; pinned by test); the `DomainEvent` append-only enum + factory + validator + `expected_ids`-exhaustiveness-pin discipline (a new event is appended at the enum END, gets a factory + a payload validator, and is registered in `test_domain_event.gd` `expected_ids` ‚Äî the pin TRIPS by design, exactly as 8.1's `run_failed` did); the 4.3 validate-then-mutate command idiom IF you model the award as a command (validate ‚Üí award ‚Üí event, ZERO events + no mutation on reject); the named-RNG rule (draw ZERO RNG ‚Äî a deterministic calculation, no roll); the difficulty-non-goal guard (the award does NOT scale by difficulty). It builds NO outpost `.tscn`, NO unlock-spend tree, NO Echo/unlock content, NO first-death narrative.
