# Story 5.2: Hero Select with Playable and Locked Classes

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want to choose from MVP classes and see future locked classes,
so that I understand current options and future goals.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 5 -> Story 5.2 (verbatim AC, with the implementation-grounded detail this story adds).

1. **The hero-select surface presents Warrior/Pyromancer/Ranger as selectable and Necromancer/Shadeblade as locked, grayed-out future classes â€” projected from `ClassRepository`, not hardcoded in UI.**
   **Given** the hero select flow opens
   **When** class definitions are loaded
   **Then** Warrior, Pyromancer, and Ranger are selectable
   **And** Necromancer and Shadeblade are visible as locked, grayed-out future classes (the roster is projected from `ClassRepository.class_ids()` / `get_class_definition(id)` through a scene-free `HeroSelectViewModel`; selectability is `ClassDefinition.is_selectable()` / `lock_state`, NOT a UI literal).

2. **A locked class cannot be selected/confirmed; the unlock hint is surfaced; no run can start with a locked class.**
   **Given** a locked class is selected or tapped
   **When** the UI responds
   **Then** it shows the unlock hint or requirement (the view model exposes the locked class's `unlock_hint` as the displayable string; "grayed-out" maps to a `selectable: false` view flag)
   **And** no run can start with the locked class (the run-start seam rejects a non-selectable class id fail-closed with a structured `ActionResult.error` BEFORE any `RunState` is built â€” never a crash, never a silently-started run).

3. **Confirming a playable class creates a new-run request carrying that class id, and run startup receives the class through DOMAIN state â€” not UI-only state.**
   **Given** a playable class is selected
   **When** the player confirms
   **Then** a new run request is created with that class id (the confirm path produces a typed new-run request â€” `RunStartCommand` extended to carry the chosen `class_id` â€” NOT a bare seed; the class id is a single lower_snake `StringName` resolved through `ClassRepository`)
   **And** route/run startup receives class selection through domain state rather than UI-only state (the started `RunState` RECORDS the selected `class_id`; the class arrives at `NEW_RUN -> ACTIVE_ROUTE` by EXTENDING `RunStartCommand`, never a parallel/forked run-start path, and never read back off a UI node).

### Acceptance-criteria interpretation notes (read before coding)

- **This story is the run-start SEAM + a scene-free hero-select VIEW MODEL â€” NOT a polished `Control`/scene HUD.** It has two real deliverables: (a) EXTEND `RunStartCommand` (the existing Story-4.6 run-start command) so a fresh run is started WITH a selected `class_id`, resolved through `ClassRepository`, rejecting an unknown/locked class fail-closed and RECORDING the class on the started `RunState`; and (b) a `HeroSelectViewModel` (`RefCounted`, scene-free) that projects the class roster (selectable + locked-with-unlock-hint) from `ClassRepository` so the future hero-select scene reads domain truth, not UI literals. Building the actual `Control` scene + scene presenter is DEFERRED to a later HUD story, exactly as Story 2.5 deferred the adaptive-layout `Control` presenter while proving the contract scene-free (see "Presentation deferral" below). This honors the project's hard rule: do NOT start with UI-heavy scenes before the domain/command/view-model contract exists. [Source: project-context.md "Do not start implementation with UI-heavy scenes before the domain model, commands/events, RNG streams, and tactical board tests exist"; Story 2.5 deferral precedent]

- **CRITICAL â€” the class accessor is `ClassRepository.get_class_definition(class_id)`, NOT `get_class(id)`.** Story 5.1 shipped the accessor as `get_class_definition` because `get_class()` is a reserved native `Object` method and overriding it is a hard Godot parse error (Story 5.1 Decision E, human-ratified). The epics.md Story 5.2 text and this story's earlier draft language say `get_class` â€” that name is IMPOSSIBLE. Every class lookup in this story MUST call `ClassRepository.get_class_definition(id)` (and `has_class(id)` / `class_ids()` / `is_selectable()` keep their real names â€” none collide). [Source: 5-1 Completion Notes Decision E; deferred-work.md Story 5-1 Decision 3 (D3 ratified); epic-5 retro-notes Story 5-1]

- **EXTEND `RunStartCommand`; do NOT fork a parallel run-start path (the single biggest design risk).** `RunStartCommand` (`scripts/core/commands/run_start_command.gd`) already takes `(root_seed, is_manual_seed, sequence_id)` in its constructor and is the ONE `run_started` emitter (the Story-4.6 "Option A" run-start factory). 5.2 adds a class to that SAME command â€” the cleanest fit per the epic-4 retro's most specific Epic-5 prep item ("extend `RunStartCommand`'s inputs ... so the class is chosen at `NEW_RUN -> ACTIVE_ROUTE`, recorded on `RunState`, and surviving the route-position save") and Story 5.1's forward-seam note ("5.1 deliberately keeps the class a by-id content lookup so 5.2 EXTENDS `RunStartCommand` (not forks it)"). Do NOT write a second run-start command, a `HeroSelectCommand` that re-implements run creation, or a service that builds a `RunState` outside the command. [Source: epic-5 retro-notes Story 5-1 ("the run-start seam is a by-id class lookup that 5.2 should EXTEND"); 5-1 Completion Notes forward-seam notes; project-context.md run-progression rules "RunStartCommand is the ONE exception ... it BUILDS and returns the live RunState"]

- **`RunStartCommand` must remain BACKWARD-COMPATIBLE â€” the class id is OPTIONAL with a default empty (`&""`).** The Story-4.6 `RunOrchestrator.start(root_seed, is_manual_seed)` and `test_run_start_command.gd` construct `RunStartCommand.new(seed)` / `.new(seed, is_manual_seed)` / `.new(seed, is_manual_seed, sequence_id)` and assert a started run with NO class. Add the class as a NEW constructor parameter that DEFAULTS to empty so every existing call site still compiles and still produces a valid run. A started run with an empty `class_id` is the legacy "no class chosen" run (still valid â€” the orchestrator's seed-only start path and all 4.6 tests must stay green). The class is REQUIRED only when the hero-select confirm path supplies one; an empty class id is the back-compat path, NOT a validation failure. Decide + document whether you place `class_id` after `is_manual_seed`/`sequence_id` or as the last param (LAST is recommended so the existing positional `.new(seed, is_manual_seed, sequence_id)` calls are untouched). [Source: run_start_command.gd; test_run_start_command.gd; run_orchestrator.gd start()]

- **Run-start class resolution = RESOLVE through `ClassRepository`, fail closed on unknown OR locked (AC2 "no run can start with the locked class").** When a NON-empty `class_id` is supplied, `RunStartCommand` must resolve it through a `ClassRepository` and REJECT before building any `RunState`:
  - unknown id (`get_class_definition(id) == null`) -> structured `ActionResult.error` (e.g. `&"unknown_class"`), ZERO events, NO run built;
  - a class that resolves but is NOT selectable (`is_selectable() == false` / `lock_state == LOCK_STATE_LOCKED`) -> structured `ActionResult.error` (e.g. `&"class_not_selectable"`), ZERO events, NO run built.
  This is the exact "validate-then-mutate, byte-identical no-mutation on reject, build the event only after success" run-command idiom the 4.3/4.4/4.5/4.6 commands follow. The class check belongs in `validate(_state)` (a pure read) so `execute` short-circuits on it, mirroring the existing `sequence_id <= 0` / `root_seed < 0` gates. [Source: project-context.md "RUN-COMMAND IDIOM"; run_start_command.gd validate(); epics.md Story 5.2 AC2]

- **How does `RunStartCommand` get a `ClassRepository`? â€” INJECT it (constructor), defaulting to the baseline repository.** Mirror `RunOrchestrator`'s repository injection (`_recipe_repository`/`_enemy_repository` default to `*.create_baseline_repository()` and are test-injectable). Add an optional `ClassRepository` constructor arg defaulting to `ClassRepository.create_baseline_repository()` (or build it lazily inside `validate`/`execute` ONLY when a non-empty class id is supplied â€” but constructor injection is cleaner and matches the orchestrator). Do NOT make `RunStartCommand` an autoload, a `Node`, or reach for a global. The orchestrator's `start(...)` should thread the chosen class through to the command (extend `RunOrchestrator.start` to accept an optional `class_id` defaulting to empty, passed into `RunStartCommand.new(...)`). [Source: run_orchestrator.gd _init/_resolve_combat repository injection; project-context.md "Keep autoloads thin ... delegate gameplay decisions to domain services"]

- **Record the class on `RunState` so it is DOMAIN state â€” and persistence is a `[Decision]` you MUST resolve.** AC3's "run startup receives class selection through domain state" means the started `RunState` must carry the selected class. Add a `selected_class_id: StringName` field to `RunState` (default `&""`), set by `RunStartCommand` after `new_run(...)`. The persistence dimension is the live trap â€” pick ONE and record it as a `[Decision]`:
  - **(Option A â€” RECOMMENDED for 5.2) record `selected_class_id` on the in-memory `RunState` ONLY; do NOT add it to the route-position save payload yet.** `RunState.to_dictionary()` / `to_run_snapshot_fields()` / `try_from_*` stay UNCHANGED, so the pinned 23-key `RunSnapshot` no-surprise-key gate and the 4.6 route-position save/resume round-trip stay green with NO re-pin. Rationale: 5.2 only needs the class in domain state at run-start (AC3); the kit is applied in 5.3, and whether the SELECTED CLASS must survive a between-node save is naturally 5.3's concern (the kit/HP it derives is what actually needs persisting). Document that 5.2 deliberately keeps `selected_class_id` a live-run field NOT yet in the save bridge, and that 5.3 (or whichever story first needs class to survive resume) extends the persistence â€” and MUST nest any new run field UNDER `route_state` (never a new top-level `RunSnapshot` key) to keep the 23-key gate green. If you add it to `to_dictionary()` (the full RunState dict, which is NOT the 23-key snapshot), keep `try_from_dictionary` lenient (default empty when absent) so old payloads still parse.
  - **(Option B) thread `selected_class_id` through the route-position save now** (nested under `route_state` via a new `RunState` key, NOT a top-level `RunSnapshot` key) with a round-trip + lenient-default-on-absent test. Only do this if you also add the resume coverage; it is more surface than AC3 strictly needs and risks perturbing the 4.6 save gate. REJECTED as the default â€” keep 5.2 minimal and let 5.3 own class persistence alongside the kit.
  Either way: if you touch `RunState.to_dictionary()`/`try_from_dictionary`, the new field must be lenient-read (default `&""` when absent) so every existing run/route/save test stays green, and you must NOT add a top-level `RunSnapshot` key. [Source: project-context.md "Do not add a new top-level RunSnapshot key for run-progression state; nest it under route_state"; run_state.gd to_run_snapshot_fields(); 4.6 retro route-position save]

- **The `HeroSelectViewModel` is a scene-free `RefCounted` projection (AC1/AC2 presentation contract) â€” NOT a `Control`, NOT a scene.** It takes a `ClassRepository` and projects the roster as serializable view data: for each class in `class_ids()` order, a small dict with `class_id`, `display_name`, `selectable` (bool, from `is_selectable()`), and `unlock_hint` (the displayable string â€” only meaningful when `selectable == false`).
