## Story

As a player,
I want Darkness to create uncertainty without cheap unseen damage,
so that caution and inspection feel rewarding rather than unfair.

**⭐ THE SINGLE MOST IMPORTANT ARCHITECTURAL FACT — Darkness's effect is a VISIBILITY/FOG/MEMORY effect, NOT a board-cell HAZARD effect. It does NOT belong in the 7.5 hazard resolver.**

7.5's `AffinityEffectResolver.resolve_board_plan` + `AffinityPreviewQuery` are a BOARD-CELL HAZARD layer (Scorched stamps `HAZARD` terrain; Flooded marks conductive/pathing CELLS). Darkness's effect is a fundamentally different KIND — it shapes how far/what the hero can SEE (line-of-sight radius) and how reliable EXPLORED MEMORY is. It is NOT a cell-hazard. **Two existing tests already PIN that Darkness produces NO board-cell effect**, and they MUST stay green (they are correct — Darkness genuinely has no hazard-cell effect):
- `godot/tests/unit/rules/test_affinity_effect_resolver.gd::_darkness_is_a_no_op_in_7_5()` — asserts `resolve_board_plan(board, &"darkness", repo).has_effects == false` and no cues.
- `godot/tests/unit/tactical/test_affinity_preview_query.gd::_neutral_preview_is_a_legal_empty_effect_preview()` — iterates `&"darkness"` asserting `has_effects == false` and no `cue_ids`.

**`[Decision]` — THE RECOMMENDED DISPOSITION:** build Darkness as a SEPARATE, scene-free, deterministic, PURE-DOMAIN **visibility/memory-pressure layer** that operates on the Epic-1 fog/visibility model (`TacticalVisibilityQuery` + `BoardCell.visible`/`explored` + `visible_facts_for_cell`) GIVEN a level's assigned affinity id — NOT by extending the 7.5 hazard `AffinityEffectResolver`. This keeps the two existing Darkness-no-op tests green BY CONSTRUCTION (the hazard resolver's Darkness branch genuinely stays a no-op — Darkness's effect lives in the new visibility layer), and it models Darkness with the correct domain seam (visibility, not terrain). Do NOT thread Darkness into `resolve_board_plan`/`apply_board_effects`/`AffinityPreviewQuery` — those are for cell-hazard affinities. Confirm this `[Decision]` in Completion Notes.

Like 7.4/7.5, the Darkness layer is **BOARD/QUERY-SCOPED + CALLER-DRIVEN, NOT run-flow-wired.** The run orchestrator AUTO-RESOLVES combat (`RunOrchestrator._resolve_combat` — no live `BoardState`, no turn loop, no live LoS recompute in the run flow). ### ⭐ THE FAIRNESS GUARDRAIL is the heart of FR58 (and the "without unavoidable damage from unseen space" half of AC1 + all of AC3)

FR58 / the GDD Darkness guardrail (lines 507-512) is explicit: Darkness must create UNCERTAINTY, NOT cheap shots — "It must not spawn unavoidable damage from unseen space. The player should feel cautious and clever, not ambushed unfairly." The architecture already has a v0 home for this fairness semantic that 7.6 EXTENDS (it does NOT reinvent it):

- `LevelValidator._check_safe_first_reveal` (`godot/scripts/generation/level/level_validator.gd`) already rejects the entrance cell being HAZARD or entity-occupied, and its header + the `FIRST_REVEAL_RADIUS = 4` constant document the SAFE-FIRST-REVEAL v0 semantic AS the FR58 protection: "FR58 protects against damage from UNSEEN space, not from a visible adjacent enemy." A threat ADJACENT to the entrance is fair because the baseline LoS radius (4, FR5) reveals it on spawn.
- The KEY FR58 fairness risk Darkness introduces: **if Darkness REDUCES the LoS radius, a threat that was "seen on first reveal" at radius 4 might now be UNSEEN at the reduced radius** — re-opening exactly the "damage from unseen space" the guardrail forbids. So the Darkness fairness check must be **affinity-aware**: it re-asserts the safe-first-reveal / no-unavoidable-unseen-damage guarantee AT THE DARKNESS-REDUCED radius (a threat must still be either seen from the entrance at the reduced radius, OR avoidable / not forced to deal turn-1 damage from an unseen cell). This is the `[Decision]` core: define the Darkness fairness predicate so a Darkness level with the reduced radius still satisfies "no unavoidable damage from unseen space," and FAIL LOUD (with a stable fairness reason + the seed + the phase) when it does not (AC3).

**`[Decision]` — the fairness check is BOARD-SCOPED + AFFINITY-AWARE, NOT a generation-pipeline modification (the 7.5 posture):** 7.6 builds a Darkness-aware fairness check that runs over a BUILT `BoardState` + the assigned affinity + the Darkness-reduced LoS radius — it does NOT change `LevelValidator`'s wiring into the generation success path, does NOT change the generator's RNG draw order, and does NOT touch any pinned seed-regression fingerprint (the generator is affinity-blind in v0; the affinity is assigned POST-generation by the orchestrator, the 7.4 contract). Prefer a NEW affinity-aware fairness query/method (mirror the `LevelValidator` check shape: a stable lower-snake fairness CODE + COMPACT diagnostics + the phase) OR an additive Darkness fairness check that the existing validator can be GIVEN an affinity context to run — say which EXPLICITLY. Whatever you choose: NO `scripts/generation/` RNG-draw-order change, every Small/Medium/route fingerprint byte-identical. The "report seed, phase, and fairness reason" (AC3) means the failure result carries the seed (decimal-string if it can exceed 2^53 — the int64 save rule; the level seed is a String), a phase (reuse `LevelValidator.phase_for_code` → `validation`, or a documented Darkness fairness phase), and a stable fairness reason code.

### ⭐ AC2: memory distortion = surface UNCERTAINTY, never corrupt the VISIBLE truth (the structural guarantee already half-exists)

AC2 has two halves and they pull against each other; the existing model already encodes the safe resolution:
- "the UI can communicate uncertainty or stale memory state" — `visible_facts_for_cell` ALREADY returns a `memory` visibility-state branch with `authoritative: false` for explored-but-not-currently-visible cells. Darkness's `fog_memory_pressure` makes this memory LESS reliable / explicitly uncertain. 7.6 surfaces that uncertainty (an explicit "stale/uncertain" flag or an `inspect_memory`-family cue) — it does NOT need new persisted cell state.
- "current visible tactical truth remains reliable" — the `visible` branch (`authoritative: true`) MUST stay byte-stable. Darkness NEVER distorts a CURRENTLY-VISIBLE cell's facts (terrain, occupant, HP). It only affects (a) how FAR you can see (radius) and (b) how reliable the NON-visible MEMORY is. **`[Decision]`: model memory distortion as a READ-LAYER uncertainty annotation on the `memory` state (NOT a mutation of stored `BoardCell`/snapshot state, NOT a change to the `visible` branch).** This keeps the save snapshot + the visible-truth contract intact and is the honest, fair reading of "distort explored memory" — the memory is FLAGGED uncertain, the live truth is untouched. Surface it via a Darkness-aware sibling of `visible_facts_for_cell` (or an additive, opt-in `affinity_id` parameter that defaults to the byte-identical neutral behavior — do NOT change the default/neutral output of the existing query, or you break every Epic-1 visibility test).
