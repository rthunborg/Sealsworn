# Story 7.2: Curse and Corruption Rules

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want cursed or corrupted choices to show clear upside and downside,
so that risky rewards feel tempting but readable.

## Acceptance Criteria

(Verbatim from `planning-artifacts/epics.md` lines 1890-1911, Story 7.2; FR55, building on FR54.)

**AC1 — Cursed/corrupted offer view model: clear upside + clear downside, honest about hidden/delayed consequences**
- **Given** a cursed or corrupted reward is offered
- **When** the offer view model is built
- **Then** it exposes the clear upside **and** clear downside before acceptance
- **And** hidden or delayed consequences are labeled honestly.

**AC2 — Accept applies BOTH benefit and penalty via domain events + updates curse/corruption save state**
- **Given** the player accepts a cursed or corrupted reward
- **When** the command resolves
- **Then** the benefit **and** the penalty are both applied through domain events
- **And** curse/corruption state is updated in the run snapshot.

**AC3 — Curse/corruption effect resolves through the rules kernel deterministically, with source-identifying explanation**
- **Given** a curse or corruption effect has a trigger
- **When** the trigger window occurs
- **Then** the rules kernel resolves the effect deterministically
- **And** its explanation identifies the curse or corruption source.

## Scope boundary (read FIRST — this story is the CURSE/CORRUPTION RULES + cursed-reward TRADEOFF + cleanse, building on 7.1 state)

7.1 shipped the live `RiskEconomyState` **state fields** (`gold` / `healing_charges` / `curse_count` / `corruption` / `oath_shard_eligible` / `risk_flags`) + the **structural setters** (`set_curse_count` / `set_corruption` / `add_risk_flag`), the direct `ApplyEconomyChangeCommand` (gold/healing only), and the unified `economy_changed` event — but it authored **NO curse RULES, NO cursed-reward view model, and did NOT wire the cleanse/curse application**. 7.1's own scope-boundary and the deferred-work ledger BOTH explicitly assign all of that to **Story 7.2 (this story)**. So 7.2 is the layer that makes curses/corruption MEAN something: a readable cursed-reward tradeoff, accept-applies-both-sides, a curse/corruption EFFECT that resolves through the rules kernel, and the cleanse hook. Do NOT redefine the 7.1 state — EXTEND it.

**IN scope (this story):**
1. **A typed, validated `CursedRewardDefinition`** (a new content definition, the exact `GoldRewardDefinition` / `DestroyOutcomeTableDefinition` typed-Resource + `validate()` + lower_snake-id shape) carrying the AC1 tradeoff contract: an UPSIDE (the benefit), a DOWNSIDE (the penalty — a curse/corruption increment and/or a resource cost), and the **honest hidden/delayed-consequence labeling** (the 6.4 `has_unknown_consequences` + `consequences_text` honest-unknown contract, VERBATIM — a blank consequences line is invalid; "honestly unknown" is valid + surfaced). A small approved baseline set authored as a code constant (NO JSON pipeline — `data/source`/`data/resources` stay empty).
2. **A cursed-reward VIEW MODEL** (AC1) — a scene-free read surface that, given a `CursedRewardDefinition`, exposes the clear upside, the clear downside, and the honest hidden/delayed-consequence label BEFORE acceptance. This is presentation-facing DATA (a view model / read), NOT a `.tscn` scene (UI-scene-last) and NOT a mutation. Mirrors the 6.4 `PassiveRewardCommitFlow` / passive-modal data-contract posture: build the read surface; the HUD scene is a later story.
3. **An `AcceptCursedRewardCommand`** (AC2) — a run-domain command (the 4.3 run-command idiom VERBATIM: extends `game_command.gd`, takes `RunState` directly, rejects `sequence_id <= 0` FIRST, validate-then-mutate, byte-identical no-mutation + ZERO events on every reject) that, on accept, applies BOTH the benefit AND the penalty to the run's `RiskEconomyState` (gold/healing via the 7.1 `apply_*_delta` API; curse/corruption via the 7.1 `set_curse_count`/`set_corruption` setters) and emits domain events recording BOTH sides (AC2 "the benefit and the penalty are both applied through domain events"). Curse/corruption state is thereby updated and rides the save (AC2 "updated in the run snapshot" — the 7.1 nested-`route_state` persistence already carries `curse_count`/`corruption`).
4. **The curse/corruption EFFECT resolution through the RULES KERNEL** (AC3) — a curse/corruption effect DECLARES the trigger window(s) it fires in (from the FIXED `RuleTrigger` vocabulary, EXACTLY like a passive), and the run's `RulesResolver` resolves it deterministically alongside passives in stable order, surfacing an explanation that **identifies the curse/corruption SOURCE** (AC3). See the CRITICAL DESIGN DECISION section below for HOW to seat a curse as a rule source on the resolver.
5. **The CLEANSE hook** (the GDD "Destroy a passive to cleanse or reduce curse", lines 359 + 471) — wire a curse-REDUCTION mutation off the existing Epic-6 `passive_destroyed` outcome (the deferred-work item the 7.1 dev phase explicitly assigned to "7.2 curse-removal half"). When a Destroy outcome is a cleanse outcome, reduce `curse_count`/`corruption` via the 7.1 setter and record it. See Task 6 for the cleanse wiring.
6. Full unit tests: definition valid/invalid; view-model upside/downside/honest-unknown; the accept command valid + every invalid-no-mutation branch + both-sides-applied + determinism + ZERO/named-RNG; the new event(s) factory/validator/round-trip/malformed + the `expected_ids` enum-exhaustiveness pin; the rules-kernel curse resolution (trigger window / stable order / source-identifying explanation / determinism); the cleanse reduction; the curse/corruption save round-trip (extends the 7.1 nested-`route_state` coverage).

### CRITICAL DESIGN DECISION — how to seat a curse as a RULES-KERNEL rule source (AC3)
This is the single hardest decision in 7.2 and the one most likely to cause a review cycle. AC3 demands the curse/corruption effect "resolves through the rules kernel deterministically" and "its explanation identifies the curse or corruption source." The rules kernel TODAY is intentionally minimal:
- `RulesResolver` (`scripts/rules/resolver/rules_resolver.gd`) holds `Array[PassiveDefinition] _registered_passives`, keyed by the trigger windows each passive declares; `resolve(window_id)` / `explain(window_id)` return the matching passives / their explanations in STABLE registration order. It is a PURE READ (no RNG, no command, no mutation).
- `RuleTrigger` (`scripts/rules/triggers/rule_trigger.gd`) is the FIXED ten-window vocabulary (`run_started`, `level_entered`, `turn_started`, `before_move`, `after_move`, `before_attack`, `damage_calculated`, `enemy_killed`, `reward_offered`, `level_completed`). Do NOT add/rename/renumber windows.
- `scripts/rules/{conditions,operations}` are EMPTY scaffolding — the per-effect OPERATION engine (RuleCondition/RuleTarget/RuleOperation + combat hook sites) is a LATER story. v0 rule sources are EXPLANATION-ONLY.

The resolver currently types its registry as `Array[PassiveDefinition]`. To resolve a CURSE alongside passives you have two clean options — pick the LEANER one and record it as a `[Decision]`:

- **(Option A — RECOMMENDED) A `CurseDefinition` that satisfies the SAME `fires_in_window(window_id)` + `explanation` shape the resolver already uses, registered into the resolver.** The resolver calls `definition.fires_in_window(window_id)` and reads `definition.explanation` (rules_resolver.gd:42-57) — a curse type with those two members + a `passive_id`-like id resolves through the EXISTING machinery with the LEAST new surface. The hazard: `_registered_passives` is typed `Array[PassiveDefinition]`, so you either (a) broaden the resolver registry to a shared rule-source interface/base both types satisfy (more surface, but the architecturally-correct "any rule source" model AC3 hints at), or (b) keep a SEPARATE `Array` of registered curse effects on the resolver with a parallel `resolve_curses`/`explain` path and MERGE them into `resolve`/`explain` in a STABLE order. Prefer the smallest change that keeps `resolve`/`explain` returning curses + passives together in a deterministic order with source-identifying explanations. The curse's explanation MUST name its source (e.g. include the `cursed_reward_id` / a curse name) so AC3's "identifies the curse or corruption source" holds.
- **(Option B) A generic rule-source registration** where the resolver collects any object exposing `fires_in_window` + `explanation` (a duck-typed or base-class rule source). This is the more general "rules kernel" model but is MORE surface than v0 needs for one curse type — only choose it if Option A's typing forces an ugly cast.

WHICHEVER you choose: (1) the resolver STAYS a pure read (no RNG/command/mutation); (2) the curse resolves in a STABLE deterministic order (registration order, the 5.4 ratified order); (3) the explanation IDENTIFIES the source (AC3); (4) you do NOT author the operation engine (`scripts/rules/{conditions,operations}` stay empty) — v0 curse resolution is EXPLANATION-ONLY, exactly like v0 passives; the economy-side penalty is applied by the COMMAND, not the resolver.
