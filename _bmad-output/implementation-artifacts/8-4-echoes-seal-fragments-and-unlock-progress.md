# Story 8.4: Echoes, Seal Fragments, and Unlock Progress

Status: review

## Story

As a player,
I want discoveries and unlock progress to be tracked,
so that meta progression expands knowledge, variety, and starting options.

## Acceptance Criteria

1. **Given** a run discovers an Echo, Seal Fragment, class mastery point, or unlock flag **When** the run ends **Then** eligible progress is merged into the profile snapshot **And** duplicate discoveries do not grant duplicate unique unlocks.
2. **Given** a manual seed run discovers progress-bearing content **When** progression is evaluated **Then** the content may appear in the run summary as discovered during replay **And** it does not grant permanent meta progress unless explicitly allowed by policy.
3. **Given** unlock progress reaches a configured threshold **When** the profile updates **Then** the unlock state changes deterministically **And** the run summary reports the unlock or progress change.

### DISCOVERY SOURCE DECISION (the primary design decision — resolve at implementation)

There is **NO in-run domain source** for Echo / Seal-Fragment / class-mastery / unlock discoveries in v0 — grep confirmed: `RunState` has no echoes/mastery/unlock/fragment/discover fields, and no discovery event exists. This exactly mirrors 8.2's situation with `passives_consumed`/`notable_loot`: those lists have NO persisted home, so `RunSummary.build` DERIVES them from the ordered `DomainEvent` list the run emitted. **Adopt the same pattern**: introduce a `content_discovered` SYSTEM event (Task 1), and both the merge command and the summary derive the discovery list from the run's event list. This keeps AC2's boundary honest ("domain state and event records rather than presentation logs as source truth" — the 8.2 AC2 rule) and means a manual-seed run's summary can REPORT discoveries (from events) while the MERGE denies the grant (Gate 1).

- Do NOT invent a persisted `RunState.discoveries` list / a new top-level `RunSnapshot` key — that is a larger save-shape decision (8.7 territory; 8.2 explicitly deferred persisting the run log). The discovery events are the source; the profile is the persisted DESTINATION.
- v0 has NO live combat / real content-discovery source that FIRES a `content_discovered` event (combat auto-resolves; the Destroy-outcome category `progress_unlock_hidden_flag` RECORDS but does not yet emit a discovery — that wiring is a later story). 8.4 ships the EVENT + the MERGE + the SUMMARY-report + the tests (driven by test-supplied event lists, exactly as `test_award_meta_progress_command.gd` supplies `run_completed` events). The live per-node discovery call site is a later HUD/run-flow story — do NOT auto-wire it into `run_to_completion`.

### SEAL FRAGMENTS HOME DECISION (resolve at implementation — story title names it, but no dedicated home exists)

`ProfileSnapshot` reserves `echoes` (Array), `class_mastery` (Dict), `unlock_progress` (Dict) — but **NO dedicated `seal_fragments` field**. The GDD (line 270) defines Seal Fragments as "major seal/story unlocks" — i.e. UNLOCK PROGRESS. **Recommended resolution: model Seal Fragments INSIDE `unlock_progress`** (e.g. `unlock_progress["seal_fragments"]` as a unique-id set or count, plus per-unlock-track progress). This honors the 8.3 explicit decision that the reserved homes let 8.4 "merge WITHOUT a migration" — adding a new top-level `ProfileSnapshot` key would force a `SCHEMA_VERSION` bump + a migration (8.7's matrix) and contradict the reserved-homes plan. If a reviewer insists Seal Fragments need their own top-level field, that is a deliberate schema-v2 decision requiring a migration test in the SAME change — do NOT do it silently. Document whichever choice in the Completion Notes as a cross-story `[Decision]`.

### IDEMPOTENCY DECISION (resolve at implementation)

AC1 forbids double-granting on both (a) re-invocation and (b) duplicate ids in one discovery list. Two layers:
- **Per-invocation**: the merge for a terminal run must be idempotent. Cleanest: reuse the run identity marker the award already sets. If 8.4's merge is called as part of the SAME run-end meta step as the award, `profile.last_awarded_run_seed` already gates re-award — but the MERGE may be a SEPARATE command invoked independently, so decide: (i) a shared `last_awarded_run_seed` gate (whole run-end meta step is one idempotent unit — simplest, but couples merge ordering to award), or (ii) a dedicated `last_merged_run_seed` marker (needs a new `ProfileSnapshot` field → a schema-v2 migration, which contradicts "merge without migration"). **Prefer (i)** unless the run-flow requires merge-before-award. Whichever, a re-merge for an already-processed run rejects with a stable code + ZERO mutation.
- **Per-item (SET semantics)**: `echoes` is a SET of unique ids — appending an id already present is a no-op; a Seal Fragment already unlocked is not re-unlocked; a class-mastery point may legitimately ACCUMULATE (a count that rises), so mastery is additive-per-run but the per-RUN merge is still idempotent via the invocation gate. This makes AC1's "duplicate discoveries do not grant duplicate unique unlocks" true even on a FIRST merge with a dirty list.
