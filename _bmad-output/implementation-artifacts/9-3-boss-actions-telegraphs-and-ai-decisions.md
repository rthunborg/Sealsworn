## Story

As a player,
I want the boss to telegraph major danger and choose explainable actions,
so that the finale is dangerous but readable.

## Acceptance Criteria

(Verbatim from `planning-artifacts/epics.md` lines 2256-2289, Story 9.3; FR63 ŌĆö "The Larval Avatar must be implemented as the only required MVP boss." The THIRD story of Epic 9 ŌĆö "Larval Avatar MVP Finale." It consumes the boss CONTENT + PHASE MODEL Story 9.2 shipped (the validated `BossDefinition` + per-phase legal-action set + `BossPhaseResolver` + `boss_phase_changed` event) and turns it into RUNTIME behavior: the boss TELEGRAPHS a major dangerous ability BEFORE damage resolves with a response window, ROUTES its movement/attack/telegraph/phase/effect actions through existing commands / rules-kernel operations / a NARROW boss command adapter (never mutating tactical state directly), SELECTS among its phase-legal actions via reproducible UTILITY SCORING with score/reason output, and ŌĆö when a telegraphed attack resolves with the player still in danger ŌĆö applies deterministic damage/effect events whose explanation log identifies the boss ability. It does NOT build the boss VICTORY + first-victory reveal (9.4), the finale regression/tuning hooks (9.5), the 9.1 encounter setup, or the 9.2 definition/phase model.)

**AC1 ŌĆö A major dangerous ability telegraphs before it resolves, with a response window**
- **Given** the boss has a major dangerous ability
- **When** it selects that ability
- **Then** a telegraph event is emitted before damage resolves
- **And** the player has a reasonable response window.

**AC2 ŌĆö Boss actions route through existing commands / rules-kernel ops / a narrow adapter, never mutating state directly**
- **Given** the boss chooses a movement, attack, telegraph, phase, or effect action
- **When** the action resolves
- **Then** it uses existing commands, rules-kernel operations, or a narrow boss command adapter
- **And** boss logic does not mutate tactical state directly.

**AC3 ŌĆö Utility scoring selects a reproducible action with score/reason output**
- **Given** the boss chooses among valid actions
- **When** utility scoring runs
- **Then** the selected action includes score or reason output
- **And** the choice is reproducible for the same seed and state.

**AC4 ŌĆö A resolved telegraphed attack applies deterministic damage/effects and the log identifies the boss ability**
- **Given** a telegraphed boss attack resolves
- **When** the player remains in danger
- **Then** damage or effect events apply deterministically
- **And** the explanation log identifies the boss ability.

## Scope boundary (read FIRST)

This story closes the **boss RUNTIME-BEHAVIOR boundary**: it makes the Larval Avatar a **decision-making, telegraph-emitting, damage-applying combatant** that reads its 9.2 content. Concretely it ships (a) a **boss AI** ŌĆö the `PrototypeEnemyAi` analogue ŌĆö that scores ONLY the active phase's legal actions (`BossDefinition.legal_action_ids(phase_index)`) and returns a reproducible `AiDecision`-shaped choice with a score + reasons; (b) a **two-turn TELEGRAPH ŌåÆ RESOLVE flow** for major dangerous abilities (the boss emits a telegraph event this turn; the damage/effect lands on a later turn only if the player is still in danger) mirroring the Ash Seer mark ŌåÆ detonation precedent; (c) a **narrow boss command ADAPTER** that turns a chosen boss action into the SAME existing domain events (`entity_moved`/`entity_attacked`/`damage_applied`/`tile_marked`/`marked_tile_detonated`/`enemy_waited`) applied through `BoardState.apply_events` ŌĆö the boss NEVER mutates board/entity/turn state directly; and (d) the **live phase re-resolution seam** that, after boss HP changes, calls 9.2's `BossPhaseResolver` and emits the `boss_phase_changed` event(s) it returns ŌĆö **closing the two 9.2-deferred Low findings (the resolverŌåÆevent integration seam + the `from_phase = -1` defensive branch)**. It is the "dangerous but readable" runtime half of Epic 9's boss.

**ŌŁÉ THE SINGLE MOST IMPORTANT ARCHITECTURAL FACT ŌĆö 9.3 makes the boss ALIVE on a board and drives its turns; this is where the long-parked LIVE-TACTICAL-LOOP seam (Epic-7 retro T1, re-flagged across 9.1/9.2) FINALLY comes due.** 9.1 shipped the encounter SETUP (`BossArenaBuilder` emits an arena with a `boss_slot` marker + an EMPTY `entities` array; `boss_slot == {x, y, entity_id: "larval_avatar", definition_id: "larval_avatar", is_placeholder: true}`). 9.2 shipped the boss CONTENT (`BossDefinition` with `max_hp = 36`, 3 phases `emergence@100 / adaptation@60 / desperation@25`, per-phase legal actions with telegraph_text + damage + damage_type + explanation) + the pure `BossPhaseResolver` + the `boss_phase_changed` event ŌĆö but authored NO live board entity and NO turn loop (9.2 is content-and-state-machine-only; its tests drive the resolver with explicit `(phase, hp)` inputs). **9.3 is the first story that instantiates a LIVE boss `TacticalEntityState` on the arena board (with `max_hp` from the `BossDefinition`) and resolves its turns.** `TacticalEntityState.validate()` requires `max_hp > 0` ŌĆö the `BossDefinition.max_hp` (36) satisfies it, so the boss becomes a real board entity (`entity_type = ENEMY`, a distinct faction if you use one). **RECORD the live-loop boundary you draw** (see the OUT-of-scope live-loop note): 9.3 needs a live boss turn to prove telegraph ŌåÆ resolve + AI + damage + the phase re-resolution, but it does NOT need to auto-wire the full run-to-completion combat loop, the victory (9.4), or a player-death path. Keep the live board headless, deterministic, and driven by explicit test turns.

**ŌŁÉ REUSE THE ASH SEER TELEGRAPH MACHINERY ŌĆö do NOT invent a parallel telegraph system.** The `mark ŌåÆ delayed detonation` two-turn flow the Ash Seer already ships is the exact "telegraph a major danger this turn, resolve it a later turn if the player stays in danger" shape AC1/AC4 demand. Study `EnemyCommandAdapter._apply_mark` / `_apply_detonation` + `PendingTelegraphState` + the `tile_marked` / `marked_tile_detonated` events (all read below in Dev Notes). The boss's "major dangerous ability" telegraph SHOULD reuse this `PendingTelegraphState` pending-telegraph vocabulary (a pending entry with `kind`, `source_entity_id`, `target_entity_id`, `marked_cell`, `created_turn_number`, `due_turn_number`, `damage`, `damage_type`, `status: "pending"`) ŌĆö either by reusing the `ash_seer_mark` kind verbatim (cleanest if the boss's telegraphed ability is genuinely a marked-tile detonation) OR by adding a NEW boss telegraph `kind` (e.g. `larval_avatar_lash` / a per-ability kind) to `PendingTelegraphState` with the same validation shape. **Record the `[Decision]`.** The "reasonable response window" (AC1) is the SAME `due_turn_number == created_turn_number + 1` one-turn gap the Ash Seer uses (the player gets their turn between the telegraph and the resolution ŌĆö that IS the response window; do not invent a real-time timer ŌĆö NFR10 forbids reflex/timing requirements).

**ŌŁÉ THE BOSS ACTION VOCABULARY IS ALREADY DEFINED ŌĆö the adapter maps it to the SAME existing events.** 9.2's `BossActionDefinition` gives each action an `action_id` (`lash` / `skitter` / `corrupt_mark` / `frenzied_lash` / `corrupt_flood`), a `telegraph_text`, a `damage` (int >= 0; 0 == a pure-telegraph/reposition read like `skitter`), a `damage_type` (`physical` / `corruption`), and an `explanation`. The boss AI selects one of these ids (constrained to the active phase); the boss adapter turns the selected action into events:
- a **movement** action (`skitter`, damage 0) ŌåÆ an `entity_moved` event (the `_apply_move` shape ŌĆö one cardinal step toward the player) applied through `apply_events`.
- a **direct attack** action (`lash` / `frenzied_lash`, damage > 0, no telegraph gate) ŌåÆ an `entity_attacked` + `damage_applied` pair (the `_apply_attack` shape) IF you choose to resolve some boss attacks immediately, OR route ALL damaging abilities through the telegraph gate ŌĆö **record which abilities telegraph and which (if any) resolve same-turn**; AC1 requires AT LEAST the "major dangerous ability" to telegraph, not necessarily every attack.
- a **telegraphed effect** action (`corrupt_mark` / `corrupt_flood`, the marked-tile danger) ŌåÆ a `tile_marked` this turn (adds a pending telegraph) then a `marked_tile_detonated` (+ `damage_applied` if the player is on the marked cell) a later turn (the `_apply_mark` / `_apply_detonation` shape).
- a **phase** "action" is NOT an AI-selected action in the same sense ŌĆö phase change is a CONSEQUENCE of HP crossing a threshold (9.2's resolver), emitted as `boss_phase_changed`. Treat "phase" in AC2's action list as "the phase-transition events the adapter/loop emits after damage," not a scored menu entry. **Record this.**
- an **effect** action (corruption flood) reuses the marked-tile/damage events; do NOT build the generic `RuleOperation` engine for it (see the OUT-of-scope rules-kernel note).

**ŌŁÉ DETERMINISM / REPRODUCIBILITY (AC3) ŌĆö the boss AI draws ZERO RNG (mirror `PrototypeEnemyAi`).** The prototype enemy AI is a PURE utility read: it scores actions from board geometry (distance, alignment, line-of-sight, pending marks) with FIXED integer scores (`attack = 100`, `detonate = 120`, `mark = 80`, `move = 50`, `wait = 0`) and NEVER draws RNG ŌĆö its reproducibility comes from being a pure function of `(board, enemy, player_id, pending_telegraphs, turn_number)`. **The boss AI follows this EXACTLY: a pure function of `(board, boss_entity, player_id, pending_telegraphs, turn_number, active_phase_index)` returning a deterministic `AiDecision`-shaped choice with a stable integer score + `reasons` array.** AC3's "reproducible for the same seed and state" is satisfied by pure determinism (same inputs ŌåÆ same decision), NOT by seeding an RNG. If ANY boss decision genuinely needs a random tie-break (it should not ŌĆö author distinct scores so ties never occur), it MUST draw the `combat` stream through the run/tactical RNG set, NEVER `randi()`/`randf()`/a fresh `RandomNumberGenerator` ŌĆö but the RECOMMENDED and simplest design is a fully deterministic scorer with no ties. Record the choice.

**ŌŁÉ CLOSE THE TWO 9.2-DEFERRED LOW FINDINGS ŌĆö they were explicitly earmarked for 9.3's live-call-site wiring.**

The live loop always starts the boss at phase 0 (always active), so this branch stays unreachable in production, but the internal contract must be consistent. Record the reconciliation choice.

**ŌŁÉ EVERY boss-outcome event is EXPLAINABLE (AC4 + the architecture Readability Rule).** Every damage/effect/move/telegraph/phase event the boss emits carries an `explanation` (and `action_id` + `score` + `reasons` where the adapter pattern does) so the combat explanation log can identify the boss ABILITY that hit the player. Do NOT redefine the established patterns ŌĆö EXTEND them. 9.3 reuses: the `PrototypeEnemyAi` pure-utility-scoring shape (fixed integer scores, `reasons` array, ZERO RNG, a pure function of board state) ŌåÆ the boss AI is its analogue scoring the 9.2 per-phase legal-action set; the `EnemyCommandAdapter` narrow-adapter shape (a `match decision.action_id` that turns a chosen action into existing `DomainEvent`s applied through `board.apply_events`, NEVER a direct board mutation) ŌåÆ the boss adapter is its analogue; the `PendingTelegraphState` + `tile_marked` / `marked_tile_detonated` two-turn telegraph flow (a pending entry with `created_turn_number` / `due_turn_number` / `status`) ŌåÆ the boss telegraph reuses/extends this kind vocabulary; the `EnemyTurnResolver` simulate-then-apply discipline (copy the context, resolve on the copy, `PendingTelegraphState.validate_events` the events, `board.apply_events`, then `PendingTelegraphState.apply_events`) ŌåÆ the boss turn resolves the same way; 9.2's `BossPhaseResolver` PURE-READ + forward-only + stable-order + idempotent resolve (the live loop CONSUMES it after each HP change); the `boss_phase_changed` append-only SYSTEM event (already fully wired in 9.2 ŌĆö 9.3 EMITS it live, adding NO new enum member unless a genuinely new runtime fact needs one ŌĆö see the event `[Decision]`); the named-RNG rule (the boss AI draws ZERO RNG; any unavoidable combat roll routes through the `combat` stream, NEVER a bare RNG); the difficulty-non-goal guard (the boss's telegraphs/damage/AI are AUTHORED phase content from 9.2, NOT a difficulty tier ŌĆö nothing scales stats/HP/damage/rewards/RNG/run length by a selectable knob).
