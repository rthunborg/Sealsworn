## 1. Epic Summary

Epic 14 is the eleven-story epic that made the MVP loop **honestly finishable, readable, and presentable** — it closed the gap between "a human can SEE the board and CLICK it (Epic 13)" and "a human can COMPLETE a run without soft-locking, understand what is happening, and see a game that looks intentional." The 2026-07-16 agent-driven desktop playtest (`main` @ `259c32b`, post-Epic-13) proved the loop was still not shippable: neither of two hands-on runs could be completed — one died unexplained, one hit a **permanent mid-fight soft-lock** (dead enemies block movement + no wait affordance + enemy turns only advance on a successful player action). The domain underneath already worked (turn engine, enemy AI, LoS/darkness, HP model, fail-closed validation) — the gap was almost entirely **presentation, feedback, and flow-navigation over a working domain**, with exactly **two** small, contract-bounded domain touches required. Epic 14 delivered that in two deliberately-ordered bands (finishable before pretty):

- **Band 1 — Finishable & Readable Loop (14.1–14.7).** The F1 soft-lock fix first: corpse-clearing (dead units leave board occupancy — the cell goes walkable + non-targetable) plus a `WaitCommand` turn-advance backstop (14.1). Then the readability layer: a visible armed-attack preview + a non-color cue for every rejected command (14.2); an in-combat event log + damage numbers + move/hit/death animation (14.3); entropy-derived per-run seed variety with the manual-seed path preserved byte-deterministic (14.4); a death/victory beat + run-summary screen + Descend-Again-through-hero-select routing (14.5); the live route map + node-choice (14.6); and the full-backpack reward escape hatch (14.7 — a `DeclineRewardCommand` that clears a stuck offer without weakening the fail-closed `inventory_full` guard).
- **Band 2 — Looks Intentional (14.8–14.11).** A rebuilt hero-select with the approved portraits, kit summaries, and visible selection state (14.8); an outpost cleanup that drops the raw `[#]`/`[!]` markers and renders real tallies including the honest oath-shards-earned count (14.9); a styled player HUD with move/attack-range highlights and a turn indicator replacing the debug readout (14.10); and a shared UI `Theme` + semantic layout applied across all six screens, folding the two 13.2 overlay defers and adding the mid-fight resize handler (14.11).

The epic held the same structural fidelity that carried Epics 5–13 — every load-bearing discipline that keeps a change additive over the pinned contracts carried through:

- **Additive presentation + exactly TWO contract-bounded domain commands; scenes own no truth.** Nine of eleven stories are pure presentation/flow over `scripts/ui/`; only 14.1 (`WaitCommand`/`hero_waited`) and 14.7 (`DeclineRewardCommand`/`reward_declined`) touch `scripts` outside `scripts/ui/`, each the ratified 4.3 command idiom (validate-before-mutate, `ActionResult`, append-only tail event, zero RNG). No SceneTree presenter test was claimed; no new autoload was added.
- **Not one durable save / RNG / VM invariant moved — the ONE justified exception is 14.1's combat-replay re-pin.** The 23-key `RunSnapshot` gate held at 23; the 16-key `TacticalBoardViewModel` gate held at 16; the 11-key `RunHudViewModel` and 12-key `TacticalHudView` gates held; `SCHEMA_VERSION == 1` (no migration); the 7 named RNG streams held (`required_streams() == 7` — Epic 14 adds NO stream and draws NO new gameplay RNG: `WaitCommand`/`DeclineRewardCommand` are zero-RNG and the per-run seed is an injected OS-entropy read, never a named stream); `scripts/rules/conditions/` stayed empty and `operations/` stayed one file; no new content family/repository. ## 4. What Was Challenging / Growth Areas

## 5. Key Insights

1. **When a domain change is combat-time movement-legality, it moves combat-replay fingerprints — not layout fingerprints — and "more mobility can only help" is FALSE.** Corpse-clear gives enemy pursuers mobility too, so a driven Medium seed can go from a win to a MAX_ROUNDS stalemate. The right move (14.1) was to identify the moved pin by re-derivation, re-verify winnability for every approved seed, and re-pin in the same change with justification — never a silent edit, and never assume the direction of the effect.
2. **Separate domain occupancy from presentation decoration on death.** Releasing the cell's `occupant_id` + `blocks_movement` while KEEPING the corpse in `_entities` is what lets the cell go walkable/non-targetable for free while the `defeated_enemy_ids` payload and the corpse decal still see the body — and it keeps the `_cells`/`_entities` invariant consistent across the snapshot round-trip and the enemy-phase simulation copy.
3. **A "render the existing thing" story is often a "wire the source" story — read the live flow before scoping it render-only.** 14.2's armed preview was empty until its VM slot was sourced from the LIVE bound session rather than the presenter's empty local state; the same defect class recurred in 14.1/14.3. The live VM slot must be sourced from the session, not presenter-owned state.
4. **The SM's create-story precision about the EXISTING codebase state drifted in a recurring, detail-level way** — 14.1 named the wrong presenter files (`gameplay_shell_presenter`/`tactical_board_grid` vs the live `tactical_board_presenter`); 14.4's AC said "the new-run flow caller" (singular) when there were TWO hardcoded-seed start sites; 14.6's F12 framing said the route map "never surfaces" when it shipped since 11.3 (verify/harden, not build); "D4 routes to 14-8" was a misread. Each was caught pre-implementation and cost nothing structural, but the class is worth a create-story habit: GREP the live surface before writing "Files to touch" or a finding's framing.
5. **The per-run seed is a seed SOURCE, not a gameplay draw — so variety and determinism coexist.** 14.4's `RunSeedSource` is a pure injected-entropy seam; the ONE impure line (the OS-entropy read) stays in the presenter, and every regression/winnability test passes an EXPLICIT seed, so only the live run varies. No named stream, no new draw site, no save key — `root_seed` was already field #5 of the 23-key snapshot.
6. **Converge on ONE live seed source deliberately, and expect transient dead code when the enabling story lands second.** 14.4 wired the outpost "Descend Again" seed; 14.5's D3 hero-select reroute then made that outpost wiring dead code and removed it. This was foreseen (D3 + D4 both in the plan) but the ordering created a story-boundary window of dead code — worth sequencing the routing story before or with the seed story next time.
7. **The first story to cross a tooling boundary inherits a project-wide convention** — 14.11 authored the repo's first Godot `Theme` `.tres`, and the safe pattern was to GENERATE it from a committed one-shot tool (`tools/build_sealsworn_theme.gd`) rather than hand-author (hand-authoring risks the compile guardrail); and every `.gd`-adding presentation-seam story must run `--headless --import` SEPARATELY to emit new `*.gd.uid`/`*.svg.import` sidecars (the `--scene` run does not).
8. **A convention that emerges mid-epic ("seams expose only what the presenter consumes") must be reconciled with the pre-existing exact-key-set spec, or it re-litigates itself every story.** The right resolution is often that a LATER story consumes the spares (14.11 consumed the 14.10 spares) — but the ambiguity should be written down, not re-decided per review.
