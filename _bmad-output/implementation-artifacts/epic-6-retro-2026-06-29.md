## 4. What Was Challenging / Growth Areas

**Amelia (Developer):** The 6.4 tightened-`validate()` regression is the recurring teeth of content work: tightening `PassiveDefinition.validate()` to require the new FR47 modal fields would have retroactively invalidated the six Story-5.4 baseline passives (referenced by 6.3's `passive_reward_choice` table) and silently null'd `create_baseline_repository()`. It was caught at create-time and the baselines were extended in the same change — and the *identical* trap recurred in 6.7 (the `ConsumableDefinition` `outcome_effect`/`explanation` tightening broke `test_consumable_repository.gd`'s 4-arg constructions). Lesson firmly captured: any additive required field on a definition must extend every baseline AND every test construction in the same change.

**Charlie (Senior Dev / Reviewer):** The forward residuals are now numerous and they cluster on **two not-yet-existent systems**. Almost every story added a "RECORDS but does not yet apply" note pointing at Epic 7's risk-economy state (live HP/wallet/curse) and a "stays explanation-only" note pointing at the later Epic-6 per-effect operation engine. None is a defect — but the volume means the value of the loot/passive layer is currently *latent*: a player sees offers, picks, consumes, destroys, and uses consumables, but no number on the board changes yet. That is correct for v0, but it is a large standing "make it felt" debt that two future work-streams own.

**Dana (QA Engineer):** The three Low `[Review][Defer]` from 6-7 are all test/content-completeness nits, but one is worth not losing: the `expected_ids` event-id pin iterates the *map's* keys, with no assertion tying the map size to the enum member count — so a future appended event the author forgets to add to the map would be *silently un-pinned* (the test still passes). It is moot today (`item_consumed` is in the map), but the no-renumber/stable-id guard that has protected five event additions this epic would be materially stronger with an enum-exhaustiveness assertion.

**Alice (Product Owner):** The `warding_salve` content gap is the one player-facing one. It ships as the uncommon baseline consumable but is weighted into **no** reward table (`standard_combat_reward`/`elite_combat_reward` reference only `minor_healing_draught` and `ember_flask`), so in v0 it is obtainable **only** via a direct `PickupItemCommand` — never rollable from a reward offer. Its "semi-rare frequency band" is therefore effectively zero-from-rewards, not merely rare. Not a code defect (explicitly inside the deferred consumable-frequency tuning), but the frequency pass must either weight it into a table or record the deliberate omission.

**Rasmus (Project Lead):** Same status-hygiene slip as last epic — 6-7 sits at `review` despite a clean APPROVE and all ACs met. Small, but it is the exact drift Epic 5 called out for 5-5. Worth a standing "flip on review-APPROVE" habit so sprint-status stays trustworthy.

### Challenge themes
1. **Tightened-`validate()`-breaks-baselines/referencing-suites recurred (6.4 then 6.7).** Anticipated and handled both times; now a firm house rule.
2. **Forward-residual volume concentrated on two not-yet-built systems.** Epic-7 economy (apply recorded outcomes) + the later Epic-6 operation engine (make passives felt) own a large latent "make it felt" debt.
3. **Event-id-map exhaustiveness gap.** The `expected_ids` pin would silently miss a future un-mapped event; hardening recommended.
4. **`warding_salve` un-rollable from any reward table** (zero-from-rewards in v0) — consumable-frequency tuning owes a decision.
5. **Status-field lag on the capstone (6-7 at `review`)** — same as Epic-5's 5-5.

## 5. Key Insights

1. **The seam strategy now demonstrably scales to content volume + a command family.** "Accept an injected dependency, add the new param last, copy the command idiom verbatim, validate-then-mutate" took the epic from zero loot content to a six-disposition capstone with the capstone touching almost no existing seam. This is the proven house pattern for adding a run-domain command + its content.
2. **Outcome-record-only is the right v0 shape for effects whose target system does not exist yet.** Gold, Destroy outcomes, and consumable-use all *record* a keyed effect (`reward_resolved.category`, `passive_destroyed.outcome_category/outcome_id`, `item_consumed.item_id/outcome_effect`) without inventing a domain field to mutate. Epic 7 wires the actual mutation off those single hooks. The same discipline that kept passives explanation-only in Epic 5 kept the whole loot layer honest here.
3. **The inert-stream trap was real, and closing it required drawing through the run, not just wiring it.** The Epic-5 retro's T2 warning was precise: until something actually ROLLS, the run-level stream is inert and the save persists a never-advanced stream with false-confidence test coverage. 6.3 closed it by *drawing a reward through the orchestrator's streams before saving* and replacing the false-confidence assertion. The level-gen-injection half remains correctly parked (no Epic-6 AC needed it).
4. **Append-only + exhaustive id-pin is a durable event-evolution pattern — with one hardening owed.** Five new events, zero renumbers, every addition pinned. The one gap (map-size vs enum-count) is the natural next hardening.
5. **Verification guards stayed load-bearing.** The false-PASS grep caught the recurring tightened-`validate()` compile breaks before they shipped; the 23-key gate stayed green through six stories that all touched run/save-adjacent code.

## 7. Next-Epic Preview — Epic 7: Risk Economy and Affinities

**Objective (epics.md line 1863):** Players encounter readable temptation through gold, scarce healing, curses/corruption, risky rewards, and tactical affinity pressure from Scorched, Flooded/Conductive, Cursed, and Darkness. **FRs covered:** FR54-FR58. **Implementation note (epics.md line 459):** *"Risk and affinity effects must be explicit enough for player understanding and deterministic enough for replay and save/resume."*

**Stories (from `sprint-status.yaml`):** 7-1 risk-economy state, 7-2 curse/corruption rules, 7-3 risk/reward event choices, 7-4 affinity definitions + assignment, 7-5 tactical affinity effects, 7-6 darkness fairness + memory pressure.

### Dependencies on Epic 6 (all satisfied and stable)
- **The recorded-outcome hooks are exactly Epic 7's wire-off points.** Epic 6 deliberately left `reward_resolved.category=gold`, `passive_destroyed.outcome_category/outcome_id`, and `item_consumed.item_id/outcome_effect` as the single keyed hooks. Epic 7's risk-economy state (the wallet, scarce healing, curse/corruption) is the system that *applies* the mutation those events record. This is the most important handoff: 7-1's economy state is the missing domain field that makes the whole 6.x loot layer *felt*.
- **The run-command idiom + the additive-lenient-`RunState`-field pattern** — 7-1's risk-economy state (gold, healing, curse/corruption, Oath-Shard eligibility, risk flags) should be a new additive `RunState` field the same way `inventory` was (deep-copied, rides the full dict; and per its AC it must also be in the save snapshots — note that lands it in the 23-key gate territory, so compose carefully).
- **The run-level `rewards`/`events` streams + the proven T2 threading** — 7-3's risk/reward events draw through the named `events` stream; the orchestrator's run-level streams are now live and proven (6.3), so event rolls route through them the same way.
- **The rules kernel + trigger windows** — 7-2's curse/corruption triggers and 7-5's tactical affinity effects resolve through `RuleTrigger`/`RulesResolver`, the same kernel the passives use.

### Preparation / risks Epic 7 must account for
1. **Epic 7 owns the "make it felt" mutation that all of Epic 6 deferred — this is the headline.** The wallet (gold), scarce healing, and curse/corruption are the live domain fields that `item_consumed`/`passive_destroyed`/`reward_resolved` already *record* against. 7-1 should introduce that state and wire the gold-reward credit, the heal/ward/burst application, and the cleanse/curse application **off the existing recorded events** (the single hooks are already in place). Budget this as the central Epic-7 thread, not an afterthought.
2. **Risk-economy state enters the SAVE — unlike Epic-6's inventory.** 7-1's AC explicitly requires gold/healing/curse/Oath-Shard eligibility/risk flags in domain state **and save snapshots**. Epic 6 kept inventory/offer/outcome off the 23-key route-position gate (full-run-dict only). Epic 7 must persist economy state — so it will touch the snapshot gate. Plan the gate change deliberately (compose into existing nested `route_state` mechanism where possible; do not fork a parallel save format), and add migration coverage for the new fields.
3. **Affinity tactical effects are the first content that genuinely alters the tactical board (FR57) — a new correctness surface.** Epic 6 never mutated a board number; 7-5 ("affinities alter tactical choices rather than only visuals") + 7-6 (Darkness visibility/memory pressure with the no-unavoidable-damage fairness guardrail, FR58) are the first systems to change tactical truth. This likely needs the per-effect operation surface the rules kernel has been deferring since Epic 5. Decide early whether 7-5 builds the minimal `RuleOperation`/`RuleCondition`/`RuleTarget` evaluation model (currently empty `scripts/rules/{conditions,operations}` scaffolding) or whether affinity effects are special-cased — and budget trigger/order/stacking/conflict/duration test design up front per the project testing rules.
4. **The level-gen-injection half of T2 may finally come due.** If any Epic-7 affinity assignment or event draws RNG through level generation, thread `orchestrator.streams` into `LevelGenerator.generate(...)` then (the deferred T2 half), so generation advances the same set the route-position save persists. Flag at create-story for 7-4/7-5.

No Epic-7 dependency from Epic 6 is incomplete or unstable.
