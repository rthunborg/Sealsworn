# Story 3.4: Tactical Wrinkles, Blockers, and Hazards

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want generated combat levels to include at least one readable tactical wrinkle,
so that generated spaces create decisions rather than empty rooms.

## Acceptance Criteria

1. Given a combat level recipe requires a tactical wrinkle, when generation runs, then it places at least one wrinkle such as hazard, door, choke point, flank route, blocker cluster, affinity placeholder, enemy formation, reward behind danger, or risky side branch, and the wrinkle type is recorded in generation diagnostics.
2. Given blockers or hazards are placed, when pathing validation runs, then mandatory entrance-to-exit progress remains possible, and no required progress depends on a specific class, weapon, or item.
3. Given a hazard is included in a generated snapshot, when the tactical board loads the snapshot, then the hazard is represented as domain data, and future presentation can mirror it without owning hazard truth.

### Wrinkle Determinism Contract (AC1 — same seed reproduces wrinkles; the `level` stream is the only randomness source)

The single most important constraint, identical in spirit to 3.2/3.3: **wrinkle placement is a pure deterministic function of (root seed, recipe, starting `level`-stream state). Every wrinkle-affecting choice flows through `GenerationRequest.draw_layout_int`/`draw_layout_float` (→ `RngStreamSet.STREAM_LEVEL`), the draw ORDER is fixed (wrinkle draws APPENDED after the existing blocker draws), and no other randomness source touches the layout.**

- Build the streams as `RngStreamSet.new(request.level_seed())` (as the phase methods already do). Draw blocker count, blocker positions, THEN the wrinkle selection + wrinkle placement, in a FIXED, documented order. Appending wrinkle draws after blocker draws preserves the existing blocker behaviour and keeps the new fingerprints reproducible. [Source: Story 3.2/3.3 fixed-draw-order; project-context.md Determinism rules]
- Different seeds must still produce meaningfully different layouts (now varying in wrinkle kinds/positions too) — keep the `>= 2 distinct` divergence assertion green.
- Stream isolation (carried from Story 1.4 / 3.1 / 3.2 / 3.3): making `cosmetic`/`combat` draws before generation must NOT change the layout incl. wrinkles. Keep the `_cosmetic_and_combat_noise_do_not_perturb_layout` and `_layout_draws_only_from_level_stream` assertions green (extend them to cover the wrinkle draws — all wrinkle draws must advance ONLY `STREAM_LEVEL`).
- No global fallback: an unknown stream or invalid range surfaces a structured `ActionResult` error from `RngStreamSet`; the generator must never silently fall back to `randi()`/`randf()`. The `GenerationRequest` helpers enforce this — just don't bypass them. [Source: project-context.md "Do not use one global RNG for gameplay"]

### AC2 Reachability-With-Wrinkles Contract (mandatory progress remains possible; no required class/item gate)
