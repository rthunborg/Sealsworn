# Story 6.4: Passive Reward Modal Data Contract

Status: review

## Story

As a player,
I want passive choices to show exact mechanics and fiction flavor,
so that Consume and Destroy decisions are deliberate.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 6 -> Story 6.4 (verbatim AC, with the implementation-grounded detail this story adds). FRs covered: **FR47** (passive rewards present awakened-memory choices with icon, evocative name, short flavor line, exact mechanical effects, Consume choice, and Destroy choice — THIS is the story that surfaces those FIELDS, after 6.3 delivered the OFFER surface) and **GDD FR77** (passive pillar validation — every passive serves at least one pillar from tactical clarity / build synergy / risk / mystery, GDD line 329; epics.md line 396 "GDD FR77: passive pillar validation -> Story 6.4"). Builds DIRECTLY on Story 5.4's `PassiveDefinition`/`PassiveRepository` (the explanation-only starting-passive schema this story EXTENDS with reward-modal fields), Story 6.3's `RewardOffer` value object + `passive_reward_choice` 3-choice table (the offered passive entries this modal projects) and the scene-free view-model + two-step-commit-flow precedents (`ClassStartSummaryViewModel`/`HeroSelectViewModel`/`TacticalAttackCommitFlow`). This story is **DATA-CONTRACT + CONTENT-VALIDATION ONLY** — it builds NO Consume/Destroy COMMAND (those are 6.5/6.6) and NO `Control`/`.tscn` modal SCENE (UI-scene-last; a later HUD story owns the real modal).

1. **The passive reward modal view model exposes icon, evocative name, one flavor line, exact mechanical effects, Consume text, Destroy text — and honestly labels unknown hidden consequences.**
   **Given** a passive reward is offered
   **When** the passive modal view model is built (a scene-free `RefCounted` projection that takes an offered passive's `content_id` — the passive id from a `RewardOffer.offered_entries` `{category: "passive", content_id}` entry — resolves it through `PassiveRepository.get_passive(id)`, and projects an EXACT-pinned-key modal dict)
   **Then** it exposes: an **icon id or placeholder** (a stable lower_snake icon-asset id, or an explicit placeholder sentinel when no icon art exists yet — placeholder, never a crash/empty surprise), an **evocative name** (the passive's `display_name`), **one short flavor line** (a new single-line fiction string on the passive definition), the **exact mechanical effects** (a player/debug-readable explicit-mechanics string — mechanics MUST be explicit even when flavor is mysterious; GDD line 340), **Consume text** (what Consume gives — power / build identity), and **Destroy text** (what Destroy gives — safety / purification / resources / refusal)
   **And** hidden consequences are labeled honestly when unknown (a structured "honest unknown" surface: when a downside/consequence is unknown the modal SAYS SO — a `has_unknown_consequences` flag + an honest unknown line — rather than presenting false certainty or hiding it; GDD line 340 "If Destroy has unknown consequences, label it honestly as unknown").

2. **On mobile, Consume/Destroy require clear confirmation or a two-step commit, and canceling confirmation leaves reward state unchanged.**
   **Given** the passive choice is presented on mobile
   **When** the player taps Consume or Destroy
   **Then** destructive or irreversible choices require clear confirmation or a two-step commit (a scene-free CONFIRM/TWO-STEP-COMMIT data contract — mirroring the EXISTING `TacticalAttackCommitFlow` pattern: a first tap arms a pending Consume/Destroy confirmation that surfaces a `confirm_available` / `cancel_available` state; a second confirming tap (or an explicit confirm) is required before the choice is COMMITTED — this story produces the COMMIT-INTENT data contract / pending-confirmation state, NOT the executed Consume/Destroy command, which is 6.5/6.6)
   **And** canceling confirmation leaves reward state unchanged (a `cancel` clears the pending confirmation and emits NO command intent; `RunState.pending_reward_offer` and all run/inventory/passive/RNG state are byte-identical — the AC2 "no-mutation on cancel" posture, exactly as the attack commit-flow `cancel()` mutates nothing).

3. **A passive definition with missing mechanics, missing choice text, or unclear downside fields FAILS validation and cannot be in approved MVP reward tables.**
   **Given** a passive definition has missing mechanics, missing choice text, or unclear downside fields
   **When** content validation runs (`PassiveDefinition.validate()`, extended)
   **Then** the passive FAILS validation (a structured `invalid_passive_definition` + `{reason: "invalid_field", field: <field>}` — REJECT, never coerce — for an empty/blank exact-mechanical-effects string, an empty Consume text, an empty Destroy text, OR a downside-disclosure field that is neither a known-downside line NOR an explicit honest-unknown marker)
   **And** it cannot be included in approved MVP reward tables (the existing fail-closed table-load path enforces this: `RewardTableRepository`/`PassiveRepository` only register VALIDATED definitions — `create_repository_from_definitions` returns null on a validation error — so a passive that fails the extended validation can never be a registered passive a `passive` reward-table entry resolves to; a regression test PROVES an invalid passive is rejected at load and that a passive reward table referencing it cannot resolve it).

4. **Every passive declares at least one served design pillar; a passive serving no pillar FAILS validation even if its mechanics are technically valid.**
   **Given** passive definitions are validated for design purpose
   **When** content validation runs
   **Then** each passive declares at least one served pillar from **tactical clarity, build synergy, risk, or mystery** (a new `served_pillars` field on `PassiveDefinition` — at least one, each from the fixed four-pillar allowlist; GDD line 329 "Every passive should serve at least one pillar ... tactical clarity, build synergy, risk, or mystery")
   **And** passives with no served pillar FAIL validation even if their mechanics are technically valid (an empty `served_pillars`, or one carrying an out-of-allowlist pillar, is `invalid_passive_definition` field `served_pillars` — a mechanically-complete-but-pillarless passive is INVALID; a dedicated negative test pins this).

5. **A passive modal dismissed without choosing executes no command and the offer can be reopened from domain state.**
   **Given** a passive modal is dismissed without choosing
   **When** the reward remains unresolved
   **Then** no Consume or Destroy command is executed (dismissing the modal — distinct from canceling a pending confirmation in AC2 — produces NO command intent and NO mutation; this is the no-op dismiss path)
   **And** the offer can be reopened from domain state (the modal is a PURE projection of `RunState.pending_reward_offer` — dismissing it discards only the transient view/confirm state, never the offer; the SAME modal view model rebuilds byte-identically from the still-`pending` offer on `RunState`, so "reopen" is just re-projecting the unchanged domain offer — there is no separate "modal open" flag on the domain that dismissal could corrupt).
