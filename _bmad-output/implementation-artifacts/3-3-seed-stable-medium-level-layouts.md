# Story 3.3: Seed-Stable Medium Level Layouts

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want Medium levels to generate with more space while remaining readable,
so that mid-run tactical problems can grow without becoming unfair.

## Acceptance Criteria

1. Given a Medium level recipe around 14x12 tiles, when generation runs for a fixed seed, then it produces a deterministic layout with entrance, exit, blockers, and enough open space for movement, and the layout supports line-of-sight play.
2. Given Medium levels are larger than Small levels, when validation checks readability constraints, then it rejects layouts with excessive blockage, unreachable exits, or unreadable first reveal, and rejected results include compact diagnostics.
3. Given Medium level seed regression tests are run, when approved fixture seeds generate levels, then output remains stable or changes only with explicit fixture updates, and the tests do not depend on rendering.

### Layout Determinism Contract (AC1 — same seed reproduces the layout; the `level` stream is the only randomness source)

The single most important constraint, identical to 3.2: **a Medium layout is a pure deterministic function of (root seed, recipe, starting `level`-stream state). Every random choice flows through `GenerationRequest.draw_layout_int`/`draw_layout_float` (which use `RngStreamSet.STREAM_LEVEL`), the draw ORDER is fixed, and no other randomness source touches the layout.**

- Build the streams as `RngStreamSet.new(request.level_seed())` (the same way `_run_small_layout_phase` does). Draw blocker count, then blocker positions, then any retry re-draws, in a FIXED, documented order. Reordering draws or adding an unrelated draw between two layout draws silently changes every approved fixture — pin the order; the seed-regression test is the tripwire. [Source: epics.md Story 3.3 AC1; project-context.md Determinism rules; Story 3.2 fixed-draw-order]
- Different seeds must be able to produce meaningfully different layouts (AC1 second half) — assert this with `>= 2` seeds whose fingerprints differ. Do NOT collapse to a single fixed layout regardless of seed.
- Stream isolation (carried from Story 1.4 / 3.1 / 3.2): making `cosmetic`/`combat` draws before generation must NOT change the `level`-affected layout. Mirror the 3.2 `_cosmetic_and_combat_noise_do_not_perturb_layout` and `_layout_draws_only_from_level_stream` assertions.
- No global fallback: an unknown stream or invalid range surfaces a structured `ActionResult` error from `RngStreamSet`; the generator must never silently fall back to `randi()`/`randf()`. (The `GenerationRequest` helpers already enforce this — just don't bypass them.) [Source: project-context.md "Do not use one global RNG for gameplay"; Story 3.1/3.2]
- If you choose bounded re-draw (3.3.3 option (b)): EVERY re-draw advances ONLY the `level` stream in a FIXED, documented way, so a re-drawn layout is STILL a pure deterministic function of the seed. The seed-regression fingerprints must be pinned to the FINAL accepted layout (post-retry), and a re-draw must not consume any non-`level` randomness.

### AC2 Readability Validation Contract (the NEW requirement — reject-with-diagnostics; scoped to three checks; NOT the full 3.6 validator)

This is the single biggest delta from 3.2 and the biggest review risk. Get the SCOPE right:

- AC2 verbatim: "when validation checks readability constraints, then it rejects layouts with **excessive blockage**, **unreachable exits**, or **unreadable first reveal**, and rejected results include **compact diagnostics**." Implement EXACTLY these three checks for Medium. Each failing candidate returns `GenerationResult.error(GenerationResult.PHASE_VALIDATION, <lower-snake code>, <reason>, <seed string>, <compact diagnostics>)`. [Source: epics.md Story 3.3 AC2]
- **Excessive blockage:** define a readable bound (e.g. interior-blocker count / interior-cell ratio, or minimum open-floor count) and reject candidates that exceed it. The bound is a generator constant derived from readability intent — NOT from any difficulty tier. Keep the recipe `blocker_budget_max` and this bound consistent (the budget band should normally keep blockage readable; the AC2 check is the backstop). Diagnostics: the offending count/ratio + the bound, NOT a full grid.
- **Unreachable exit:** deterministic flood fill / BFS from entrance over non-WALL cells; reject if the exit cell is not reached. Use `BoardCell.Terrain.WALL` (= `terrain_blocks_occupancy()`) as the only "blocked" predicate so "walkable" matches the board model. 4-neighbour connectivity is sufficient and matches the tactical movement model (document the neighbourhood). Diagnostics: entrance + exit coordinates + reachable-cell count. This is a PURE query (no RNG, no mutation).
- **Unreadable first reveal:** guard against "you spawn boxed in / cannot orient." Reuse the line-of-sight query service on a built `BoardState` if it is cleanly callable, OR approximate with a bounded flood / Chebyshev-radius count from the entrance over non-WALL cells using the baseline LoS radius (4 — see FR5), rejecting if the visible/reachable-within-radius cell count is below a readable minimum. Pick one, keep it deterministic + scene-free, document it. Diagnostics: the visible-cell count + the minimum + the radius used.
- **SCOPE GUARD:** AC2 covers ONLY these three. Do NOT add legal-enemy-placement or reachable-reward checks (no entities/rewards exist until 3.5), and do NOT build a general no-soft-lock/no-required-gate validator or a general bounded-retry engine — those are Story 3.6, which will likely REPLACE this focused validator with the comprehensive one. Keep this validator small, named, and easy for 3.6 to subsume; document in Completion Notes what 3.6 should replace. [Source: epics.md Story 3.6; game-architecture.md "Entrance-to-exit reachability / No soft-locks / Safe first reveal / Legal enemy placement / Reachable required rewards / Fog/readability constraints"]
- **Rejection MUST be reachable and TESTED.** If your Medium shape is readable-by-construction (3.3.3 option (a)) so approved seeds never fail, you STILL must exercise each rejection path by feeding a deliberately-malformed candidate straight into the validator (e.g. a hand-built terrain grid that walls off the exit, or one that is almost entirely WALL, or one that boxes in the entrance) and asserting the correct structured `PHASE_VALIDATION` error + diagnostics. AC2 is not satisfied by an unreachable rejection branch.

- Read, write, or mutate `user://run_autosave.json` or `user://settings.json` (generation has no save WRITE work this story). If a test builds a `TacticalSnapshot`, it does so in-memory only.

### Latest Technical Information

Official Godot 4.6 sources relevant to deterministic generation, the board model, reachability, and JSON transport (these inform correctness, not gameplay):

- `RandomNumberGenerator` (seed/state) backs `RngStreamSet`; `randi_range`/`randf` advance per-stream state deterministically from a seed, so a fixed seed + fixed draw order reproduces a layout exactly. - `Resource` + `@export` typed properties back `LevelRecipeDefinition` (read its fields directly). Source: https://docs.godotengine.org/en/4.6/classes/class_resource.html
- `Vector2i` is the board's cell-coordinate type (`BoardCell.position`); board snapshot cell dicts use `{"x": int, "y": int}` (NOT a packed Vector2i) per `BoardCell.to_dictionary()`. A 4-neighbour flood fill over `Vector2i` cells is the standard deterministic reachability approach (iterate neighbours in a fixed order; use a `Dictionary`/visited set). ### Completion Notes List

- **Medium dimensions / jitter (3.3.2):** Fixed `14x12` (`MEDIUM_WIDTH=14`, `MEDIUM_HEIGHT=12`), NOT jittered — consistent with the 3.2 fixed-footprint choice. Height 12 is even, so `corridor_row = 6` is strictly interior.
- **Entrance/exit placement:** NOT seed-randomized — fixed entrance `(1, 6)` / exit `(12, 6)` on the central row, mirroring 3.2. All seed-to-seed divergence comes from interior blocker draws (AC1 "meaningfully different layouts" satisfied via blocker variety; 5 approved seeds yield 5 distinct fingerprints).
- **Fixed draw order (AC1):** FIXED DRAW #1 = blocker count over `[blocker_budget_min..max]` clamped to candidate count (one draw, always taken when blockers are allowed); FIXED DRAW #2.. = shrinking-pool blocker positions (no rejection loop). Byte-identical discipline to the Small generator. Every draw routes through `request.draw_layout_int` (→ `RngStreamSet.STREAM_LEVEL`); no `randi()`/`randf()`, no other stream. Verified by the level-stream-only + stream-isolation tests.
- **`wall_density` decision (3.3.4) — OPTION (b): intentionally UNUSED in v0 for BOTH generators; budget band authoritative.** Rationale: `medium_combat_basic.wall_density = 0.28`; the 14x12 interior is `(14-2)*(12-2) = 120` cells, so a density target ≈ 34 vastly exceeds the band max 8 and would clamp to a CONSTANT 8 blockers every seed, collapsing the seed-to-seed blocker-COUNT variety AC1 relies on — adding no readability value the band does not already give. The AC2 excessive-blockage bound (NOT `wall_density`) is the readability backstop. Documented in the `WALL_DENSITY DECISION` block of `medium_level_layout_generator.gd` and a matching cross-reference comment added to `small_level_layout_generator.gd::_draw_blocker_count` (comment-only; no behaviour/fingerprint change — Small fingerprints unchanged, not re-pinned). This closes the 3.2-deferred Med "`wall_density` silently ignored / undocumented" item for both generators.
- **AC2 validation approach (3.3.3) — OPTION (a): reject-by-construction (NO bounded re-draw).** The border-ring + open-interior + reserved-corridor shape with a `3..8` blocker budget over 120 interior candidates cannot wall off the exit or exceed the openness bound, so approved seeds pass all three AC2 checks by construction (verified by `_approved_layout_passes_readability_validation` across 6 seeds incl. the 5 pinned). The three rejection paths are therefore exercised by deliberately-malformed hand-built candidates fed straight into `validate_readability`. **What Story 3.6 should replace:** the focused `validate_readability` method (three AC2 checks) — 3.6 owns the comprehensive validator (reachability across all sizes, no-soft-lock, no-required-gate, legal enemy placement, reachable rewards, safe first reveal, fog/readability) + bounded deterministic retry + per-phase reporting, and will subsume this method.
- **AC2 checks + bounds:** (a) **excessive_blockage** — reject if `interior_wall_count / interior_cell_count > MAX_INTERIOR_WALL_RATIO (0.35)`, where "interior" excludes the always-WALL border ring (border walls are not readability blockers); baseline band ≈ 0.067, far below the bound. (b) **unreachable_exit** — deterministic **4-neighbour BFS flood fill** over non-WALL cells from the entrance; reject if the exit is not in the visited set. `BoardCell.Terrain.WALL` (= `terrain_blocks_occupancy()`) is the sole "not walkable" predicate, matching the board model. (c) **unreadable_first_reveal** — approximated as a **bounded 4-neighbour flood capped at Chebyshev distance `FIRST_REVEAL_RADIUS (4)`** (the baseline LoS radius, FR5) over non-WALL cells from the entrance; reject if the reachable-within-radius count `< MIN_FIRST_REVEAL_CELLS (8)`. No scene/LoS service is used — pure, deterministic, scene-free. Each rejection returns a structured `ActionResult.error(<code>, <compact diagnostics>)`; `_run_medium_layout_phase` maps it onto `GenerationResult.error(PHASE_VALIDATION, <code>, &"readability_validation_failed", <seed string>, <diagnostics>)`. Diagnostics carry counts/ratios/coordinates only (asserted to NOT contain a `terrain` grid dump).
- **Validation check order is fixed:** excessive_blockage → unreachable_exit → unreadable_first_reveal; the first failing check returns. (This ordering is why the unreadable-first-reveal test candidate keeps the exit reachable and the interior sparse, so it reaches check (c) rather than tripping (a)/(b).)
- **Board conversion (3.3.5):** `build_board_snapshot` reuses the 3.2 approach verbatim (exact `BoardState.to_snapshot()` shape, `entities: []`, routed through the STRICT `BoardState.try_from_snapshot`) AND adds the 3.2-deferred-Low `invalid_layout_shape` leading shape guard (width/height vs terrain-grid dimensions) to the **Medium** generator only. The Small generator's shape guard remains a deferred Low for its owning task. Successful Medium `GenerationResult.ok` diagnostics carry `phase: "validation"` (a successful Medium result cleared both the layout phase and the AC2 readability pass); a layout-construction or board-conversion failure reports `PHASE_LAYOUT`.
