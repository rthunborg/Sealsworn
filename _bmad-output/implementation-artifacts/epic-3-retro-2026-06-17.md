## 3. What Went Well

Amelia (Developer): "Let's open with the wins, because this epic had a lot of them — and they were structural, not lucky."

Cloud Dragonborn (Game Architect): "The headline is the invariant we landed and never broke: **level = pure function of (seed, recipe)**. Same seed and recipe produce a byte-identical layout AND an identical final `GenerationResult` — same attempt count, same payload or same error. Everything downstream — seed regression, manual-seed replay, bounded retry — rides on that one property, and it held across all seven stories. Generation draws layout randomness through exactly one path, `GenerationRequest.draw_layout_int/float` into `STREAM_LEVEL`, and a test asserts no other stream's `draw_index` ever moves. That discipline is why a generation epic produced zero determinism bugs."

Charlie (Senior Dev): "And the seed-regression fingerprints were the tripwire that made it real. Every story pinned a compact terrain fingerprint per approved seed. The rule was absolute: a fingerprint changes ONLY through an intentional generator change re-pinned in the same PR, via the dump tools — never hand-edited to make a drifting test pass. Across the whole epic that fired exactly once, on 3.4, when wrinkles and the first HAZARD cell legitimately changed the terrain grid. That was a deliberate, logged re-pin with a comment in each fixture. Every other story — placement in 3.5, the validator and retry in 3.6, the loader in 3.7 — held the fingerprints GREEN with no re-pin. When 3.5 added enemies to the board and the terrain fingerprint *didn't* move, that was itself the proof that placement didn't perturb the draw order."

Dana (QA Engineer): "Quality bar stayed high and the reviews were genuinely adversarial. The standout testing pattern: reachability was always verified with an **independent** flood written fresh in the test, never by re-asserting the generator's own BFS against itself. The validator could claim a reward is reachable and the test would prove it with its own 4-neighbour flood. Same-seed determinism, ≥2-distinct-layout divergence, and stream-isolation were asserted on every generator. And every generated board rode the *real* serialization path — strict `BoardState.try_from_snapshot`, then a real `JSON.stringify`/`parse_string` round-trip, then the `TacticalSnapshot.from_domain`/`parse` path. That's the Epic 2 int64 lesson applied from story one."

Alice (Product Owner): "From a design standpoint, the fairness model is the quiet triumph. Instead of generating-then-validating-then-rerolling for fairness, the generators are **fair by construction**: a reserved blocker-free central corridor, a fixed entrance and exit on that row, and a shrinking candidate pool that pre-excludes the corridor, entrance, exit, and walls. So a blocker can never seal the path and a hazard can never land on the safe mandatory route. The player gets seed-to-seed variety from interior blockers and wrinkles while the path is provably always there. That's the anti-frustration pillar baked into the geometry."

Amelia (Developer): "The breakthrough story was 3.6 — the comprehensive `LevelValidator` plus bounded retry. The hard problem there was beautiful."

Cloud Dragonborn (Game Architect): "Right. Bounded retry COLLIDES head-on with the determinism invariant: if you reroll the same seed you reproduce the same rejected candidate, so retry needs a per-attempt seed mix. But perturbing the seed would drift the pinned fingerprints. The resolution is the hard invariant `_attempt_seed(seed, 0) == seed` — **attempt 0 is byte-identically unperturbed**, only attempts 1..7 mix in the attempt index. That preserved every 3.2–3.5 fingerprint and layout test with zero re-pin while still giving the retry engine somewhere to go. And the validator is a *pure query* — it draws no RNG and mutates nothing, the same purity contract as a snapshot — proven by per-stream `draw_index` invariance. That's the kind of constraint that looks impossible until someone finds the one move that satisfies both sides."

Charlie (Senior Dev): "The other thing 3.6 nailed was the test seam. Generated candidates are valid *by construction*, so the retry-failure branch and most validator-rejection branches are literally unreachable end-to-end. The story added an injected-validator seam — an optional 4th param defaulting to the real validator — so tests can inject an always-fail stand-in to exercise the cap (`attempts == MAX == 8`) and a fail-then-pass stand-in (`attempts == k+1`). That's how you test an unreachable-by-construction branch honestly instead of pretending it's covered."

Dana (QA Engineer): "And model-diversity review paid off. On 3.6 and 3.7 we ran a second independent reviewer with a different model on the same diff. Round 2 of 3.7 explicitly re-probed the Round 1 patch at every int64 boundary with a throwaway harness and confirmed convergence with zero new findings. That's real verification, not a rubber stamp."

Rasmus (Project Lead): "This is exactly what I wanted the generation epic to be — deterministic, fair, and provable. Good."

## 5. Key Insights / Lessons Learned

1. **One randomness path, one draw order, one tripwire.** Routing every layout draw through `STREAM_LEVEL` via `GenerationRequest.draw_layout_int/float`, fixing the draw order, and pinning a compact terrain fingerprint per seed is what made a generation epic produce zero determinism bugs. The fingerprint that *doesn't* move when you add a feature (enemies in 3.5) is as informative as the one that does.
2. **Fair by construction beats validate-and-reroll for the common case.** The reserved corridor + shrinking candidate pool means baseline candidates pass validation on attempt 0 every time. The validator and retry exist for *robustness and future content*, not to rescue routine generation. (Codified in `project-context.md`.)
3. **When a hard invariant collides with a new feature, find the single move that satisfies both — don't relax the invariant.** Bounded retry vs. fingerprint stability was resolved by `attempt 0 unperturbed`, not by re-pinning. Negative-seed loading vs. the `root_seed >= 0` rule was resolved by normalizing at the loader boundary (`& 0x7fffffffffffffff`), not by relaxing `GenerationRequest.validate()`.
4. **A story's Dev Notes are a hypothesis; the strict validator is the truth.** The 3.5 occupant-invariant note was wrong; the dev verified against the actual board validator and corrected it. Trust executable contracts over prose, including prose in the story spec.
5. **Test unreachable-by-construction branches with an injected seam, and log the rest as deferred.** 3.6's injected-validator seam exercised the cap and fail-then-pass paths honestly. The recurring 'correct but unreachable' Lows are a coherent class — they should be tracked as such, with the tripwire being a future jittered/degenerate board.
6. **Model-diversity review converges and catches real edges.** Running a second independent reviewer with a different model on 3.6 and 3.7 surfaced genuine patch items (the int64 seed-string saturation) and then *confirmed* convergence. Worth keeping for the gnarly stories.
7. **Carry the int64/real-JSON round-trip rule into every serialization, forever.** Generated boards rode the strict snapshot + real-JSON path from story one — the Epic 2 lesson applied proactively, not re-learned.

## 9. Significant Discovery / Epic-Update Assessment (Planning Drift)

Three detail-level traceability items are worth a light re-sync note — all are deliberate, named, architecture-endorsed v0 narrowings within the FRs as written, and none blocks Epic 4.**

1. **`epics.md` FR35 (combat levels include "at least one tactical wrinkle such as hazard, door, choke point, flank route, blocker cluster, affinity placeholder, enemy formation, reward behind danger, or risky side branch") — DETAIL-LEVEL.** Epic 3 realizes **four** of the nine listed kinds as terrain (`choke_point`/`flank_route`/`blocker_cluster` → WALL, `hazard` → HAZARD) and explicitly does **not** realize `door`/`affinity_placeholder`/`enemy_formation`/`reward_behind_danger`/`risky_side_branch` in v0. Re-sync recommendation: a one-line note that the remaining five kinds are downstream (doors/affinity ride Epic 7's affinity + Epic 4's node-seal work; enemy_formation/reward_behind_danger/risky_side_branch ride later generation tuning). Not a scope change.

2. **`epics.md` FR36 / Story 3.6 ("safe first reveal") — DETAIL-LEVEL (ratified v0 semantic).** The validator's `unsafe_first_reveal` check rejects only the entrance cell being HAZARD or occupied by an entity; a threat merely *adjacent* to the entrance is **permitted** (it's seen on spawn within LoS radius 4, FR5; FR58 protects against *unseen* damage, not a visible adjacent enemy). A stricter `Chebyshev<=1` guard was deliberately rejected because the baseline generator legitimately places enemies adjacent to the entrance and enforcing it would force attempt-0 rerolls that drift the pinned fingerprints. This is a defensible interpretation of FR36's "safe first reveal," documented in-code and confirmed across both 3.6 review rounds. Re-sync recommendation: none required for the FR; flagged only so a future first-reveal/safety-tuning story treats "entrance-cell-only" as the ratified v0 contract rather than re-litigating it.

3. **`epics.md` Story 3.5 / FR52 reward modeling + `RewardTableDefinition` — DETAIL-LEVEL (expected deferral).** The architecture's runtime-definition list names a `RewardTable`, and Story 3.1's task list contemplated a reward table. Epic 3 decided **no standalone `RewardTableDefinition` in v0**: reward-placement rules live on `LevelRecipeDefinition` and rewards are abstract payload markers `{x, y, optional}`, not board entities (there is no reward entity type). Concrete loot tables + repository are explicitly Epic 6 (Loot, Passives, Consume/Destroy). FR52 routes loot "through data boundaries" and is mapped to Epic 6 in the FR coverage map, so this is the requirement resolving where the plan assigned it — not drift. Re-sync recommendation: none required; flagged so Epic 6 knows it owns the first real reward content + the reward-table decision, and that the `rewards`/`loot` RNG streams (reserved, unused in Epic 3) resurface there for runtime resolution.

**Verdict: build matched plan. Re-sync is optional and detail-level only** — three "the v0 narrowing is intentional and named" clarifications, zero structural changes. No PRD/architecture/epics rewrite is warranted before Epic 4.
