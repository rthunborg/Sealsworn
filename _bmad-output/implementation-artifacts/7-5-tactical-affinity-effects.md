## Story

## Acceptance Criteria

(Verbatim from `planning-artifacts/epics.md` lines 1959-1992, Story 7.5; FR57 — "Affinities must alter tactical choices rather than only visuals". The SECOND of the three affinity stories: 7.4 shipped the DEFINITIONS + deterministic ASSIGNMENT + the neutral contract; 7.5 = the TACTICAL EFFECTS that READ a level's assigned affinity and turn its recorded `tactical_rules` into real, deterministic, explainable tactical pressure; 7.6 = the Darkness fairness/visibility layer. **AC1 — Scorched: fire-hazard / burning-terrain rules resolve damage/hazard pressure through domain events and are explainable in previews and logs**
- **Given** Scorched affinity is active
- **When** fire hazards or burning terrain rules trigger
- **Then** damage-over-time or hazard pressure resolves through domain events
- **And** affected cells are explainable in previews and logs.

**AC2 — Flooded/Conductive: water/conductive danger zones resolve deterministically with non-color-only danger communication**
- **Given** Flooded/Conductive affinity is active

- **Then** pathing pressure or electric interaction placeholders resolve deterministically
- **And** critical danger information is not color-only.

**AC3 — Cursed: risk/reward or penalty hooks resolve through the rules kernel with pre/on-effect clarity**
- **Given** Cursed affinity is active
- **When** risk/reward or penalty hooks trigger

**AC4 — Flooded/Conductive MVP placeholders are tracked for the Epic-10 readiness gate, with distinct (non-final) cue/visual/explanation ids**
- **Given** Flooded/Conductive uses placeholder interactions during MVP implementation
- **When** Epic 10 readiness reviews affinity content

- **And** placeholder cue ids, visual ids, and explanation text remain distinct from final production identifiers.

### ⭐ PIVOTAL SCOPE DECISION — how far does the operation/effect engine open? `[Decision]` (the single highest-stakes 7.5 call; resolve it FIRST)

The architectural reality the dev MUST internalize before writing code (verified against the merged tree at the baseline commit):

1. **The tactical board FULLY EXISTS (Epic 1) and is the natural seam for effects:** `BoardState` (grid + cells + entities), `BoardCell.Terrain.HAZARD` (a board-VALID, WALKABLE, sight-TRANSPARENT terrain — only WALL blocks occupancy/LOS; HAZARD's danger is "the rules kernel's job" per the 3.4 ratified contract), `TacticalEntityState` (`current_hp`/`max_hp`/`is_alive()`), the `DAMAGE_APPLIED` / `TILE_MARKED` / `MARKED_TILE_DETONATED` domain events (the Ash-Seer mark→delayed-detonation telegraph is the EXACT precedent for hazard/DoT pressure resolving through events — see `pending_telegraph_state.gd`), `AttackPreviewQuery` (already returns `warnings`/`effects`/`explanation` arrays — the "explainable in previews" seam), and `CombatExplanationLog` (builds readable entries from domain events — the "explainable in logs" seam).
2. **BUT there is NO live tactical play loop wired into the run.** `RunOrchestrator._resolve_combat` AUTO-RESOLVES a combat node ("level generated successfully → node cleared → exit"); it never instantiates a live `BoardState`, never runs a turn loop, never fires `before_attack`/`damage_calculated`/`turn_started` windows. The `RulesResolver` is a PURE-READ explanation registry (passives + curses); `scripts/rules/{conditions,operations}` are EMPTY scaffolding (0 `.gd` each — VERIFIED). The combat hook sites that would FIRE the rules-kernel windows do NOT exist yet (they are deferred to "the later operations story" across 5.4/6.x/7.x). So there is no live, run-driven board for an affinity effect to mutate IN THE RUN FLOW.
3. **The 7.4 `tactical_rules` are DATA the 7.5 layer reads** (`AffinityRepository.tactical_rules_for(id)` returns the recorded markers; `AffinityDefinition.tactical_rules_copy()` deep-copies them). Each MVP affinity carries real markers — Scorched: `fire_hazard_cells`/`burning_terrain_spread`/`damage_over_time_pressure`; Flooded: `water_electric_interaction`/`pathing_pressure`/`danger_zone_marking`; Cursed: `risk_reward_bargains`/`curse_penalty_pressure`; Darkness: `reduced_visibility`/`hidden_threats`/`fog_memory_pressure` (Darkness is 7.6's, NOT 7.5's).

**`[Decision]` — THE RECOMMENDED DISPOSITION (the BOARD-SCOPED EFFECT LAYER, NOT a run-flow wiring + NOT a full generic operation engine):** 7.5 builds a deterministic, scene-free, PURE-DOMAIN **affinity tactical-effect layer that operates on a `BoardState` (the Epic-1 tactical model) given a level's assigned affinity** — NOT a wiring into the auto-resolving run loop, and NOT the full `RuleCondition`/`RuleTarget`/`RuleOperation` generic operation engine. Concretely, the recommended shape (each `[Decision]`-confirm in Completion Notes):

- **An affinity tactical-effect resolver/service (`scripts/rules/operations/` — THIS is the legitimate fence move):** a scene-free `RefCounted` (e.g. `AffinityEffectResolver` / `AffinityTacticalEffects`) that, GIVEN a `BoardState` + the level's assigned affinity id (resolved through `AffinityRepository`), produces the affinity's tactical effects DETERMINISTICALLY. This is the FIRST real occupant of `scripts/rules/operations/` — author it here, narrowly (an affinity-effect resolver), NOT a generic operation/condition evaluation engine. It draws NO RNG unless an effect explicitly needs a `combat`-stream roll (see the named-RNG note); it is a PURE function of `(board, affinity, [combat stream])`.
- **Scorched (AC1) — hazard placement + a DoT/hazard-damage RESOLUTION that emits `DAMAGE_APPLIED` (or `MARKED_TILE_DETONATED`):** the resolver STAMPS Scorched fire-hazard cells onto the board as `BoardCell.Terrain.HAZARD` (the 3.4 contract: HAZARD is board-valid + walkable + sight-transparent; its danger is the rules kernel's — that IS this layer) keyed off the `fire_hazard_cells` marker, and resolves DoT/hazard pressure as a domain-event-producing command/path: an entity standing in / entering a Scorched hazard cell takes hazard damage modeled through the EXISTING `DAMAGE_APPLIED` event (reuse the attack-command damage shape: `hp_before`/`hp_after`/`final_damage`/`damage_type`), or the mark→detonate `TILE_MARKED`→`MARKED_TILE_DETONATED` telegraph for a TELEGRAPHED burning-terrain pulse (the Ash-Seer precedent — PendingTelegraphState). **`[Decision]`: choose the DoT model** — (a) a direct `DAMAGE_APPLIED` on enter/occupy via a small validated effect command, or (b) a telegraphed `TILE_MARKED`→`MARKED_TILE_DETONATED` pulse.
