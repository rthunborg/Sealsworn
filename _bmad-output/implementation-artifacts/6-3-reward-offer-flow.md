## Story

As a player,
I want rewards to appear as clear offers after eligible encounters or nodes,
so that each run can begin developing a build identity.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 6 -> Story 6.3 (verbatim AC, with the implementation-grounded detail this story adds). FRs covered: FR47 (passive rewards present awakened-memory choices — the OFFER surface; the Consume/Destroy FIELDS are 6.4/6.5/6.6), FR52/FR53 (loot categories + consumable scarcity as the table draws), and GDD FR71 (3-choice passive moments — AC4). Builds DIRECTLY on Story 6.1's reward CONTENT + `RewardOfferBuilder` + `RewardTableRepository` and Story 6.2's `InventoryState` + `PickupItemCommand` + `item_gained` event — it threads them into the FIRST LIVE reward roll through the run. This is the FIRST story that ROLLS a reward through the run, so it OWNS the inert run-level `RngStreamSet` threading (the deferred-work "code review of 4-6" T2 item) AND the reward-table validate-before-draw posture (the 6.1 `[Review][Decision]`, owner Story 6.3).

1. **A completed combat/reward node deterministically generates a reward offer from approved reward tables, stored in domain state until resolved.**
   **Given** a combat or reward node completes
   **When** reward generation runs
   **Then** it creates a DETERMINISTIC reward offer from approved reward tables (drawn through the Story-6.1 `RewardOfferBuilder.build_offer(streams, stream_name, table)` against a `RewardTableRepository`-resolved `RewardTableDefinition`, using the run-level `RngStreamSet`'s `STREAM_REWARDS`/`STREAM_LOOT` — NEVER `randi()`/`randf()`, NEVER a local `RandomNumberGenerator`, NEVER the `map`/`level`/`combat`/`events`/`cosmetic` stream)
   **And** the offer is stored in DOMAIN STATE until resolved (a new additive `RunState` reward-offer field holds the pending, unresolved offer — a serializable record carrying the offered choice(s), the table id, the draw provenance, and a `pending`/`resolved` status — exactly as `starting_kit`/`rules_resolver`/`inventory` are additive run-domain fields; same seed + same pre-draw stream state reproduce a byte-identical offer).

2. **Selecting a reward applies it through domain events and discards/marks-resolved the unselected rewards.**
   **Given** the player selects a reward
   **When** the reward command executes
   **Then** the selected reward is applied through DOMAIN EVENTS (a validate-then-mutate run-domain reward-RESOLVE command: it applies the chosen reward by its category — a backpack-category reward routes through the EXISTING `PickupItemCommand`/`item_gained` consumption model, gold/passive get their documented v0 handling — and emits a deterministic reward-RESOLVED system event; on success the offer's status flips to `resolved`)
   **And** unselected rewards are discarded or marked resolved (the pending offer is consumed: after resolution the offer is `resolved` and no second reward can be applied from it — the unchosen entries are dropped/marked, not silently re-offered).

3. **A duplicate selection against an already-resolved offer fails with a stable error and mutates nothing.**
   **Given** a reward offer is already resolved
   **When** a duplicate selection command is submitted
   **Then** it FAILS with a stable error (a stable lower_snake `ActionResult.error` — e.g. `reward_offer_already_resolved` / `no_pending_reward_offer` — carrying the offer/table id in metadata; validate-then-mutate, ZERO events)
   **And** inventory, currency, passive, and RNG state remain unchanged (the load-bearing no-double-apply guarantee: a re-submitted selection leaves the backpack byte-identical, draws NO RNG / advances NO stream, applies NO second reward, and the run is byte-identical — exactly the AC3 "no-mutation on reject" posture the 6.2 pickup command already proves).

4. **A 3-choice passive table yields three distinct passive choices, unless an explicit, visible MVP test-scope exception is recorded.**
   **Given** a passive reward table is configured for a 3-choice moment
   **When** a passive offer is generated
   **Then** the offer contains THREE DISTINCT passive choices (three distinct passive `content_id`s drawn without replacement from the passive table) UNLESS the table explicitly records an MVP test-scope exception (a stable marker ON the reward-table definition — e.g. a `mvp_choice_count_exception` / reduced `choice_count` with a reason — mirroring the 6.1 `*_mvp_deferred` posture, never a silent reduction)
   **And** exceptions are visible to content validation and tuning notes rather than silently reducing choice density (a table whose declared choice count is below 3 is only VALID if it carries the explicit exception marker; the exception is surfaced — a `RewardTableDefinition.validate()` check + a Completion-Notes tuning note — not buried).
