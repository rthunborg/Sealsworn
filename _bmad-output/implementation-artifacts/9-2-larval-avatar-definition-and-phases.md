- **When** content validation runs
- **Then** it includes HP, phase thresholds or triggers, legal actions, telegraph definitions, damage rules, and explanation text
- **And** invalid boss content fails validation.

- **Given** the boss changes phase
- **When** the phase transition trigger resolves
- **Then** a deterministic phase-change event is emitted

- **And** repeated triggers do not duplicate phase changes.

### ‚≠ê The phase-transition MODEL ‚Äî deterministic, stable-ordered, idempotent (AC2/AC3)

- **PHASES ARE ORDERED + FORWARD-ONLY.** The Larval Avatar has ‚â•2 phases (a boss with one phase has no phase model ‚Äî the finale needs a readable escalation; 2-3 phases is the right MVP scope, matching FR63 "without requiring a full boss roster"). Phases transition in **declaration order** (phase 0 ‚Üí 1 ‚Üí 2 ‚Ä¶), each gated by an HP THRESHOLD (e.g. "at ‚â§ 60% HP enter phase 1") and/or a scripted trigger. **Forward-only:** once the boss is in phase N, it never returns to phase < N (healing back above a threshold does NOT revert ‚Äî mirror the `RouteState` reveal monotonicity: transitions are monotonic). This is the AC3 "stable order" + "repeated triggers do not duplicate phase changes" guarantee: crossing a threshold that is already behind the current phase is a NO-OP.
- **THE RESOLVE IS A PURE FUNCTION OF `(current_phase_index, current_hp, max_hp)` (+ any scripted trigger flags).** A small helper ‚Äî the boss analogue of `RulesResolver.resolve(window)` ‚Äî takes the boss's current phase + HP and returns the ordered list of phase transitions that should fire (typically 0 or 1; MORE than one only if a single large damage event crosses MULTIPLE thresholds at once ‚Äî decide + test the multi-threshold-in-one-hit case: either fire each crossed transition in order, or jump to the deepest crossed phase and emit ONE change to the final phase; RECOMMENDED ‚Äî advance to the deepest crossed phase and emit ONE `boss_phase_changed` per phase actually entered, in order, so the record is honest and no phase is skipped in the log; record your choice). It draws **ZERO RNG**, mutates nothing on the source (a PURE READ ‚Äî the snapshot/`RulesResolver`/`LevelValidator` purity contract), and is deterministic: same `(phase, hp, max_hp)` ‚Üí same transition list + same explanation.
- **IDEMPOTENCY (AC3) is structural, not a dedup cache.** Because the current phase index is the state and transitions are forward-only, re-evaluating at the same or a higher phase yields NO transition. A test MUST assert: reaching a threshold fires exactly one change; RE-reaching it (or staying below it) fires NONE; a threshold already behind the current phase fires NONE.
- **WHO OWNS THE PHASE STATE?** 9.2 does NOT wire a live board/turn loop (that is 9.3/9.4), so 9.2's tests drive the resolver with explicit `(phase, hp)` inputs ‚Äî it does NOT need to seat phase state on `RunState` or a live entity yet. Provide the resolver + the definition + the event; the LIVE call site (a boss entity carrying current HP + current phase, re-resolved after each damage event) is 9.3's live-loop concern. **Record the boundary you draw:** 9.2 ships the pure phase MODEL (definition + deterministic resolver + event) that 9.3's live loop consumes; it does NOT persist phase state, add a `RunSnapshot` key, or instantiate a live boss entity. (If you introduce a tiny value-object to carry `(phase_index, current_hp)` for the resolver's convenience, keep it a pure `RefCounted` DTO, not seated on `RunState`.)
