# Story 4.3: Route Choice and Forward Commitment

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want to choose one available route node and commit forward,
so that the Labyrinth descent has irreversible tactical stakes.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` → Epic 4 → Story 4.3 (verbatim, with the implementation-grounded detail this story adds).

1. **Only legal forward nodes are selectable.**
   **Given** the player is at a route choice (the run is in `RunState.PHASE_ACTIVE_ROUTE`, parked on `route.current_node_id`)
   **When** available nodes are queried
   **Then** only legal forward nodes are selectable — i.e. the current node's `outgoing_link_ids` that are REVEALED (`RouteNode.REVEAL_REVEALED`) and NOT in `cleared_node_ids`
   **And** hidden (`REVEAL_HIDDEN`) or unreachable future nodes (not a current-node outgoing link) cannot be selected. (This is the choice-eligibility FILTER that Story 4.1 explicitly deferred to 4.3; see the deferred-work overlap below.)

2. **Committing to a legal next node advances run state and emits a route-advanced event.**
   **Given** the player commits to a legal next node (a node in the eligible-choice set from AC1)
   **When** the route command executes
   **Then** run state advances to that node (`route.current_node_id` becomes the chosen node id) and emits a `route_advanced` `DomainEvent`
   **And** previous nodes are marked unavailable for backtracking (the node the player just left is recorded in `cleared_node_ids`, and because every route edge is strictly forward — proven in 4.2 — no edge can ever target a now-cleared lower-depth node).

3. **Selecting a cleared, hidden, or non-linked node returns a stable error and mutates nothing.**
   **Given** the player attempts to select a cleared, hidden, or non-linked node (a node id NOT in the AC1 eligible set: already in `cleared_node_ids`, `REVEAL_HIDDEN`, not an outgoing link of the current node, an unknown id, the current node itself, or any selection made in the wrong run phase)
   **When** the route command executes
   **Then** it returns a stable error (a structured `ActionResult.error` with a stable lower_snake code distinguishing the rejection reason)
   **And** route state remains unchanged (the `RouteState` / `RunState` is byte-identical before and after the rejected command — the no-mutation contract; zero events emitted).

### Acceptance-criteria interpretation notes (read before coding)

- **This story is the `RouteAdvanceCommand` (choose-and-commit) + its choice-eligibility filter + a NEW `route_advanced` `DomainEvent`.** It is the FIRST run-domain command. It does NOT generate routes (4.2, DONE), does NOT enter/exit nodes or create a level request (4.4), does NOT resolve per-node-type behavior (4.5), and does NOT build the full start-to-end shell/pacing (4.6).
- **"only legal forward nodes are selectable" = a reveal-gated, cleared-excluded choice FILTER, layered ON TOP of the 4.1 graph + the 4.2 forward-only edge guarantee.** 4.1's `RouteState.available_choice_ids()` returns `current.outgoing_link_ids` minus `cleared_node_ids` with NO reveal-state gating — a `hidden` linked node is still surfaced. 4.2 proved the GRAPH cannot contain a backtracking edge (every edge strictly increases depth). 4.3 adds the SELECTION-time filter the other two deferred: a node is an ELIGIBLE choice iff it is (a) an outgoing link of the current node, (b) `REVEAL_REVEALED`, and (c) NOT in `cleared_node_ids`. Add this as a NEW derivation (e.g. `RouteState.eligible_choice_ids()` next to `available_choice_ids()`), and have the command validate the chosen id against THAT set. Do NOT silently change `available_choice_ids()`'s existing behavior (4.1's tests pin it: `test_route_state.gd::_available_choices_exclude_cleared_nodes` expects a REVEAL_CLEARED-but-not-in-cleared-ids subtlety to be excluded only by cleared-id membership, and a revealed node to be included). Add a sibling method; keep the old one as-is.
- **"run state advances to that node" + "reveal-on-arrival" is load-bearing and easy to miss → soft-lock if skipped.** The 4.2 generator marks ONLY depth-0 (start) and depth-1 nodes `REVEAL_REVEALED`; everything at depth ≥ 2 is `REVEAL_HIDDEN`. So after the FIRST commit (start → a depth-1 node), that depth-1 node's outgoing links (depth-2) are all `REVEAL_HIDDEN` and the AC1 filter would return an EMPTY eligible set — the run soft-locks with no legal choice (and the boss is never reachable). Therefore committing to a node MUST reveal that node's immediate `outgoing_link_ids` (flip each linked node `REVEAL_HIDDEN → REVEAL_REVEALED`) as part of the advance, so the next tier becomes selectable. This "reveal the chosen node's forward neighbors on arrival" is the mechanic that makes the reveal-gating filter coherent over a multi-step descent. Decide and document: reveal the arrived node's direct forward neighbors (depth+1) on commit (recommended, minimal, deterministic). Do NOT reveal the whole graph; reveal exactly the next reachable tier. (GDD line 249 "locked once visible": once revealed, a node's facts are stable — revealing is monotonic, never un-revealed.)
- **"emits a route-advanced event" = a NEW `route_advanced` `DomainEvent`.** There is no route-advanced event yet — you must ADD `Type.ROUTE_ADVANCED` + `EVENT_ID_ROUTE_ADVANCED := &"route_advanced"` + a `DomainEvent.route_advanced(sequence_id, payload)` factory + a payload validator + the `id_for_type`/`type_for_id` map arms, mirroring the existing `run_started` SYSTEM event (no actor — route advance is a system/run-level transition, not an entity action; so it is NOT added to `_event_requires_actor`). Suggested payload (all serializable, JSON-round-trippable): `from_node_id` (String), `to_node_id` (String, non-empty), `to_node_type` (lower_snake, from `RouteNode.TYPE_*`), `to_node_depth` (non-negative integral), `cleared_node_id` (String — the node just left), `revealed_node_ids` (Array[String] — the neighbors newly revealed on arrival, may be empty). Use the existing payload-validator helpers (`_has_nonempty_string_payload`, `_has_lower_snake_payload`, `_has_nonnegative_integral_payload`, `_has_string_array_payload`, etc.). Node ids contain hyphens (`node-1-0` from 4.2's `_mint_node_id`), so node ids are NOT lower_snake `_is_lower_snake_id` values — validate them as plain non-empty strings, and put node ids in payload/metadata, NEVER in an error code (error codes reject hyphens via `ActionResult._is_valid_error_code`). `to_node_type` IS lower_snake and may use the lower_snake validator.
- **"previous nodes are marked unavailable for backtracking" = move the left node into `cleared_node_ids` + mark it `REVEAL_CLEARED`.** When the command advances start→next, the node being LEFT (the old `current_node_id`) is appended to `cleared_node_ids` and its `reveal_state` set to `RouteNode.REVEAL_CLEARED` (the 4.1 vocabulary). This is the run-state truth of "the path sealed behind you." NOTE the boundary: the deterministic `door_sealed`/`route_sealed` PRESENTATION event with the `door_sealed_placeholder` cue id (GDD line 210 "doors seal behind the hero as a containment law") is **Story 4.4** (node EXIT resolution), NOT 4.3. 4.3 records the route-state fact (cleared + unavailable) and emits `route_advanced`; 4.4 emits the door-sealed cue on node exit. Do NOT add a door-sealed event here.
- **Phase handling: the command requires `PHASE_ACTIVE_ROUTE`; whether it transitions to `PHASE_NODE_RESOLUTION` is a documented decision.** The 4.1 `RunState` machine is `NEW_RUN → ACTIVE_ROUTE → NODE_RESOLUTION → (ACTIVE_ROUTE | COMPLETED | FAILED)`. The route-choice/commit happens IN `PHASE_ACTIVE_ROUTE`. AC2 says "run state advances to that node." Two defensible readings: (A) 4.3 advances the route POINTER (`current_node_id`) and leaves the run in `ACTIVE_ROUTE`, deferring the `ACTIVE_ROUTE → NODE_RESOLUTION` transition to 4.4 node-entry (the transition belongs with "create/load the level request"); or (B) 4.3 both advances the pointer AND transitions `ACTIVE_ROUTE → NODE_RESOLUTION` (committing to a node = entering node resolution). RECOMMENDED: (A) — keep 4.3 to the route-pointer advance + reveal + clear-left-node + `route_advanced` event, and require the phase be `ACTIVE_ROUTE` (reject in any other phase with a stable `wrong_run_phase`-style code), leaving the phase TRANSITION to 4.4 where node entry actually happens. Whichever you choose, RECORD it in Completion Notes and make the wrong-phase rejection (AC3) explicit and tested. Do NOT pull node-entry / level-request creation forward (that is unambiguously 4.4).
- **The command draws NO RNG and is a pure deterministic state transition.** Route generation already drew the `map` stream (4.2). Choosing and committing a node is deterministic given (run state, chosen id) — it mutates the route pointer / cleared set / reveal flags and emits one event; it draws ZERO RNG (no stream touched), executes no sub-command, and consumes nothing from `RngStreamSet`. Mirror the 4.1 "run-state init/transition draws no RNG" discipline (`test_run_state.gd::_new_run_draws_no_rng`): hold an `RngStreamSet`, snapshot it, run the advance, assert the streams are byte-identical.
- **Command shape mirrors the Epic-1 command precedent.** Extend `GameCommand` (`scripts/core/commands/game_command.gd`): `validate(state) -> ActionResult` then `execute(state) -> ActionResult` that calls `validate` first and returns `ActionResult.ok([route_advanced_event], metadata)` on success. The command operates on a RUN context (the `RunState`, or a thin run-action context holding it) the way `MoveCommand` operates on a `TacticalActionContext`. Decide the context shape (pass the `RunState` directly, OR introduce a minimal `RunActionContext` holding the `RunState`) and document it; the tactical commands use a context object, so a small `RunActionContext` is the consistent choice but passing `RunState` directly is acceptable for a single-field context — keep it simple and typed.

## Dev Notes

### What this story IS (and is NOT)

- **IS:** the route CHOICE / forward-commitment slice — a `RouteAdvanceCommand` (the FIRST run-domain command) that validates a chosen next node against a reveal-gated, cleared-excluded eligibility filter, commits the advance (move `current_node_id`, append the left node to `cleared_node_ids` + mark it `REVEAL_CLEARED`, REVEAL the arrived node's forward neighbors so the next tier is selectable), and emits a NEW `route_advanced` `DomainEvent` — plus the `RouteState.eligible_choice_ids()` selection filter that 4.1 explicitly deferred to this story.
- **IS NOT:** route GENERATION. `RouteGenerator`/`RouteValidator` (`scripts/generation/route/`) and the `RouteState`/`RouteNode`/`RunState` model (`scripts/run/`) already exist (Stories 4.1 + 4.2, DONE). 4.3 CONSUMES a generated route; it does NOT regenerate it, does NOT touch the `map` stream, and does NOT change the generator or its pinned fingerprints.
- **IS NOT:** node ENTRY/EXIT, level-request creation, the `GenerationResult`(level) consumption, or the deterministic `door_sealed`/`route_sealed` PRESENTATION event with the `door_sealed_placeholder` cue id — that is **Story 4.4**. 4.3 records the route-state TRUTH of "the path sealed behind you" (cleared + unavailable for backtracking) and emits `route_advanced`; it does NOT emit a door-sealed cue, does NOT call `LevelGenerator`/`ManualSeedLoader`, and does NOT create a level `GenerationRequest`. Per the AC interpretation, the `ACTIVE_ROUTE → NODE_RESOLUTION` phase transition is RECOMMENDED to stay with 4.4 (node entry), not 4.3.
- **IS NOT:** per-node-type RESOLUTION behavior or the boss placeholder run-end semantics (**Story 4.5**), nor the start-to-end playable shell + pacing + the `run_started`/`run_ended` emission (**Story 4.6**). 4.3 advances the pointer through revealed forward nodes regardless of type; it does NOT implement what entering a `shop`/`elite_combat`/`boss` DOES.
- **IS NOT:** the run-start COMMAND or the `run_started` emission. 4.3 is the route-ADVANCE command, not the run-START command. Do NOT emit `run_started` here, and do NOT fix the two `run_started` payload defers (`node_count` raw-JSON-number, `_has_decimal_string_payload` loose `is_valid_int()`) — those belong to whatever story first EMITS `run_started` (the run-start flow, 4.6 per the current split). See the deferred-work overlap below. (Adding the SEPARATE `route_advanced` event is in scope; touching `run_started` is not.)

Keeping these boundaries tight is the single biggest risk for this story, exactly as it was for 4.1 and 4.2. The epics Story 4.3 AC is "choose one available route node and commit forward" — a command + its eligibility filter + its event. The temptation is to pull node entry / level-request creation (4.4) or per-type resolution (4.5) forward. Resist it.

### Architecture patterns and constraints

- **Gameplay actions are validated commands returning `ActionResult`; successful commands emit deterministic past-tense `DomainEvent` records.** `RouteAdvanceCommand` follows the Epic-1 command precedent exactly: `validate()` then `execute()` (which re-validates), returning `ActionResult.ok([route_advanced_event], metadata)` on success and a structured `ActionResult.error(code, metadata)` with ZERO events + no mutation on failure ([Source: `project-context.md` §Determinism & Simulation Rules; `godot/scripts/core/commands/move_command.gd`]).
- **Route advance is a deterministic, RNG-free state transition.** Choosing a node is a pure function of (run state, chosen id). It draws NO RNG (no stream touched), runs no sub-command, and consumes nothing from `RngStreamSet` — mirroring the 4.1 "run-state init/transitions draw no RNG" rule. The `map` stream was already fully consumed by 4.2 generation ([Source: `godot/scripts/run/run_state.gd`; `godot/tests/unit/run/test_run_state.gd::_new_run_draws_no_rng`]).
- **Scene-independent, autoload-free domain ownership.** The command + the `eligible_choice_ids` filter live in the scene-free domain (`scripts/core/commands/`, `scripts/run/`) as `RefCounted` services/DTOs. No `Node`, no `get_tree`/`get_node`, no new autoload. The autoload set is unchanged through Epic 3 + 4.1/4.2 (`GameSession`, `SceneManager`, `SaveManager`, `AudioManager`, `SettingsManager`, `Diagnostics`) — do NOT register a route-command autoload. UI submits this command through the command bridge later; 4.3 builds only the domain command ([Source: `project-context.md` §Engine-Specific Rules, §Code Organization Rules]).
- **Validate-then-mutate, no coercion, no-mutation-on-reject.** An ineligible / wrong-phase / unknown-id selection surfaces a structured `ActionResult.error` (stable lower_snake code + diagnostics in metadata) and mutates NOTHING — the `RunState`/`RouteState` is byte-identical before and after. This is the Epic-1 command no-mutation contract + the 4.1 illegal-transition contract ([Source: `godot/scripts/core/commands/move_command.gd`; `godot/tests/unit/run/test_run_state.gd::_assert_illegal_transition`]).
- **Reveal is monotonic ("locked once visible").** Committing reveals the arrived node's forward neighbors (HIDDEN → REVEALED); a revealed node is never un-revealed, and a node's facts are stable once visible (GDD line 249). The left node moves HIDDEN/REVEALED → CLEARED. Reveal-state stays orthogonal to type/depth (4.1 rule) ([Source: `_bmad-output/planning-artifacts/gdds/gdd-Game-2026-05-31/gdd.md` line 249; `godot/scripts/run/route_node.gd` REVEAL_* constants]).
- **Compose, do not fork; keep the save bridge coherent.** The advance mutates the live `RouteState` inside the `RunState`. `RunState.to_run_snapshot_fields()` (4.1) derives `current_route_node_id` + `revealed_route_node_ids` from the route — after an advance those reflect the new state automatically, with the phase still nested under `route_state` (`RUN_PHASE_KEY`) and NO new top-level `RunSnapshot` key. Do NOT add a top-level run/route key or fork a parallel save format ([Source: `godot/scripts/run/run_state.gd` `to_run_snapshot_fields`; `project-context.md` §Save Rules; epic-4 retro Story 4-1 item on nested `run_phase`]).
- **New domain event = full wiring.** A `route_advanced` event must be wired end-to-end like the existing events: `Type` enum value, `EVENT_ID_*` const, factory, payload validator, `id_for_type`/`type_for_id` arms.
