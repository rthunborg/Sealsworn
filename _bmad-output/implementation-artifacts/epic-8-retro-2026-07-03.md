## 1. Epic Summary

Epic 8 is the epic that lets a run **end** and the meta profile **begin**. It closed the roguelite return loop in seven deliberate steps: **8.1** the run-END boundary (the first command to actually drive `PHASE_FAILED`, a new `run_failed` event carrying a cause, `run_completed` broadened past the boss-placeholder marker, and a scene-free `RunEndOutcome` carrying `next_destination: "outpost"`); **8.2** the DERIVED, non-persisted `RunSummary` read surface (FR60 — cause of death/victory, nodes cleared, consumed/destroyed passives, notable loot, seed, manual-seed eligibility); **8.3** the FIRST persistent cross-run state in the project — the versioned `ProfileSnapshot` + `ProfileRepository` (`SCHEMA_VERSION == 1`, atomic temp→backup→replace, structured recovery) with the Oath-Shard AWARD command and the migration-reject path baked in from day one; **8.4** Echoes / Seal-Fragments / class-mastery / unlock-progress content + the profile-MERGE command + capped `UnlockProgressRules` thresholds, merged into the homes 8.3 reserved WITHOUT a schema bump; **8.5** the first-death FLAG + `FirstDeathNarrativeBeat` DTO + the `first_death_recorded` event (the FR28 narrative-vs-meta boundary precedent, human-ratified Option A); **8.6** the pure-read `OutpostViewModel` + named-space metadata (Memory Archive, Hall of Oaths, Seal Table, Gate/Descent Stair) + the start-another-descent request + the recovery/fresh-profile path; and **8.7** the comprehensive cross-cutting meta/summary SAVE-LOAD + MIGRATION TEST MATRIX proving every 8.1–8.6 contract survives a real save → app-restart → load round-trip.

The epic held to its plan with the same structural fidelity that carried Epics 5–7 — but this time while introducing the project's first cross-run persistent state, which is exactly the kind of change that usually forces a migration treadmill. It did not:

- **The meta profile was correctly given its OWN save shape**, distinct from the run autosave. `ProfileSnapshot`/`ProfileRepository` live at `user://profile.json`; the run autosave (`RunSnapshot`, its own `SCHEMA_VERSION`, the 23-key gate) stays at `user://run_autosave.json`. The **23-key `RunSnapshot` gate stayed exactly 23 all epic** — no meta state was smuggled into the route snapshot. This is the meta-save-shape decision Epic 7's retro (T2) said to make early, made correctly at 8.3 and proven separable at 8.7.
- **`ProfileSnapshot.SCHEMA_VERSION` stayed 1 across 8.3 → 8.4 → 8.5.** 8.3 reserved the `class_mastery` / `echoes` / `unlock_progress` / `first_death_recorded` homes on day one; 8.4 merged Echoes/Seal-Fragments/mastery/unlock-flags into them and 8.5 set the first-death latch — both WITHOUT a bump or a migration step. The migration-REJECT path (`unsupported_profile_schema` + `{expected, actual}`) and lenient legacy-parse-with-clean-defaults were baked in from 8.3, so the "does not grant unintended progress" clause was provable at 8.7 without ever having shipped a v2.
- **The append-only `DomainEvent.Type` discipline held for SIX new/changed event surfaces** (`run_failed` 8.1, `oath_shards_awarded` 8.3, `content_discovered` + `profile_progress_merged` 8.4, `first_death_recorded` 8.5, plus the `run_completed` outcome-allowlist broadening 8.1) — every one appended at the enum end, `expected_ids`-pinned, JSON-round-tripped, per-field malformed-negative tested. No renumber anywhere.
- **Zero gameplay RNG was introduced.** Run-end resolution, meta award, discovery merge, and the first-death latch are all deterministic phase-transition + command + event work — no roll. The 7 named RNG streams and `required_streams()` were untouched.
- **FR28 (manual-seed grants no meta progression) was enforced with a clean, ratified boundary.** The award (8.3) and the merge (8.4) both DENY a manual-seed run (`run_not_meta_eligible`); the first-death latch (8.5) is deliberately eligibility-INDEPENDENT (a manual-seed first death still shows the line and sets the boolean, because a boolean latch grants zero Oath Shards / unlocks / mastery). That fork was flagged at 8.5 create-story and human-ratified as **Option A** — and it is now the project's FR28 narrative-vs-meta precedent that Epic 9's first-VICTORY reveal will build on.

## 5. Key Insights

1. **When new state is a different LIFETIME than existing state, give it its own save — do not nest.** Epic 7 correctly nested the risk economy under `route_state` because it *is* run-position state. Epic 8 correctly did the OPPOSITE: the meta profile is run-SPANNING, not route-position state, so it got its own `ProfileSnapshot` + `ProfileRepository` + path. That single seam choice is why the 23-key `RunSnapshot` gate never moved, the two saves are provably separable, and the profile schema could evolve independently. The lesson is not "nest" or "don't nest" — it is "match the save boundary to the state's lifetime."
2. **Reserve the schema homes on day one, merge into them without a bump.** 8.3 declared the `class_mastery`/`echoes`/`unlock_progress`/`first_death_recorded` fields at `SCHEMA_VERSION == 1` before any of them had content; 8.4 and 8.5 then filled them without a migration. Reserving empty homes early is what turned three save-touching stories into zero migration steps. A dedicated top-level field for Seal Fragments would have been a deliberate schema-v2 + migration decision — 8.4 correctly folded them into `unlock_progress` instead, honoring the discipline.
3. **Bake the migration-REJECT and lenient-legacy paths in with the FIRST version, so "no unintended progress" is provable before a v2 exists.** 8.3 shipped `unsupported_profile_schema` + lenient-parse-with-clean-defaults on the first profile commit. That is why 8.7 could assert a rejected/leniently-parsed profile carries zero currency, zero unlocks, first-death `false` — the safety net was already load-bearing before there was anything to migrate FROM.
4. **A boolean latch is progression-free, so a narrative flag can be eligibility-independent without violating FR28.** The sharpest scope call of the epic: the first-death line fires on a manual-seed death (narrative flavor) even though the award and merge deny manual-seed runs — because setting a boolean grants zero Oath Shards / unlocks / mastery. Distinguishing "narrative state" from "meta progression" by *what it grants* (nothing vs. currency/unlocks) is the FR28 boundary, and it was ratified as precedent rather than decided silently.
5. **Independent per-run idempotency markers beat a shared one when caller order is not guaranteed.** The award (`last_awarded_run_seed`), the merge (`unlock_progress["_last_merged_run_seed"]`), and the first-death latch (the `first_death_recorded` bool itself) are THREE independent idempotency mechanisms, each safe in any caller order. A shared marker would make whichever run-end command runs first block the second. Deliberately diverging from the story's "prefer a shared marker" was the right call — 8.6/8.7 must (and do) treat them as independent.
6. **The JSON boundary lies about int types on nested data — normalize before comparing.** Godot's `Dictionary ==` is type-strict and JSON decodes ints as floats, so `{"warrior": 3} != {"warrior": 3.0}` after a round-trip. A whole-dict byte-identity assertion FALSE-FAILS on any nested int-valued dict. Int-coercion-aware normalization is now mandatory for every nested-save round-trip test. (This is why `last_awarded_run_seed` is a decimal STRING, not an int — JSON doubles truncate a full int64 seed beyond 2^53.)
7. **FR numbers are not stable across artifacts — trace obligations by design intent, not by number.** The 8.6 "named outpost/meta spaces" obligation traces to a *design-time GDD* "FR63" via the 2026-06-04 readiness patch — but canonical `epics.md` FR63 is the Larval Avatar boss (Epic 9). Citing "FR63" naively would mis-scope. Same class of trap as the 8.7 AC1 "class unlock states restore" wording (v0 has no profile→class-selectability wiring, so the AC was correctly redefined as "profile STATE round-trips"). Read the obligation, not the label.
