This is the FIFTH and FINAL story of Epic 9 — "Larval Avatar MVP Finale." It CONSUMES everything Epic 9 shipped — 9.1's encounter SETUP (`BossNodeEnterCommand` / `BossEncounterRequest` / `BossArenaBuilder` / the `boss_encounter_started` event / `RunOrchestrator._resolve_boss` + the boss-setup terminus), 9.2's DEFINITION/phase model (`BossDefinition` / `BossPhaseResolver` / the `boss_phase_changed` event), 9.3's FIGHT behavior (`BossAi` / `BossCommandAdapter` / `BossTurnResolver` + the two-turn telegraph→resolve flow + the live phase re-resolution seam), and 9.4's VICTORY resolution (the `detect_boss_defeat` seam + `boss_defeated` event + the `victory` `run_completed` outcome + `resolve_run_end(victory)` + the `first_victory_recorded` latch/command/event + `FirstVictoryRevealBeat`) — and ties them into ONE comprehensive, deterministic finale: an approved-BOSS-SEED regression suite proving setup/phases/telegraphs/victory/defeat all stay deterministic (failures report seed + phase), a FULL run reaching the boss THROUGH the run shell whose defeat OR death records boss progress into the run summary + delivers the correct completion event to the outpost/meta flow, and LOCAL/OFFLINE run-length + boss tuning diagnostics (turn count / damage taken / major telegraphs / outcome). It builds NO new boss content, NO new gameplay mechanic, and NO new event — it is the regression/integration/diagnostics closeout that makes the finale a FAIR CULMINATION.)

**AC1 — Approved boss-seed regression keeps setup/phases/telegraphs/victory/defeat deterministic**
- **Given** boss encounter seed regression tests run
- **When** approved boss seeds are executed
- **Then** setup, phases, telegraphs, victory, and defeat paths remain deterministic
- **And** failures report seed and phase.

**AC2 — A full run through the shell records boss progress + delivers the correct completion event**
- **Given** a full run reaches the boss through the run shell
- **When** the boss is defeated or the player dies
- **Then** run summary records boss progress and outcome
- **And** outpost/meta flow receives the correct completion event.

**AC3 — Local tuning diagnostics capture the finale attempt without telemetry**
- **Given** tuning diagnostics are enabled for development builds
- **When** boss attempts complete
- **Then** diagnostics capture turn count, damage taken, major telegraphs, and outcome
- **And** diagnostics remain local/offline and do not introduce telemetry dependencies.

## Scope boundary (read FIRST)

This story is the **finale REGRESSION + full-run INTEGRATION + tuning DIAGNOSTICS closeout** of Epic 9 — it makes the Larval Avatar finale a *provably fair, deterministic culmination of the full MVP run*, WITHOUT adding any new boss content, mechanic, event, or save key. Concretely it ships three deliverables, one per AC:

1. **(AC1) A finale SEED-REGRESSION SUITE** — a headless regression harness (the `test_seed_batch_regression.gd` model) that drives an APPROVED, ANNOTATED, PRESERVED catalog of boss seeds through the WHOLE Epic-9 boss chain (9.1 SETUP → 9.2 phases → 9.3 telegraphs/AI → 9.4 victory + defeat) and asserts each path is DETERMINISTIC (the same seed → byte-identical / reproducible setup, phase transitions, telegraph→resolve behavior, and victory/defeat resolution), with a **failure report carrying seed + phase** (the `GenerationResult`-style compact discipline — NEVER a grid dump). It covers BOTH terminal outcomes (a boss defeated → victory, AND a hero death mid-fight → defeat — the death is the seeded-fixture death, NOT a new live hero-death SOURCE; see the OUT-of-scope note).
2. **(AC2) A FULL-RUN-THROUGH-THE-SHELL INTEGRATION** — the comprehensive integration test that 9.1/9.4 EXPLICITLY earmarked to 9.5: drive `RunOrchestrator.start(...)` → `run_to_completion()` to the boss-setup terminus (the run parks non-terminal in `NODE_RESOLUTION` with `boss_encounter_pending == true`), then DRIVE THE BOSS FIGHT on the live arena board to 0 HP (or to a hero death), then resolve the run END through the 9.4 seam (`detect_boss_defeat` → `resolve_run_end(victory)` for a defeat; `resolve_run_end(<death cause>)` for a death), then build `RunSummary` off the terminal run + the collected events and assert it RECORDS boss progress (`boss_cleared` / the outcome) AND the outpost/meta flow receives the CORRECT completion event (`run_completed` + `outcome == victory`, or `run_failed` + a cause). **⭐ THE LOAD-BEARING RECONCILIATION: `RunSummary.boss_cleared` derives from a CLEARED `TYPE_BOSS` route node, but the live 9.4 victory chain drives the run to `PHASE_COMPLETED` WITHOUT clearing the boss route node — so the integration MUST reconcile this (mark the boss node cleared on victory, or record boss progress from the `boss_defeated` event) or `boss_cleared` reads FALSE after a live victory. See the ⭐ reconciliation note below.**
3. **(AC3) LOCAL/OFFLINE tuning DIAGNOSTICS** — a build-profile-gated diagnostics recorder (the `LocalTimingRecorder` / `Diagnostics` autoload model — inert unless `OS.is_debug_build()`) that captures a boss ATTEMPT's **turn count + damage taken + major telegraphs + outcome** as LOCAL in-memory / stdout records, introducing ZERO telemetry / network / cloud dependency (the `TelemetrySink`-stays-local rule).
