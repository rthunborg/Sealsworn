## ⭐ AC1: a saved profile restores every cross-run field; current-run autosave stays SEPARATE from profile/meta data

AC1 has TWO halves — the full-field restore, and the separability from the run autosave:

**Half 1 — full-field profile restore round-trip.** Build a FULLY-POPULATED `ProfileSnapshot` exercising EVERY field the 8.3/8.4/8.5 systems can set, then `ProfileRepository.write_profile(snapshot, TEST_PATH)` → **simulate an app restart** by discarding the in-memory objects and constructing a FRESH `ProfileRepository` → `read_profile(TEST_PATH)` → assert every field byte-identical:
- `oath_shards` (a non-trivial awarded total, e.g. 12) — restores exactly.
- `last_awarded_run_seed` (a FULL-int64 decimal string, e.g. `"9223372036854775000"`) — restores losslessly (NO JSON-double truncation — the decimal-string encoding is why; assert the exact string).
- `echoes` (a non-empty Array[String], e.g. `["echo_of_salt", "echo_of_tide"]`) — restores in order.
- `unlock_progress` — the COMPOSITE 8.4 home: `seal_fragments` (a unique-id SET, e.g. `["seal_a", "seal_b"]`), the threshold flags (`seal_gate_1_unlocked: true`), the class-mastery is a SEPARATE top-level field, AND the dedicated merge marker (`_last_merged_run_seed: "<seed>"`) — all restore verbatim (Seal Fragments live INSIDE `unlock_progress`, NOT a top-level key — the 8.4 home decision).
- `class_mastery` (a Dictionary of per-class counts, e.g. `{"warrior": 3}`) — restores (NOTE JSON has no int/float distinction: an int dict VALUE decodes as a float via JSON — assert with `int(...)` coercion, the `test_profile_snapshot.gd` precedent).
- `first_death_recorded` (`true`) — restores as the set latch.
- `schema_version` stays `1`, `content_version` / `profile_id` restore.

**Half 2 — the profile save is SEPARATE from the current-run autosave.** The profile is its OWN save file (`user://profile.json`) at a DISTINCT path from the run autosave (`user://run_autosave.json`). Prove separability structurally:
- `ProfileRepository.DEFAULT_PROFILE_PATH != SaveRepository.DEFAULT_RUN_PATH` (the paths differ — `test_profile_repository.gd` already asserts the profile path; 8.7 asserts the CONTRAST against the run-autosave path explicitly).
- Write BOTH a profile (via `ProfileRepository`) AND a run autosave (via `SaveRepository.write_run_snapshot`) to their respective test paths, then read each back — each restores its OWN data, and reading/writing one leaves the OTHER file untouched (no cross-contamination). The profile OUTLIVES a run; the run autosave is run-scoped. A NEW descent (a fresh `RunSnapshot`) does NOT overwrite the accumulated profile, and awarding to the profile does NOT touch the run autosave.

**"Class unlock states restore correctly" (AC1) — READ THIS CAREFULLY (a scoping nuance):** In v0 there is NO wiring from the profile to `ClassDefinition.lock_state`. Classes are STATIC CONTENT — `warrior`/`pyromancer`/`ranger` are `LOCK_STATE_SELECTABLE` and `necromancer`/`shadeblade` are `LOCK_STATE_LOCKED` on the `ClassDefinition` itself, NOT derived from `profile.unlock_progress` or `class_mastery` (grep-confirmed: `HeroSelectViewModel` has NO profile/unlock coupling; the unlock-SPEND/apply system that would turn profile state into a class unlock is EXPLICITLY DEFERRED — see the scope fence). Therefore "class unlock states restore correctly" in v0 means: **the PROFILE fields that REPRESENT unlock/class progress — `unlock_progress` (incl. the threshold flags + `seal_fragments`) and `class_mastery` — persist and restore correctly.** Do NOT interpret this AC as requiring a live re-derivation of `ClassDefinition.lock_state` from the profile, and do NOT build a profile→class-unlock wiring (that is the deferred meta-spend/apply system). Test that the unlock-progress + class-mastery STATE round-trips; note in the test/Completion Notes that the SPEND/APPLY that would consume that state to unlock a playable class is deferred (Epic 9 / a later meta-spend story).

## ⭐ AC2: migration — a supported older snapshot migrates OR fails with a clear unsupported-version result; migration grants NO unintended progress

AC2 is the MIGRATION MATRIX. `ProfileSnapshot.SCHEMA_VERSION == 1` is the ONLY shipped schema (8.4 + 8.5 merged into reserved homes WITHOUT a bump — `SCHEMA_VERSION` STAYS 1). The migration path is the versioned `parse(data)` contract baked in from 8.3. Prove:
- **Current-schema round-trip:** a `schema_version == 1` snapshot writes + reads back at the SAME version (NO migration needed — the shipped case). `parse` succeeds; `schema_version` is unchanged.
- **Unsupported-schema REJECT (the migration-reject path):** a snapshot dict with `schema_version` != 1 (e.g. `2`, or `SCHEMA_VERSION + 10`) → `ProfileSnapshot.parse(...)` returns `ActionResult.error(&"unsupported_profile_schema")` with `{expected_schema_version: 1, actual_schema_version: <that value>}`. And through the REPOSITORY: a `user://` file with an unsupported schema → `ProfileRepository.read_profile(...)` surfaces `unsupported_profile_schema` (NOT a crash, NOT a partial profile). The AC's "fail with a clear unsupported-version result" is exactly this stable code + the version metadata.
- **Legacy / partial dict LENIENT parse (a pre-8.4 / pre-8.5 profile):** a `schema_version == 1` dict MISSING the 8.4/8.5 fields (e.g. only `schema_version` + `oath_shards`) parses cleanly with defaults — `class_mastery == {}`, `echoes == []`, `unlock_progress == {}`, `first_death_recorded == false`, `last_awarded_run_seed == ""`. This is the "older SUPPORTED snapshot migrates" half (a same-version leniently-forward-compatible parse — a legacy profile that predates the 8.4/8.5 homes still loads at v1 with empty new homes). NOTE: `test_profile_snapshot.gd` already covers `_partial_legacy_dict_parses_leniently` + the pre-8.5 legacy-first-death default — 8.7 EXTENDS this into the migration NARRATIVE (a legacy → current forward-compat parse through the REPOSITORY + a JSON file), it does not merely re-assert the unit case.
- **"Migration does not grant unintended progress" (AC2 — the load-bearing safety clause):** assert that NEITHER a leniently-parsed legacy profile NOR a rejected unsupported-schema read grants any phantom progress. A leniently-parsed legacy profile carries the fields it HAD + clean-zero defaults for the rest (a missing `oath_shards` → 0, NOT a phantom award; a missing `first_death_recorded` → false, NOT a spuriously-set latch; a missing `seal_fragments` → empty, NOT a phantom unlock). A rejected read yields NO profile at all (the caller falls back to `ProfileSnapshot.fresh()` — a brand-new 0-shard profile). Crucially: a NEGATIVE / garbage `oath_shards` clamps to 0 (never a negative or inflated total — the `_nonnegative_int_or_zero` floor), a garbage `unlock_progress`/`class_mastery` defaults empty (never a coerced foreign shape). The migration/lenient-parse layer can ONLY lose or zero fields, NEVER invent progress.

**`SCHEMA_VERSION` stays 1 (do NOT bump it):** 8.7 does NOT introduce a v2 schema or a real `migrate()` step. The migration MATRIX tests the EXISTING v0 migration contract (the version reject + the lenient forward-compat parse). A future schema bump (a genuine v1→v2 migration) is a LATER story that WILL add a `migrate` step — 8.7 proves the SCAFFOLDING (the version gate + the reject metadata + the lenient parse) is correct and ready, exactly as 8.3 designed it ("a future schema bump adds a real migrate step; Story 8.7 owns the comprehensive migration matrix"). If a test wants to exercise a hypothetical FUTURE version, it constructs a dict with a bumped `schema_version` and asserts the REJECT — it does NOT add a real v2 schema to production.

## ⭐ AC3: headless grant/deny matrix across eligible / manual-seed / death / completion; grants+denials match the rules; NO scene nodes serialized

AC3 is the GRANT-vs-DENY MATRIX. The three run-end profile-mutation commands (award / merge / first-death) each have gates; 8.7 proves the FULL cross-product of {eligible, manual-seed} × {completion, death} produces the correct grant/deny, THEN that the resulting profile SAVES + RELOADS with the granted (or withheld) state. Build TERMINAL `RunState` fixtures (the 4.3 idiom — a validated terminal run; see the fixture note) for each case:

| Case | Award (Oath Shards) | Merge (Echoes/Seal/mastery/unlock) | First-death flag |
|---|---|---|---|
| **Eligible + COMPLETED** | GRANTED — `min(1 + nodes_cleared, 5)` (>0) | GRANTED — discoveries merged, thresholds evaluated | N/A (`run_not_failed` reject — a completion is not a death; ZERO mutation) |
| **Eligible + FAILED (death)** | GRANTED but 0 (a death yields 0 currency this story) — `oath_shards` UNCHANGED, but the marker + event still fire (an eligible death IS awarded, amount 0) | GRANTED — discoveries merged (a death can still have discovered content) | GRANTED — the FIRST death latches `first_death_recorded == true` + emits |
| **Manual-seed + COMPLETED** | DENIED — `run_not_meta_eligible`, ZERO mutation, profile byte-identical | DENIED — `run_not_meta_eligible`, ZERO mutation | N/A (a completion is not a death) |
| **Manual-seed + FAILED (death)** | DENIED — `run_not_meta_eligible` | DENIED — `run_not_meta_eligible` | ⭐ GRANTED — the first-death latch is ELIGIBILITY-INDEPENDENT (Option A, human-ratified): a manual-seed first death STILL records + shows the line (a boolean narrative latch grants ZERO Oath Shards/unlocks/mastery, so it does NOT violate FR28) |

Then, for each GRANTED case, SAVE the mutated profile via `ProfileRepository.write_profile` → reload via a fresh repository → assert the granted state (the raised `oath_shards`, the merged `echoes`/`seal_fragments`/`class_mastery`/flags, the set `first_death_recorded`) survives byte-identical; and for each DENIED case, assert the profile is UNCHANGED from its pre-command state (a denied award/merge leaves the profile byte-identical — no phantom grant persisted). This is the "grants and denials match expected rules" clause proven end-to-end THROUGH the save layer.

**Key grant/deny facts to pin (the rules — do NOT re-derive them; READ them from the shipped commands):**
- A **manual-seed** run has `meta_progression_eligible == false` (the `RunState.validate()` lockstep: `meta_progression_eligible == not is_manual_seed`, in lockstep with `RiskEconomyState.oath_shard_eligible`). Both the award AND the merge reject it with the EXACT SAME code `run_not_meta_eligible`. Build the manual-seed fixture with `is_manual_seed = true` + `meta_progression_eligible = false` so `run.validate()` passes (a mismatched pair FAILS validation → `invalid_context`, not the eligibility reject — build the fixture correctly).
- A **death** (`PHASE_FAILED`) awards 0 (`MetaAwardRules` returns 0 for a non-`PHASE_COMPLETED` terminal run) but is still a VALID award target (an eligible death runs the award command successfully with amount 0 + sets `last_awarded_run_seed` + emits) — do NOT assert an eligible death REJECTS the award; assert it awards 0.
- The **first-death** latch is DEATH-ONLY (`run_not_failed` for a completed run) + ONCE-ONLY (`first_death_already_recorded` on a second death) + ELIGIBILITY-INDEPENDENT (fires on a manual-seed death too). The three run-end markers are INDEPENDENT (award = `last_awarded_run_seed`; merge = `unlock_progress["_last_merged_run_seed"]`; first-death = the `first_death_recorded` bool itself) — safe in ANY caller order; a 8.7 test MAY assert running award→merge→first-death in different orders on the same profile is order-independent (each separately idempotent).

**"No scene nodes serialized" (AC3, NFR15):** structural — `ProfileSnapshot.to_dictionary()` / `RunSnapshot.to_dictionary()` project plain serializable data (String/int/bool/Array/Dictionary), never a `Node`/`Object`/scene reference; the save truth is a versioned domain snapshot. A test MAY additionally assert the serialized dict contains no `Object`-typed value (walk the dict; assert every value is a JSON-primitive/Array/Dictionary) as an explicit NFR15 guard.
