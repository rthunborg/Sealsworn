## 5. Key Insights

1. **"Open the fence exactly as far as the current story needs, and not one file further" is the right way to grow a deferred subsystem.** The operation engine had been deferred since Epic 5. Rather than build a speculative generic `RuleCondition`/`RuleTarget`/`RuleOperation` kernel when the affinity stories arrived, 7.5 opened `operations/` to exactly one narrow resolver and 7.6 didn't touch it at all. `conditions/` stayed empty. The generic engine is still correctly deferred to a later/broader story that has a concrete need for it. This kept the blast radius of the riskiest architectural move of the epic to a single new file.
2. **Choosing the right domain seam is worth more than reusing the nearest existing one.** 7.6's pivotal call was recognizing that Darkness is a *visibility/fog/memory* effect, not a *board-cell hazard* effect — so it built a separate `fog/` layer instead of extending the 7.5 hazard resolver. That decision kept two existing Darkness-no-op tests green *by construction*, modeled the effect honestly, and avoided conflating two different effect kinds. The wrong (easier) choice would have broken tests and muddied the model.
3. **A guardrail is only as trustworthy as its alignment with the system it guards — so prove the alignment, don't assume it.** The FR58 fairness predicate is sound specifically because its "seen at the reduced radius" condition is byte-identical to what the LoS query actually marks visible. The model-diverse Round-2 review independently re-derived that alignment. That is the difference between a fairness check that *looks* right and one that *is* right.
4. **Nesting + reusing placeholders kept a save-touching epic off the migration treadmill entirely.** Economy state nested under `route_state`; affinity assignment reused the `affinities` placeholder; effects/visibility added nothing (live re-derivable). The 23-key gate never moved and there was no migration step in the whole epic — despite this being the epic the Epic-6 retro predicted "will touch the snapshot gate." It didn't have to, because the placeholders were already there and the effects are re-derivable.
5. **Record-only → applied is a clean two-epic handoff when the hooks are single and keyed.** Epic 6 deliberately left `reward_resolved`/`item_consumed`/`passive_destroyed` as single keyed hooks; Epic 7's economy state applied the mutation off exactly those hooks. The discipline that kept Epic 6 honest (record the outcome, defer the mutation) is what made Epic 7's wire-off trivial.
6. **The same "live-loop-later" residual, when it recurs every epic, eventually becomes the headline debt.** It was a footnote in Epic 5, a paragraph in Epic 6, and is now the single largest coherent body of latent value in the project. It is correctly deferred — but it is no longer a footnote.
