# Story 6.1: Item, Loot, and Reward Definitions

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want loot and reward options to come from approved definitions,
so that runs can offer readable build choices without hardcoded item logic.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 6 -> Story 6.1 (verbatim AC, with the implementation-grounded detail this story adds). FRs covered: FR46 (passive pool *definitions* — the content the class passive ids point at), FR51/FR52/FR53 (loot categories, item handling, consumable scarcity/value); GDD FR75/FR76 (item character-level requirement + roll model).

1. **MVP loot/reward definitions for every category are available through typed repository lookups; gameplay never reads source files directly.**
   **Given** MVP loot definitions are authored
   **When** the content repository loads them
   **Then** weapons, armor, jewelry, support items, consumables, pickups, passives, and gold-reward definitions are available through typed lookups (each category resolved through its repository -> `ContentRepository`, never `FileAccess`/`load()`/`JSON.parse` in a gameplay/hot path)
   **And** gameplay systems do not read source files directly.

2. **A definition referencing a missing id, invalid value range, or unsupported category fails closed with a structured content error.**
   **Given** a definition references a missing id, invalid value range, or unsupported category
   **When** content validation runs
   **Then** validation fails with a structured content error (a stable lower_snake `invalid_*_definition` / `invalid_*_repository` `ActionResult.error` with `{reason, field}` metadata — never a crash, a coerced field, or a silently-fabricated default)
   **And** invalid content is not available to gameplay (a fail-closed factory returns `null` / a fail-closed lookup returns `null`; no partial registration survives a rejected definition).

3. **Reward-offer rolls use the named `rewards`/`loot` stream and reproduce from the same seed and state.**
   **Given** reward selection uses randomness
   **When** reward offers are generated (in this story: through deterministic FIXTURES that prove the stream contract — the live reward-offer command/flow is Story 6.3)
   **Then** rolls use the named `rewards` or `loot` stream as appropriate (drawn through `RngStreamSet.rand_int`/`rand_float` on `STREAM_REWARDS`/`STREAM_LOOT`, NEVER `randi()`/`randf()`, NEVER a local `RandomNumberGenerator`, NEVER the `map`/`level`/`combat`/`events`/`cosmetic` stream)
   **And** the same seed and state reproduce the same reward offer (a re-run from the same `root_seed` + the same pre-draw stream state yields a byte-identical offer; a different seed diverges).

4. **Equippable item definitions declare a character-level requirement (or explicit `none`); run-depth is never the equip gate.**
   **Given** equippable item definitions are authored
   **When** content validation runs
   **Then** each equippable item declares a character-level requirement (a small non-negative int) OR an explicit `none` requirement (a named sentinel that resolves and means "no level gate")
   **And** minimum run-depth requirements are NOT used as the item equip gate (no `min_run_depth`/`min_node_depth`/route-depth field gates equipping — character level is the only equip gate; GDD FR75/FR76).

5. **Item instance variation uses roll ranges / affix-enhancement ids / affinity tags, not fixed item levels; un-implemented advanced roll families are explicitly deferred while preserving the data boundary.**
   **Given** MVP item instances can vary
   **When** item stat, affix, affinity, or enhancement data is defined
   **Then** the data model uses roll RANGES (min/max bands), affix/enhancement IDs, and affinity TAGS rather than fixed item levels (no `item_level` field drives stats)
   **And** any advanced roll family NOT implemented for MVP is explicitly deferred with a stable marker on the definition/data model (an MVP-deferral flag/note per advanced family) while PRESERVING the repository/data boundary for later rollout (the field/shape exists and validates inert, so Epic 7+/post-MVP can light it up without a schema fork).

6. **Duplicate-id registration fails loud across ALL content repositories (closes the carried Epic-5 cross-cutting hardening item).**
   **Given** two content definitions are registered under the same id in ANY content repository (carried cross-cutting `[Review][Defer]` from Epic 5 — see `deferred-work.md` "code review of 5-1")
   **When** the repository ingests them
   **Then** registration fails loud with a structured `duplicate_*` error (e.g. `duplicate_weapon` / `duplicate_class` / `duplicate_loot` carrying the offending id in metadata) instead of silently last-write-wins (where `*_ids()` keeps the FIRST and `get_*` returns the SECOND, so the id-list and the resolver disagree)
   **And** the guard is applied UNIFORMLY — retrofitting the six existing repositories that currently accept duplicates (`class`, `enemy`, `level_recipe`, `weapon`, `support`, `passive`) AND the new Epic-6 loot/reward repositories — done in this first-data-pipeline story so a duplicate id can never silently shadow another, with negative tests on every repo.
