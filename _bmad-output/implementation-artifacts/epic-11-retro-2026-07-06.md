## 4. What Was Challenging / Growth Areas

**Amelia (Developer):** The load-bearing subtlety of the whole epic was the gap between "provable in an integration test" and "playable hands-off," and it bit in two related places. First, the scripted live-combat hero is deterministic but NOT universally-winning across arbitrary generated seeds — a mutually-unreachable straggler hits the round cap and fails-loud; the live loop is the seed-batch/AC-proof driver, not a shipped hands-off game loop. A universally-winning live run needs a stronger hero driver (LoS-aware ranged targeting) or class-kit→combat-loadout wiring, and the `StartingKit.baseline_hp` (warrior 18) is a BALANCE number, not a viable live-driver HP (the live driver uses `LiveCombatResolver.DEFAULT_HERO_HP` 60; the HUD displays the class baseline between levels — two distinct concerns). Second, the 11.3 H1: the on-screen path silently diverged from the domain's tested resolve-then-advance order and auto-cleared the depth-0 opener. Both are the same root: when the presentation re-implements a sequence the domain already encodes, it can diverge invisibly unless the shared sequencing seam is the thing tested.

**Charlie (Senior Dev / Reviewer):** 11.3 was the epic's one genuinely bumpy story — three review rounds (Changes Requested → Changes Requested → Approve), one High, two Meds — and it's worth being honest about why: it's the story where the near-zero scene layer met the scene-free harness, so the surface most in need of an integration test was the surface the harness structurally can't reach. We handled it correctly (steer logic into RefCounted seams, verify scenes by construction, add the shared-seam test the H1 exposed), but it means the on-screen combat surface has a real coverage boundary: L4 recorded that the shell never renders the LIVE board on a combat node — `resolve_combat_node_live` returns terminal-only metadata with no `"board"` key, so the in-run "board hosts a generated level" surface renders the empty-board VM under auto-resolve. That's a direct consequence of the accepted auto-resolve scope line (the shell auto-plays to a terminal outcome rather than holding an interactive board across taps), so it's not a defect — but the interactive tap-loop handoff (L3) that would make the board actually playable-by-hand, and the L4 live-board render that pairs with it, remain the largest single piece of run-flow behavior still deferred.

**Dana (QA Engineer):** Two coverage boundaries are honest v0 states worth naming as forward signals. First, the scene surface is verified by construction, not by `SceneTree` tests (the harness is scene-free) — correct and disclosed, but it means the actual on-device layout/anchor behavior is unverified by the suite (11.3's L2 flagged a `Node2D`-hosting-a-`Control` anchor risk that was re-rooted to `Control`, but only a real on-device smoke can confirm the rendered result). Second, several live paths are proven under AUTO-RESOLVE as the headless testability stand-in — the affinity board renders correctly under auto-resolve, but a human PLAYING the fight via the wired tap-loop is the deferred L3 follow-up. And the run-level event store gap has a visible tail: 11.5's bridge builds `RunSummary.build(run, [])` with an empty events list, so the embedded summary's passives/loot/discovery lists come out empty AND `outcome_or_cause` is BLANK — a summary-render MUST key victory/death off `phase`, not `outcome_or_cause`, until a run-level event store lands.

**Sally (UX Designer):** The appendix is dense and internally cross-referenced, and that density cost us twice in review — two section-number cross-refs drifted (§0.2→§16, §0.4→§14) even after dev self-verification, and a field-source attribution mis-sourced hero HP on `RunState` (there is no run-level HP field — which is exactly WHY the G1 projection exists). Neither had contract-accuracy impact, but they're a signal: a dense spec that downstream scene stories bind against needs a final mechanical "resolve every §N against the section map" sweep, and its field-source attributions deserve the same pinned-key rigor as its contract bindings. ## 5. Key Insights

1. **Build the scene/UI layer ON TOP of already-testable RefCounted seams, verify the scenes BY CONSTRUCTION, and no scene work moves a domain invariant.** With a scene-free harness, the way to ship six stories of scenes without destabilizing the model is to steer every piece of testable logic into read-only RefCounted projections (the G1/G2/recovery VMs, `RunFlowController`, the delegated route resolver, the layout-invariance VM) and verify the `.tscn`/`Control` wiring with a scene-load compile guardrail + the read-only-projection discipline — never claiming a `SceneTree` test the harness can't run. The semantic layout plan stays the source of truth; the scene honors it.
2. **When a presenter re-implements a sequence the domain already encodes, TEST THE SHARED SEQUENCING SEAM — not just the domain driver.** The 11.3 H1 divergence (on-screen advance-then-resolve silently skipped the guaranteed depth-0 combat opener that the tested `run_to_completion_live` resolve-then-advance driver always plays) was invisible because the only walk test drove the domain driver. The fix centralized the decision on a shared `current_node_needs_board()` query and tested THAT. General lesson: a presentation re-implementation of domain-encoded ordering needs a test at the shared seam both paths consume.
3. **Make every live wiring opt-in and prove the DEFAULT path byte-identical, and determinism survives a live layer.** 11.2's death source + boss auto-play and 11.4's post-generation affinity are all opt-in drivers; the default `run_to_completion` stayed fingerprint-preserving, the generator stayed affinity-blind, and the resolver's new params default to neutral so existing callers are byte-identical. A live playable layer added zero risk to the deterministic base the seed regressions guard.
4. **State that is EPHEMERAL or lives inside an EXISTING container never moves the save gate.** The in-node fight stays ephemeral (23-key gate = 23, no in-node save), and the spend state (unlock flags + the spent ledger) lives INSIDE the existing `unlock_progress` Dictionary home — so a whole meta-spend system shipped with no new top-level `RunSnapshot` key, no schema bump, no migration (the seal-fragments precedent). Reach for an existing container or ephemerality before a new save key.
5. **A profile-aware OVERLAY beats mutating static content.** 11.6 made `ClassDefinition`'s STATIC `lock_state` effectively profile-aware WITHOUT mutating static content: a single `MetaSpendRules.unlocked_class_ids_for` source both the VM affordance and the authoritative gate read, with a null profile yielding byte-identical static behavior. The static roster stays immutable; the unlock is a derived overlay — which is why the pinned VM key sets didn't change (the `selectable` VALUE became profile-aware; no new key).
6. **A structural constraint is not a new gate — don't add a redundant guard for something that's already impossible.** FR28 is structural for the spend: manual-seed runs never award shards, so there is nothing manual-seed-earned to spend — so NO redundant manual-seed eligibility gate was added to the spend command. Recognizing when an invariant is enforced by the shape of the system (not by a check) avoids dead defensive code.
7. **A UX contract that records its gaps as OWNED notes turns into the next stories' task list.** The 11.1 appendix mapped every screen to an existing contract and recorded G1/G2/G3/G4 as explicit owned gaps rather than silently expanding scope — and those four notes became exactly what 11.3 (G1/G2), 11.5 (G3), and the parked settings owner (G4) built. Recording a gap against an owner is more useful than either inventing the surface early or leaving the gap implicit.
8. **The WIP-checkpoint recovery is the right response to a mid-run host-process exit — don't assume the work landed.** When the process died mid-11.3, the discipline held: commit the partial work as a checkpoint, push, write resume_notes, run the suite as a DIAGNOSTIC (not proof of doneness), then delegate a resume-verify continuation. Treating a process exit as "possibly-incomplete, verify" rather than "probably-done" is what kept the run honest.
