# Story 6.2: Small Inventory and Equipment Model

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want a small inventory and equipped items to be tracked clearly,
so that loot choices remain tactical rather than a flood of comparisons.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 6 -> Story 6.2 (verbatim AC, with the implementation-grounded detail this story adds). FRs covered: FR51 (small MVP inventory ‚Äî ~6 backpack items, no stacking by default). Builds DIRECTLY on Story 6.1's loot/reward content (the `armor`/`jewelry`/`consumable`/`pickup`/`gold_reward` definitions + repositories and the reused `weapon`/`support` repos) ‚Äî the inventory/equipment model TRACKS items defined there; it does NOT re-author content.

1. **The MVP backpack supports a configured small capacity (defaulting to 6) and stacking is OFF unless a specific item definition allows it later.**
   **Given** the hero has an MVP backpack
   **When** inventory is initialized
   **Then** it supports the configured small capacity, defaulting to 6 backpack items (a named `DEFAULT_BACKPACK_CAPACITY := 6` constant; capacity is a small bounded positive int, NOT a difficulty/run-depth knob)
   **And** item stacking is disabled unless a specific item definition allows it later (each occupied slot holds exactly ONE item instance/id; there is NO per-slot `quantity > 1` in v0 ‚Äî a "stackable" item flag is a LATER additive opt-in the model must not preclude, but v0 authors NO stackable item and the model defaults `quantity == 1` / one-item-per-slot).

2. **Picking up an item with capacity records it per the domain model and emits an item-gained event.**
   **Given** an item is picked up
   **When** there is capacity
   **Then** the inventory records the item instance or item id according to the domain model (the backpack appends ONE slot entry ‚Äî an item-id reference plus the minimal instance fields the model tracks ‚Äî through a validate-then-mutate command; the slot count increases by exactly one)
   **And** an item-gained event is emitted (a NEW `item_gained` SYSTEM `DomainEvent` appended to the END of the event enum, wired end-to-end: factory normalizer + payload validator + `id_for_type`/`type_for_id` map + a real round-trip test + a malformed-payload negative).

3. **A full backpack returns a stable inventory-full error or a replacement-choice state on another pickup, deleting nothing silently.**
   **Given** the backpack is full
   **When** another backpack item is picked up
   **Then** the command returns a stable inventory-full error or a replacement choice state (a stable lower_snake `inventory_full` `ActionResult.error`, OR a structured replacement-choice result ‚Äî PICK ONE and record it as a `[Decision]`; the recommended v0 is the `inventory_full` error, deferring the replacement-choice UX to a later story)
   **And** no existing item is silently deleted (validate-then-mutate: the rejected pickup mutates NOTHING ‚Äî the backpack is byte-identical, the item-gained event is NOT emitted, and no slot is overwritten/dropped; the "no silent delete" guarantee is the load-bearing correctness property).
