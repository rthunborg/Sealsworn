# Story 7.3: Risk/Reward Event Choices

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want events to offer tempting choices with known risks,
so that greed and safety compete during a run.

## Acceptance Criteria

(Verbatim from `planning-artifacts/epics.md` lines 1913-1934, Story 7.3; FR54 — "Gold, healing, curses/corruption, Oath Shards, passives, and loot form the risk economy". Builds on 7.1 `RiskEconomyState` + 7.2 curse/cursed-reward layer.)

**AC1 — A risk/reward node offers a deterministic event choice from approved definitions, drawn through the `events` stream**
- **Given** a risk/reward node is entered
- **When** event generation runs
- **Then** it offers a deterministic event choice using approved event definitions
- **And** the event uses the named `events` RNG stream.

**AC2 — A chosen risk option records BOTH the reward and the risk through domain events; future systems can query the resulting risk flags**
- **Given** the player chooses a risk option such as gold now for future danger, strong passive for max HP loss, or cheap reforge with corruption
- **When** the event command resolves
- **Then** both the reward and the risk are recorded through domain events
- **And** future systems can query the resulting risk flags.

**AC3 — An already-resolved or invalid event choice fails with a stable error and applies no extra reward or penalty**
- **Given** the event is already resolved or the choice is invalid
- **When** another event command is submitted
- **Then** it fails with a stable error
- **And** no extra reward or penalty is applied.

## Scope boundary (read FIRST — this story is the RISK/REWARD EVENT NODE: definitions + the `events`-stream offer roll + the choose-command that records reward AND risk AND populates `risk_flags`, building on 7.1 state + 7.2 curse layer)

7.1 shipped the live `RiskEconomyState` (`gold` / `healing_charges` / `curse_count` / `corruption` / `oath_shard_eligible` / `risk_flags`) + the structural setters (`apply_gold_delta` / `apply_healing_delta` / `set_curse_count` / `set_corruption` / **`add_risk_flag`** — the latter EMPTY in v0), the `economy_changed` event, and the gold wire-off. 7.2 shipped the curse/corruption RULES + the cursed-reward TRADEOFF view model + `AcceptCursedRewardCommand` + the `curse_applied` event + the cleanse hook. BOTH stories' scope-boundaries AND the deferred-work ledger explicitly assign the **risk/reward EVENT node + the `events`-stream event ROLL + the `risk_flags` PRODUCERS** to **Story 7.3 (this story)**. This is the FIRST Epic-7 story to draw through the named `events` stream. 7.3 is the layer that makes the `event` node type — which has been a pure no-op placeholder since 4.5 — actually OFFER a tempting, readable risk/reward choice. Do NOT redefine the 7.1/7.2 domain — EXTEND it (reuse `add_risk_flag`, `apply_gold_delta`, `set_curse_count`/`set_corruption`, `economy_changed`, `curse_applied`; do NOT add a parallel economy/curse model).

**IN scope (this story):**
1. **A typed, validated `EventDefinition`** (a new content definition, the exact `GoldRewardDefinition` / `CursedRewardDefinition` / `DestroyOutcomeTableDefinition` typed-`Resource` + `validate() -> ActionResult` + lower_snake-id shape) carrying an approved risk/reward event: a stable `event_id`, a `display_name`, a `prompt` (the readable situation the player faces), and a small set of **CHOICES** — each choice a readable option carrying its REWARD side (gold/healing benefit) AND its RISK side (curse/corruption increment, a resource cost, and/or a `risk_flag` id the choice raises) plus a clear `choice_text` (the GDD "tempting choices with known risks" / "the player understands the trade before accepting"). At least one choice must be a genuine tradeoff (a reward AND a risk) so the node is a real decision; a "safe/decline/leave" choice MAY be offered (it raises no risk and grants little/nothing — the GDD "If the player always chooses safety, rewards are too weak"). A small approved baseline set authored as a code constant through a fail-closed repository (NO JSON pipeline — `data/source`/`data/resources` stay EMPTY). The baseline choices must realise the AC2 GDD examples (lines 466-471): "gain a strong passive but lose max HP" (modeled as gold/healing benefit + a recorded penalty + a risk flag — v0 has NO live max-HP to decrement; see the max-HP note below), "accept gold now and increase elite chance later" (gold benefit + an `elite_chance` risk flag — the FLAG is the future-danger record), "reforge for cheap but add corruption" (gold benefit + corruption increment).
2. **A deterministic `events`-stream OFFER roll** (AC1) — when a risk/reward node is entered, GENERATE a deterministic event offer by drawing through the **run-level `RngStreamSet` on the `STREAM_EVENTS` stream** (`&"events"`) via the orchestrator (the EXACT 6.3 `RunOrchestrator.generate_reward_offer` pattern — through `streams.rand_int(RngStreamSet.STREAM_EVENTS, ...)`, NEVER `randi`/`randf`/a new `RandomNumberGenerator`). The offer SELECTS one approved `EventDefinition` (and/or its choice set) deterministically from the run-level streams, stores it as a PENDING event offer on `RunState`, and emits an `event_offered` record. Deterministic + reproducible from the seed/state (same seed + same route position -> same offered event). Resolves the event id through the repository (validated definitions only; fail-closed on a miss — the validate-before-use posture).
3. **An event-CHOOSE command** (AC2/AC3) — a run-domain command (the 4.3 run-command idiom VERBATIM: extends `game_command.gd`, takes `RunState` directly, rejects `sequence_id <= 0` FIRST, validate-then-mutate, byte-identical no-mutation + ZERO events on every reject) that, given the player's chosen choice id, applies BOTH the REWARD and the RISK to the run's `RiskEconomyState` (gold/healing via the 7.1 `apply_*_delta` API; curse/corruption via the 7.1 `set_curse_count`/`set_corruption` setters; **the risk flag(s) via the 7.1 `add_risk_flag` setter — the `risk_flags` PRODUCER this story owns**) and emits domain events recording BOTH sides (AC2). It flips the pending event offer to `resolved` and is **fail-closed no-double-apply** (AC3): a second choose against a `resolved` offer, or an off-offer/invalid choice id, fails with a stable error and applies ZERO extra reward/penalty (the 6.3 `ResolveRewardCommand` no-double-apply pattern VERBATIM). The choose command draws ZERO RNG (the choice amounts are AUTHORED on the definition, not rolled at choose — the offer roll at GENERATE is the ONLY `events` draw).
4. **`risk_flags` POPULATION** (AC2 "future systems can query the resulting risk flags") — a chosen risk option raises its declared `risk_flag` id(s) via `RiskEconomyState.add_risk_flag(...)` so `has_risk_flag(...)` returns true afterward and the flag rides the save (the 7.1 nested-`route_state` economy already persists `risk_flags`). This is the explicit deferred-work item 7.1 and 7.2 both routed to 7.3.
5. **New SYSTEM event(s)** wired end-to-end, append-only (after `CURSE_APPLIED`) — at minimum an `event_resolved` (or `event_choice_resolved`) deterministic event recording the chosen event + choice + the risk flag(s) raised (AC2's "the risk are recorded"), reusing the 7.1 `economy_changed` (for the gold/healing reward side) and the 7.2 `curse_applied` (for any curse/corruption risk side) at DISTINCT sequence ids. An `event_offered` record at GENERATE (mirroring `reward_offered`) is also appropriate. Decide the exact event-shape split as a `[Decision]` (see Task 4).
6. Full unit tests: definition valid/invalid; repository fail-closed + duplicate-id; the GENERATE offer (deterministic for a seed, draws through the `events` stream, named-stream isolation — only `events` advances); the choose command valid + every invalid-no-mutation branch + both-sides-applied + the risk-flag raised + determinism + ZERO RNG; the new event(s) factory/validator/round-trip/malformed + the `expected_ids` enum-exhaustiveness pin; the `risk_flags` save round-trip (extends the 7.1 nested-`route_state` coverage); the no-double-apply (AC3).
