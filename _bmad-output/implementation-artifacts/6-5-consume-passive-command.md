# Story 6.5: Consume Passive Command

## Story

As a player,
I want to Consume a passive for power and build identity,
so that awakened memories can change how I solve fights.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 6 -> Story 6.5 (verbatim AC, with the implementation-grounded detail this story adds). FRs covered: **FR48** (Consume must provide power and build identity — THIS is the story that ADOPTS the passive into the run), **FR82** (the Consume/Destroy resolution commands — epics.md line 397 lists the 70/20/10 Destroy distribution under 6.6; 6.5 is the Consume half), and it consumes the FR47 commit-INTENT (`{choice:"consume", passive_content_id, table_id}`) that Story 6.4's `PassiveRewardCommitFlow.confirm()` produces. Builds DIRECTLY on: Story 5.4's `PassiveDefinition`/`PassiveRepository`/`RulesResolver`/`RuleTrigger` (the explanation-only passive system + the stable-order resolver this story ADDS the consumed passive into); Story 6.3's `RewardOffer` value object + the `passive` offered-entry shape + the `ResolveRewardCommand` run-command idiom + the `reward_resolved` SYSTEM event (the offer-resolution surface 6.5 mirrors); Story 6.4's `PassiveRewardCommitFlow` commit-intent (the data structure a 6.5 caller hands in). This is a **COMMAND + EVENT + RUN-STATE-MUTATION story** (UNLIKE 6.4's data-contract-only story): it builds the real `ConsumePassiveCommand`, emits a new `passive_consumed` SYSTEM event, and ADOPTS the passive into the run's `RulesResolver`. It does NOT build `DestroyPassiveCommand` / the 70/20/10 Destroy outcome distribution (Story 6.6), the per-effect passive OPERATION engine (a later Epic-6 operations story — `scripts/rules/{conditions,operations}` stay EMPTY; v0 passives are EXPLANATION-ONLY), or any `.tscn` modal scene (UI-scene-last).

1. **An unresolved passive offer + a Consume choice -> `ConsumePassiveCommand` validates the offer, adds the passive to active run state, and emits a deterministic passive-consumed event.**
   **Given** a passive reward offer is unresolved (`RunState.pending_reward_offer != null` AND `is_pending()`, carrying a `{category: "passive", content_id: <passive_id>}` offered entry)
   **When** the player chooses Consume (a caller hands the `{passive_content_id, table_id}` from the FR47 commit-intent to `ConsumePassiveCommand.new(passive_content_id, table_id, sequence_id)`)
   **Then** `ConsumePassiveCommand` VALIDATES the offer (validate-then-mutate, the run-command idiom: reject `sequence_id <= 0` FIRST with `invalid_event_sequence_id`; reject a non-`RunState`/structurally-invalid run with `invalid_context`; reject no pending offer with `no_pending_reward_offer`; reject an already-`resolved` offer with `reward_offer_already_resolved`; reject a selected passive id that is NOT a `passive`-category offered entry on the pending offer with `invalid_reward_selection`; reject a passive id that does not resolve through `PassiveRepository.get_passive(id)` with `unknown_passive`) and **ADDS the passive to active run state** (the v0 "adopt" surface: REGISTER the resolved `PassiveDefinition` into the run's `RulesResolver` via `register_passive(def)` — creating a fresh `RulesResolver` and seating it on `run.rules_resolver` if the run has none yet, exactly as `RunStartCommand` seats the starting resolver — so the consumed passive is now resolvable through its declared trigger windows alongside the starting passives; flip the offer to `resolved` + record the selected entry, the `ResolveRewardCommand` offer-resolution posture)
   **And** a DETERMINISTIC passive-consumed event is emitted (a NEW `passive_consumed` SYSTEM `DomainEvent` appended at the enum end, carrying the lower_snake `passive_id` + `table_id` — emitted ONLY AFTER the registration + offer flip; the command draws ZERO RNG — Consume is deterministic, there is no roll here, the Destroy-outcome RNG is Story 6.6).

2. **The consumed passive's rules resolve through the kernel's stable ordering with player/debug-readable explanations.**
   **Given** the consumed passive has rules (it declares one or more trigger windows from the fixed `RuleTrigger` vocabulary)
   **When** the relevant trigger window occurs (the run's `RulesResolver.resolve(window_id)` / `explain(window_id)` is queried — the EXISTING explanation-only resolution surface)
   **Then** the rules kernel resolves the passive through STABLE ordering (the consumed passive appears in `resolve(window_id)` in REGISTRATION order — registered AFTER the starting passives the resolver already holds, so the deterministic resolution order is starting-passives-then-consumed; `RulesResolver` returns the matching passives in registration order, the EXISTING Story-5.4 stable-order contract)
   **And** generated outcomes include player/debug-readable explanations (`RulesResolver.explain(window_id)` surfaces the consumed passive's `explanation` line in the same stable order — the architecture's Readability Rule surface. **SCOPE CLARIFICATION (READ): v0 passives are EXPLANATION-ONLY rule-benders** — "resolves through the kernel" means the consumed passive is REGISTERED and SURFACES via `resolve`/`explain`, NOT that it mutates an HP/movement/damage number. The per-effect `RuleCondition`/`RuleTarget`/`RuleOperation` engine + the combat HOOK sites that would make a consumed passive CHANGE a combat number are the later Epic-6 operations story — `scripts/rules/{conditions,operations}` stay EMPTY. This AC is satisfied by the consumed passive being resolvable + explainable in stable order, exactly as the starting passives are, NOT by a felt combat-number change.).

3. **Passive trigger-order fixtures verify trigger timing, resolver order, stacking, conflict handling, and explanation output across starting/item/affinity/consumed-passive rules; new passive content cannot bypass those fixtures.**
   **Given** passive trigger-order fixtures exist
   **When** multiple starting, item, affinity, or consumed-passive rules can trigger in the same window
   **Then** tests verify trigger timing, resolver order, stacking, conflict handling, and explanation output (a dedicated trigger-order fixture suite that registers a STARTING passive AND a CONSUMED passive into the SAME resolver in the SAME trigger window and asserts: both resolve in REGISTRATION order — starting first, consumed second — for that window; `explain(window)` surfaces BOTH explanations in the same stable order; two consumed passives in the same window resolve in their consume order (stacking = both present, stable order); the resolution is byte-identical for the same registration sequence (determinism). **SCOPE: item/affinity rules + the conflict-RESOLUTION engine (priority/duration/operation) do NOT exist in v0** — item-effect rules are a later Epic-6/7 story, affinity rules are Epic 7, and the per-effect operation/conflict engine is the later Epic-6 operations story. This AC is satisfied at the v0 EXPLANATION-ONLY level: the fixtures prove trigger-window timing (a passive resolves ONLY in its declared windows), resolver ORDER (stable registration order across starting + consumed passives), STACKING (multiple passives in one window all surface), and EXPLANATION output (each surfaces its line) — for the rule types that exist (starting + consumed passives). Record the item/affinity/conflict-resolution-engine scope as a forward residual; do NOT build them.)
   **And** new passive content cannot bypass those fixtures (the trigger-order fixtures exercise the `RulesResolver` + `ConsumePassiveCommand` registration path that ALL consumed passive content flows through — a new passive is registered + resolved through the SAME seam the fixtures pin, so it cannot bypass the stable-order/explanation contract; the validated-only `PassiveRepository` load is the content gate — a passive that fails `validate()` is never registered and so can never be consumed).

4. **An already-consumed/destroyed/not-offered passive Consume fails with a stable error and mutates nothing.**
   **Given** the passive was already consumed, destroyed, or is not in the current offer
   **When** Consume is submitted
   **Then** the command FAILS with a stable error (validate-then-mutate, ZERO events: a Consume against an offer that is already `resolved` — the offer was consumed/destroyed/resolved — fails `reward_offer_already_resolved`; a Consume of a passive id that is not a `passive`-category entry on the current pending offer fails `invalid_reward_selection`; a Consume with no pending offer fails `no_pending_reward_offer`; a Consume of an offered passive id that does not resolve through `PassiveRepository` fails `unknown_passive`)
   **And** passive state, reward state, and RNG state remain unchanged (the load-bearing no-double-consume guarantee: a re-submitted Consume against a `resolved` offer leaves `run.rules_resolver` byte-identical — no second registration — the offer byte-identical, draws NO RNG / advances NO stream, applies NO second passive, and the whole `RunState` is byte-identical — exactly the AC3 "no-mutation on reject" posture `ResolveRewardCommand`/`PickupItemCommand` already prove).
