## ŌŁÉ AC3: start-another-descent creates a NEW run request; prior completed run state is NOT reused

AC3 requires the outpost's "start run" to create a NEW run request with the selected class + seed eligibility, and NEVER reuse the prior completed/failed run as active state. Make this STRUCTURAL via the existing start seam:
- The outpost's start action produces a REQUEST (Decision A above) validated against `HeroSelectViewModel.is_class_selectable(class_id)` (the fail-closed pre-gate ŌĆö a locked/unknown class cannot start; the AUTHORITATIVE gate is still `RunStartCommand`'s class validation, so even a mis-enabled request cannot start a locked run). The caller hands the request to a FRESH `RunOrchestrator.start(root_seed, is_manual_seed, class_id)`.
- **Prior run NOT reused is STRUCTURAL:** `RunOrchestrator.start(...)` ŌåÆ `RunStartCommand.execute(null)` builds a FRESH `RunState` via `RunState.new_run(root_seed, is_manual_seed, route)` which RESETS `current_node_id` + `cleared_node_ids` (the documented 4.3 fixture trap ŌĆö new_run resets fresh-run state). The prior terminal run is a SEPARATE `RunState` object; the outpost view model holds it (for the SUMMARY) but never threads it into the start path. A new run is a pure function of a NEW `(root_seed, is_manual_seed)` ŌĆö so AC3's "prior completed run state is not reused as active state" holds by construction. A test asserts: building a start request from an outpost holding a terminal prior run, then starting via a fresh orchestrator, produces a NEW run in `PHASE_ACTIVE_ROUTE` with empty `cleared_node_ids` (not the prior run's cleared set), and the prior terminal run is left byte-identical (untouched).
- **Seed eligibility settings (AC3):** the `is_manual_seed` flag carried into the request ŌåÆ `start(...)` ŌåÆ a manual-seed run reports `meta_progression_eligible == false` (the existing lockstep; do NOT re-implement eligibility). A normal-seed start is eligible. 8.6 threads the flag; it does NOT change the FR28 eligibility model.
- The view model itself does NOT call `start(...)` (Decision A) ŌĆö it produces the request; the caller starts the run. This keeps "do not let UI mutate domain state directly" intact (AC1 source-truth).

## ŌŁÉ AC4: missing/incompatible outpost data uses a structured recovery / fresh-profile path; no crash, no invalid meta state

AC4 requires the outpost to handle missing/incompatible outpost data via a structured recovery or fresh-profile path, without crashing or creating invalid meta state. This is ALREADY the documented "8.6 fresh-profile path":
- **Missing profile (first launch / no `user://profile.json`):** `ProfileRepository.read_profile()` returns `profile_not_found`; the caller starts `ProfileSnapshot.fresh()` (a brand-new 0-Oath-Shard profile ŌĆö the deferred-work-anchored "8.6 fresh-profile path"). The outpost view model built from a fresh profile is a VALID surface (0 Oath Shards, empty Echoes/unlock progress, first-death not recorded, the full class roster). No crash, no invalid state.
- **Incompatible profile (schema mismatch):** `ProfileRepository.read_profile()` ŌåÆ `ProfileSnapshot.parse(...)` returns `unsupported_profile_schema` (the migration reject). The outpost surfaces this as a STRUCTURED recovery state (a flag + the structured code), NOT a crash ŌĆö a later HUD story renders a "profile incompatible / recover" affordance. 8.6 does NOT build the comprehensive migration matrix (8.7 owns it ŌĆö see the scope fence); it consumes the EXISTING structured `unsupported_profile_schema` result and represents it as a recovery state in the view model.
- **Failed profile WRITE (AC5 of 8.3, relevant here):** `ProfileRepository.write_profile()` returns a structured profile-scoped error (`profile_save_open_failed` / `..._backup_failed` / `..._replace_failed`) ŌĆö the outpost surfaces the failure + a retry affordance, and NEVER loses the current run summary (the summary is a DERIVED read that does not read the profile file ŌĆö 8.3's structural guarantee). 8.6 renders the recovery; it does NOT change the repository.
- **Model the recovery STRUCTURALLY:** the `OutpostViewModel` construction path takes the profile (or a recovery signal) and projects a valid surface for BOTH a real profile AND a fresh/recovery profile (a `has_profile` / `recovery_state` flag distinguishes them). A null/absent profile projects the fresh-profile default (the `RunSummary._empty()` fail-closed discipline). A test asserts: a `profile_not_found` ŌåÆ fresh-profile outpost builds without crashing (valid 0-shard surface); an `unsupported_profile_schema` read surfaces a structured recovery state (no crash, no invalid meta state); the fresh profile has 0 Oath Shards / empty homes (no unintended progress granted ŌĆö the AC4 "does not create invalid meta state").
