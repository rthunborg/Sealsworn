# Story 8.6: Outpost Menu and Start Another Descent

Status: review

## Story

As a player,
I want the outpost to show summary, progress, and start options,
so that I can quickly begin another run after death or victory.

The SIXTH story of Epic 8 — "Outpost, Meta Progression, and Run Summary". Epic 8 splits the return loop into: **8.1** run-END boundary (`run_failed`/`run_completed` + terminal phase + `next_destination: outpost` + the `RunEndOutcome` flow signal, DONE) → **8.2** the DERIVED `RunSummary` read surface (DONE) → **8.3** the META PROFILE + Oath-Shard AWARD, the FIRST persistent cross-run state + `ProfileRepository` + the AC5 structured save-failure recovery (DONE) → **8.4** Echoes / Seal Fragments / class-mastery / unlock-progress DOMAIN content + profile-MERGE + `UnlockProgressRules` thresholds (DONE) → **8.5** the first-death narrative FLAG + the scene-free `FirstDeathNarrativeBeat` DTO + the `first_death_recorded` event (DONE) → **8.6 (THIS STORY)** the OUTPOST view-model that AGGREGATES all five prior read surfaces + the named-space metadata + the start-another-descent request + the structured recovery/fresh-profile path → **8.7** comprehensive save-load/migration matrix. 8.6 is the "assemble the outpost surface and start the next run" story — it is a **view-model / read-projection assembly + a start-run seam wiring**, NOT new domain truth.

**AC1 — the outpost view-model exposes summary, Oath Shards, Echoes, unlock progress, class options, and start-run actions; domain/profile snapshots stay source truth**
- **Given** the player returns to the outpost
- **When** the outpost view model is built
- **Then** it exposes run summary, current Oath Shards, discovered Echoes, unlock progress, class options, and start-run actions
- **And** domain/profile snapshots remain source truth.

**AC2 — named-space metadata with stable ids or explicit deferred placeholders (Memory Archive, Hall of Oaths, Seal Table, Gate/Descent Stair); names support nav/unlock/layout without making UI authoritative**
- **Given** the outpost is represented in UI/view-model data
- **When** named space metadata is built
- **Then** it includes stable ids or explicit deferred placeholders for Memory Archive, Hall of Oaths, Seal Table, and Gate/Descent Stair
- **And** those names support navigation, unlock affordances, or future layout notes without making UI state authoritative.

**AC3 — start-another-descent creates a NEW run request with selected class + seed eligibility settings; prior completed run state is NOT reused as active state**
- **Given** the player starts another descent
- **When** start-run is selected
- **Then** a new run request is created with selected class and seed eligibility settings
- **And** prior completed run state is not reused as active state.

**AC4 — missing/incompatible outpost data uses a structured recovery or fresh-profile path; no crash, no invalid meta state**
- **Given** outpost data is missing or incompatible
- **When** the outpost loads
- **Then** it uses a structured recovery or fresh-profile path
- **And** it does not crash or create invalid meta state.

## Scope boundary (read FIRST)

This story ships the **OUTPOST VIEW-MODEL DATA half** of the return loop: a scene-free, pure-read PROJECTION/ASSEMBLY (`OutpostViewModel`, a `RefCounted` DTO with an exact pinned key set) that (1) AGGREGATES the five prior Epic-8 read surfaces — `RunSummary` (8.2/8.4), `ProfileSnapshot` (Oath Shards / Echoes / unlock progress / first-death flag — 8.3/8.4/8.5), the class-options roster (`HeroSelectViewModel`, 5.2), the `FirstDeathNarrativeBeat` (8.5), and the `RunEndOutcome` `next_destination` flow signal (8.1) — into ONE serializable outpost surface a later outpost `.tscn` (a future HUD story) renders (AC1); (2) exposes NAMED-SPACE metadata (stable ids / explicit deferred placeholders for Memory Archive, Hall of Oaths, Seal Table, Gate/Descent Stair) that supports nav/unlock/layout WITHOUT making UI state authoritative (AC2); (3) exposes a START-ANOTHER-DESCENT request seam — the outpost's "start run" action creates a NEW run via the EXISTING `RunOrchestrator.start(root_seed, is_manual_seed, class_id)` / `RunStartCommand`, with the selected class + seed-eligibility settings, and NEVER reuses the prior completed/failed `RunState` as active state (AC3); (4) the structured RECOVERY / FRESH-PROFILE path for missing/incompatible outpost data (`ProfileRepository.read_profile` → `profile_not_found` → `ProfileSnapshot.fresh()`; `unsupported_profile_schema` → a surfaced structured result, never a crash or invalid meta state) (AC4).

## ⭐ AC1: the outpost view-model AGGREGATES the five prior read surfaces; domain/profile snapshots stay source truth

AC1 requires the outpost view model to expose run summary + current Oath Shards + discovered Echoes + unlock progress + class options + start-run actions, with domain/profile snapshots as source truth. Make this an ASSEMBLY of existing surfaces:
- **Oath Shards / Echoes / unlock progress / class mastery / first-death flag** come from the `ProfileSnapshot` (source truth — read from the profile the outpost loaded via `ProfileRepository.read_profile`, NOT from the run summary). The AWARDED Oath-Shard total is `profile.oath_shards`; discovered Echoes are `profile.echoes`; unlock progress is `profile.unlock_progress` (including the seal-fragment set + the threshold state flags + class mastery). ⭐ `RunSummary.profile_meta.oath_shards_earned` STAYS 0 / not-yet-supported (the summary is a run-scoped read that reads NO profile) — the outpost's Oath-Shard display reads the PROFILE (see the deferred-work fold: "The Oath-Shard EARNED-count summary wiring" stays deferred).
- **Run summary** comes from the (optional) `RunSummary` the run that JUST ended produced (or an empty projection when the player opens the outpost with no just-ended run — e.g. a fresh session). The outpost RENDERS `RunSummary.to_dictionary()` DIRECTLY: `notable_loot` is already single-sourced/deduped from `item_gained` (render it verbatim — no second dedup); `not_yet_supported` is the honest-limitation note the outpost surfaces (AC1/AC5 of 8.2 — the outpost renders an honest limitation note from it).
- **Class options** come from `HeroSelectViewModel.classes()` (the selectable/locked roster + unlock hints) — DELEGATE to it (compose a `HeroSelectViewModel`), do NOT re-project the roster (no duplicate class-projection logic).
- **Start-run actions** are the AC3 seam (the selectable class ids + the request producer) — see AC3 below.
- **Source-truth guarantee:** the view model is a PURE READ — it NEVER writes the profile, NEVER mutates a run, NEVER awards/merges/records anything. The profile + the run + the events are the source truth; the view model projects them. A test asserts building/reading the outpost view model leaves the profile + any run byte-identical (the `HeroSelectViewModel` / `RunSummary` mutation-free discipline).

## ⭐ AC3: start-another-descent creates a NEW run request; prior completed run state is NOT reused

AC3 requires the outpost's "start run" to create a NEW run request with the selected class + seed eligibility, and NEVER reuse the prior completed/failed run as active state. Make this STRUCTURAL via the existing start seam:
- The outpost's start action produces a REQUEST (Decision A above) validated against `HeroSelectViewModel.is_class_selectable(class_id)` (the fail-closed pre-gate — a locked/unknown class cannot start; the AUTHORITATIVE gate is still `RunStartCommand`'s class validation, so even a mis-enabled request cannot start a locked run). The caller hands the request to a FRESH `RunOrchestrator.start(root_seed, is_manual_seed, class_id)`.
- **Prior run NOT reused is STRUCTURAL:** `RunOrchestrator.start(...)` → `RunStartCommand.execute(null)` builds a FRESH `RunState` via `RunState.new_run(root_seed, is_manual_seed, route)` which RESETS `current_node_id` + `cleared_node_ids` (the documented 4.3 fixture trap — new_run resets fresh-run state). The prior terminal run is a SEPARATE `RunState` object; the outpost view model holds it (for the SUMMARY) but never threads it into the start path. A new run is a pure function of a NEW `(root_seed, is_manual_seed)` — so AC3's "prior completed run state is not reused as active state" holds by construction. A test asserts: building a start request from an outpost holding a terminal prior run, then starting via a fresh orchestrator, produces a NEW run in `PHASE_ACTIVE_ROUTE` with empty `cleared_node_ids` (not the prior run's cleared set), and the prior terminal run is left byte-identical (untouched).
- **Seed eligibility settings (AC3):** the `is_manual_seed` flag carried into the request → `start(...)` → a manual-seed run reports `meta_progression_eligible == false` (the existing lockstep; do NOT re-implement eligibility). A normal-seed start is eligible. 8.6 threads the flag; it does NOT change the FR28 eligibility model.
- The view model itself does NOT call `start(...)` (Decision A) — it produces the request; the caller starts the run. This keeps "do not let UI mutate domain state directly" intact (AC1 source-truth).

## ⭐ AC4: missing/incompatible outpost data uses a structured recovery / fresh-profile path; no crash, no invalid meta state

AC4 requires the outpost to handle missing/incompatible outpost data via a structured recovery or fresh-profile path, without crashing or creating invalid meta state. This is ALREADY the documented "8.6 fresh-profile path":
- **Missing profile (first launch / no `user://profile.json`):** `ProfileRepository.read_profile()` returns `profile_not_found`; the caller starts `ProfileSnapshot.fresh()` (a brand-new 0-Oath-Shard profile — the deferred-work-anchored "8.6 fresh-profile path"). The outpost view model built from a fresh profile is a VALID surface (0 Oath Shards, empty Echoes/unlock progress, first-death not recorded, the full class roster). No crash, no invalid state.
- **Incompatible profile (schema mismatch):** `ProfileRepository.read_profile()` → `ProfileSnapshot.parse(...)` returns `unsupported_profile_schema` (the migration reject). The outpost surfaces this as a STRUCTURED recovery state (a flag + the structured code), NOT a crash — a later HUD story renders a "profile incompatible / recover" affordance. 8.6 does NOT build the comprehensive migration matrix (8.7 owns it — see the scope fence); it consumes the EXISTING structured `unsupported_profile_schema` result and represents it as a recovery state in the view model.
- **Failed profile WRITE (AC5 of 8.3, relevant here):** `ProfileRepository.write_profile()` returns a structured profile-scoped error (`profile_save_open_failed` / `..._backup_failed` / `..._replace_failed`) — the outpost surfaces the failure + a retry affordance, and NEVER loses the current run summary (the summary is a DERIVED read that does not read the profile file — 8.3's structural guarantee). 8.6 renders the recovery; it does NOT change the repository.
- **Model the recovery STRUCTURALLY:** the `OutpostViewModel` construction path takes the profile (or a recovery signal) and projects a valid surface for BOTH a real profile AND a fresh/recovery profile (a `has_profile` / `recovery_state` flag distinguishes them). A null/absent profile projects the fresh-profile default (the `RunSummary._empty()` fail-closed discipline). A test asserts: a `profile_not_found` → fresh-profile outpost builds without crashing (valid 0-shard surface); an `unsupported_profile_schema` read surfaces a structured recovery state (no crash, no invalid meta state); the fresh profile has 0 Oath Shards / empty homes (no unintended progress granted — the AC4 "does not create invalid meta state").
