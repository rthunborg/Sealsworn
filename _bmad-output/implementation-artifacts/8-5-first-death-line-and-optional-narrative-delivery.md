# Story 8.5: First-Death Line and Optional Narrative Delivery

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want short narrative beats to support the loop without blocking play,
so that story flavor remains optional and skippable.

## Acceptance Criteria

(Verbatim from `planning-artifacts/epics.md` lines 2132-2153, Story 8.5; FR61 — "First death must be able to show the line 'Good. You remembered how to die.'", FR64 — "keep story discovery optional and must not require lore reading to understand the gameplay loop", FR65 — "Cutscenes and control-loss narrative moments must be skippable". The FIFTH story of Epic 8 — "Outpost, Meta Progression, and Run Summary". Epic 8 splits the return loop into: **8.1** run-END boundary (`run_failed`/`run_completed` + terminal phase + `next_destination: outpost`, DONE) → **8.2** the DERIVED `RunSummary` read surface (DONE) → **8.3** the META PROFILE + Oath-Shard AWARD, the FIRST persistent cross-run state (DONE) → **8.4** Echoes / Seal Fragments / class-mastery / unlock-progress DOMAIN content + profile-MERGE (DONE) → **8.5 (THIS STORY)** the FIRST-DEATH narrative flag + the optional/skippable narrative-delivery DATA contract → 8.6 outpost menu scene → 8.7 comprehensive save-load/migration matrix. 8.5 is the "narrative flavor without blocking play" half. 8.3 explicitly RESERVED an empty `first_death_recorded: bool` home in `ProfileSnapshot` for exactly this story, so **8.5 records the flag by merging into the existing shape WITHOUT a schema bump / migration**.)

**AC1 — First death can show the line, and it is tracked so it is not repeated as a first-death event**
- **Given** the player dies for the first time
- **When** the outpost return flow displays narrative text
- **Then** it can show "Good. You remembered how to die."
- **And** the line is tracked so it is not repeated as a first-death event.

**AC2 — A narrative / control-loss moment is skippable, and skipping changes nothing**
- **Given** a narrative or control-loss moment appears
- **When** the player skips or dismisses it
- **Then** gameplay flow continues to the outpost or next menu
- **And** skipping does not change rewards, unlocks, or tactical state.

**AC3 — Lore is optional; ignoring it never blocks understanding or starting another descent**
- **Given** a player ignores lore
- **When** they view run summary and outpost options
- **Then** required gameplay choices remain understandable
- **And** lore reading is not required to start another descent.

## Scope boundary (read FIRST)

This story ships the **DOMAIN + DATA half** of the first-death narrative beat: (1) a DETERMINISTIC, IDEMPOTENT first-death FLAG on the persistent `ProfileSnapshot` (`first_death_recorded`, the empty home 8.3 reserved), set exactly once behind the run-end seam so the first-death LINE is shown ONCE and never repeated (AC1); (2) a scene-free, skippable NARRATIVE-BEAT DATA CONTRACT (a `RefCounted` value object / read projection, exact pinned key set) that surfaces the first-death line + a skippability marker so a later outpost UI (8.6) can PRESENT and DISMISS it, and so a skip is a pure no-op that grants/mutates nothing (AC2); (3) the structural guarantee that this narrative surface is OPTIONAL and OFF the critical path — the run summary + start-another-descent path (8.2 `RunSummary`; 8.6 outpost) do NOT depend on it, so ignoring lore never blocks understanding or a new descent (AC3).

## ⭐ AC1: the DEATH-ONLY GATE + the ONCE-ONLY idempotency (the two invariants that keep the line un-repeated)

Two independent guards make "the line is tracked so it is not repeated as a first-death event" (AC1) true:
- **DEATH-ONLY GATE** — the first-death flag is set ONLY when the terminal run is a DEATH (`PHASE_FAILED`). A COMPLETED run (a victory) is NOT a first death — reject a completed run with a stable code (e.g. `run_not_failed`) + ZERO mutation. Read the terminal phase off the `RunState` (`run.phase == RunState.PHASE_FAILED`, mirroring how `RunSummary._derive_outcome_or_cause` branches on the terminal phase). This is the discriminator between the 8.5 first-death beat and the Epic-9 first-VICTORY reveal — they key off opposite terminal phases.
- **ONCE-ONLY IDEMPOTENCY** — the `first_death_recorded` bool IS its own idempotency marker (unlike the award/merge, which need a run-seed marker because they can re-award/re-merge different amounts; the first-death flag is a monotonic latch). On a first death (`profile.first_death_recorded == false`) SET it true + emit the event. On EVERY subsequent death (`profile.first_death_recorded == true`) reject with a stable `first_death_already_recorded` code + ZERO mutation + ZERO event — so a second, third, Nth death NEVER re-emits the first-death event and NEVER re-shows the line. This is a THIRD independent run-end idempotency mechanism alongside the award's `last_awarded_run_seed` and the merge's `unlock_progress["_last_merged_run_seed"]` (the 8.4 two-marker invariant) — 8.5 does NOT reuse either seed marker (the first-death latch is per-PROFILE-lifetime, not per-run: it fires on the FIRST death across ALL runs, then never again, whereas the award/merge fire once PER eligible run). Keep the three mechanisms independent; 8.6/8.7 treat all three as separate.

**ELIGIBILITY — a first-death beat is NARRATIVE, not meta progression (resolve at implementation).** The award (8.3) + the merge (8.4) DENY a manual-seed run at their eligibility gate (FR28 — a manual-seed run grants NO meta progression). The first-death FLAG is arguably narrative flavor, not "meta progression" in the FR28 sense — but it IS persisted cross-run profile state. `[Decision]` — pick one and record it with rationale:
- **(A, RECOMMENDED) the first-death flag is NOT gated on `meta_progression_eligible`** — it fires on the first real death whether or not the run was manual-seed, because the line is pure narrative flavor (FR61/FR64) and setting a boolean latch is not "granting meta progression" (no Oath Shards / unlocks / mastery). This keeps the narrative beat available in a manual-seed practice death too (a player practicing their first-ever death still "remembers how to die"). Document that the first-death latch is deliberately eligibility-INDEPENDENT and explain WHY it does not violate FR28 (it is a narrative marker, not progression currency/unlocks).
- **(B) gate it on eligibility** (mirror the award/merge Gate) if a reviewer argues the profile flag is meta state that a manual-seed run must not touch. This is defensible but means a manual-seed first death shows no line — weigh against FR64 ("story discovery optional" cuts the other way: the line is flavor, not a reward). If you pick (B), reuse the EXACT gate shape from `AwardMetaProgressCommand` (reject `run_not_meta_eligible`).
Whichever: surface the decision + rationale in the Completion Notes as a cross-story `[Decision]` (8.6/8.7 need to know whether the first-death latch is eligibility-gated when they wire the outpost + the save-load matrix).

## ⭐ AC2: SKIP is a pure no-op (the structural guarantee)

AC2 requires skipping/dismissing the narrative moment to continue flow AND change NOTHING (rewards, unlocks, tactical state). Make this STRUCTURAL, not behavioral:
- The narrative BEAT DTO (`FirstDeathNarrativeBeat` or your chosen name) is a PURE READ value object — building it, reading it, and "dismissing" it draw ZERO RNG, run NO command, emit NO event, and mutate NOTHING. A skip/dismiss is simply the presentation layer NOT rendering it further; there is no "skip command" that could mutate state. The DTO owns no truth to change.
- The first-death FLAG is set by the RUN-END command (when the death resolves), INDEPENDENT of whether the beat is ever shown or skipped — so the flag records the first-death FACT (for the once-only guard) whether or not the player reads the line. Showing/skipping the line is a separate presentation act with no domain effect. (This is why the flag mutation lives in the command and the line delivery lives in the read DTO — separation keeps skip a no-op.)
- v0 has NO reward/unlock/tactical mutation reachable from a narrative dismiss (there is no narrative-choice command). The "skipping does not change rewards/unlocks/tactical state" clause is satisfied STRUCTURALLY because the narrative surface is read-only and the only mutation (the flag) is not triggered by the skip. A test asserts: building + reading the beat, and a simulated dismiss, leave the profile + run + any economy byte-identical.

## ⭐ AC3: narrative is OPTIONAL and OFF the critical path (the load-bearing non-dependency)

AC3 requires that ignoring lore never blocks understanding the run summary / outpost options / starting another descent. Make this STRUCTURAL:
- The `RunSummary` (8.2) is COMPLETE without the narrative beat — it already aggregates cause-of-death/victory, nodes, loot, passives, economy, discoveries, eligibility. 8.5 does NOT add a narrative field to `RunSummary` and does NOT make the summary depend on the beat (keep `RunSummary.DICTIONARY_KEYS` UNCHANGED — no new top-level key; the 8.2/8.4 exact-key pins stay green). The summary is understandable with the narrative surface entirely absent.
- The start-another-descent path (8.6 outpost / a new `RunStartCommand`) does NOT read the first-death flag or the beat as a precondition — a player can start a new run whether or not the line was shown or skipped. 8.5 adds NO gate to the run-start path.
- The narrative beat is a SEPARATE, OPTIONAL surface (its own DTO) that a later outpost UI (8.6) MAY render alongside the summary — never a blocker. This mirrors the retro-notes/deferred-work discipline: "Narrative stays off the summary CRITICAL path (retro §7 risk 2)" (deferred-work.md, 8.1 + 8.2 first-death defers). Do NOT fold the narrative into the summary or the run-end resolution's required path.

### THE TWO-INDEPENDENT-MARKERS INVARIANT (from 8.4) + the THIRD 8.5 mechanism — the cross-story constraint

### RUN-END META STEP ORDERING (heads-up for the first-death command coexisting with award + merge)

8.3's award, 8.4's merge, and 8.5's first-death command are THREE run-domain commands that may all mutate the profile at run end behind the SAME 8.1 idempotency seam. They are independent caller-driven commands (no auto-wire). Because each uses an INDEPENDENT idempotency mechanism (see above), the CALLER may order them in any sequence and each tolerates the others having already run. The first-death command is additionally gated to FAILED runs only (the award/merge are gated on eligibility + terminal, and the award awards 0 for a failed run — a failed run still merges discoveries and now also records first-death). So on a first DEATH the run-end sequence may run: award (0 for a death) → merge (discoveries) → first-death (set flag) in any order, each idempotent. Test that the first-death command succeeds even when the award marker + merge marker already match the run (they gate DIFFERENT things). This coupling is the reason the markers are independent — surface the exact chosen semantics in the Completion Notes.
