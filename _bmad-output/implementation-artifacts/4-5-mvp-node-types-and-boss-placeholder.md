# Story 4.5: MVP Node Types and Boss Placeholder

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want the run map to include MVP node types even before every system is fully deep,
so that the route shell reflects the intended roguelite structure.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 4 -> Story 4.5 (verbatim AC, with the implementation-grounded detail this story adds). Readiness traceability: the 2026-06-04 readiness patch lists Stories 4.5/7.5/8.2/10.7 as the "placeholder replacement/de-scope checkpoints" (epics line 401) -> AC4 here OWNS recording the placeholder ids + affected node types for the readiness/de-scope checkpoint.

1. **Route generation can assign every MVP node type, and unsupported implementations resolve safely (no broken gameplay).**
   **Given** route generation assigns node types
   **When** an MVP route is created
   **Then** it can include `combat`, `elite_combat`, `shop`, `reforge`, `gambling`, `event` (risk/reward), `secret` (secret/lore), and `boss` placeholder nodes
   **And** unsupported node implementations route to **safe placeholder resolution** rather than broken gameplay (entering a non-combat node no longer fails with `unsupported_node_entry` and never soft-locks the run).

2. **A placeholder non-combat node, when entered, communicates placeholder completion in domain/debug terms and lets the player continue forward.**
   **Given** a placeholder non-combat node is entered (the run is in `PHASE_ACTIVE_ROUTE`, parked on a `shop`/`reforge`/`gambling`/`event`/`secret` node)
   **When** it resolves
   **Then** it communicates placeholder completion in domain/debug terms (a deterministic `node_placeholder_resolved` `DomainEvent` carrying the node id + type + a stable `placeholder` resolution marker, plus a result-metadata flag) and transitions `PHASE_ACTIVE_ROUTE -> PHASE_NODE_RESOLUTION` so the existing `NodeExitCommand` can then exit it
   **And** the player can continue forward without soft-locking the run (after placeholder resolution + exit, the node is `REVEAL_CLEARED`, the run is back in `PHASE_ACTIVE_ROUTE`, and `eligible_choice_ids()` still surfaces a legal forward choice through to the boss).

3. **The boss placeholder, when reached and resolved before Epic 9 content exists, can end the run with a placeholder completion event — and the same node boundary is replaceable.**
   **Given** the boss placeholder is reached (parked on the `boss` node in `PHASE_ACTIVE_ROUTE`)
   **When** it resolves before Epic 9 content exists
   **Then** the run can END with a placeholder completion event (resolving the boss placeholder transitions `PHASE_ACTIVE_ROUTE -> PHASE_NODE_RESOLUTION -> PHASE_COMPLETED` and emits a deterministic `run_completed` `DomainEvent` carrying an `outcome` of a stable placeholder marker, the boss node id, and the cleared-node count)
   **And** future Larval Avatar implementation can replace the placeholder through the **same node boundary** (the boss node resolves through the SAME `NodeResolvePlaceholderCommand` entry point; Epic 9 swaps the boss's resolution behavior — running the real boss level + a real victory — for the placeholder WITHOUT changing the route model, the `boss` node type, or the run-completed boundary).

4. **Placeholder/boss behavior is explicitly recorded for the readiness checkpoint (replaced / de-scoped / blocking), with placeholder ids and affected node types listed.**
   **Given** placeholder node or boss behavior remains after the route shell is playable
   **When** Epic 7, Epic 9, or Epic 10 readiness work begins
   **Then** each placeholder is either replaced, explicitly de-scoped with an approved MVP limitation, or treated as blocking full MVP readiness
   **And** placeholder ids and affected node types are listed in readiness notes (this story RECORDS the placeholder catalog — the stable placeholder marker/cue ids + the exact affected node-type set — in `_bmad-output/implementation-artifacts/deferred-work.md` as the MVP placeholder/de-scope checkpoint, owned forward by Epic 7/9/10, so the checkpoint is auditable rather than scattered).

### Acceptance-criteria interpretation notes (read before coding)

- **This story is ONE new run-domain command — `NodeResolvePlaceholderCommand` — plus a small surgical change to `NodeEnterCommand` (stop rejecting non-combat/boss types), plus TWO new system `DomainEvent`s (`node_placeholder_resolved` for non-combat placeholder nodes, `run_completed` for the boss-placeholder run-end), plus the AC4 readiness-catalog ledger entry.** It is the per-node-type RESOLUTION + boss-placeholder run-end slice. It does NOT generate routes (4.2 DONE — `RouteGenerator` already ASSIGNS all 8 node types incl. one terminal `boss`), does NOT implement the route CHOICE/commit (4.3 DONE — `RouteAdvanceCommand`), does NOT re-implement combat node entry/exit (4.4 DONE — `NodeEnterCommand`/`NodeExitCommand`/`route_sealed`), and does NOT build the full start-to-end playable shell, pacing measurement, save/resume of route position, or the `run_started` emission (4.6). It also does NOT build any real shop/reforge/gambling/event/secret system (those are Epics 6/7) or the real Larval Avatar boss + victory (Epic 9). Resist pulling any of that forward.

- **CRITICAL — the node types ALREADY EXIST and are ALREADY GENERATED; AC1 is mostly satisfied by 4.1+4.2 already.** `RouteNode` (4.1) defines all 8 `TYPE_*` constants (`combat`, `elite_combat`, `shop`, `reforge`, `gambling`, `event`, `secret`, `boss`) and `supported_types()`. `RouteGenerator` (4.2) ALREADY draws a depth-banded weighted type for every non-boss node (`_draw_node_type` / `_type_weights_for_band`) and places exactly ONE terminal `boss` node. So a generated route ALREADY "can include" every MVP type. The REAL work of AC1 is the **"unsupported node implementations route to safe placeholder resolution rather than broken gameplay"** clause: today `NodeEnterCommand.validate` REJECTS any non-combat type with `unsupported_node_entry` (4.4 scoped entry to combat/elite). 4.5 must give those nodes — and the boss — a safe RESOLUTION path so a run can pass through every node type and reach the end. **Do NOT change `RouteGenerator`, the `map` stream, or the route fingerprints** — the type vocabulary and assignment are done; 4.5 only adds resolution behavior.

- **THE TWO RESOLUTION PATHS — keep them cleanly separated by node type:**
  - **Combat / elite_combat** (`NodeEnterCommand`, 4.4, UNCHANGED behavior): build + validate a level `GenerationRequest`, transition `ACTIVE_ROUTE -> NODE_RESOLUTION`, emit `node_entered`. The caller runs the tactical level. Exit via `NodeExitCommand`. **4.5 must NOT alter this path's behavior for combat/elite** — only relax `NodeEnterCommand` so a NON-combat node is no longer its concern (it routes to the placeholder command instead). See the `NodeEnterCommand` change note below.
  - **Non-combat (`shop`/`reforge`/`gambling`/`event`/`secret`) AND `boss`** (`NodeResolvePlaceholderCommand`, NEW): there is NO tactical level and NO `GenerationRequest` for these in v0. Resolve the node as a PLACEHOLDER: transition `ACTIVE_ROUTE -> NODE_RESOLUTION`, emit `node_placeholder_resolved`, and (for a non-boss placeholder) STOP there so `NodeExitCommand` exits it like a combat node; (for the `boss` node) ALSO transition `NODE_RESOLUTION -> COMPLETED` and emit `run_completed` (the run ends — there is no exit-back-to-route from the boss). Draw ZERO RNG. Build no `GenerationRequest`.

- **AC1 "unsupported node implementations route to safe placeholder resolution" — the `NodeEnterCommand` change is SURGICAL and behavior-preserving for combat.** Today `NodeEnterCommand.validate` returns `unsupported_node_entry` for any node whose type is not in `NODE_TYPE_RECIPE` (i.e. not combat/elite). 4.5 KEEPS combat/elite entry exactly as-is in `NodeEnterCommand`, but the non-combat/boss types must now resolve through `NodeResolvePlaceholderCommand` instead of being a dead end. **Pick ONE and document it:**
  - **(Option A — RECOMMENDED) Leave `NodeEnterCommand` as the COMBAT-ONLY entry, add `NodeResolvePlaceholderCommand` as the PLACEHOLDER entry, and let the CALLER dispatch by node type.** `NodeEnterCommand` keeps rejecting non-combat with `unsupported_node_entry` (it is the combat-level entry; a non-combat node is genuinely not a level entry). The run orchestrator (the dispatcher — a later 4.6 concern, but the COMMANDS exist now) picks `NodeEnterCommand` for combat/elite and `NodeResolvePlaceholderCommand` for everything else. This keeps each command single-responsibility and is the cleanest mirror of the 4.4 idiom. **The AC1 "no broken gameplay / no soft-lock" guarantee is then proven by the walk test**: a generated route is walked with the type-dispatch (enter combat, placeholder-resolve non-combat/boss) and reaches `COMPLETED` for every seed. `unsupported_node_entry` stays a real code (it now means "you used the COMBAT entry on a non-combat node" — a caller error, not a dead end), and a test asserts the placeholder command ACCEPTS exactly the node types `NodeEnterCommand` rejects. **This is the recommended split** — it adds no behavior change to the settled 4.4 command and keeps the determinism/no-mutation contract trivially intact.
  - **(Option B) Make `NodeEnterCommand` itself dispatch** (build a request for combat, do placeholder resolution for non-combat) — REJECTED: it overloads the combat-level-entry command with reward/event/boss-end semantics, muddies its "produces a level request" contract, and forces it to emit three different event types. Do NOT do this.
  - Whichever you choose, the load-bearing AC1/AC2/AC3 proof is the **multi-seed start-to-COMPLETED walk** (see Testing): dispatch by type across a 4.2-generated route, asserting `run.validate()` is green at every step, the phase ping-pongs correctly, `cleared_node_ids` never duplicates, NO step soft-locks, and the run reaches `PHASE_COMPLETED` with a `run_completed` event for every seed.

- **AC3 boss-placeholder run-end — THE 4.6 BOUNDARY (read this twice).** Story 4.4's notes repeatedly state the `run_started`/`run_ended` emission belongs to 4.6 (the playable shell). AC3 of THIS story, however, EXPLICITLY requires that resolving the boss placeholder "can END with a placeholder completion event." That is a genuine, irreducible 4.5 requirement at the BOSS NODE boundary. Resolve the overlap precisely:
  - **4.5 OWNS the boss-NODE resolution -> run-end boundary:** resolving the `boss` placeholder transitions to `PHASE_COMPLETED` and emits ONE NEW `run_completed` `DomainEvent` (a placeholder-outcome run-completion record). This is the node-boundary event AC3 demands, and "future Larval Avatar implementation can replace the placeholder through the same node boundary" (AC3) means Epic 9 reuses THIS `run_completed` boundary, swapping only the boss's pre-completion behavior (run a real boss level + a real victory) — NOT the run-completed event itself.
  - **4.6 OWNS the full START-to-end shell:** emitting `run_started` AT RUN START, the start-to-end pacing measurement, the route-position save/resume, and wiring the type-dispatch into a run orchestrator/HUD. 4.6 CONSUMES the `run_completed` boundary 4.5 ships (its AC "a deterministic run-ended event is emitted ... later meta progression can consume that event" is satisfied by 4.5's `run_completed`). **4.5 does NOT emit `run_started`, does NOT add the run-START command, does NOT measure pacing, and does NOT touch the two `run_started` payload defers (`node_count` raw-JSON-number, `_has_decimal_string_payload` loose `is_valid_int()`).** Name the event `run_completed` (a clean past-tense node-boundary record); do not retrofit/rename `run_started`. Record this 4.5/4.6 split explicitly in Completion Notes as a `[Decision]` so the 4.6 owner knows the run-end boundary already exists and to consume (not re-create) it.
  - **Why a NEW `run_completed` event (not `node_exited`/`route_sealed`):** the boss does NOT exit back to a route choice (there is no forward node after the boss) and is NOT "sealed behind" the player like a cleared mid-run node — it ENDS the run. Reusing `node_exited` would (a) imply a return-to-route that does not happen and (b) require the `NODE_RESOLUTION -> ACTIVE_ROUTE` transition that is wrong for a terminal boss. A dedicated `run_completed` event with an `outcome` marker is the correct, replaceable boundary.

- **"communicates placeholder completion in domain/debug terms" = a deterministic event + metadata flag, NOT a real shop/event/reward/boss system.** There is NO shop UI, NO reforge logic, NO gambling, NO event-choice system, NO secret/lore reveal, NO reward entity, NO `RewardTableDefinition`, and NO boss combat in v0 (project-context Generation Rules: rewards are abstract markers, concrete loot is Epic 6; the risk economy + affinities are Epic 7; the Larval Avatar is Epic 9). For 4.5, "placeholder completion" means the resolution emits a deterministic `node_placeholder_resolved` event (carrying a stable `resolution` marker like `placeholder_completed`) + a metadata flag, so a later real-system story (shop = Epic 6/7, event/risk = Epic 7, boss = Epic 9) hooks the SAME node boundary. Do NOT build any real per-type behavior, modal, currency, or loot here.

- **Phase transitions use ONLY existing legal edges; do NOT change the transition TABLE.** The 4.1 machine is `NEW_RUN -> ACTIVE_ROUTE -> NODE_RESOLUTION -> (ACTIVE_ROUTE | COMPLETED | FAILED)` (`run_state.gd` `_legal_next_phases`). `NodeResolvePlaceholderCommand` for a non-boss placeholder performs `ACTIVE_ROUTE -> NODE_RESOLUTION` (legal — same as `NodeEnterCommand`); for the boss it performs `ACTIVE_ROUTE -> NODE_RESOLUTION` THEN `NODE_RESOLUTION -> COMPLETED` (both legal edges already in the table). Use `RunState.transition_to(...)` for each step; do NOT mutate `phase` directly and do NOT edit `_legal_next_phases` (every edge you need already exists). Wrong-phase rejection (AC mirror): the placeholder command rejects any phase != `ACTIVE_ROUTE` with the stable `wrong_run_phase` code carrying the actual + expected phase.

- **Boss exit semantics — the boss does NOT use `NodeExitCommand`.** A non-boss placeholder node is exited by the EXISTING `NodeExitCommand` (it is `REVEAL_CLEARED` + cleared + back to `ACTIVE_ROUTE` exactly like a combat node — confirm `NodeExitCommand` already handles ALL node types on exit, which it does: its `REWARDS_PLACEHOLDER_TYPES` flag is computed defensively and an unknown type just yields `rewards_placeholder: false`; it does not gate on type). The BOSS node, however, ends in `PHASE_COMPLETED` (terminal), from which `NodeExitCommand` would (correctly) reject with `wrong_run_phase` — the boss is never exited back to a route choice. So: non-combat placeholder -> resolve (placeholder cmd) -> exit (`NodeExitCommand`); boss -> resolve-to-COMPLETED (placeholder cmd) -> DONE. Document that the boss is the only node that does not round-trip back to `ACTIVE_ROUTE`.

- **Should the boss node mark itself `REVEAL_CLEARED` / append to `cleared_node_ids` on completion?** YES — for consistency and so the AC3 "cleared-node count" in the `run_completed` payload is correct and `run.validate()` stays green. `NodeResolvePlaceholderCommand`, when resolving the BOSS, should mark the boss node `REVEAL_CLEARED` and append it to `cleared_node_ids` (idempotently — guard `if not route.cleared_node_ids.has(boss_id)`) BEFORE transitioning to `COMPLETED`, mirroring how `NodeExitCommand` clears a resolved node. The `run_completed` payload's `cleared_node_count` then reflects the full path including the boss. For a NON-boss placeholder node, do NOT clear on resolve (the existing `NodeExitCommand` clears it on exit, exactly as it clears a combat node — keep the clear in ONE place to avoid a double-clear, mirroring the 4.4 cleared-set discipline). So: placeholder resolve of a non-combat node = NO cleared-set mutation (exit clears it); placeholder resolve of the boss = clear the boss + complete (no exit follows).

- **Boss-path transition ORDERING (the 4.4 "mutate-before-infallible-transition" Low, applied here).** The boss path performs TWO mutations the others do not: it clears the boss AND runs two `transition_to` calls (`ACTIVE_ROUTE -> NODE_RESOLUTION`, then `NODE_RESOLUTION -> COMPLETED`). `validate()` already pins `ACTIVE_ROUTE`, and BOTH edges are always-legal from their respective states, so neither transition can actually fail at runtime — the boss-clear-then-complete sequence cannot leave a half-completed run in practice. Still, for robustness/symmetry: run the two transitions and check each result; if EITHER ever returns an error, return a structured `wrong_run_phase` error WITHOUT having emitted any event (build the events AFTER both transitions succeed, exactly as `NodeExitCommand`/`NodeEnterCommand` build their events only after the transition). Do NOT emit `run_completed` and then transition — emit nothing until the run is actually in `COMPLETED`. Document the ordering choice in a one-line comment so the next reviewer sees it was deliberate (this is the exact `[Review][Patch]` Low the 4.4 reviewer raised on `NodeExitCommand`).

- **Two NEW system events, each fully wired like `route_advanced`/`node_entered`.** Add `node_placeholder_resolved` and `run_completed` to `domain_event.gd` exactly the way `route_advanced` / `node_entered` are wired: a `Type` enum value (append at the END — never renumber), an `EVENT_ID_*` const, a `static func <name>(sequence_id, payload)` factory (SYSTEM event, `actor_id = &""`, NOT added to `_event_requires_actor`), a `_validate_<name>_payload(...)` validator, the `_validate_payload_for_event` arm, and the `id_for_type`/`type_for_id` arms. Suggested payloads (all serializable + JSON-round-trippable; node ids carry hyphens so they are PLAIN non-empty strings via `_has_nonempty_string_payload`, NOT lower_snake; node TYPE / resolution markers / outcome are lower_snake via `_has_lower_snake_payload`; counts via `_has_nonnegative_integral_payload`):
  - `node_placeholder_resolved`: `node_id` (String, hyphenated), `node_type` (lower_snake — the actual placeholder node type, e.g. `shop`), `node_depth` (non-negative integral), `resolution` (lower_snake — MUST equal a named constant e.g. `placeholder_completed`; validate present + lower_snake, assert the exact value in the command/test).
  - `run_completed`: `outcome` (lower_snake — MUST equal a named placeholder constant e.g. `placeholder_completed` / `boss_placeholder`; validate present + lower_snake, assert exact value), `boss_node_id` (String, hyphenated — the boss node), `cleared_node_count` (non-negative integral — `route.cleared_node_ids.size()` AFTER the boss is cleared). Mirror the `level_victory_reached`/`level_defeat_reached` `outcome`-value-equality validator pattern (those validators assert `outcome == "victory"`/`== "defeat"`).
  Real-JSON round-trip BOTH new events in `test_domain_event.gd` + at least one malformed-payload rejection per event (e.g. missing `node_id` / bad `node_type` / missing `resolution`; missing `outcome` / wrong `outcome` value / negative `cleared_node_count`). Add the two new ids to the stable-machine-id map. Keep ALL existing event cases green.

- **Sequence id is caller-supplied (the run still has no event sequencer).** Mirror `RouteAdvanceCommand`/`NodeEnterCommand`/`NodeExitCommand`: the command takes the run-level `sequence_id` via the constructor (default 1, gated `> 0`). For the BOSS path the command emits TWO events (`node_placeholder_resolved` + `run_completed`) — give them distinct sequence ids (e.g. `sequence_id` and `sequence_id + 1`, centralized in a helper like `NodeExitCommand._route_sealed_sequence_id`); validate BOTH ids `> 0` in `validate()` BEFORE reading/mutating state, so a success path can never emit an event its own validator rejects (the exact Round-1 finding 4.3/4.4 guard against — `invalid_event_sequence_id`). For a NON-boss placeholder the command emits ONE event (`node_placeholder_resolved`) at `sequence_id`.

- **The command draws ZERO RNG and is a pure deterministic state transition.** Resolving a placeholder node (combat-less) draws nothing: no `GenerationRequest` is built (there is no level), no `map`/`level`/`rewards` stream is touched, marking cleared / transitioning phase / emitting events draws nothing. Mirror the 4.1/4.3/4.4 no-RNG discipline: hold an `RngStreamSet`, snapshot it, run a placeholder resolve (boss AND non-boss; success AND reject), assert the streams are byte-identical.

- **Context shape: take the `RunState` directly (mirror 4.3/4.4).** `validate(state)` / `execute(state)` accept `Variant`, reject a non-`RunState` / null-route with `invalid_context`, and (mirroring 4.3/4.4) `run.validate()` the run first, surfacing the inner structural error code in the `invalid_context` metadata. A single-field context wrapper adds no value.

- **Which node types does the placeholder command ACCEPT? Define it as the COMPLEMENT of the combat set, plus boss.** The placeholder command accepts `shop`, `reforge`, `gambling`, `event`, `secret`, AND `boss` (every `RouteNode.supported_types()` value that is NOT `combat`/`elite_combat`). Define this as a named constant set on the command (e.g. `PLACEHOLDER_NODE_TYPES`), derived to stay in lockstep with `NodeEnterCommand.NODE_TYPE_RECIPE` (combat set) so the two commands partition the 8 types with no gap and no overlap. A `combat`/`elite_combat` node passed to the placeholder command is rejected with a stable code (e.g. `node_not_placeholder` — combat nodes use `NodeEnterCommand`); a test asserts the two commands' accepted-type sets are exact complements covering all 8 `supported_types()`.

## Dev Notes

### What this story IS (and is NOT)

- **IS:** the per-node-type RESOLUTION + boss-placeholder run-end slice — ONE new run-domain command (`NodeResolvePlaceholderCommand`) that safely resolves the five non-combat MVP node types (`shop`/`reforge`/`gambling`/`event`/`secret`) as no-op placeholders (transition `ACTIVE_ROUTE -> NODE_RESOLUTION`, emit `node_placeholder_resolved`, then `NodeExitCommand` exits them) AND resolves the `boss` placeholder as a run-END (transition to `COMPLETED`, emit `run_completed`). Plus TWO new system events (`node_placeholder_resolved`, `run_completed`), a comment-only clarification on `NodeEnterCommand`, and the AC4 placeholder-catalog ledger entry. This makes a generated route fully traversable start-to-end through EVERY node type with no soft-lock and no broken gameplay.
- **IS NOT:** route GENERATION or the node-type vocabulary (`RouteNode` 8 types + `RouteGenerator` depth-banded assignment + the single terminal boss are Stories 4.1/4.2, DONE). 4.5 CONSUMES the generated route + its assigned types; it does NOT change the generator, the `map` stream, the type weights, or the route fingerprints. AC1 "the route can include every MVP type" is ALREADY true from 4.2 — 4.5 adds the RESOLUTION half.
- **IS NOT:** the route CHOICE / forward-commitment command (`RouteAdvanceCommand`, 4.3 DONE) or combat node entry/exit (`NodeEnterCommand`/`NodeExitCommand`/`route_sealed`, 4.4 DONE). 4.5 adds a SIBLING resolution command for non-combat/boss and REUSES `NodeExitCommand` for non-boss placeholder exit + `RouteAdvanceCommand` for the forward step. The only touch to a 4.4 command is a one-line clarifying comment on `NodeEnterCommand`'s `unsupported_node_entry` branch (no behavior change).
- **IS NOT:** running a tactical level. Non-combat placeholder nodes have NO level and build NO `GenerationRequest` (a shop/event is not a board). The boss placeholder ALSO runs no real boss level in v0 — it is a placeholder run-end. Real boss combat + victory is Epic 9; it will reuse the SAME boss-node `run_completed` boundary.
- **IS NOT:** the full start-to-end playable shell, pacing measurement, route-position save/resume, or the `run_started` emission (**Story 4.6**). 4.5 ships the per-type resolution COMMANDS + the `run_completed` boundary; 4.6 wires them into the run orchestrator/HUD, emits `run_started` at run start, measures pacing, and persists/resumes route position — CONSUMING 4.5's `run_completed` boundary (do not re-create it). 4.5 does NOT emit `run_started` and does NOT fix the two `run_started` payload defers (those belong to the run-start emitter, 4.6).
- **IS NOT:** any real shop / reforge / gambling / event / secret / reward / boss system. "placeholder completion" is a deterministic event + metadata flag. Concrete loot/rewards are Epic 6; the risk economy (gambling, curses, risk events) + affinities are Epic 7; the Larval Avatar boss is Epic 9. No `RewardTableDefinition`, no reward/currency entity, no boss combat path here.

Keeping these boundaries tight is the single biggest risk for this story, exactly as it was for 4.1/4.2/4.3/4.4. The temptation is to build a real shop/event/boss, pull the full 4.6 shell forward, or change the generator. Resist it.

### Architecture patterns and constraints

- **Gameplay actions are validated commands returning `ActionResult`; successful commands emit deterministic past-tense `DomainEvent` records.** `NodeResolvePlaceholderCommand` follows the Epic-1 + 4.3/4.4 command precedent exactly: `validate()` then `execute()` (which re-validates), returning `ActionResult.ok([events], metadata)` on success and a structured `ActionResult.error(code, metadata)` with ZERO events + no mutation on failure ([Source: `project-context.md` §Determinism & Simulation Rules; `godot/scripts/core/commands/node_enter_command.gd`; `node_exit_command.gd`; `route_advance_command.gd`]).
- **Placeholder resolution is a deterministic, RNG-free state transition.** Resolving a non-combat node (no level) or the boss (no real combat) draws NO RNG (no stream touched), runs no sub-command, builds no `GenerationRequest`, and consumes nothing from `RngStreamSet`. Mirror the 4.1/4.3/4.4 "run-state transition draws no RNG" discipline ([Source: `godot/tests/unit/core/test_node_exit_command.gd::_exit_draws_no_rng_on_success_and_reject`]).
- **Scene-independent, autoload-free domain ownership.** The new command + the new events live in the scene-free domain (`scripts/core/commands/`, `scripts/core/events/`) as `RefCounted` services/DTOs. No `Node`, no `get_tree`/`get_node`, no new autoload. The autoload set is unchanged through Epic 3 + 4.1-4.4 (`GameSession`, `SceneManager`, `SaveManager`, `AudioManager`, `SettingsManager`, `Diagnostics`) — do NOT register a node-flow autoload. UI/orchestration dispatches these commands in the 4.6 shell story; 4.5 builds only the domain command ([Source: `project-context.md` §Engine-Specific Rules, §Code Organization Rules]).
- **Validate-then-mutate, no coercion, no-mutation-on-reject, stable codes.** A wrong-phase / not-parked / not-a-placeholder-type / invalid-context / bad-sequence-id rejection surfaces a structured `ActionResult.error` (stable lower_snake code + diagnostics in metadata) and mutates NOTHING — the `RunState`/`RouteState` is byte-identical before and after. Error codes are lower_snake with no hyphens/dots/spaces (`ActionResult._is_valid_error_code`) — so hyphenated node ids go in METADATA, never in a code ([Source: `godot/scripts/core/commands/node_enter_command.gd`; `godot/scripts/core/results/action_result.gd`]).
- **Phase machine: use the existing legal edges; do not change the table.** `ACTIVE_ROUTE -> NODE_RESOLUTION` (placeholder resolve), `NODE_RESOLUTION -> ACTIVE_ROUTE` (non-boss exit via `NodeExitCommand`), and `NODE_RESOLUTION -> COMPLETED` (boss run-end) are ALL already in `RunState._legal_next_phases`. Call `run.transition_to(...)`; do not mutate `phase` directly and do not edit the table ([Source: `godot/scripts/run/run_state.gd` `_legal_next_phases`, `transition_to`; `_bmad-output/game-architecture.md` §State Management line 330 — `RunState` phases new_run -> active_route -> node_resolution -> completed/failed]).
- **Reveal is monotonic; cleared is the run-state truth of "the path sealed behind you"; the boss-clear is the only clear this command does.** A NON-boss placeholder is cleared by the EXISTING `NodeExitCommand` on exit (keep the clear in ONE place — do NOT also clear it in the placeholder resolve, or the exit would double-clear and `RouteState.validate()` rejects `duplicate_cleared_node`). The BOSS is cleared by `NodeResolvePlaceholderCommand` itself (idempotent append) because no exit follows it. A revealed/cleared node is never un-revealed (GDD line 249) ([Source: `_bmad-output/planning-artifacts/gdds/gdd-Game-2026-05-31/gdd.md` line 249; `godot/scripts/core/commands/node_exit_command.gd` cleared-set idempotency; `godot/scripts/run/route_node.gd` REVEAL_* constants]).
- **Compose, do not fork; keep the save bridge coherent.** The command mutates the live `RouteState`/`RunState`. It adds NO new top-level `RunSnapshot` key (the boss-clear flows through the EXISTING 4.1 bridge fields — `cleared_node_ids`, nested `run_phase`). The `run_completed` event is a pure record; it does NOT introduce a parallel run-save format ([Source: `godot/scripts/run/run_state.gd` `to_run_snapshot_fields`; `project-context.md` §Save Rules; epic-4 retro Story 4-1 on nested `run_phase` + the 23-key gate]).
- **New domain event = full wiring.** Each new event (`node_placeholder_resolved`/`run_completed`) must be wired end-to-end like `node_entered`/`route_advanced`: `Type` enum value (appended at END), `EVENT_ID_*` const, factory, payload validator, `_validate_payload_for_event` arm, `id_for_type`/`type_for_id` arms. A half-wired event is a latent bug. The `run_completed` `outcome` validator mirrors the `level_victory_reached`/`level_defeat_reached` value-equality pattern (assert the exact outcome value) ([Source: `godot/scripts/core/events/domain_event.gd` `node_entered`/`route_advanced` precedent + `_validate_level_victory_reached_payload` value-equality]).
- **The boss is a placeholder run-END, not a sealed mid-run node.** It does NOT use `node_exited`/`route_sealed` (those imply a return-to-route that does not happen for a terminal boss). It uses a dedicated `run_completed` event with an `outcome` marker, and the `NODE_RESOLUTION -> COMPLETED` transition. Epic 9 replaces the boss's pre-completion behavior through the SAME node + the SAME `run_completed` boundary ([Source: `_bmad-output/planning-artifacts/epics.md` Story 4.5 AC3 "future Larval Avatar implementation can replace the placeholder through the same node boundary"; Story 4.6 AC "a deterministic run-ended event is emitted ... later meta progression can consume that event"]).
