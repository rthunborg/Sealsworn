# Story 4.6: Playable Run Shell from Start to End

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want to start a run, clear nodes, and reach an endpoint,
so that the rough roguelite descent exists before classes, loot, and meta progression deepen it.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 4 -> Story 4.6 (verbatim AC, with the implementation-grounded detail this story adds). Readiness traceability: the 2026-06-04 readiness patch maps "GDD FR47 and NFR5: early/mid/late/finale pacing and run-length targets -> Story 4.6 and Story 10.4" (epics line 390) -> AC5 here OWNS the FIRST start-to-end pacing measurement from the playable shell.

1. **The player can start a new MVP shell run; route + first node are generated; route/level/RNG state are tracked together.**
   **Given** the player starts a new MVP shell run (a fresh run from a root seed via the NEW run-start command)
   **When** the route and first node are generated
   **Then** the player can enter at least one combat level from the route (the start node is always `combat` — Story 4.2 — so `NodeEnterCommand` produces a valid level `GenerationRequest` and `LevelGenerator.generate(...)` returns a playable Small level)
   **And** route state, level state, and RNG stream state are tracked together (one orchestrator holds the live `RunState` + the in-flight level `GenerationResult`/`BoardState` + the run-level `RngStreamSet`, so the three advance in lockstep — not three disconnected pieces).

2. **The player clears multiple nodes; run state persists; between-level save/resume restores the route position.**
   **Given** the player clears multiple nodes
   **When** the run advances between nodes
   **Then** cleared nodes, current node, available choices, and manual-seed eligibility persist in run state (the orchestrator threads ONE `RunState` through `NodeEnterCommand`/`NodeExitCommand`/`NodeResolvePlaceholderCommand`/`RouteAdvanceCommand`, and `run.validate()` stays green at every boundary)
   **And** between-level save/resume can restore the **route position** (a between-node autosave writes the run's `route_state` + `current_route_node_id` + `revealed_route_node_ids` through `SaveRepository`/`RunSnapshot`, and resume rehydrates a `RunState` parked on the same node with the same cleared set, manual-seed eligibility, and RNG streams — proven by a save-restore-replay determinism test against the uninterrupted path).

3. **Reaching boss placeholder or shell end emits a deterministic run-ended event that later meta progression can consume without changing earlier route logic.**
   **Given** the run reaches boss placeholder or shell end
   **When** completion resolves
   **Then** a deterministic run-ended event is emitted (the orchestrator resolves the boss node through `NodeResolvePlaceholderCommand`, which Story 4.5 already wired to transition `ACTIVE_ROUTE -> NODE_RESOLUTION -> COMPLETED` and emit the `run_completed` `DomainEvent` carrying `outcome = boss_placeholder`)
   **And** later meta progression can consume that event without changing earlier route logic (4.6 **CONSUMES** the existing Story 4.5 `run_completed` boundary — it must NOT re-create, rename, or duplicate that event; the orchestrator surfaces the run-ended outcome from the command result + the emitted `run_completed` event, leaving the 4.3/4.4/4.5 route/node/boss commands byte-for-byte unchanged).

4. **An internal tester plays the route shell; at least one route-information-changed-my-decision moment is recorded; missing tradeoffs become route tuning notes.**
   **Given** an internal tester plays the route shell
   **When** they choose across multiple route nodes
   **Then** they record at least one moment where route information (the reveal-gated `eligible_choice_ids()` + per-node clue tags from Story 4.2) changed their decision
   **And** missing tradeoffs are captured as route tuning notes before loot/meta systems deepen the run (recorded in a tracked notes location — the Completion Notes "Playtest / pacing notes" subsection of this story file and/or `deferred-work.md` — NOT untracked).

5. **MVP run pacing is evaluated from the playable shell; pacing failures produce tuning notes, not untracked scope expansion.**
   **Given** MVP run pacing is evaluated from the playable shell
   **When** route length, node resolution time, reward cadence, and endpoint timing are measured
   **Then** successful MVP runs target **20-35 minutes**, failed runs commonly target **5-15 minutes**, and **45 minutes** is treated as an upper stretch for careful/completionist runs (the headless shell measures the STRUCTURAL pacing surface available today — route length / tier count / node-type mix / node-count distribution across seeds — and records it; the human-felt minute targets are a tester-note overlay on that structural data, since v0 has no real per-node content time yet)
   **And** pacing failures produce tuning notes for node count, encounter length, reward frequency, or boss approach rather than untracked scope expansion (recorded as tracked notes; NO generator/route-weight/recipe change is made in 4.6 to "fix" pacing — that is Epic 10 / a tracked follow-up).

### Acceptance-criteria interpretation notes (read before coding)

- **This story is the SHELL that wires the finished 4.1-4.5 run domain into one end-to-end playable loop — it is mostly COMPOSITION + ONE genuinely new command (the run-START command, the first `run_started` emitter) + the route-position SAVE/RESUME bridge + the two `run_started` payload defer fixes + pacing/playtest measurement seams.** Almost every primitive already exists and is settled:
  - `RouteGenerator.generate(seed)` -> a deterministic 8-12-non-boss-node route + one terminal boss (Story 4.2). 4.6 calls it; it does NOT change it.
  - `RunState`/`RouteState`/`RouteNode` (Story 4.1) — the phase machine (`NEW_RUN -> ACTIVE_ROUTE -> NODE_RESOLUTION -> COMPLETED/FAILED`), the route graph, the bridge into `RunSnapshot`.
  - `RouteAdvanceCommand` (4.3 — choose+commit forward with reveal-on-arrival), `NodeEnterCommand`/`NodeExitCommand` (4.4 — combat level entry/exit + the autosave seam + `route_sealed`), `NodeResolvePlaceholderCommand` (4.5 — non-combat placeholder resolve + the boss `run_completed` run-END).
  - `LevelGenerator.generate(request)` (Epic 3) — the deterministic Small/Medium level pipeline the combat nodes feed.
  - `SaveRepository`/`RunSnapshot`/`RunResumeService`/`SaveManager` (Epic 2) — the between-level save/resume primitives (today board-centric; 4.6 extends them for route position).
  - The single genuinely NEW domain artifact is the **run-START command** that builds a fresh `RunState` from `(root_seed, is_manual_seed)`, generates+rehydrates the route, parks on the start node, transitions `NEW_RUN -> ACTIVE_ROUTE`, and emits the EXISTING-but-UNTIL-NOW-UNEMITTED `run_started` event. Plus a thin **run orchestrator** (a `RefCounted` domain service) that dispatches the per-node commands by node type and threads one `RunState` + one `RngStreamSet` + the in-flight level state through the whole loop.

- **DO NOT re-create, rename, or duplicate `run_completed` (Story 4.5 OWNS it).** AC3's "a deterministic run-ended event is emitted ... later meta progression can consume that event" is SATISFIED by Story 4.5's `run_completed` (`outcome = boss_placeholder`), emitted by `NodeResolvePlaceholderCommand` on the boss path. 4.6 CONSUMES it: the orchestrator resolves the boss via `NodeResolvePlaceholderCommand` and surfaces the `run_completed` event/outcome from that result. Do NOT add a second run-ended event, do NOT rename `run_completed`, and do NOT move the boss run-end logic out of `NodeResolvePlaceholderCommand`. (This is the explicit 4.5->4.6 boundary the 4.5 dev recorded as a `[Decision]` and that the epic-4 retro flags: "Confirm at epic-4 retro that 4.6 consumes rather than duplicates it.")

- **The run-START command is the FIRST `run_started` emitter — and it MUST close the two open 4.1 `run_started` defers in the same change (they are explicitly OWNED by "whatever story first emits `run_started`", i.e. THIS story).** Two coupled fixes, both in `godot/scripts/core/events/domain_event.gd`:
  1. **`node_count` raw-JSON-number defer (4.1 Round-1):** `run_started.node_count` is a raw JSON number, not decimal-string-encoded. For the MVP route the count is the non-boss node count in `[8, 12]` (`RouteGenerator` `MIN_NON_BOSS_NODES`=8 .. `MAX_NON_BOSS_NODES`=12), so it can NEVER exceed 2^53 — the cleanest disposition is to **re-confirm `node_count` stays small and bounded** (this is what the 4.1 defer asks: "Owner: whatever story first EMITS `run_started` — re-confirm `node_count` stays small and bounded"). The run-start command sets `node_count` from the route's non-boss count, and a test pins it to the `[8, 12]` range across seeds. This is a permanently-benign close (per the 4.2 retro note: "4.2 structurally bounds the route to 8-12 non-boss nodes ... so the eventual run-start emitter (4.3/4.6) can close the two `run_started` int64/lossless-decimal defers as permanently benign rather than re-litigating them"). Do NOT change `node_count` to a decimal string (it is a bounded count, not a seed) — just prove the bound.
  2. **`_has_decimal_string_payload` loose-`is_valid_int()` defer (4.1 Round-2):** the `run_started.root_seed` validator (`_has_decimal_string_payload`, `domain_event.gd:991-997`) uses bare `is_valid_int()`, which ACCEPTS an out-of-int64 decimal string that `String.to_int()` silently saturates/wraps (e.g. `"9223372036854775808"` or `"99999999999999999999"`). The epic-4 retro RATIFIES the fix: "any new seed-string validator must adopt the 3.7 lossless/canonicalize idiom, not bare `is_valid_int()`." Tighten `_has_decimal_string_payload` to ALSO require lossless round-trip (`str(value.to_int())` equals the canonicalized input), reusing the EXACT 3.7 idiom in `ManualSeedLoader._decimal_string_round_trips_losslessly` / `_canonical_decimal_string` (`godot/scripts/generation/level/manual_seed_loader.gd:187-211`). Add an out-of-range rejection assertion to `test_domain_event.gd::_run_started_serializes_and_parses_stable_payload`. NOTE: `_has_decimal_string_payload` is used ONLY by `_validate_run_started_payload` today (grep-confirm before editing) — tightening it cannot regress another event. The run-start command itself encodes `root_seed` via `str(run.root_seed)` (always in-range), so the live emit path is unaffected; the fix hardens the VALIDATOR against hand-edited/foreign payloads, matching the 3.7 discipline.

- **The run-START command shape — mirror the 4.3/4.4/4.5 run-command idiom EXACTLY, but note it starts from `NEW_RUN` (the others start from `ACTIVE_ROUTE`/`NODE_RESOLUTION`).** Suggested: `RunStartCommand` in `godot/scripts/core/commands/run_start_command.gd` extending `game_command.gd`, `command_id = &"run_start"`, constructor `_init(root_seed: int, is_manual_seed: bool = false, sequence_id: int = 1)`.
  - `validate(state)`: gate `sequence_id > 0` (`invalid_event_sequence_id`) BEFORE anything; accept a `RunState` in `PHASE_NEW_RUN` (reject any other phase with `wrong_run_phase` + actual/expected) OR (cleaner) accept the seed directly and build the run internally — PICK ONE and document it (see the "context shape" note below). Reject a negative `root_seed` with a stable code (`invalid_run_seed`) BEFORE generating (mirror `RouteGenerator.generate`'s `root_seed < 0` guard; the run domain elsewhere assumes a non-negative seed). Manual-seed eligibility invariant: `meta_progression_eligible == not is_manual_seed` (the `RunState.validate()` rule).
  - `execute(state)`: generate the route via `RouteGenerator.generate(root_seed)` (handle the structured error path — a failed generation returns `route_generation_failed` carrying the inner phase/code, ZERO events, no half-built run); rehydrate via `RouteGenerator.route_from_result(result)`; park `current_node_id` on the start node (`route.nodes()[0].id` — depth-0 start); transition `NEW_RUN -> ACTIVE_ROUTE` (a legal edge); build ONE `run_started` event (`root_seed` decimal-string, `is_manual_seed`, `node_count` = the route's non-boss count from `result.payload[PAYLOAD_NODE_COUNT_KEY]` / `result.metadata.node_count`); return `ActionResult.ok([run_started_event], metadata)` with the live `RunState` + the non-boss count in metadata.
  - Build the event ONLY AFTER the transition succeeds (the 4.4/4.5 mutate-before-infallible-transition discipline); ZERO RNG drawn by the COMMAND itself beyond the route generation it delegates (route generation legitimately draws the `map` stream — that is the ONE place a run-affecting draw happens, and it is owned by `RouteGenerator`, not re-implemented here). No-mutation-on-reject: a rejected `RunStartCommand` leaves its input byte-identical with zero events.

- **Context shape for the run-START command — the run does not exist yet, so this is the ONE place the "take `RunState` directly" idiom does not cleanly fit.** The 4.3/4.4/4.5 commands take an EXISTING `RunState`. A run-START command CREATES the run. Two defensible options — PICK ONE and document it as a `[Decision]`:
  - **(Option A — RECOMMENDED) The command takes the seed + flags in its constructor and `execute(_state)` BUILDS and returns the new `RunState` in metadata** (the `state` arg is unused / accepts `null`, or accepts a fresh `RunState.new_run(seed, is_manual_seed)` and drives it from `NEW_RUN`). This keeps the command a self-contained "start a run" factory whose output is the live run, and is the cleanest fit for an orchestrator that has no run yet. If you accept a passed-in fresh `RunState`, validate it is in `PHASE_NEW_RUN` with a matching seed/eligibility and reject otherwise.
  - **(Option B) A non-command service** (`RunStartService` / a static `RunOrchestrator.start_run(...)`) builds the run + emits `run_started`. REJECTED for the EVENT-emitting path: the project's discipline is that an emitting action is a `GameCommand` returning `ActionResult` (so `run_started` rides the same validate/execute/no-mutation/result contract as every other event). Keep the start a COMMAND; the orchestrator below CALLS it.
  - Whichever you pick, the load-bearing proof is the start-to-end orchestrated run test (see Testing): start -> walk every node by type -> reach `COMPLETED` with exactly one `run_started` (at the start) and exactly one `run_completed` (at the boss), `run.validate()` green throughout.

- **The run ORCHESTRATOR (the type-dispatch driver) — a thin `RefCounted` DOMAIN service, NOT an autoload, NOT a scene.** Suggested: `RunOrchestrator` in `godot/scripts/run/run_orchestrator.gd` (the `run` domain, alongside `RunState`/`RouteState`). It is the dispatcher Story 4.5 said "the COMMANDS exist now; the run orchestrator (the dispatcher — a later 4.6 concern)". Responsibilities:
  - **Dispatch by node type:** for the parked node, call `NodeEnterCommand` if the type is in `NodeEnterCommand.NODE_TYPE_RECIPE` (combat/elite), else `NodeResolvePlaceholderCommand` (the complement set + boss). This is EXACTLY the partition Story 4.5 proved (`PLACEHOLDER_NODE_TYPES` = complement of `NODE_TYPE_RECIPE`, no gap/overlap over all 8 types) — REUSE those constants; do NOT re-derive the dispatch table.
  - **For a combat/elite node:** after `NodeEnterCommand` returns the live `GenerationRequest` in `metadata.level_request`, run `LevelGenerator.generate(request)` (the run-level `RngStreamSet`/`level` stream draws here — this is the FIRST 4.x consumer of the success-path level `GenerationResult`).   - **For the boss node:** `NodeResolvePlaceholderCommand` resolves it to `COMPLETED` + emits `run_completed`; the run ENDS (no exit, no advance). The orchestrator detects `run.is_terminal()` and stops.
  - **Between nodes (non-terminal):** `RouteAdvanceCommand` to the first `eligible_choice_ids()` entry (or a caller-supplied chosen id), moving forward with reveal-on-arrival.
  - **Sequence ids:** the run domain still has NO event sequencer — the orchestrator OWNS supplying a monotonically increasing run-level `sequence_id` to each command (mirroring how each 4.3/4.4/4.5 command takes `sequence_id` via its constructor).   - The orchestrator draws NO RNG itself except via the commands it calls (route generation in the start command, level generation for combat nodes). It is deterministic given `(root_seed, is_manual_seed, the choices made)`.

- **Route-position SAVE/RESUME (AC2's load-bearing new bridge) — the run shell saves/restores the ROUTE position, which is DIFFERENT from the Epic-2 board-centric save.** This is the single most error-prone part of 4.6. Read it twice:

## Dev Notes

### What this story IS (and is NOT)

- **IS:** the playable run SHELL — wiring the finished 4.1-4.5 run domain + the Epic-3 level pipeline + the Epic-2 save layer into ONE end-to-end loop. Concretely: (1) the NEW `RunStartCommand` (the first `run_started` emitter, closing the two 4.1 `run_started` defers); (2) a thin `RunOrchestrator` domain service that dispatches per-node commands by type (combat/elite -> enter+generate+exit; non-combat -> placeholder-resolve+exit; boss -> placeholder-resolve-to-COMPLETED) and threads ONE `RunState` + one `RngStreamSet` start-to-end; (3) the board-free ROUTE-POSITION save/resume bridge so a between-node autosave persists+restores the route position; (4) pacing + playtest measurement seams. This makes a generated route fully PLAYABLE from a fresh seed to a `run_completed` endpoint with route-position save/resume.
- **IS NOT:** route GENERATION / the node-type vocabulary / weights / fingerprints (4.1/4.2, DONE — 4.6 CONSUMES `RouteGenerator.generate(seed)` and the assigned types unchanged). It does NOT touch the `map` stream or the route fingerprints.
- **IS NOT:** the per-node-type RESOLUTION commands or the boss run-end event (`RouteAdvanceCommand`/`NodeEnterCommand`/`NodeExitCommand`/`NodeResolvePlaceholderCommand`/`run_completed`/`node_*`/`route_*` are 4.3/4.4/4.5, DONE). 4.6 ORCHESTRATES them and CONSUMES `run_completed` (it does NOT re-create, rename, or duplicate it). The only event-layer change is the two `run_started` payload defer fixes + the run-start command's emit.
- **IS NOT:** a real shop / reforge / gambling / event / secret / reward / boss SYSTEM (Epics 6/7/9), a class/starting-kit (Epic 5), or meta-progression LOGIC (Epic 8). The shell starts a class-less run, resolves non-combat nodes as 4.5 no-op placeholders, ends the boss as a 4.5 placeholder run-end, and only EMITS/SURFACES `run_completed` so Epic 8 can later consume it (it grants no progression and writes no profile/meta snapshot).
- **IS NOT:** a real tactical play loop wired into the headless orchestrator, or a run-map/HUD scene as authoritative state. The headless shell proves the route<->level handoff + determinism + route-position save/resume scene-free; combat nodes are v0 auto-resolved on successful level generation. A real HUD/run-map + in-orchestrator tactical play is a later presentation story.
- **IS NOT:** a pacing/route-weight TUNING change. 4.6 MEASURES the structural pacing surface and RECORDS tuning notes; it makes NO generator/recipe/weight change to "fix" pacing (Epic-10 / tracked-follow-up).

Keeping these boundaries tight is the single biggest risk for this story, exactly as it was for 4.1-4.5. The temptation is to build a real shop/event/boss, a real tactical play loop, a HUD, or to "fix" pacing by editing the generator. Resist it — 4.6 is the SHELL + the start command + the save/resume bridge.

The orchestrator is the CALLER that sequences commands; it owns no gameplay decision a command does not ([Source: `project-context.md` §Determinism & Simulation Rules; `godot/scripts/core/commands/node_enter_command.gd`; `node_exit_command.gd`; `node_resolve_placeholder_command.gd`; `route_advance_command.gd`]).
