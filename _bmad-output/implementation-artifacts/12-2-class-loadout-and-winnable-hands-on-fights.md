# Story 12.2: Class Loadout and Winnable Hands-On Fights

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want my chosen class's kit to arm me for a fight I can actually win,
so that hands-on combat is fair and classes feel tactically distinct.

## Context and Scope Authority

This story was created by the 2026-07-07 sprint change proposal (`_bmad-output/planning-artifacts/sprint-change-proposal-2026-07-07.md`, committed `3a8f3e3`) that inserted **Epic 12 (Interactive Tactical Combat)** between Epic 10's Stories 10.3 and 10.4. It is the **SECOND and FINAL story of Epic 12** and closes the **Epic-11 retro T2 (a winnable hands-on hero path, MED-HIGH)** — a human must be able to WIN an arbitrary generated fight by hand, and the three classes must feel tactically distinct. This is the direct input to Epic 10's Story 10.4 class-comparison AC and the "die or win" loop gate in 10.6.

**Authoritative scope inputs (already folded into this file — do not re-derive):**
- `epics.md` Epic 12 / Story 12.2 (canonical ACs, Epic List lines 505-511; Story block lines 2866-2891).
- The 2026-07-07 sprint change proposal §1 (T1/T2 residual), §2 (technical impact — determinism/save gates), §4.5 (Story 12.2 ACs). **The proposal deliberately REVISES the Story 11.2 "class-kit → combat-loadout is a later story" boundary; this story is the one that closes it, and updates `project-context.md`'s 11.2-boundary rule in the SAME change (AC1-pinned).**
- The Epic-11 retro T2 + the 11.2/11.3 "hero HP is two concerns" boundary (`_bmad-output/implementation-artifacts/epic-11-retro-2026-07-06.md` §8 T2, §9 residual).
- `project-context.md` run-flow-scene-hud-rules (lines 450-456) — in particular the **load-bearing line 453 "do not arm the live combat driver with `StartingKit.baseline_hp`"** boundary that THIS story revises, and the class_kit_rules (line 262 in the "hero HP is two concerns" body).
- The **completed 12-1 story** (`_bmad-output/implementation-artifacts/12-1-interactive-combat-tap-loop-and-live-board-render.md`, done, merged PR #62) — the surface this story builds on.

**The one-sentence essence:** 12-1 shipped the tap-loop but kept the **driver-supplied loadout** (`DEFAULT_HERO_HP` 60 / sword) that 11.2/11.3 use — a flat scripted default, class-blind. **This story wires the selected class's starting kit (HP, weapon, support, passives) into the live combat loadout so the hero is armed from `run.starting_kit` instead of the flat 60/sword default, and proves — via a strengthened reference driver (retro T2, LoS-aware targeting) — that an approved live-combat seed batch is winnable by every playable class, with unwinnable seeds failing loud (seed + class + reason) for triage before 10.4.** The default hands-off auto-resolve driver stays available and byte-identical for the seed-batch/AC proofs; any intentionally-changed live-combat fixture/fingerprint is re-pinned in the same change.

**Current baseline after 12-1 (PR #62):** the scene-free `InteractiveCombatSession` + the additive `RunOrchestrator.begin_interactive_combat_node` / `finish_interactive_combat_node` orchestrator seams exist and are the surface 12-2 builds on. Suite baseline **187 PASS / 0 FAIL (~49s)**.

## Acceptance Criteria

**AC1 — the live loadout derives from the class starting kit (and the 11.2 boundary is formally revised, `project-context.md` updated in the same change).**
**Given** a run starts with a selectable class (warrior / pyromancer / ranger)
**When** a live combat node begins under the tap-loop (the 12-1 interactive path)
**Then** the hero's live loadout (HP, weapon, support, passives) derives from the class starting kit (`run.starting_kit` — `StartingKit.baseline_hp` / `weapon_id` / `support_id` / the two passive ids) rather than the flat scripted default (`LiveCombatResolver.DEFAULT_HERO_HP` 60 / sword)
**And** the Story 11.2 "class-kit → combat-loadout is a later story" boundary is formally revised by this story, with `project-context.md` updated in the same change (the load-bearing line 453 boundary rule + the line 262 "hero HP is two concerns" body edited to record that the loadout is now class-derived; a run with NO kit still falls back to the driver default).

**AC2 — the approved live-combat seed batch is winnable by every playable class via the strengthened reference driver; unwinnable seeds fail loud.**
**Given** the approved live-combat seed batch (an inline annotated catalog of Small/Medium combat-node seeds, the finale-catalog discipline)
**When** the strengthened reference driver (retro T2 — e.g. a LoS-aware / ranged-aware targeting hero, stronger than the current pure focus-fire `LiveCombatResolver` driver) plays each playable class with its class loadout
**Then** every approved seed is winnable by at least one legal line of play per class (the driver reaches `CombatOutcomeState.STATE_VICTORY` within the round cap)
**And** an unwinnable seed fails LOUD with seed + class + reason (the structured `live_combat_did_not_resolve` fail-loud, never a fabricated outcome) so it can be triaged BEFORE Story 10.4 — a failing seed is a hard combat error (no partial run progression), exactly like a generation failure.

**AC3 — the three classes are tactically distinct on the same seed (the direct input to 10.4's class-comparison AC).**
**Given** the three playable classes on the SAME seed
**When** their live loadouts are compared
**Then** each class changes at least one combat decision through equipment, passive, or preview behavior (e.g. warrior sword+shield trades melee with a block chance; pyromancer staff+tome is ranged with bonus damage; ranger bow is ranged with no support) — a demonstrable per-class difference in the resolved fight (round count, damage taken, target order, block/bonus events, or preview), not merely a different label
**And** this per-class distinctness is proven in a headless test as the direct input to 10.4's class-comparison AC.

**AC4 — determinism + save gates hold; the hands-off default stays byte-identical; any changed fixture is re-pinned.**
**Given** determinism and save gates
**When** the loadout wiring lands
**Then** NO new RNG stream or unnamed draw site is added (`RngStreamSet.required_streams()` stays 7; no `randi`/`randf`/fresh `RandomNumberGenerator`), the 23-key `RunSnapshot` gate stays 23, and the in-node fight state remains EPHEMERAL (no in-node fight save key)
**And** any intentionally-changed live-combat fixture/fingerprint is re-pinned in the SAME change, with the hands-off default path (the neutral `DEFAULT_HERO_HP`/sword `LiveCombatResolver.resolve(...)` + the DEFAULT `run_to_completion` + every generator/route/finale fingerprint) otherwise byte-identical. **NOTE (intentional-change territory):** threading a class **support** into the live loadout MAY draw the `combat` stream where the flat sword default drew zero — a warrior `shield` support triggers a `shield_block` roll on the `combat` stream in `AttackCommand`, and a `tome` attacker support adds bonus damage. That is an INTENTIONAL, seeded, reproducible change on the class path (re-pin any live-combat fixture it moves); the NEUTRAL default-driver / auto-resolve / generator paths stay byte-identical (they never carry a class support). No new autoload.
