# Story 3.2: Seed-Stable Small Level Layouts

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want Small levels to generate as readable tactical spaces,
so that early combat stays compact and fair.

## Acceptance Criteria

1. Given a Small level recipe around 8x8 tiles, when generation runs for a fixed seed, then it produces the same entrance, exit, floor, wall, and blocker layout every time, and different seeds can produce meaningfully different layouts.
2. Given a generated Small level is converted to a level snapshot, when the snapshot is loaded into the tactical domain model, then board bounds, blockers, entrance, and exit are represented correctly, and no scene nodes are required to understand the layout.
3. Given Small level seed regression tests are run, when approved fixture seeds generate levels, then output remains stable or changes only with explicit fixture updates, and regressions include the failing seed in test output.

### Layout Determinism Contract (AC1 — same seed reproduces the layout; the `level` stream is the only randomness source)

The single most important constraint: **a Small layout is a pure deterministic function of (root seed, recipe, starting `level`-stream state). Every random choice flows through `GenerationRequest.draw_layout_int`/`draw_layout_float` (which use `RngStreamSet.STREAM_LEVEL`), the draw ORDER is fixed, and no other randomness source touches the layout.**

- Build the streams as `RngStreamSet.new(request.level_seed())` (or accept an injected `RngStreamSet` so a caller can supply a mid-run stream — but the determinism guarantee is "same starting `level`-stream state -> same layout"). Draw blocker count, then blocker positions, then any entrance/exit randomization, in a FIXED, documented order. Reordering draws or adding an unrelated draw between two layout draws silently changes every approved fixture — so pin the order and keep the seed-regression test as the tripwire. [Source: epics.md Story 3.2 AC1; project-context.md Determinism rules; Story 3.1 AC1 `level`-stream contract]
- Different seeds must be able to produce meaningfully different layouts (AC1 second half) — assert this with >=2 seeds whose fingerprints differ. Do NOT collapse to a single fixed layout regardless of seed.
- Stream isolation (carried from Story 1.4 / 3.1): making `cosmetic`/`combat` draws before generation must NOT change the `level`-affected layout. Add an assertion mirroring the 3.1 `test_generation_request.gd` stream-isolation proof.
- No global fallback: an unknown stream or invalid range surfaces a structured `ActionResult` error from `RngStreamSet`; the generator must never silently fall back to `randi()`/`randf()`. (The `GenerationRequest` helpers already enforce this — just don't bypass them.) [Source: project-context.md "Do not use one global RNG for gameplay"; Story 3.1]

### Latest Technical Information

Official Godot 4.6 sources relevant to deterministic generation, the board model, and JSON transport (these inform correctness, not gameplay):

- `RandomNumberGenerator` (seed/state) backs `RngStreamSet`; `randi_range`/`randf` advance per-stream state deterministically from a seed, so a fixed seed + fixed draw order reproduces a layout exactly. - `JSON.stringify`/`JSON.parse_string` round-trip is the real transport the seed-stability tests must use; JSON numbers are IEEE-754 doubles (52-bit mantissa), so a full-64-bit seed persisted as a raw number truncates past 2^53 (string-encode it — `GenerationResult.seed` already is a String). Small bounded ints (dims, counts, coords) round-trip exactly. Source: https://docs.godotengine.org/en/4.6/classes/class_json.html
- `Resource` + `@export` typed properties back `LevelRecipeDefinition` (read its fields directly). Source: https://docs.godotengine.org/en/4.6/classes/class_resource.html
- `Vector2i` is the board's cell-coordinate type (`BoardCell.position`); board snapshot cell dicts use `{"x": int, "y": int}` (NOT a packed Vector2i) per `BoardCell.to_dictionary()`. Source: https://docs.godotengine.org/en/4.6/classes/class_vector2i.html

### Completion Notes List

- **Layout phase implemented.** `SmallLevelLayoutGenerator` (`RefCounted`, data-layer only) turns a validated `GenerationRequest` + a Small `LevelRecipeDefinition` + an `RngStreamSet` into a deterministic layout (dimensions, row-major terrain grid, entrance, exit, interior blocker cells), then converts it into a `BoardState.to_snapshot()`-shaped board dict validated through the strict `BoardState.try_from_snapshot()`.
- **GenerationResult payload-shape decision (3.2.3).** A Small success now returns `GenerationResult.ok(payload, {phase: "layout", ...})` where `payload = {"board": <board_snapshot_dict>, "entrance": {x,y}, "exit": {x,y}, "blockers": [{x,y}...], "size_class": "small", "recipe_id": "...", "level_seed": "<decimal string>"}`. The payload is pure serializable data (no `BoardState`/`RefCounted`/scene refs) and survives a real `JSON.stringify`/`parse_string` round-trip + re-conversion via `try_from_snapshot`. The full-64-bit seed is carried as a decimal string (`level_seed`); board dims/terrain/coords stay numeric (small bounded ints). The 3.1 `{placeholder: true}` payload is fully obsolete and removed from the assertions.
- **Non-Small handling.** A Medium recipe (Story 3.3) resolves successfully but carries NO layout — payload is `{"layout_pending": true, "recipe_id":..., "size_class":"medium"}` with a `note: "layout_unsupported_for_size_class"` diagnostic. No crash; clearly marked so it is never mistaken for a finished level. (`SmallLevelLayoutGenerator.generate_layout` itself returns a structured `unsupported_size_class_for_layout` error if handed a non-Small recipe directly.)
- **Small-dimension decision.** Fixed `8x8` for Small v0 (no jitter). A fixed footprint keeps entrance/exit positions stable and the fairness corridor trivially valid; `SMALL_WIDTH`/`SMALL_HEIGHT` constants make a future jittered size (derived from a `level`-stream draw) a localized change. The recipe size class is the source of WHICH size path runs.
- **Entrance-to-exit reachability assumption (for Story 3.6 to tighten).** Layout shape is a full border WALL ring + open FLOOR interior. Entrance is fixed at `(1, height/2)` and exit at `(width-2, height/2)` — both interior FLOOR-adjacent cells on the central row. The ENTIRE central corridor row (the interior span of `y == height/2`) is RESERVED as blocker-free, so the entrance can always reach the exit along a clear floor corridor REGARDLESS of blocker draws. This is a deliberate simplifying guarantee in place of a formal reachability validator; Story 3.6 may replace the reserved-corridor heuristic with a real reachability/no-soft-lock validator + bounded retry and then relax the corridor reservation. Entrance/exit are NOT seed-randomized this story; seed-to-seed divergence comes entirely from the interior blocker layout (satisfies AC1 "meaningfully different layouts" while staying provably fair).
- **AC1 determinism / draw contract.** EVERY layout-affecting draw routes through `GenerationRequest.draw_layout_int` (which uses `RngStreamSet.STREAM_LEVEL`). FIXED DRAW ORDER: (1) one `draw_layout_int` for the blocker count over the recipe `blocker_budget_min..max` band (clamped to available interior-floor candidate count); (2) `blocker_count` shrinking-pool `draw_layout_int` calls for blocker positions (each picked index removed from the candidate pool, so no collisions and no rejection loop). No `randi()`/`randf()`, no `RandomNumberGenerator.new()`, no other stream. A test asserts non-`level` streams stay at `draw_index == 0` after generation, and that cosmetic/combat pre-noise does not perturb the layout (stream isolation). `allow_blockers == false` or a zero budget skips the count draw entirely and places no interior blockers.
- **Validate-then-reject (Story 1.3 precedent, now first applied to generation).** The generated board is built to the strict `BoardState.to_snapshot()` shape and routed through `BoardState.try_from_snapshot()`; a malformed cell would surface the validator's structured error (a generator bug), never be silently coerced. Tests also ride the generated board through `TacticalSnapshot.from_domain` -> JSON round-trip -> `TacticalSnapshot.parse` to prove it passes the same strict embedded-tactical path the save/resume layer uses. `entities` is EMPTY this story (Story 3.5 adds entities).
- **Seed regression (AC3).** `test_small_level_layout_seed_regression.gd` pins five approved `root_seed` -> fingerprint pairs for `small_combat_basic`, asserts byte-identical regeneration per seed, and asserts >= 2 distinct layouts across the set. A drift fails with the failing `root_seed` in the assert message. The pinned fingerprints are documented to change ONLY with an intentional generator/recipe change, re-pinned via `tools/dump_small_layout_fingerprints.gd`.
