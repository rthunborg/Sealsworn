# Story 3.6: Generator Validation and Bounded Retry

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want generated levels to reject soft-locks and unfair starts,
so that seeded generation creates fair uncertainty rather than unavoidable punishment.

## Acceptance Criteria

1. Given generation produces a candidate level, when validation runs, then it checks entrance-to-exit reachability, no soft-locks, legal enemy placement, reachable intended rewards, fog/readability constraints, and safe first reveal, and validation emits a pass/fail report.
2. Given validation fails and retries are allowed, when bounded retry runs, then generation attempts a limited number of deterministic retries, and final failure returns a structured `GenerationResult` error with seed, phase, reason, and diagnostics.
3. Given validation tests run headlessly, when known bad fixture layouts are tested, then each bad layout fails for the expected reason, and known good fixture layouts pass.
4. Given generation phase fixtures are tested, when route context, recipe selection, layout/pathing, blocker placement, enemy placement, reward placement, and final validation run, then each phase has at least one focused regression assertion, and phase failures are reported separately from final-map fingerprint failures.

### Scope Boundary

This story delivers a **comprehensive deterministic level validator** (`LevelValidator`) that checks a built candidate against ALL the FR36/Story-3.6 fairness constraints (entrance->exit reachability, no soft-lock, no required class/item gate, legal enemy placement, entity-aware reachable intended rewards, fog/readability, safe first reveal) and emits a structured pass/fail report; plus a **bounded deterministic retry** loop in `LevelGenerator` that re-attempts a limited number of deterministically-perturbed candidates when validation fails and returns a structured `GenerationResult` error (seed + phase + reason + diagnostics + attempt count) on final failure; plus **per-phase fixture assertions** reported separately from the seed-regression terrain fingerprints.

In scope:

- **`LevelValidator`** — a pure-query `RefCounted` running every named check as a separate method with a stable lower-snake code, returning a compact pass/fail report, size-agnostic (both Small + Medium), short-circuiting on the first failure.
- **Entity-aware reachability** — a flood over the built `BoardState` treating non-WALL AND non-blocking-entity-occupied cells as traversable; used by the no-soft-lock, enemy-placement-reachability, and mandatory-reward-reachability checks. Closes the 3.5 terrain-only reward-reachability deferred Low.
- **Bounded deterministic retry** — `LevelGenerator` re-attempts up to `MAX_GENERATION_ATTEMPTS` candidates, attempt 0 using the UNPERTURBED `request.level_seed()` (fingerprints unchanged) and attempt `i` mixing `i` into the seed deterministically; final failure -> structured `GenerationResult.error` with `attempts`. An injected-validator test seam exercises the cap + fail-then-pass without a genuinely-bad baseline seed.
- **Per-phase fixture assertions (AC4)** — one focused regression per phase (recipe/layout-pathing/blockers/enemies/rewards/final-validation), each asserting a DISTINCT `failed_phase`/`error_code`, reported separately from the terrain fingerprint tests.

### Bounded-Retry Determinism Contract (AC2 — the central design decision)

A generated candidate is TODAY a pure function of `(root_seed, recipe)` (each phase builds `RngStreamSet.new(request.level_seed())`). So re-running the same seed yields the SAME candidate — naive retry would loop forever on a genuinely-bad seed. The retry must produce DIFFERENT-but-DETERMINISTIC candidates:

- **Per-attempt seed mix:** for attempt index `i` (0-based), derive `attempt_seed` from `request.level_seed()` mixed with `i` using a documented stable mix (MIRROR `RngStreamSet._derive_seed`'s style: xor + LCG-style multiply/add, mask to 31 bits, avoid 0). Build `RngStreamSet.new(attempt_seed)` and run the layout+placement+validation for that attempt. [Source: rng_stream_set.gd `_derive_seed`; project-context.md Determinism rules]
- **Attempt 0 is UNPERTURBED:** `attempt_seed(0)` MUST equal `request.level_seed()` exactly (e.g. the mix returns the base seed for `i == 0`), so the existing seed-regression terrain fingerprints + all 3.2-3.5 generated-layout/placement tests stay byte-identical. This is the hard invariant — verify fingerprints do NOT drift. [Source: retro-notes epic-3 Story 3-5 "placement draws APPENDED ... fingerprints stay stable"]
- **Bounded:** cap at `MAX_GENERATION_ATTEMPTS` (a NAMED constant; recommend a small bound such as 8 — document the choice; NFR4 < 3s load budget means a small cap is safe and a large one risks the budget). On exhaustion, return a structured `GenerationResult.error` with the LAST failing check's phase + code + reason + `attempts == MAX_GENERATION_ATTEMPTS` + compact diagnostics. [Source: epics.md Story 3.6 AC2; NFR4]
- **Deterministic final result:** the SAME `(root_seed, recipe)` produces the SAME final result every time (same attempt count, same final passing candidate OR same final error). A retry that succeeds on attempt `k` reports `attempts == k+1` and the successful candidate. [Source: epics.md Story 3.6 AC2]
- **Test seam (no genuinely-bad baseline seed needed):** baseline candidates pass on attempt 0, so to exercise the retry loop + cap, INJECT a validator outcome. Add an OPTIONAL injected-validator param to `generate` (real `LevelValidator` default) OR factor the retry loop to accept a validator object/callable. The test injects: (a) an always-fail validator -> assert the cap is reached + a structured error + `attempts == MAX`; (b) a fail-for-the-first-k-then-pass validator -> assert success + `attempts == k+1`; (c) assert attempt `i>0`'s candidate fingerprint differs from attempt 0's (deterministic perturbation). [Source: deferred-work.md 3-5 "add a test seam"]
