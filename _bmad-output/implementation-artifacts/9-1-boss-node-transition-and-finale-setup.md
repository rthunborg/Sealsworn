## Story

As a player,
I want the final route node to transition into the Larval Avatar encounter,

## Acceptance Criteria

(Verbatim from `planning-artifacts/epics.md` lines 2210-2231, Story 9.1; FR31 — "The MVP victory condition must be defeating the Larval Avatar at the final node" — and FR63 — "The Larval Avatar must be implemented as the only required MVP boss." The FIRST story of Epic 9 — "Larval Avatar MVP Finale." It establishes the boss-ENCOUNTER SETUP layer: reaching the terminal boss node produces a Larval Avatar boss encounter REQUEST (mirroring how a combat node produces a level `GenerationRequest`), the encounter's level snapshot carries entrance / boss arena / player start / a boss entity slot / finale constraints, setup is deterministic for the same `(seed, run state)`, and a failed boss setup fails LOUD with seed + phase + reason + compact diagnostics without leaving the run in a broken boss state. It replaces ONLY the boss node's PRE-completion behavior at the SAME node boundary the run-progression model already owns — it does NOT build the boss's phases/HP (9.2), its actions/telegraphs/AI (9.3), its victory + the first-victory reveal (9.4), or the finale regression/tuning hooks (9.5).)

**AC1 — Reaching the boss node creates a Larval Avatar encounter request through existing boundaries**
- **Given** the player reaches the boss node
- **When** node entry resolves
- **Then** it creates a Larval Avatar boss encounter request
- **And** it uses existing run, level, save, and RNG boundaries.

**AC2 — Boss setup produces a deterministic finale level snapshot**
- **Given** the boss encounter is generated or loaded
- **When** setup completes
- **Then** the level snapshot includes entrance, boss arena, player start, boss entity, and any finale constraints
- **And** setup is deterministic for the same seed and run state.

**AC3 — Boss setup failure is structured and does not corrupt the run**
- **Given** boss setup fails validation
- **When** the failure is reported
- **Then** it includes seed, phase, reason, and compact diagnostics
- **And** the run does not enter a broken boss state.

## Scope boundary (read FIRST)

This story closes the **boss node ENTRY/SETUP boundary**: it makes the terminal `boss`-type route node, when reached, produce a **Larval Avatar boss ENCOUNTER REQUEST** (a deterministic, validated request built from the run seed + the boss node — the direct sibling of the combat-node level `GenerationRequest`), carrying a level snapshot with **entrance / boss arena / player start / a boss ENTITY slot / finale constraints**, made **deterministic for the same `(root_seed, run state)`**, with a **structured seed+phase+reason+diagnostics failure that leaves the run un-broken**. It is the run-progression setup half of Epic 9's "reach and defeat the Larval Avatar."

**⭐ THE SINGLE MOST IMPORTANT ARCHITECTURAL FACT — the boss node boundary ALREADY EXISTS as a v0 PLACEHOLDER; 9.1 REPLACES its PRE-completion behavior at the SAME boundary. It does NOT change the route model, the boss node type, or the `run_completed` boundary.** The Epic-4 run-progression model already ships:
- **The terminal boss node.** `RouteGenerator` (`scripts/generation/route/route_generator.gd`) assigns exactly ONE terminal `RouteNode.TYPE_BOSS` node at a fixed shallow depth (boss always at `boss_depth`; route depth is constant at 8 tiers, boss at depth 7). The boss node id is hyphenated (`node-7-0` style) like every route node. `RouteState`/`RunState` already carry it; `RouteAdvanceCommand` reveals it; the run parks on it in `PHASE_ACTIVE_ROUTE`.
- **The boss node RESOLUTION path — a v0 PLACEHOLDER run-END.** Today, reaching the boss resolves through `NodeResolvePlaceholderCommand._resolve_boss` (`scripts/core/commands/node_resolve_placeholder_command.gd:211-279`): it marks the boss `REVEAL_CLEARED`, runs the two-step `ACTIVE_ROUTE → NODE_RESOLUTION → COMPLETED` transition, and emits `node_placeholder_resolved` + `run_completed` (with `outcome == "boss_placeholder"`, the boss node id, and the cleared-node count). **There is NO real boss LEVEL, NO boss arena, NO boss ENTITY, and NO tactical play — the boss "resolves" as a placeholder that immediately ends the run in victory.** The orchestrator dispatches to it via `RunOrchestrator._resolve_boss` (`scripts/run/run_orchestrator.gd:823-840`), which STOPS the run (no exit/advance).
- **The explicit forward contract from Story 4.5 (in `node_resolve_placeholder_command.gd:24-27`):** "Epic 9 replaces the boss's PRE-completion behavior (run a real boss level + a real victory) through the SAME node + the SAME `run_completed` boundary — WITHOUT changing the route model, the boss node type, or the run-completed boundary." **This story is that replacement, for the SETUP half.**
- **The combat-node ENTRY template.** `NodeEnterCommand` (`scripts/core/commands/node_enter_command.gd`) is the exact pattern 9.1 mirrors: it validates (sequence-id-first, context, phase `ACTIVE_ROUTE`, parked, supported type), derives a lower_snake `GenerationRequest.node_id` from the hyphenated route id (`node-1-0 → node_1_0`), builds + validates a deterministic `GenerationRequest` from `(run.root_seed, node)`, transitions `ACTIVE_ROUTE → NODE_RESOLUTION`, emits ONE `node_entered` event, and returns the live request in `result.metadata["level_request"]` for the caller to run generation. **It draws ZERO RNG (building a request is pure).**

**`[Decision]` — THE RECOMMENDED DISPOSITION (confirm in Completion Notes): a dedicated boss-ENTRY command that produces a boss encounter REQUEST, mirroring `NodeEnterCommand`, dispatched by the orchestrator for the boss type BEFORE the placeholder completion.** The cleanest shape — pick one and record it:
- **(A, RECOMMENDED) A dedicated `BossNodeEnterCommand`** (`scripts/core/commands/`, the 4.3 run-command idiom VERBATIM) that: takes the live `RunState` directly; rejects `sequence_id <= 0` FIRST; validates the run structurally sound, phase `ACTIVE_ROUTE`, parked on the current node, and the node is `TYPE_BOSS`; builds + validates a deterministic boss encounter REQUEST from `(run.root_seed, boss node)`; transitions `ACTIVE_ROUTE → NODE_RESOLUTION`; emits ONE boss-entered event (RECOMMENDED new `boss_encounter_started`, appended at the enum end — see the event note); and returns the live request in `result.metadata`. **It does NOT complete the run** — completion + victory is 9.4. On any reject: structured `ActionResult.error`, ZERO events, byte-identical no-mutation `RunState`. Draws ZERO RNG. The orchestrator's `_resolve_boss` dispatch changes to call THIS (the setup) instead of / before the placeholder completion — the boss no longer auto-completes on arrival in the live setup path.
- **(B) Extend `NodeEnterCommand` to also accept `TYPE_BOSS`** with a boss recipe/size branch. Cleaner in file count but couples the combat-level entry to the boss finale and complicates the `NODE_TYPE_RECIPE`/`PLACEHOLDER_NODE_TYPES` partition invariant (`test_node_resolve_placeholder_command.gd::_placeholder_and_combat_sets_partition_all_node_types` cross-checks that combat/placeholder sets partition all 8 types with no gap/overlap — moving boss out of `PLACEHOLDER_NODE_TYPES` requires re-deriving that partition + its test). Acceptable only if the boss recipe is genuinely a level recipe; RECOMMENDED (A) to keep the boss finale a distinct, self-documenting command.

Whatever you choose: the boss SETUP is a run-domain COMMAND under `godot/scripts/core/commands/` (NOT under `scripts/run/`), the orchestrator's boss dispatch (`_resolve_boss`) grows to call it (mirroring `_resolve_combat`'s enter→generate shape), and you MUST NOT change the route model, the `boss` node type, or the `run_completed` event shape/boundary (9.4 reuses it for the real victory).

### ⭐ The boss ENCOUNTER REQUEST — what shape? (the load-bearing `[Decision]`)

AC1 says "creates a Larval Avatar boss encounter request ... using existing run, level, save, and RNG boundaries," and AC2 says the level snapshot includes "entrance, boss arena, player start, boss entity, and any finale constraints." `GenerationRequest` (`scripts/generation/level/generation_request.gd`) is the existing level-request boundary, BUT it validates `size_class ∈ {SIZE_SMALL, SIZE_MEDIUM}` and `difficulty_band == standard` ONLY, and the level pipeline is a `(root_seed, recipe)` pure function that produces an all-FLOOR combat board with enemies/rewards from the residual pool. A boss ARENA with a designated boss-entity slot + finale constraints is NOT a generic combat level. Pick ONE and record it:
- **(A, RECOMMENDED) A dedicated boss encounter REQUEST + a deterministic boss ARENA setup, built ON the existing generation boundaries but NOT the generic combat-level pipeline.** Reuse `GenerationRequest` where it fits (root_seed + the lower_snake boss node id + `node_type == boss`), OR introduce a small dedicated `BossEncounterRequest` DTO (the `GenerationRequest` shape: seed + node id + validate()) if the generic request's size/difficulty validation does not fit a boss arena. The boss ARENA level snapshot is produced DETERMINISTICALLY from the seed (a fixed or seed-stable boss arena — entrance + arena bounds + a deterministic player start cell + a designated boss-entity CELL/slot + finale constraint markers). It emits a PURE serializable payload (a board snapshot dict + entrance/player-start/boss-slot + finale constraints) that survives a JSON round-trip — the same "payload is serializable data, never a live `RefCounted`/`BoardState`" rule the level pipeline follows. Whether the arena is a fixed hand-authored layout or a seed-stable generated one is your call — but it MUST be deterministic for the same `(root_seed, run state)` (AC2) and MUST fail loud with a structured `GenerationResult`-style error (seed + phase + reason + compact diagnostics, NEVER a grid dump) on invalid setup (AC3).
- **(B) Add a boss size class / a `boss_arena` recipe to the existing level pipeline** and run `LevelGenerator.generate` for it. Only viable if the boss arena is genuinely expressible as a `LevelRecipeDefinition` (a size class + terrain/blocker/wrinkle budget). Adding a `SIZE_*`/difficulty value touches `GenerationRequest.validate()` + every size/difficulty assertion + potentially the seed-regression fingerprints — a deliberate, re-pinned change. RECOMMENDED (A) unless the boss arena is a natural recipe; if you take (B), re-pin every affected fingerprint in the SAME change and do NOT perturb the existing Small/Medium/route fingerprints.

⭐ **CRITICAL — the BOSS ENTITY itself is Story 9.2, not 9.1.** AC2 names "boss entity" in the snapshot, but the Larval Avatar DEFINITION (HP, phase thresholds, legal actions, telegraph definitions, damage rules, explanation text) is Story 9.2 ("Larval Avatar Definition and Phases"), and its actions/AI are 9.3. **9.1 sets up the ENCOUNTER + the boss's PLACE in the arena (a boss-entity SLOT / a placeholder boss entity referenced BY id), NOT the boss's stats/behavior.** Provide a stable boss-entity id / slot in the setup so 9.2 can attach the real definition at the SAME point — mirror how the route generator reserves the boss node BY type in Epic 4 and Epic 9 fills it. Do NOT author the Larval Avatar's HP/phases/actions here (that duplicates 9.2 and will diverge). If you need a minimal placeholder boss entity to make the arena valid, mark it explicitly deferred/placeholder (the visible-exception-marker discipline) and note that 9.2 replaces it — never silently ship a "real-looking" boss stat block.
