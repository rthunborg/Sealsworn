# Story 5.5: Class Start Playable Smoke Slice

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want each MVP class to start a tactical run and immediately feel a different nudge,
so that class choice matters before deeper talent systems exist.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 5 -> Story 5.5 (verbatim AC text, with the implementation-grounded detail this final smoke-slice story adds). Epic-level intent: "Players can choose Warrior, Pyromancer, or Ranger ... and begin a run with class-specific passive/build nudges" + "Starting passives must use the rules kernel rather than ad hoc class logic" (epics.md Epic 5 + implementation notes); FR42 (selectable Warrior/Pyromancer/Ranger), FR44 (each playable class starts with one class passive + one equipment-synergy passive), FR45 (no level-1 active class skills), FR1/FR70 (every milestone preserves a playable build that launches a small test loop).

This is the EPIC-5 CLOSING smoke slice: it ties the 5.1 class content -> 5.2 hero-select/run-request -> 5.3 starting-kit -> 5.4 rules-kernel passive REGISTRATION together into ONE observable, deterministic "start a run as a class and immediately see the class identity + its passive nudges at the first tactical level" surface. It builds NO new content and NO new effect engine — it ASSEMBLES + SURFACES + PROVES what 5.1-5.4 already deliver, plus closes the consolidated 5.4 obligation (re-derive BOTH kit AND resolver from the restored class id on a route-position resume).

1. **Starting a run as Warrior/Pyromancer/Ranger and entering the first tactical level surfaces the correct starting kit + visible class identity data, and the level stays playable with the Epic-1 movement/attack/enemy/outcome loop.**
   **Given** Warrior, Pyromancer, or Ranger is selected
   **When** a run starts and enters the first tactical level
   **Then** the hero has the correct starting kit and visible class identity data (the started run's `RunState.starting_kit` carries the resolved weapon/support ids + baseline_hp + the two passive-id refs from 5.3, and the run carries `selected_class_id` + the seated `RunState.rules_resolver` from 5.4 — the smoke slice ASSEMBLES a scene-free "class start" view surface that projects this run-domain identity (class id/display name, weapon/support ids, baseline_hp, the two passive ids + their explanations) at the point the run enters its first tactical level)
   **And** the level remains playable with movement, attack, enemy turns, and outcomes from earlier epics (the first level GENERATES through the existing `LevelGenerator` pipeline and is a valid playable `BoardState` — the Epic-1 `MoveCommand`/`AttackCommand`/enemy-turn/outcome loop is UNCHANGED and still operates on the generated level; the class start does NOT break, gate, or alter that loop).

   **AC1 SCOPE — "enters the first tactical level" vs the v0 auto-resolve boundary (read carefully):** the headless `RunOrchestrator` v0 AUTO-RESOLVES combat ("level generated successfully -> node cleared"); there is NO real live tactical play loop wired into the orchestrator yet (run_orchestrator.gd V0 COMBAT-AUTO-RESOLVE BOUNDARY [Decision]; project-context.md "v0 combat is AUTO-RESOLVED ... there is no real tactical play loop wired into the headless orchestrator yet (a later HUD concern)"). 5.5 honors the architecture's HEADLESS/SCENE-FREE + view-model-first precedent (Story 2.5 / 5.2): it proves the first level GENERATES into a playable `BoardState` and that the Epic-1 combat loop still RUNS on it (a headless drive of move/attack/enemy-turn/outcome on the generated first level is sufficient evidence of "remains playable") — it does NOT require building a `Control`/HUD scene or wiring a real interactive tactical play loop into the orchestrator. The "playable" proof is: (a) the first level generates valid and (b) the Epic-1 commands operate on it deterministically. Surfacing the class identity is a scene-free view-model projection, NOT a scene.

2. **Each class's passive EXPLANATIONS appear in debug/player-readable output when preview/combat events (trigger windows) occur, and an unrelated class does NOT receive another class's effects.**
   **Given** class-specific passive effects are active
   **When** preview or combat events occur
   **Then** relevant passive explanations appear in debug/player-readable output (resolving the relevant trigger window through the run's `RulesResolver` yields the registered starting passives for THIS class in stable order, each surfacing its readable `explanation` — e.g. for a Warrior run, resolving `before_attack` surfaces "Unbreakable Guard (warrior class passive) ..." and resolving `run_started` surfaces "Blade and Board (warrior equipment synergy) ...")
   **And** unrelated classes do not receive those effects (a Pyromancer run's resolver surfaces ONLY the pyromancer passives, a Ranger run ONLY the ranger passives, etc. — no class's resolver ever returns another class's passive at any window; an empty-class/legacy run surfaces NO passive explanations).

   **AC2 SCOPE CLARIFICATION (the single biggest scope trap — INHERITED VERBATIM from 5.4):** v0 starting passives are **EXPLANATION-ONLY** rule-benders. 5.5 surfaces and asserts the passive **EXPLANATIONS** + the resolver presence + class differentiation — it does NOT assert (and does NOT build) a MUTATED combat number (HP/damage/movement/preview value). The per-effect OPERATIONS (`RuleCondition`/`RuleTarget`/`RuleOperation`) + the combat HOOK sites that would FIRE a passive's window inside `AttackCommand`/`MoveCommand`/attack-preview and CHANGE a number are **Epic 6** (the effect model + the 20-30 passive POOL, FR46) — NOT 5.5 (consolidated 5.4 [Review][Decision] forward note; deferred-work.md "Tracked from: dev of 5-4" -> "Per-effect passive OPERATIONS + the combat HOOK sites remain deferred ... Story 5.5 (the felt-in-combat smoke slice) + Epic 6"). The "felt-in-combat smoke slice" 5.5 owns is making the class's passive nudges VISIBLE/OBSERVABLE at run start + first level (the explanation surface + the resolver's window resolution), and PROVING each class differs — NOT wiring a combat-effect operation. Resolve a trigger window -> get the explanation entries -> assert they are the right class's. The `RulesResolver` stays a PURE READ (no RNG, no commands, no tactical mutation), exactly as 5.4 built it.

   Note: the "preview or combat events" the AC names map onto the FIXED trigger-window vocabulary 5.4 wired (rule_trigger.gd): the equipment-synergy passive declares `run_started` (its synergy is established when the run begins — so it surfaces AT START), and the class passive declares `before_attack` (it bends an attack — so it surfaces when an attack-window resolution is requested). 5.5 surfaces explanations by RESOLVING these declared windows on the run's resolver; it does NOT need a NEW combat hook site (that is Epic 6).

3. **The smoke run is deterministic under the same seed, and the build stays playable after this story (the milestone-playable gate).**
   **Given** class smoke tests run
   **When** each playable class completes the first tactical test loop
   **Then** the run remains deterministic under the same seed (same (root_seed, class_id) -> byte-identical started run + byte-identical resolver output/explanations + byte-identical first-level generation + byte-identical Epic-1 combat-loop outcome on that level; the determinism the 4.6 orchestrator + 5.2/5.3/5.4 already guarantee, now asserted for the class smoke slice across all three classes)
   **And** the build remains playable after the class story is complete (FR70 — the full headless suite stays green; an empty-class/legacy run + every 1.x-4.x-5.x path stays byte-identical; the milestone build still launches and validates the small test loop).

4. **An internal tester comparing Warrior/Pyromancer/Ranger first-loop starts finds each class changes at least one combat decision through equipment, passive, or preview behavior — and any stat-only reskin is flagged.**
   **Given** an internal tester plays the first tactical loop with Warrior, Pyromancer, and Ranger
   **When** they compare the class starts
   **Then** each class must change at least one combat decision through equipment, passive, or preview behavior (the STRUCTURAL differentiation surface 5.5 makes observable: each class enters with a DIFFERENT starting weapon (Warrior `sword`, Pyromancer `staff`, Ranger `bow`) + support (Warrior `shield`, Pyromancer `tome`, Ranger `none`) shaping the attack/preview, and a DIFFERENT pair of passive explanations — so the first-loop start genuinely differs per class in a way a tester can attach a "this changed my decision" note to)
   **And** any class that feels like a stat-only reskin is flagged before more class content is added (a TRACKED-NOTE deliverable — exactly like the 4.6 AC4/AC5 playtest notes: the dev shell makes the per-class differentiation OBSERVABLE; the human-felt "did class choice change a combat decision?" pass is recorded as a local playtest/tracked note in Completion Notes, with any stat-only-reskin concern flagged there before Epic 6 adds the passive pool. v0 passives are explanation-only, so the "felt difference" at this milestone is primarily the equipment shape + the passive explanation surface, NOT a mutated combat number — that mutation is Epic 6; the tester note should frame the comparison honestly against that v0 boundary).
