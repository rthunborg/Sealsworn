# Story 5.4: Starting Passive Rule Integration

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want class passives to nudge builds without requiring active skills at level 1,
so that class identity appears early while MVP combat stays focused.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 5 -> Story 5.4 (verbatim AC, with the implementation-grounded detail this story adds). Epic-level intent: "Starting passives must use the rules kernel rather than ad hoc class logic." (epics.md Epic 5 implementation notes); FR44 (each playable class starts with one class passive + one equipment-synergy passive), FR45 (no level-1 active class skills).

1. **A playable class starting a run REGISTERS its class passive + equipment-synergy passive into the rules kernel, exposed through explicit trigger windows — WITHOUT bypassing command/result/event flow.**
   **Given** a playable class starts a run
   **When** its class passive and equipment-synergy passive are registered
   **Then** they are available to the rules kernel through explicit trigger windows (the two passive ids on the started run's `StartingKit` — `class_passive_id` / `equipment_synergy_passive_id` — are RESOLVED through a NEW fail-closed `PassiveRepository` to typed `PassiveDefinition` resources, each of which declares its trigger window(s); the run-start path REGISTERS the resolved starting passives into a NEW rules-kernel resolver keyed by trigger window, so a later trigger-window evaluation can collect them)
   **And** they do not bypass command/result/event flow (the registration happens INSIDE the `RunStartCommand` validate/execute contract — fail-closed on an unresolvable passive id with a stable `ActionResult.error` and ZERO mutation BEFORE any run is built; nothing reaches into domain state outside the command; the passive resolver is seeded as run domain state on the started `RunState`, not as a side-channel global).

2. **A starting passive that modifies movement, targeting, damage, healing, or preview output emits deterministic events or explanation entries when its trigger resolves, and the effect is testable without UI scenes.**
   **Given** a starting passive modifies movement, targeting, damage, healing, or preview output
   **When** its trigger resolves
   **Then** the outcome emits deterministic events or explanation entries as appropriate (resolving the relevant trigger window through the rules-kernel resolver yields the registered starting passives in a STABLE deterministic order, each producing a player/debug-readable EXPLANATION entry — the architecture's readability rule, e.g. "Unbreakable Guard (warrior class passive) applies at before_attack" — and a deterministic `passive_registered` / passive-explanation record; same run + same trigger window -> byte-identical resolver output)
   **And** the passive effect is testable without UI scenes (every part — `PassiveDefinition`, `PassiveRepository`, the resolver, the registration, the trigger-window query, the explanation output — is scene-free `RefCounted`/`Resource` proven by headless unit tests; NO `Control`/scene/HUD).

   **AC2 SCOPE CLARIFICATION (read carefully — this is the single biggest scope trap):** v0 MVP combat has NO movement/attack/damage/healing/preview hook site wired to the rules kernel yet (Epic 1 `MoveCommand`/`AttackCommand` resolve damage directly; `scripts/rules/` is EMPTY). This story does NOT retrofit the rules kernel into `AttackCommand`/`MoveCommand`/preview — that integration is **Story 5.5** (the playable smoke slice that makes a passive felt in combat) + the per-effect operations Epic 6 authors. What 5.4 OWNS for AC2 is the **trigger-window + explanation SEAM**: a starting passive DECLARES the trigger window(s) it would fire in (from the architecture's fixed vocabulary — `run_started`, `level_entered`, `turn_started`, `before_move`, `after_move`, `before_attack`, `damage_calculated`, `enemy_killed`, `reward_offered`, `level_completed`), is REGISTERED at run-start, and the resolver, when asked to resolve a trigger window, returns the matching registered passives IN STABLE ORDER each with a deterministic readable explanation entry. The "outcome" 5.4 proves is the deterministic, explainable RESOLUTION of the window (the registered passive surfaces with its explanation), NOT a mutated HP/movement number (that mutation is the per-operation work of Story 5.5 / Epic 6). The MVP starting passives are authored as PASSIVE, EXPLANATION-ONLY rule-benders for v0 (no active class skill — AC3) whose mechanical operation is a later-epic concern; 5.4 wires the kernel scaffolding + the registration + the explainable resolution, scope-disciplined to the STARTING passive.

3. **Level-1 active class skills are out of scope — class definitions carry NO active-skill data, and no class can start with an active skill at level 1.**
   **Given** level-1 class active skills are out of scope
   **When** class definitions are validated
   **Then** active skill fields are absent, empty, or disabled for MVP starting classes (this is satisfied BY CONSTRUCTION today — `ClassDefinition` has NO active-skill field and the five baselines declare none — so 5.4 must NOT add one; if 5.4 adds any passive-shape metadata it must be PASSIVE-only and must NOT introduce an active-skill/level-1-skill field, and a regression test must PIN the absence of any active-skill concept on `ClassDefinition`/`PassiveDefinition`)
   **And** no class can start with an active skill at level 1 (the started run registers only PASSIVE rule-benders; nothing in the start path grants an active, player-triggered, level-1 class skill — a test asserts the resolver holds only passives and exposes no active-skill activation).

### Acceptance-criteria interpretation notes (read before coding)

- **This story is the FIRST story in the project to build ANY passive system or rules-kernel wiring — it introduces them from an EMPTY scaffold.** Empirically verified at story-creation (2026-06-25, filesystem `find`): `scripts/rules/{conditions,operations,resolver,triggers}` contain ZERO `.gd` files (pure empty scaffolding), and NO passive definition / passive repository exists ANYWHERE under `scripts/`. Through Stories 5.1/5.2/5.3, `class_passive_id` / `equipment_synergy_passive_id` were carried deliberately as lower_snake STRING-shape forward references with NO resolver — 5.1 explicitly scoped them "resolved later (Epic 6 + Story 5.4)", 5.3 RECORDED them verbatim on `RunState.starting_kit` and resolved them against NOTHING. **5.4 OWNS introducing the `PassiveDefinition` + `PassiveRepository` + the rules-kernel hook that RESOLVES and APPLIES (registers) a class's STARTING passive.** [Source: epic-5 retro-notes Story 5-1 ("class_passive_id/equipment_synergy_passive_id are scoped as lower_snake string-shape forward references resolved later (Epic 6 + Story 5.4)"); 5-3 story Decision B / "Passive ids STAY string-shape references"; deferred-work.md "passive-id resolution + rules-kernel wiring is Story 5.4 / Epic 6, NOT 5.3"; filesystem verification 2026-06-25]

- **CRITICAL — the class accessor is `ClassRepository.get_class_definition(class_id)`, NEVER `get_class(id)`.** `get_class()` is a reserved native `Object` method; defining/calling a custom `get_class` is a hard Godot parse error (Story 5.1 Decision E, human-ratified D3 — it compile-failed two 5.1 test files and produced a FALSE PASS). Any class lookup in this story MUST call `get_class_definition(id)`. By the SAME rule, when you name the NEW passive accessor, AVOID any reserved native `Object`/`Resource` method name — use `PassiveRepository.get_passive(passive_id)` (the architecture's own name, `content_repository.get_passive(passive_id)` at game-architecture.md:1307; no native collision), NOT `get` / `get_class` / `get_script` / etc. [Source: 5-1 Decision E; deferred-work.md Story 5-1 D3; epic-5 retro-notes Story 5-1; game-architecture.md:1307]

- **WHERE the passive ids come from = the started run's `StartingKit` (Story 5.3), NOT a re-read of `ClassDefinition`.** Story 5.3 applies the kit on a successful selectable-class start: `RunState.starting_kit: StartingKit` (a `RefCounted` DTO at `scripts/run/starting_kit.gd`) carries `class_id`, `weapon_id`, `support_id`, `baseline_hp`, `class_passive_id`, `equipment_synergy_passive_id`. The two passive ids 5.4 resolves are `RunState.starting_kit.class_passive_id` and `RunState.starting_kit.equipment_synergy_passive_id`. (You MAY equivalently read them from the `ClassDefinition` during `RunStartCommand.execute` since the command already resolves the class — but the canonical home of the APPLIED starting passive ids is the kit. Resolve from whichever is cleanest in the command; the kit is the domain-state record.) An empty-class / legacy run has NO kit and NO starting passives — it registers nothing (back-compat). [Source: starting_kit.gd; run_state.gd `starting_kit`; 5-3 story]

- **The five baseline classes' starting passive ids are FIXED (5.1 baselines) — author a `PassiveDefinition` for EACH of the six distinct ids.** From `class_repository.gd::_baseline_definitions()`: Warrior -> `warrior_unbreakable_guard` (class) + `warrior_blade_and_board` (equipment-synergy); Pyromancer -> `pyromancer_kindling_focus` + `pyromancer_arcane_conduit`; Ranger -> `ranger_steady_aim` + `ranger_hunters_quiver`. The NEW `PassiveRepository` baseline MUST register exactly these six ids as `PassiveDefinition`s so every selectable class's two starting passives resolve. These six are the v0 starting-passive set this story authors (the broader 20-30 MVP passive pool with 3-5 weird rule-benders is FR46 / Epic 6 — do NOT author the full pool here). [Source: class_repository.gd baselines; FR44; FR46 -> Epic 6]

- **`PassiveDefinition` is a typed Godot `Resource` mirrored from an APPROVED static baseline through the `ContentRepository` boundary — EXACTLY like `ClassDefinition` / `SupportDefinition` / `EnemyDefinition`.** It is in the architecture's runtime-definition list (`PassiveDefinition`, game-architecture.md:376). Follow the established content discipline to the letter: `class_name PassiveDefinition extends Resource`; a `DEFINITION_TYPE := &"passive"` const; `@export` fields; a `validate() -> ActionResult` returning a stable `invalid_passive_definition` + `{reason: "invalid_field", field: ...}` on a bad field (mirror `SupportDefinition._invalid`); `_is_lower_snake_id` for id-shape validation (copy the helper used by every sibling definition). The repository: `class_name PassiveRepository extends RefCounted` holding a `ContentRepository`, with `_baseline_definitions()`, `create_baseline_repository()`, `create_repository_from_definitions(...)`, `register_passive(...)`, `get_passive(id)` (null on a miss = fail-closed), `has_passive(id)`, `passive_ids()`, `content_repository()` — a BYTE-FOR-STRUCTURE clone of `SupportRepository` / `ClassRepository`. Place definition at `godot/scripts/content/definitions/passive_definition.gd` and repository at `godot/scripts/content/repositories/passive_repository.gd`. [Source: support_definition.gd / support_repository.gd / class_repository.gd as the exact templates; project-context.md "Static Content & Asset Rules"; game-architecture.md:370-383, 957]

- **`PassiveDefinition` minimal v0 schema — enough to register + explain + assert no-active-skill, NOT the full effect engine.** Required fields (keep it lean; this is the STARTING-passive seam, not the Epic-6 effect model):
  - `passive_id: StringName` (lower_snake; validated by `_is_lower_snake_id`).
  - `display_name: String` (non-empty; the evocative name — FR47's "evocative name" surfaces later; here it feeds the explanation entry).
  - `passive_kind: StringName` (a stable lower_snake category, e.g. `&"class"` vs `&"equipment_synergy"` — distinguishes the two FR44 starting-passive roles; validate against a small allowlist).
  - `trigger_windows: Array[StringName]` (one or more trigger-window ids from the FIXED architecture vocabulary — validate EACH against the allowlist of the ten named windows; a starting passive must declare at least one; this is the "explicit trigger windows" AC1 demands).
  - `explanation: String` (non-empty; the player/debug-readable line the resolver surfaces — the readability rule).
  - Do NOT add an active-skill field, a level field, a cooldown, a `randi()`/RNG field, or a concrete operation/effect-amount model — those are NOT needed to satisfy 5.4's ACs and would pull in Epic-6 scope (and AC3 forbids any active-skill concept). The effect OPERATION (the actual damage/movement mutation) is deferred to Story 5.5 / Epic 6; v0 starting passives are PASSIVE + EXPLANATION-ONLY rule-benders. Document this lean schema as a `[Decision]`. [Source: game-architecture.md Rules-Kernel "Required concepts" + "Readability Rule"; FR45/FR46/FR47; project-context.md rules-kernel rules]

- **The rules kernel: build the MINIMAL deterministic resolver scaffolding under `scripts/rules/`, scope-disciplined to STARTING-passive registration + trigger-window collection + explanation — NOT the full combat rules engine.** The architecture's full kernel is `RuleTrigger` / `RuleCondition` / `RuleTarget` / `RuleOperation` / `RulesResolver` / `RuleContext` (game-architecture.md:1043-1086). 5.4 builds ONLY what the STARTING-passive seam needs and leaves the rest as the empty scaffolding it found:
  - **Trigger-window vocabulary** — a single source of truth for the ten fixed window ids (`run_started`, `level_entered`, `turn_started`, `before_move`, `after_move`, `before_attack`, `damage_calculated`, `enemy_killed`, `reward_offered`, `level_completed`). Author it as `UPPER_SNAKE` consts holding lower_snake ids on a `RuleTrigger`-style file under `scripts/rules/triggers/` (e.g. `godot/scripts/rules/triggers/rule_trigger.gd`, `class_name RuleTrigger`), with an `is_valid_window(id)` allowlist check that `PassiveDefinition.validate()` reuses.   - **The resolver** — a `class_name RulesResolver extends RefCounted` under `scripts/rules/resolver/` (e.g. `godot/scripts/rules/resolver/rules_resolver.gd`) that holds registered passives and, given a trigger-window id, returns the matching passives in a STABLE deterministic order (registration order, or sorted by `passive_id` — pick one and document it; registration order is fine and matches the architecture's "stable queue and ordering"). Document the scoped resolver as a `[Decision]`.
  - Do NOT build `RuleCondition`/`RuleTarget`/`RuleOperation` evaluation, a combat resolution path, stacking/conflict/duration handling, or any per-effect mutation — those are Epic 6 (and the per-window combat HOOK sites are Story 5.5 + Epic 6). - **WHERE registration happens = INSIDE `RunStartCommand` (EXTEND it, do NOT fork), on a successful selectable-class start, AFTER the 5.3 kit is recorded — and the resolver is seeded as run domain state.** The run-start factory is the ONE `run_started` emitter and the ONE place a `RunState` is born; 5.2 added the class gate, 5.3 added the kit gate + records `run.starting_kit`. 5.4 adds the NEXT step in the SAME command:

### Starting-Passive Resolution + Registration Contract (AC1/AC2 — the load-bearing correctness)

The single most important correctness constraint: **`RunStartCommand`, on a successful SELECTABLE-class start, RESOLVES the class's two starting passive ids (`class_passive_id` + `equipment_synergy_passive_id`) through the NEW fail-closed `PassiveRepository` and REGISTERS the resolved `PassiveDefinition`s into a `RulesResolver` keyed by trigger window, SEATING the resolver on `RunState` as run domain state — failing CLOSED (REJECT, byte-identical no-mutation, zero events, no run built) on an unresolvable passive id, and drawing ZERO RNG.**

- The passive gate runs ONLY when `class_id` is non-empty AND the resolved class `is_selectable()` (the 5.2 gate rejected unknown/locked, the 5.3 gate rejected a missing weapon/support, BEFORE this — passive resolution never runs for a locked/unknown class or a kit-invalid class). [Source: run_start_command.gd validate() ordering; AC3 "no class can start with an active skill"]
- `PassiveRepository.get_passive(def.class_passive_id) == null` -> `ActionResult.error(&"unknown_class_passive", {command, class_id, passive_id})`. `PassiveRepository.get_passive(def.equipment_synergy_passive_id) == null` -> `ActionResult.error(&"unknown_equipment_synergy_passive", {command, class_id, passive_id})`. Both lower_snake; the offending id rides metadata, never the code. [Source: AC1; action_result.gd; the 5.3 kit-gate metadata pattern]
- On success `execute` builds a `RulesResolver`, registers the two resolved `PassiveDefinition`s into it (each keyed by its `trigger_windows`), and seats it on the run AFTER `run.starting_kit = applied_kit` (the "record after kit" ordering). The route, the single `run_started` event, and determinism are UNCHANGED by the passive registration. [Source: run_start_command.gd execute() ordering]
- The registration is the AC1 "available to the rules kernel through explicit trigger windows" application — the resolver is the rules kernel; the passive's `trigger_windows` are the explicit windows; the seated resolver is reachable from run domain state (NOT a side-channel global). [Source: AC1; game-architecture.md rules-kernel]
- The AC2 "outcome ... emits deterministic events or explanation entries" is satisfied by the resolver's deterministic, stable-order, explainable resolution of a trigger window (each registered passive surfaces its `explanation`). v0 starting passives are EXPLANATION-ONLY (no HP/movement mutation — that operation is Story 5.5 / Epic 6). [Source: AC2 SCOPE CLARIFICATION; game-architecture.md Readability Rule]
- The empty/legacy run (empty `class_id`) resolves NO passives and seats NO resolver — byte-identical to today's seed-only start. [Source: run_start_command.gd back-compat path]
- ZERO RNG: passive resolution + registration is a content lookup + a stable-order collection + a field set. Do NOT add a passive-affecting RNG draw. If a future passive rolls, it uses a named stream through the run-level `RngStreamSet` (Story 5.5 / Epic 6), NEVER global/un-named RNG. [Source: project-context.md determinism rules; epic-5 retro-notes named-RNG constraint]
- Manual-seed interaction is UNCHANGED: a manual-seed run is still not meta-eligible regardless of class/kit/passives; the passives do not touch the eligibility invariant. [Source: run_state.gd validate()]
