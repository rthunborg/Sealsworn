## 4. What Was Challenging

Amelia (Developer): "Now the honest part. No blame — where did we carry risk or hit friction?"

Charlie (Senior Dev): "The most instructive *correctness* trap was the **cross-command cleared-set double-clear**, surfaced at 4.4 create-story and confirmed in dev. 4.3's `RouteAdvanceCommand` unconditionally appends the *left* node to `cleared_node_ids`, and `RouteState.validate()` rejects `duplicate_cleared_node`. So once 4.4's `NodeExitCommand` also cleared the node on exit, the *next* advance re-cleared it and the run failed validation mid-walk. The fix was a one-line idempotency guard in the 4.3 command — guard the append with `if not cleared.has(from_node_id)`. It's small, but it's a genuine epic-4 gotcha: the responsibility for 'clear a node' is split across two commands that run in sequence, and the invariant 'clear in exactly ONE place per node' had to be held deliberately through 4.4, 4.5, and 4.6. 4.5 leaned on it directly — a non-boss placeholder resolve does *no* clearing (exit does it), and only the terminal boss clears itself because no exit follows."

Cloud Dragonborn (Game Architect): "The **reveal-on-arrival coupling** is the same shape of trap from the other direction, and it's the single easiest thing in this epic to get silently wrong. Because the generator hides everything past depth 1, if any advance/entry path *forgets* to reveal the arrived node's forward neighbors, the run soft-locks after the first choice and the boss is never reachable — with no error, just an empty eligible set. It's not catchable by a unit test that commits once. The discipline that saved us was making 'reveal forward neighbors on commit' a load-bearing, multi-step-walk-tested mechanic from 4.3, and every later story keeping advance + reveal + clear in lockstep and *not* shuffling those responsibilities between commands. This is the epic's #1 'do not refactor casually' surface."

Dana (QA Engineer): "The scariest *process* find was a **false-PASS in the headless runner**, caught during 4.6 dev. `RunOrchestrator.compose_route_position_snapshot()` was declared `-> RunSnapshot` but returned an `ActionResult`. That's a return-type mismatch — a Parse Error — and it emitted a transient 'Compile Error at line 0' yet the runner *still printed PASS* for that script, because the test body never actually ran. The PASS sentinel alone was not proof the script loaded. It was only caught by grepping `SCRIPT ERROR|Parse Error` directly. That's now a convention for heavy cross-domain test files: verify with a direct error grep, not just the PASS/FAIL summary line. This is the most important new lesson of the epic because it undermines the thing we trust most — a green suite."

Alice (Product Owner): "The traceability flag I want on the record: Epic 4 is titled 'Run Map and Forward Progression' and AC5 lists real minute targets — 20–35 min successful runs, 5–15 min failed, 45 as a stretch — but what we *built* is an honest v0 **structural** shell. There is no real per-node content time yet: placeholder nodes resolve instantly and combat is *auto-resolved* on a successful level generation. So the literal wall-clock minute targets genuinely *cannot* be measured from the headless shell — they're recorded as a tester-note overlay on the structural data. What 4.6 actually measured across an 18-seed sample is the structural surface: constant 8-tier route depth, the `[8,12]` non-boss count distribution, and the node-type mix. Anyone reading AC5 expecting validated minute targets needs to know those wait for a human play pass and real content."

Cloud Dragonborn (Game Architect): "Related, and worth a clear line: the **`RunOrchestrator`'s threaded run-level `RngStreamSet` is inert in v0**. The orchestrator builds one `RngStreamSet` and threads it start-to-end, which is what AC1's 'route/level/RNG advance in lockstep' asks for — but *nothing actually draws from it during a run*. `RouteGenerator` mints and draws its own set internally, `LevelGenerator` mints its own per attempt off `request.level_seed()` (which is just `root_seed`), and the node/exit/advance commands draw zero RNG. So the stream is threaded but never advanced. Determinism is *still* correct today because everything keys off `root_seed` — which is exactly why it was accepted as a documented v0 limitation rather than a blocking fix. But it's a real latent trap: the moment Epic 6/7/9 draws run-affecting RNG (combat resolution, reward rolls, event outcomes), those draws MUST route through the orchestrator's stream and the generators MUST accept an injected set — otherwise the route-position save will persist a stream disconnected from where draws actually happen, and interrupted==uninterrupted determinism breaks. The 4.6 test that round-trips the stream gives *false confidence* right now: it would pass even if the stream were removed entirely. That's tracked to Epic 6/7/9 with an explicit owner."

Charlie (Senior Dev): "Two recurring *technical* themes, named as patterns rather than scattered notes. First, the **hyphen trap**: route node ids are hyphenated (`node-1-0`), but error codes reject hyphens (`ActionResult._is_valid_error_code`) and `GenerationRequest.node_id` is validated lower_snake. So every event/command that touches a node id had to put the id in *metadata or a plain-string payload field*, never in a code, and derive a lower_snake id (`node_1_0`) for the level request. It bit 4.3, 4.4, 4.5, and 4.6 — each handled it correctly, but it's a tax every run-domain event paid. Second — carried straight from Epic 3 — the **`new_run()` fixture trap**: `RunState.new_run()` resets `current_node_id`/`cleared_node_ids` to fresh state, so any test needing a parked/mid-run/boss-parked route must build `RunState.new(PHASE_*, ...)` directly. Every command test in 4.3–4.6 hit it."

Dana (QA Engineer): "Last one, smaller and familiar: the **AC4 human-feel playtest is still autonomous-inspection only**, exactly as it was for 3.7. The 4.6 shell is genuinely playable and observable — a 4.2 route is fully traversable to `run_completed`, the reveal-gated choices + clue tags are a real information surface, and the walk test records the structural choice points. But the 'route information *changed my decision*' moment (AC4) and the felt minute targets (AC5) are honestly logged as a human deliverable pending a real play pass. That's correctly Epic 10 territory, but 'is this run actually fun to choose through' remains unanswered by a human — for the second epic running."

## 5. Key Insights / Lessons Learned

1. **A green suite is not proof a test ran — verify heavy cross-domain test files with a direct `SCRIPT ERROR|Parse Error` grep.** A return-type Parse Error in 4.6 produced a "Compile Error at line 0 → PASS" — the test body never executed yet the runner reported PASS. The PASS sentinel alone is insufficient for files that load many cross-domain scripts. This is the highest-value new lesson of the epic.
2. **Split-responsibility invariants must be held deliberately and tested across the seam.** "Clear a node in exactly ONE place" spans `RouteAdvanceCommand` (clear the left node) and `NodeExitCommand` (clear the resolved node); the fix was a one-line idempotency guard plus a multi-step walk that proves no duplicate across advance→enter→exit→advance. Don't shuffle clear/reveal responsibilities between commands without re-running the walk.
3. **Reveal-on-arrival is a load-bearing mechanic, not a detail — and only a multi-step walk proves it.** Because the generator hides depth ≥ 2, a forgotten forward-reveal soft-locks the run silently after the first choice. A single-commit test passes while the run is broken; the start→boss walk across seeds is the tripwire.
4. **The `map`-stream fingerprint + fixed-draw-order discipline transferred from `level` with zero new determinism bugs.** Pinned once at the generator's birth (4.2), never re-pinned again across four consuming stories. Run-time reveal/clear mutations operate on a route *instance* and never touch the generator fingerprint — by design.
5. **Compose into the existing save bridge; never fork.** The whole run-progression state (phase, cleared set, reveal flags, route, RNG streams) rides the existing `RunSnapshot` fields nested under `route_state`. The pinned 23-key no-surprise gate stayed GREEN untouched across all six stories. 4.6's board-free `from_route_position` reused the 4.1 bridge rather than minting a parallel route-save format.
6. **A defer opened in story one can be closed correctly six stories later if each step is recorded.** `run_started` was wired-but-unemitted (4.1), structurally bounded (4.2), then emitted with both payload defers closed (4.6) — every handoff captured in the retro notes so the eventual owner knew exactly what to close. Same for the two cross-story deferrals (`available_choice_ids` filter → 4.3; phaseless-payload resume → 4.4).
7. **Threading a resource is not the same as exercising it — guard against false-confidence tests.** The orchestrator's run-level `RngStreamSet` is threaded but inert in v0; the round-trip test would pass even if it were removed. When the resource becomes load-bearing (Epic 6/7/9 run-affecting RNG), the test must draw through the run *before* saving.
8. **Model-diversity review keeps converging and keeps catching real edges.** Second-model passes on 4.1/4.5/4.6 verified fixes against live code and surfaced a genuine additional Low (the read-side seed-mismatch asymmetry). Worth keeping for foundation/contract/shell stories.

## 7. Next Epic Preview — Epic 5: Classes and Starting Kits

**Objective:** Players can choose Warrior, Pyromancer, or Ranger, see future locked classes, and begin a run with class-specific passive/build nudges.

- **The MVP node/level pipeline (4.4 combat entry → Epic-3 generation)** — present. Class-specific passive/build nudges will most likely express through the combat/level path the run shell already drives. No change to the route model is needed for classes.
- **Manual-seed eligibility invariant (`meta_progression_eligible == not is_manual_seed`)** — settled and pinned. Class choice does not interact with it; do not reintroduce `manual_seed_eligible_for_progression` (test-pinned absent).

**Preparation needs (light — no prep sprint required):**

- **Decide where class selection seats into the run-start flow before Epic 5 story one.** The cleanest fit is extending `RunStartCommand`'s inputs (or a sibling) so the class is chosen at `NEW_RUN → ACTIVE_ROUTE`, recorded on `RunState`, and surviving the route-position save. Flag this so the Epic-5 dev does not fork a parallel run-start path.
- **Static-content discipline applies the moment classes carry data.** Class definitions / starting kits are static content → JSON/CSV source + typed Godot Resource through a repository/import boundary, with validation + human approval (the project's content rule + the Epic-3 recipe/enemy repository precedent). This is the first new content domain since enemies/recipes; mirror that boundary, do not hot-read files.
- **Carry the `RngStreamSet` inert-stream warning into Epic 5 if class nudges introduce run-affecting RNG.** If a class passive ever rolls anything, route it through the orchestrator's run-level stream (the Epic 6/7/9 obligation applies the instant *any* run-affecting draw appears).

The one genuinely new surface Epic 5 introduces is a **class/kit content domain + its repository boundary** and **seating class selection into the run-start command** — both build on settled patterns.
