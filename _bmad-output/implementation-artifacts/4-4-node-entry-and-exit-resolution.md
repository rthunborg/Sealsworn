# Story 4.4: Node Entry and Exit Resolution

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want entering and exiting nodes to transition cleanly between route and level flow,
so that a run can progress through multiple encounters.

## Acceptance Criteria

Source: `_bmad-output/planning-artifacts/epics.md` -> Epic 4 -> Story 4.4 (verbatim AC, with the implementation-grounded detail this story adds). Readiness traceability: GDD FR46 "door sealing / containment-law feedback" maps to this story (epics line 390) -> the `door_sealed_placeholder` cue (AC3).

1. **Entering a combat node creates/loads the level request and makes level state active.**
   **Given** a combat node is selected (the run is in `RunState.PHASE_ACTIVE_ROUTE`, parked on `route.current_node_id`, and that node is `RouteNode.TYPE_COMBAT` or `TYPE_ELITE_COMBAT`)
   **When** node entry resolves
   **Then** the run creates or loads the corresponding level `GenerationRequest` (deterministic from the run `root_seed` + the node), transitions `PHASE_ACTIVE_ROUTE -> PHASE_NODE_RESOLUTION`, and emits a `node_entered` `DomainEvent`
   **And** level state becomes active for tactical play (the entry result carries the buildable level `GenerationRequest` in metadata so the caller can run `LevelGenerator.generate(...)`; this story does NOT itself run the tactical level — it produces the request + the phase transition).

2. **Exiting a completed node marks it cleared, records a rewards-placeholder, and returns to route choice.**
   **Given** the active node is completed (the run is in `PHASE_NODE_RESOLUTION`, parked on the node entered in AC1)
   **When** node exit resolves
   **Then** the node is marked cleared (`reveal_state -> RouteNode.REVEAL_CLEARED`, id appended to `route.cleared_node_ids`), a rewards-placeholder flag is recorded in the exit result metadata if applicable, the run transitions `PHASE_NODE_RESOLUTION -> PHASE_ACTIVE_ROUTE` so route choice state becomes active again, and a `node_exited` `DomainEvent` is emitted
   **And** a between-level autosave can be requested (the exit result advertises `autosave_requested: true` + the route-side snapshot fields via `RunState.to_run_snapshot_fields()`; the actual file write through `SaveRepository` is performed by the caller / a later story — see the autosave-feasibility note).

3. **Node exit seals the completed path with a deterministic door-sealed cue event.**
   **Given** node exit seals the completed path behind the player
   **When** the exit transition resolves
   **Then** a deterministic `route_sealed` (door-sealed) `DomainEvent` is emitted carrying a stable cue id `door_sealed_placeholder` (in the payload `cue_id` field)
   **And** future presentation/audio can mirror the containment-law feedback (GDD line 210 "doors seal behind the hero as a containment law") without owning route state — the event is a pure SYSTEM record (no actor), JSON-round-trippable, and presentation reads it; it never drives domain control flow.

4. **Node entry or exit in the wrong run phase fails with a structured error and mutates nothing.**
   **Given** node entry or exit is requested in the wrong run phase (entry requested when not in `PHASE_ACTIVE_ROUTE`; exit requested when not in `PHASE_NODE_RESOLUTION`; or any structural problem — not parked on a node, unknown node, a non-combat node for entry where this story scopes combat entry, etc.)
   **When** the command executes
   **Then** it fails with a structured `ActionResult.error` (stable lower_snake code + diagnostics in metadata)
   **And** run and level state remain unchanged (the `RunState`/`RouteState` is byte-identical before and after the rejected command; zero events emitted).

### Acceptance-criteria interpretation notes (read before coding)

- **This story is TWO run-domain commands — `NodeEnterCommand` and `NodeExitCommand` — plus three NEW system `DomainEvent`s (`node_entered`, `node_exited`, `route_sealed`).** It is the node ENTRY/EXIT slice that bridges route flow and level flow. It does NOT generate routes (4.2, DONE), does NOT implement the route CHOICE/commit (4.3, DONE — `RouteAdvanceCommand` already advances the pointer), does NOT implement per-node-type RESOLUTION behavior or the boss placeholder run-end (4.5), and does NOT build the full start-to-end shell / pacing / `run_started`+`run_ended` emission (4.6). It also does NOT itself run a tactical level (no board play here) — it produces the level `GenerationRequest` and the phase transition; tactical play and the full board-bearing autosave land in 4.6.

- **CRITICAL — the cleared-set boundary with 4.3 (read this twice). Story 4.3's `RouteAdvanceCommand` ALREADY clears the LEFT node on commit** (`route_advance_command.gd` step 1: the node being LEFT moves to `cleared_node_ids` + `REVEAL_CLEARED`, and reveal-on-arrival flips the arrived node's forward neighbors). So when the player advances start -> node-A, **node-A becomes the new `current_node_id` and is NOT yet cleared** (the START node was cleared). 4.4's node EXIT is what clears node-A — the node the player just RESOLVED. The clean model:
  - `RouteAdvanceCommand` (4.3): choose a revealed forward node, advance the pointer to it, clear the node you LEFT, reveal the next tier. Run stays `ACTIVE_ROUTE`.
  - `NodeEnterCommand` (4.4): you are parked on the chosen node in `ACTIVE_ROUTE`; entering it transitions `ACTIVE_ROUTE -> NODE_RESOLUTION` and produces the level request. NO cleared-set mutation on ENTER.
  - `NodeExitCommand` (4.4): the node resolved; mark the CURRENT node (the one you entered) cleared + `REVEAL_CLEARED`, transition `NODE_RESOLUTION -> ACTIVE_ROUTE`, emit `node_exited` + `route_sealed`, surface the autosave seam. The pointer STAYS on the just-cleared node — the NEXT `RouteAdvanceCommand` moves off it (and because 4.3 clears the LEFT node, advancing off a node already cleared by exit must be idempotent on the cleared set).
  - **CONSEQUENCE you MUST handle:** after `NodeExitCommand` marks the current node `REVEAL_CLEARED` + adds it to `cleared_node_ids`, the subsequent `RouteAdvanceCommand` will try to clear that SAME node again (its "clear the left node" step). 4.3's command appends unconditionally and would create a DUPLICATE entry in `cleared_node_ids`, which `RouteState.validate()` REJECTS (`duplicate_cleared_node`). This is a REAL cross-command interaction bug if 4.4 clears on exit AND 4.3 re-clears on the next advance. **You MUST resolve this** — pick ONE and document it in Completion Notes:
    - **(Option A — RECOMMENDED) Make 4.3's left-node clear idempotent.** Surgically change `RouteAdvanceCommand.execute` so appending the left node to `cleared_node_ids` and setting `REVEAL_CLEARED` is a no-op when the node is ALREADY cleared (guard `if not route.cleared_node_ids.has(from_node_id)`). This is a tiny, safe, behavior-preserving change (a fresh route never double-clears today, so all 4.3 tests stay green), and it makes the 4.3+4.4 sequence (advance -> enter -> exit-clears -> advance) valid. Add a regression in the 4.4 test proving advance-after-exit does not duplicate-clear. Note in the story file that you touched `route_advance_command.gd` for idempotency and re-run the 4.3 test (`test_route_advance_command.gd`) to confirm it stays green.
    - **(Option B) 4.4 exit does NOT add to `cleared_node_ids`; it only sets `REVEAL_CLEARED`, and leaves the `cleared_node_ids` membership to the next advance.** This keeps 4.3 untouched but means a node is "cleared" (reveal) yet absent from `cleared_node_ids` until the next advance — which contradicts AC2 ("the node is marked cleared" should mean the run-state truth, i.e. membership in `cleared_node_ids`) and breaks `available_choice_ids()`/`eligible_choice_ids()` (they exclude by `cleared_node_ids` membership, not reveal). NOT recommended.
    - Whichever you choose, add a multi-step integration walk test: advance -> enter -> exit -> advance -> enter -> exit ... to the boss, asserting `run.validate()` stays green at every step and `cleared_node_ids` never duplicates.

- **"creates or loads the corresponding level request" = build a valid `GenerationRequest` from the run + node — and MIND THE HYPHEN TRAP.** `GenerationRequest.node_id` and `node_type` are validated as **lower_snake** (`_is_lower_snake_id`, rejecting hyphens). Route node ids are HYPHENATED (`node-1-0` from `RouteGenerator._mint_node_id`). So you CANNOT pass the raw route node id as `GenerationRequest.node_id`. Derive a lower_snake request node id from the route node (e.g. replace hyphens: `node-1-0 -> node_1_0`, or use `"%s_%d_%d" % [...]`) and carry the ORIGINAL hyphenated route id separately in the entry result metadata + the `node_entered` event payload (as a plain non-empty string, NOT lower_snake — same hyphen handling `route_advanced` uses for node ids). `node_type` IS already lower_snake (`RouteNode.TYPE_*` values are `combat`/`elite_combat`/... — all valid lower_snake), so it passes directly. Map the node to a recipe + size class: this story scopes COMBAT/ELITE_COMBAT entry to the v0 level recipes (`small_combat_basic` / `medium_combat_basic`) — pick ONE deterministic mapping (recommended: `combat -> small_combat_basic`/`SIZE_SMALL`; `elite_combat -> medium_combat_basic`/`SIZE_MEDIUM`) and document it. The request `root_seed` is the run `root_seed` (full int64; do not mangle it). VALIDATE the built request via `request.validate()` before returning it; a request that fails to validate is a structured entry error, never a silently-broken request.

- **This story produces the level REQUEST; it does NOT run `LevelGenerator.generate(...)`.** AC1 says "creates or loads the corresponding level request" + "level state becomes active for tactical play." The minimal, scope-correct reading: `NodeEnterCommand` builds + validates the `GenerationRequest`, transitions to `NODE_RESOLUTION`, and returns it in the result metadata (`level_request`) so the caller (the run orchestrator / a later HUD story) runs generation and plays the level. Running `LevelGenerator.generate(...)` here would (a) require an `EnemyRepository` + `LevelRecipeRepository` wired into the run command (a dependency this story should not introduce), (b) draw the `level` stream inside a run command (route commands have been RNG-free; mixing level generation into the route-flow command muddies the determinism boundary), and (c) pull tactical play into a route-flow story. KEEP node entry to: build the request + transition the phase + emit `node_entered`. If you judge that AC1 needs the actual board active, STILL keep generation OUT of the command — at most provide an optional helper/seam the caller invokes — and document the decision. Do NOT make `NodeEnterCommand` depend on the content repositories.

- **"rewards placeholder state is recorded if applicable" = a metadata flag, not a reward system.** There is NO reward entity, no `RewardTableDefinition`, and no loot pipeline in v0 (project-context Generation Rules: rewards are abstract markers; concrete loot is Epic 6). For 4.4, "rewards placeholder" means the EXIT result records a simple flag/marker in metadata (e.g. `rewards_placeholder: true` for combat/elite nodes) so a later reward story can hook the exit boundary. Do NOT build reward resolution, a reward modal, or any loot logic here. Per-node-type resolution (what a shop/event/secret node actually DOES) is 4.5; concrete rewards are Epic 6.

- **"a between-level autosave can be requested" — feasibility note (do NOT over-build).** `RunSnapshot.from_between_level(board_state, streams, options)` REQUIRES a live `BoardState` + `RngStreamSet` and embeds a strict `TacticalSnapshot` — it cannot compose a save without a tactical board. This story has NO board (it doesn't run the level). So 4.4 does NOT write a full board-bearing autosave. AC2 "a between-level autosave CAN be requested" is satisfied by: the exit result advertising `autosave_requested: true` AND exposing the route-side snapshot fields via `RunState.to_run_snapshot_fields()` (the existing 4.1 bridge — current node, cleared/revealed ids, nested `run_phase`), so the caller can compose+write the autosave when it has the board (4.6, the playable shell). Do NOT call `SaveRepository.save_run_snapshot` from the command, do NOT fabricate a fake board, and do NOT change `RunSnapshot.from_between_level`'s required-board contract. Record this scoping in Completion Notes (it is the kind of thing the 4.6 owner needs to know).

- **Phase transitions are exactly the two edges 4.3 deferred to this story.** The 4.1 machine is `NEW_RUN -> ACTIVE_ROUTE -> NODE_RESOLUTION -> (ACTIVE_ROUTE | COMPLETED | FAILED)` (`run_state.gd` `_legal_next_phases`). `NodeEnterCommand` performs `ACTIVE_ROUTE -> NODE_RESOLUTION` (a legal edge); `NodeExitCommand` performs `NODE_RESOLUTION -> ACTIVE_ROUTE` (a legal edge). Use `RunState.transition_to(...)` — do NOT mutate `phase` directly and do NOT change the transition TABLE (both edges already exist). Wrong-phase rejection (AC4): enter rejects any phase != `ACTIVE_ROUTE`; exit rejects any phase != `NODE_RESOLUTION` — each with a stable `wrong_run_phase` code carrying the actual + expected phase in metadata (mirror `RouteAdvanceCommand`'s wrong-phase rejection).

- **Three NEW system events, each fully wired like `route_advanced`.** Add `node_entered`, `node_exited`, and `route_sealed` to `domain_event.gd` exactly the way `route_advanced` is wired: a `Type` enum value (append at the END — never renumber), an `EVENT_ID_*` const, a `static func <name>(sequence_id, payload)` factory (SYSTEM event, `actor_id = &""`, NOT added to `_event_requires_actor`), a `_validate_<name>_payload(...)` validator, the `_validate_payload_for_event` arm, and the `id_for_type`/`type_for_id` arms. Suggested payloads (all serializable + JSON-round-trippable; node ids carry hyphens so they are PLAIN non-empty strings, NOT lower_snake; node TYPE is lower_snake; reuse the existing helpers `_has_nonempty_string_payload`, `_has_lower_snake_payload`, `_has_nonnegative_integral_payload`):
  - `node_entered`: `node_id` (String, hyphenated), `node_type` (lower_snake), `node_depth` (non-negative integral), `level_request_node_id` (lower_snake — the derived request id), `recipe_id` (lower_snake), `size_class` (lower_snake).
  - `node_exited`: `node_id` (String), `node_type` (lower_snake), `node_depth` (non-negative integral), `rewards_placeholder` (bool).
  - `route_sealed`: `node_id` (String — the sealed node), `cue_id` (lower_snake — MUST equal `door_sealed_placeholder`; validate it is present + lower_snake, and assert the exact value in tests). Keep `door_sealed_placeholder` as a named constant on the command (e.g. `const DOOR_SEALED_CUE := &"door_sealed_placeholder"`).
  Real-JSON round-trip each new event in `test_domain_event.gd` + at least one malformed-payload rejection per event (mirror the `route_advanced` test).

- **Sequence id is caller-supplied (the run still has no event sequencer).** Mirror `RouteAdvanceCommand`: both commands take the run-level `sequence_id` via the constructor (default 1, gated `> 0`). `NodeExitCommand` emits TWO events (`node_exited` + `route_sealed`) — give them distinct sequence ids (e.g. `sequence_id` and `sequence_id + 1`), or take a base + offset; document the choice. Validate `sequence_id > 0` (and any second id > 0) in `validate()` BEFORE reading/mutating state, so a command's success path can never emit an event its own validator rejects (this is the exact Round-1 finding the 4.3 command guards against — `route_advance_command.gd` `invalid_event_sequence_id`).

- **Both commands draw ZERO RNG and are pure deterministic state transitions.** Building a `GenerationRequest` is pure (it does not draw the `level` stream — generation does, later). Marking cleared / transitioning phase / emitting events draws nothing. Mirror the 4.1/4.3 no-RNG discipline: hold an `RngStreamSet`, snapshot it, run enter and exit (success AND reject), assert the streams are byte-identical.

- **Context shape: take the `RunState` directly (mirror 4.3).** `RouteAdvanceCommand` accepts the `RunState` directly (no `RunActionContext`). Do the same for both new commands for consistency — a single-field context wrapper adds no value. `validate(state)` / `execute(state)` accept `Variant`, reject a non-`RunState` with `invalid_context`, and (mirroring 4.3) `run.validate()` the run first, surfacing the inner structural error code in the `invalid_context` metadata.

## Dev Notes

### What this story IS (and is NOT)

- **IS:** the node ENTRY/EXIT slice — two run-domain commands (`NodeEnterCommand`, `NodeExitCommand`) that bridge route flow and level flow, plus three NEW system `DomainEvent`s (`node_entered`, `node_exited`, `route_sealed` with the `door_sealed_placeholder` cue). Enter: build + validate the level `GenerationRequest` and transition `ACTIVE_ROUTE -> NODE_RESOLUTION`. Exit: mark the resolved node cleared, transition `NODE_RESOLUTION -> ACTIVE_ROUTE`, emit `node_exited` + `route_sealed`, surface the between-level autosave seam. ### Architecture patterns and constraints

- **Gameplay actions are validated commands returning `ActionResult`; successful commands emit deterministic past-tense `DomainEvent` records.** Both new commands follow the Epic-1 + 4.3 command precedent exactly: `validate()` then `execute()` (which re-validates), returning `ActionResult.ok([events], metadata)` on success and a structured `ActionResult.error(code, metadata)` with ZERO events + no mutation on failure ([Source: `project-context.md` §Determinism & Simulation Rules; `godot/scripts/core/commands/route_advance_command.gd`; `move_command.gd`]).
- **Node entry/exit are deterministic, RNG-free state transitions.** Building a `GenerationRequest`, marking a node cleared, transitioning the phase, and emitting events draw NO RNG (no stream touched), run no sub-command, and consume nothing from `RngStreamSet`. The `level` stream is drawn by `LevelGenerator.generate(...)` LATER (when the caller runs generation), NOT inside these route-flow commands. Mirror the 4.1/4.3 "run-state transition draws no RNG" discipline ([Source: `godot/tests/unit/core/test_route_advance_command.gd::_advance_draws_no_rng_on_success_and_reject`]).
- **Scene-independent, autoload-free domain ownership.** Both commands + the new events live in the scene-free domain (`scripts/core/commands/`, `scripts/core/events/`) as `RefCounted` services/DTOs. No `Node`, no `get_tree`/`get_node`, no new autoload. The autoload set is unchanged through Epic 3 + 4.1/4.2/4.3 (`GameSession`, `SceneManager`, `SaveManager`, `AudioManager`, `SettingsManager`, `Diagnostics`) — do NOT register a node-flow autoload. UI submits these commands through the command bridge in a later HUD story; 4.4 builds only the domain commands ([Source: `project-context.md` §Engine-Specific Rules, §Code Organization Rules]).
- **Validate-then-mutate, no coercion, no-mutation-on-reject, stable codes.** A wrong-phase / not-parked / unsupported-node / invalid-context / bad-request rejection surfaces a structured `ActionResult.error` (stable lower_snake code + diagnostics in metadata) and mutates NOTHING — the `RunState`/`RouteState` is byte-identical before and after. Error codes are lower_snake with no hyphens/dots/spaces (`ActionResult._is_valid_error_code`) — so hyphenated node ids go in METADATA, never in a code ([Source: `godot/scripts/core/commands/route_advance_command.gd`; `godot/scripts/core/results/action_result.gd`]).
- **Phase machine: use the existing legal edges; do not change the table.** `ACTIVE_ROUTE -> NODE_RESOLUTION` (enter) and `NODE_RESOLUTION -> ACTIVE_ROUTE` (exit) are ALREADY in `RunState._legal_next_phases`. Call `run.transition_to(...)`; do not mutate `phase` directly and do not edit the table ([Source: `godot/scripts/run/run_state.gd` `_legal_next_phases`, `transition_to`; `_bmad-output/game-architecture.md` §State Management line 330]).
- **Reveal is monotonic ("locked once visible"); cleared is the run-state truth of "the path sealed behind you."** Exit moves the resolved node HIDDEN/REVEALED -> CLEARED and into `cleared_node_ids`; a revealed/cleared node is never un-revealed (GDD line 249). The door-sealed PRESENTATION cue (`door_sealed_placeholder`, GDD line 210 "doors seal behind the hero as a containment law") is emitted as a system event for presentation to mirror; it does not own route state ([Source: `_bmad-output/planning-artifacts/gdds/gdd-Game-2026-05-31/gdd.md` lines 210, 249; `godot/scripts/run/route_node.gd` REVEAL_* constants]).
- **Compose, do not fork; keep the save bridge coherent.** Both commands mutate the live `RouteState`/`RunState`. `RunState.to_run_snapshot_fields()` (4.1) derives `current_route_node_id` + `revealed_route_node_ids` from the route, with the phase nested under `route_state` (`RUN_PHASE_KEY`) and NO new top-level `RunSnapshot` key. The exit autosave seam exposes those fields; it does NOT add a top-level run/route key or fork a parallel save format ([Source: `godot/scripts/run/run_state.gd` `to_run_snapshot_fields`; `project-context.md` §Save Rules; epic-4 retro Story 4-1 on nested `run_phase`]).
- **New domain event = full wiring.** Each new event (`node_entered`/`node_exited`/`route_sealed`) must be wired end-to-end like `run_started`/`route_advanced`: `Type` enum value, `EVENT_ID_*` const, factory, payload validator, `_validate_payload_for_event` arm, `id_for_type`/`type_for_id` arms. A half-wired event is a latent bug ([Source: `godot/scripts/core/events/domain_event.gd` `route_advanced` precedent]).
- **The level request is a pure DTO with strict lower_snake id fields.** `GenerationRequest.validate()` requires `node_id`/`node_type`/`recipe_id`/`affinity_placeholder` to be lower_snake and `root_seed >= 0`, with `size_class` in {`small`, `medium`} and `difficulty_band == standard`. Derive a lower_snake request `node_id` from the hyphenated route id; the request `root_seed` is the run `root_seed` ([Source: `godot/scripts/generation/level/generation_request.gd` `validate`, `_is_lower_snake_id`, `SIZE_SMALL`/`SIZE_MEDIUM`, `DIFFICULTY_STANDARD`, `AFFINITY_NONE`]).
