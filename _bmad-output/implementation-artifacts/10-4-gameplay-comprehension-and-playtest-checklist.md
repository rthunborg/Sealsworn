## Acceptance Criteria

Sourced verbatim from `epics.md` (Epic 10, Story 10.4). Seven AC groups (Given/When/Then + And).

1. **Checklist covers the comprehension surface, no telemetry (AC1).** GIVEN a playtest checklist is prepared, WHEN it is reviewed, THEN it covers **movement comprehension, attack preview clarity, preview/commit distinction, damage/death explanation, Consume/Destroy clarity, positioning importance, and quit/resume success**, AND **it does not require live-service telemetry**.

2. **Session-record format + ≥5-session target (AC2).** GIVEN MVP playtest sessions are scheduled, WHEN the checklist format is prepared, THEN it records **tester id or alias, date, device/form factor, build id, seed, class, session length, run outcome, nodes cleared, notable confusion, memorable moment, desire for another descent, and any blocked action**, AND **at least five observed sessions across available mobile and desktop form factors are targeted before final readiness, with fewer sessions treated as a documented readiness limitation**.

3. **Death/victory/run-end feedback capture, stored locally (AC3).** GIVEN a playtester completes death, victory, or run end, WHEN feedback is captured, THEN the checklist records **whether the player wants another descent, what confused them, and what moment they remember**, AND **feedback can be stored as local playtest notes**.

4. **Repeated-issue → tuning-task rules (AC4).** GIVEN repeated feedback identifies the same issue, WHEN tuning tasks are created, THEN they **reference the relevant epic/story, seed if applicable, and observed player impact**, AND **scope remains focused on MVP readiness**.

5. **Acceptance thresholds (AC5).** GIVEN playtest acceptance thresholds are reviewed, WHEN final MVP readiness is assessed, THEN **no more than one of five observed sessions may fail to understand basic movement/attack commit after the first encounter, no more than one of five may be unable to explain the main cause of death or major damage, and at least three of five should name a memorable build, passive, risk, enemy, or moment**, AND **any repeated blocker in two or more sessions creates a tuning or UX task before readiness can pass**.

6. **Consumable-frequency review (AC6).** GIVEN consumables appear during playtest or smoke runs, WHEN frequency, use rate, and player-perceived value are reviewed, THEN **consumables are flagged for tuning if they appear too often, never appear across the approved sample, are ignored in three or more relevant sessions, or are described as not worth using by repeated testers**, AND **tuning notes reference reward tables, consumable definitions, and observed context**.

7. **Per-class comparison (AC7).** GIVEN a playtester plays the first tactical loop with each MVP class — **Warrior, Pyromancer, and Ranger** — and (once Epic 6+ ships them) any additional class/build content, WHEN they compare the class starts, THEN **each class must change at least one combat decision through equipment, passive, or preview behavior**, AND **any class that feels like a stat-only reskin is flagged for tuning before more class content is added, referencing the relevant class definition, starting kit, and passive explanation surface**.

## Dev Notes

### What this story is (and is not)

Epics 1-9 shipped a complete headless deterministic domain; Epic 11 wired the first on-screen hands-off run-flow/HUD/outpost layer; Epic 12 wired the INTERACTIVE tap-loop + a class-armed, winnable, tactically-distinct hero. What the project has NEVER had is a **structured playtest checklist that validates whether the core loop is UNDERSTANDABLE** — the FR47/FR48/FR73 comprehension half of the Epic-10 readiness mandate. **Story 10.4 is the checklist-plus-honest-gaps that closes it**: it authors the comprehension checklist + session format + triage rules + acceptance thresholds, runs every headless-verifiable slice the checklist rests on (class distinctness, consumable-frequency-from-rewards), and records the human-observed-session dimension as an explicit availability gap.

The single most important discipline (mirroring 10.1 and 11.1): **author what CAN be authored + verify what CAN be verified now; record honest gaps for the human-observed rest — do NOT fabricate tester sessions/feedback, and do NOT silently expand into a gameplay/scene change.** This is a readiness-planning story; the simulation and the scenes are untouched.

### The measurement reality (why observed human sessions are availability gaps, not blockers)

- The ACs were written to be dischargeable WITHOUT a live tester lab: AC2 explicitly permits "fewer sessions treated as a **documented readiness limitation**." So a headless autonomous run legitimately produces: the complete checklist, the session-record format, the tuning-triage rules, the acceptance thresholds, the headless-verifiable readiness facts (class distinctness, `warding_salve` reward-table absence, winnability inputs), and an availability-gap ledger for the observed-human dimension (the ≥5 sessions, the felt distinctness/pacing).
- This is the SAME honesty posture 10.1 used (record the gap against an owner rather than invent the data). The gate story (10.6) is where the project decides whether a still-open observed-session gap is an acceptable documented readiness LIMITATION or must be discharged by a real observed-playtest pass before MVP-readiness passes. 10.4's job is to make the checklist + format + thresholds + gap explicit + dischargeable, not to run the sessions.
- **Do NOT stop and ask a human to run playtest sessions as the primary outcome.** The story is completable and valuable without them (the checklist + format + thresholds + verified readiness facts + gap ledger). Only escalate to `needs-human` if a required deliverable is genuinely un-authorable without an external secret/service — which is not the case here. The ONE place a genuine `needs-human` could surface is Task 5's Medium/affinity winnability extension IF a seed proves unwinnable and needs a balance decision beyond re-picking seeds — and even then the fallback (scope to Small-neutral + hand to 10.6) keeps the story completable.
