## Dev Notes

### What this story is (and is not)

Epics 1–9 shipped a complete headless deterministic domain; Epic 11 wired the LIVE run-flow scenes + hands-off loop; Epic 12 wired the INTERACTIVE hands-on tap-loop + class loadout; the Epic-10 first tranche (10-1/10-2/10-3/10-8) shipped the readiness measurement + consolidated regression + fairness harnesses; 10-4 shipped the comprehension checklist; 10-5 shipped the accessibility audit. **What the project has never had is the single FINAL MVP-readiness GATE that rolls all of it up, verifies the loop end-to-end, runs the suite + pre-export validation, and dispositions every open gap** — the FR70 playable-build-preservation half of the Epic-10 readiness mandate. **Story 10.6 is that gate.**

The single most important discipline (mirroring 10.1, 10.4, 10.5, 11.1): **verify what CAN be verified now + roll up the sibling ledgers + DECIDE each open gap as a documented limitation for the v0 candidate; record honest availability gaps for the physical-device-and-human rest — do NOT rebuild the loop, fix a deferral, produce a signed binary, or silently expand into a feature/scene change.** This is a readiness/gate story; the domain, the scenes, the ledgers, and the deferrals are untouched (this gate DISPOSITIONS them, it does not implement them).

### The verification reality (why physical-device / human passes are availability gaps, not blockers)

- The ACs are dischargeable WITHOUT a physical-device lab, a signed mobile binary, or a human ship-sign-off because the loop is LIVE and testable, the suite is runnable, and the pre-export invariants are shipped + tested: the gate legitimately produces the per-step loop verification (grounded in the live drivers + the scene-load guardrail), the fresh suite run + exceptions, the pre-export validation (grounded in `test_export_setup.gd` + the build-profile gating), the difficulty-non-goal confirmation (grounded in the regression test), the four-ledger roll-up, and a gap-disposition ledger for the physical-device + settings-scene + asset dimensions.
- This is the SAME honesty posture 10.1 used (record each gap against an owner rather than invent the data) and 10.5 used (the human-eyes availability gaps → the 10.6 gate). 10.6 is the gate those two handed their physical-device / human passes to — and the gate's job is to DECIDE each is an acceptable documented readiness LIMITATION for the offline-first v0 CANDIDATE (with an owner + target discharge path), NOT to run the device lab itself.
- **Do NOT stop and ask a human to run a physical-device pass, produce a signed binary, or approve the ship as the primary outcome.** The story is completable and valuable without them (the loop verification + the suite run + the pre-export validation + the four-ledger roll-up + the gap-disposition ledger + the build-candidate manifest). Only escalate to `needs-human` if a required deliverable is genuinely un-authorable without an external secret/service — which is not the case here.
